# API Design - Complete Guide

> **Core idea:** API design is one of the most important skills for a backend engineer. Before writing any business logic, you MUST design your API interface - how clients will interact with your server, what data they send, and what responses they receive. By following REST standards, you eliminate guesswork, reduce bugs, and make integration seamless for everyone.

## The History - Where REST Came From

### The Birth of the Web (1990)

In **1990**, Tim Berners-Lee started a project called the **World Wide Web** - to share knowledge with the whole world. Within approximately one year, he invented:

| Invention | What It Is | Still Used Today? |
|-----------|-----------|-------------------|
| **URI** (Uniform Resource Identifier) | A way to identify any resource on the web | ✅ Yes (every URL is a URI) |
| **HTTP** (HyperText Transfer Protocol) | The protocol for client-server communication | ✅ Yes (HTTP/1.1 → 2.0 → 3.0) |
| **HTML** (HyperText Markup Language) | The markup language for web pages | ✅ Yes (HTML5) |
| **First Web Server** | Software to serve web pages | ✅ Yes (Apache, Nginx, etc.) |
| **First Web Browser** | Software to view web pages | ✅ Yes (Chrome, Firefox, etc.) |
| **First WYSIWYG Editor** | HTML editor built into the browser | ✅ Yes (built into modern browsers) |

All six of these technologies we still use today - in far more advanced forms.

### The Scalability Crisis (~1993)

The web grew **exponentially**. Tim Berners-Lee hadn't accounted for this scale. The system was heading towards breakdown.

```
The problem:
  → Thousands of new users every day
  → The original design couldn't handle this growth
  → New standards, techniques, and components were needed
  → The web needed to be re-architected for scalability
```

### Roy Fielding's Contribution

Around **1993**, **Roy Fielding** - co-founder of the Apache HTTP Server project - became concerned about web scalability. To address it, he proposed **six constraints** that would make the web scalable.

Fielding later worked WITH Tim Berners-Lee to:
- Standardize the web's design
- Write the specification for **HTTP/1.1** (the first major standard version)

### The REST Paper (2000)

In the year **2000**, after the scalability crisis was averted, Fielding named and described the web's architectural style in his **PhD dissertation**:

> **REST - Representational State Transfer**

This dissertation is the origin of everything we call "REST APIs" today.

> **Must-read for backend engineers:** Search for "Roy Fielding REST paper" - reading the original dissertation gives you deep context about why these patterns exist and where they came from.

## Why Is It Called REST?

REST = **Representational State Transfer**. Each word has specific meaning:

### R - Representational

Resources on the web (data, objects) are **represented** in specific formats. The same resource can have different representations depending on who's asking:

```
A "user" resource can be represented as:

For an API client (another server):
  → JSON format
  {
    "id": "abc-123",
    "name": "Alice",
    "email": "alice@example.com",
    "created_at": "2024-03-15T14:30:00Z"
  }

For a web browser (end user):
  → HTML format
  <div class="user-card">
    <h2>Alice</h2>
    <p>alice@example.com</p>
  </div>

For a legacy system:
  → XML format
  <user>
    <id>abc-123</id>
    <name>Alice</name>
  </user>

Same resource. Different representations. That's "Representational."
```

**Real-world example:** When you visit Amazon's product page in a browser, you get HTML. When Amazon's mobile app fetches the same product, it gets JSON. Same product resource, different representations.

### S - State

**State** = the current condition or attributes of a resource at a point in time.

```
Shopping Cart State:
  {
    "items": [
      { "product": "Laptop", "quantity": 1, "price": 999.99 },
      { "product": "Mouse", "quantity": 2, "price": 29.99 }
    ],
    "total": 1059.97,
    "item_count": 3
  }

This is the STATE of the shopping cart resource.
It changes as you add/remove items.
Each API call transfers the CURRENT state.
```

**Real-world example:** Your Netflix watchlist has a state - it includes specific movies/shows. When you open the app, the server transfers this state to your device. When you add a movie, the state changes and the new state is transferred back.

### T - Transfer

**Transfer** = the movement of resource representations between client and server using HTTP methods.

```
The TRANSFER happens over HTTP:

Client ──GET──→ Server
  "Give me the list of books"

Client ←──200──  Server
  { "books": [...] }    ← State transferred from server to client

Client ──POST──→ Server
  { "name": "New Book" } ← State transferred from client to server

Client ←──201──  Server
  { "id": 1, "name": "New Book" }  ← New state transferred back
```

### Combined

```
REST = Representational State Transfer

1. Resources have REPRESENTATIONS (JSON, HTML, XML)
2. Resources have STATE (current attributes/condition)
3. State is TRANSFERRED between client and server via HTTP

This describes an architectural style where:
  → Resources are represented in various formats
  → The state of resources is transferred between client and server
  → The system follows specific constraints for scalability
```

## The Six REST Constraints

These are the constraints Roy Fielding proposed to make the web scalable:

### 1. Client-Server Separation

```
The SEPARATION OF CONCERNS between client and server:

Client (Frontend):
  → Handles user interface
  → Handles user experience
  → Runs in the browser or mobile app

Server (Backend):
  → Manages data storage
  → Handles business logic
  → Processes requests and sends responses

Why this matters:
  → Each component can evolve INDEPENDENTLY
  → Frontend team can redesign the UI without touching the backend
  → Backend team can change the database without touching the frontend
  → Better scalability - scale client and server independently
```

**Real-world example:** Instagram's mobile app (client) is completely separate from their backend servers. They can release a new app design without changing any API. They can migrate their database from one technology to another without the app knowing.

### 2. Uniform Interface

The most important constraint - a **standardized way** for components to communicate:

```
Sub-constraints:

1. Resource Identification
   → Every resource has a unique identifier (URI)
   → /api/v1/users/abc-123 identifies a specific user

2. Resource Manipulation Through Representations
   → Clients modify resources by sending representations
   → Send JSON with updated fields to change a user

3. Self-Descriptive Messages
   → Each message contains enough info to understand itself
   → HTTP method (GET/POST), Content-Type header (application/json), status codes (200, 404)

4. HATEOAS (Hypermedia as the Engine of Application State)
   → Responses include links to related actions
   → A user response might include links to their posts, profile, etc.
```

**Real-world example:** Every REST API in the world uses the same pattern - `GET /users` fetches users, `POST /users` creates one. A developer who has used Twitter's API can immediately understand Stripe's API because they follow the same interface conventions.

### 3. Layered System

```
Architecture is composed of HIERARCHICAL LAYERS:

Client → Load Balancer → API Gateway → Application Server → Database

Each layer can only see and interact with the IMMEDIATE layer below it.

Benefits:
  → Add load balancers without changing the app
  → Insert caching layers (Redis) between app and database
  → Add proxy servers for security
  → Scale individual layers independently
```

**Real-world example:** Netflix uses hundreds of microservices in layers. Your Netflix app talks to an API Gateway, which routes to various services (recommendation engine, user service, content service), each talking to their own databases. You can add a new caching layer without the mobile app knowing.

### 4. Cacheability

```
Server responses MUST be labeled as cacheable or non-cacheable:

Cacheable response (product listing):
  HTTP/1.1 200 OK
  Cache-Control: max-age=3600  ← "Cache this for 1 hour"
  
  → Client stores the response
  → Next request within 1 hour? Use cached version
  → No server call needed → faster, less load

Non-cacheable response (bank balance):
  HTTP/1.1 200 OK
  Cache-Control: no-cache  ← "Always fetch fresh data"
  
  → Client MUST ask server every time
  → Data changes frequently, caching would show stale data
```

**Real-world example:** When you browse Amazon, product images are cached (they rarely change). But your cart total is never cached (it changes with every add/remove). Amazon's servers explicitly tell your browser what to cache and what not to.

### 5. Statelessness

```
Each request MUST contain ALL information needed to process it.
The server does NOT remember previous requests.

Stateful (BAD):
  Request 1: "I'm Alice"          → Server remembers "current user = Alice"
  Request 2: "Show my orders"     → Server knows it's Alice from memory
  
  Problem: What if Request 2 lands on a DIFFERENT server?
  That server doesn't know Alice! → Fails.

Stateless (GOOD):
  Request 1: "I'm Alice, my token is xyz123, show my orders"
  Request 2: "I'm Alice, my token is xyz123, update my profile"
  
  Each request contains EVERYTHING the server needs.
  ANY server can handle ANY request. → Scalable!

Benefits:
  → Any server in a cluster can handle any request
  → Easy to add more servers behind a load balancer
  → No server-side session storage needed
  → Better reliability - no "sticky sessions"
```

**Real-world example:** When you use Google Search, your search query, language preference, and authentication token are all included in every request. Google has thousands of servers - your requests can land on ANY of them because each request is self-contained.

### 6. Code on Demand (Optional)

```
Servers CAN temporarily extend client functionality by
transferring executable code (like JavaScript) to the client.

This is the ONLY optional constraint.

Example:
  → Server sends a JavaScript file to the browser
  → Browser executes it to render interactive UI
  → This is literally how every modern web app works!
```

**Real-world example:** When you visit Gmail, the server sends JavaScript files to your browser. Your browser executes them to create the entire email interface. The "code" was transferred "on demand" from server to client.

### Summary Table

| Constraint | Purpose | Required? |
|-----------|---------|-----------|
| **Client-Server** | Separation of concerns | ✅ Required |
| **Uniform Interface** | Standardized communication | ✅ Required |
| **Layered System** | Hierarchical architecture | ✅ Required |
| **Cacheability** | Reduce server load, faster responses | ✅ Required |
| **Statelessness** | Scalable, reliable requests | ✅ Required |
| **Code on Demand** | Extend client functionality | ⚪ Optional |

## URL Structure

### Anatomy of a URL

```
https://api.example.com/v1/books?sort=title&limit=20#section1
 ──┬──  ───────┬───────  ─┬─ ──┬── ────────┬─────── ───┬────
   │          │          │    │           │            │
Scheme    Authority   Version Path    Query Params  Fragment
(protocol) (domain)          (resource)
```

| Part | Example | Purpose |
|------|---------|---------|
| **Scheme** | `https://` | Protocol - HTTP or HTTPS (encrypted) |
| **Authority** | `api.example.com` | Domain + optional subdomain |
| **Path** | `/v1/books` | Resource being accessed |
| **Query Parameters** | `?sort=title&limit=20` | Key-value pairs for filtering, sorting, etc. |
| **Fragment** | `#section1` | Navigate to a section of a page (client-side only) |

### The Forward Slash = Hierarchical Relationship

```
/books                    → The collection of ALL books
/books/harry-potter       → ONE specific book within books
/books/harry-potter/reviews  → Reviews OF that specific book

Each / represents a parent-child relationship:
  books → harry-potter → reviews
  (collection → item → sub-resource)
```

## API URL Conventions

### Convention 1: Use `api.` Subdomain

```
Website:  https://example.com           (serves HTML pages)
API:      https://api.example.com       (serves JSON data)

Why separate?
  → Different servers can handle website vs API traffic
  → Different scaling strategies
  → Different security policies (CORS, rate limiting)
  → Clear separation for developers
```

**Real-world examples:**

| Company | API URL |
|---------|---------|
| **GitHub** | `https://api.github.com` |
| **Stripe** | `https://api.stripe.com` |
| **Twitter** | `https://api.twitter.com` |
| **Spotify** | `https://api.spotify.com` |

### Convention 2: Version Your API

```
https://api.example.com/v1/books    ← Version 1
https://api.example.com/v2/books    ← Version 2 (breaking changes)

Why version?
  → You need to change the API response format
  → Old clients still work with v1
  → New clients use v2
  → Gradual migration, no one breaks

Real-world:
  Stripe API: https://api.stripe.com/v1/charges
  GitHub API: https://api.github.com/v3/repos
```

### Convention 3: Plural Resource Names (ALWAYS)

```
✅ CORRECT - Plural:
  GET /api/v1/books          → List ALL books
  GET /api/v1/books/123      → Get ONE book (still plural!)
  POST /api/v1/books         → Create a book
  PATCH /api/v1/books/123    → Update a book
  DELETE /api/v1/books/123   → Delete a book

❌ WRONG - Singular:
  GET /api/v1/book           → Inconsistent
  GET /api/v1/book/123       → Confusing
```

**Why plural even for single resources?**

```
The resource NAME represents the COLLECTION, not the individual item.

/books        = "from the BOOKS collection..."
/books/123    = "from the BOOKS collection, give me item 123"

The identifier (123) specifies WHICH item from the collection.
The path segment (books) specifies WHICH collection.

Collections are ALWAYS plural. So the path is ALWAYS plural.
```

### Convention 4: Lowercase with Hyphens

```
✅ CORRECT:
  /api/v1/project-members     → Lowercase, hyphens for word separation
  /api/v1/user-profiles       → Readable, consistent
  /api/v1/order-items          → Clean

❌ WRONG:
  /api/v1/projectMembers      → CamelCase in URLs
  /api/v1/project_members     → Underscores in URLs
  /api/v1/Project-Members     → Capital case in URLs
  /api/v1/project members     → Spaces in URLs (will break!)
```

**Why hyphens over underscores?**

```
1. URLs travel through many environments (browsers, servers, 
   operating systems, proxies)
2. Different systems handle case and special characters differently
3. Hyphens are universally safe and readable
4. Google recommends hyphens for SEO (treats them as word separators)
5. Underscores can be hidden by underline styling in links
```

### Convention 5: Slugs for Human-Readable URLs

```
A slug is a human-readable, URL-safe representation of a property:

Original title: "Harry Potter and the Philosopher's Stone"

Steps to create a slug:
  1. Lowercase everything → "harry potter and the philosopher's stone"
  2. Remove special characters → "harry potter and the philosophers stone"
  3. Replace spaces with hyphens → "harry-potter-and-the-philosophers-stone"

Final URL:
  /api/v1/books/harry-potter-and-the-philosophers-stone

vs using an ID:
  /api/v1/books/a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11

Both are valid, but slugs are more user-friendly and SEO-friendly.
```

**Real-world examples:**

| Platform | URL Style |
|----------|-----------|
| **Medium** | `medium.com/understanding-rest-apis` (slug) |
| **Dev.to** | `dev.to/user/my-first-api-post-3k2j` (slug + ID) |
| **GitHub** | `github.com/user/repository-name` (slug) |
| **Stack Overflow** | `stackoverflow.com/questions/12345/how-to-design-rest-api` (ID + slug) |

## Idempotency

### What Is Idempotency?

> **Idempotency** = performing the same operation multiple times produces the **same effect** as performing it once. The side effects on the server don't change with repeated calls.

```
Think of it like a light switch:

IDEMPOTENT - "Turn light ON" button:
  Press once  → Light is ON
  Press twice → Light is still ON (same result)
  Press 100x  → Light is still ON (same result)
  
  No matter how many times you press, result is the same.

NOT IDEMPOTENT - "Toggle light" button:
  Press once  → Light is ON
  Press twice → Light is OFF (different!)
  Press 3x   → Light is ON again
  
  Each press causes a DIFFERENT effect.
```

### Why Does Idempotency Matter?

```
Real-world scenario:

You click "Pay Now" on an e-commerce site.
The request is sent... but your internet is slow.
You don't see a response.
You click "Pay Now" again.

If the payment API is IDEMPOTENT:
  → First click → charges $50 ✅
  → Second click → recognizes it's the same payment → no additional charge ✅
  → You're only charged $50 total

If the payment API is NOT IDEMPOTENT:
  → First click → charges $50
  → Second click → charges another $50!
  → You're charged $100 total! 💀
```

### Idempotency by HTTP Method

| Method | Idempotent? | Explanation |
|--------|------------|-------------|
| **GET** | ✅ Yes | Fetching data doesn't change anything on the server |
| **PUT** | ✅ Yes | Replacing a resource with the same data = same result |
| **PATCH** | ✅ Yes | Updating a field to the same value = same result |
| **DELETE** | ✅ Yes | Deleting something already deleted = same state (it's gone) |
| **POST** | ❌ No | Creating a new resource = new entity each time |

### Detailed Breakdown

#### GET - Idempotent ✅

```
API: GET /api/v1/books

Call 1: Returns [Book A, Book B, Book C] → No side effect
Call 2: Returns [Book A, Book B, Book C] → No side effect
Call 1000: Returns same list → No side effect

The server state never changes because of a GET call.
You're only READING, not WRITING.

"But what if someone ELSE adds a book between my calls?"
  → That's a different operation by a different client.
  → YOUR call didn't cause any side effect.
  → Idempotency is about the effect of YOUR specific action.
```

#### PUT/PATCH - Idempotent ✅

```
API: PATCH /api/v1/users/123
Body: { "name": "Alice" }

Call 1: name changes from "Bob" to "Alice" → Side effect: name = Alice
Call 2: name is already "Alice", set to "Alice" → Same state: name = Alice
Call 1000: name is still "Alice" → Same state: name = Alice

After the first call, every subsequent call produces the SAME state.
The side effect is identical regardless of how many times you call it.
```

#### DELETE - Idempotent ✅

```
API: DELETE /api/v1/users/123

Call 1: User 123 is deleted → Side effect: user gone
Call 2: User 123 doesn't exist → 404 error → No side effect
Call 3: User 123 still doesn't exist → 404 error → No side effect

The server state after call 1 = after call 2 = after call 1000.
  → User 123 is always "not there."
  → The response may differ (200 vs 404), but the STATE is the same.
```

#### POST - NOT Idempotent ❌

```
API: POST /api/v1/books
Body: { "name": "Harry Potter", "author": "J.K. Rowling" }

Call 1: Creates book with id: 1 → Now 1 book exists
Call 2: Creates book with id: 2 → Now 2 books exist (DIFFERENT!)
Call 3: Creates book with id: 3 → Now 3 books exist (DIFFERENT!)

Each call creates a NEW resource with a NEW ID.
The side effect is DIFFERENT with each call.
That's why POST is NOT idempotent.
```

> **Practical tip:** In production, make payment-related POST endpoints idempotent by requiring an **idempotency key** - a unique identifier the client sends. If the server sees the same key twice, it returns the cached result instead of processing again. Stripe uses this pattern.

## HTTP Methods - Deep Dive

### The Five Core Methods

```
┌──────────┬────────────────┬────────────────┬──────────────┐
│ Method   │ Purpose        │ Idempotent?    │ Has Body?    │
├──────────┼────────────────┼────────────────┼──────────────┤
│ GET      │ Read/Fetch     │ ✅ Yes         │ ❌ No        │
│ POST     │ Create         │ ❌ No          │ ✅ Yes       │
│ PUT      │ Replace entire │ ✅ Yes         │ ✅ Yes       │
│ PATCH    │ Partial update │ ✅ Yes         │ ✅ Yes       │
│ DELETE   │ Remove         │ ✅ Yes         │ ❌ Usually no│
└──────────┴────────────────┴────────────────┴──────────────┘
```

### GET - Retrieve Data

```
Purpose: Fetch data from the server WITHOUT causing any side effects.

Examples:
  GET /api/v1/users          → List all users
  GET /api/v1/users/123      → Get user with id 123
  GET /api/v1/users?role=admin → List users filtered by role

Rules:
  → Never use GET to modify data
  → GET requests should have NO body (data goes in query params)
  → Should be cacheable by default
  → Safe and idempotent
```

### POST - Create a Resource

```
Purpose: Create a NEW resource on the server.

Examples:
  POST /api/v1/users         → Create a new user
  POST /api/v1/books         → Create a new book
  POST /api/v1/orders        → Create a new order

Rules:
  → Request data goes in the BODY (not query params)
  → Server generates the ID for the new resource
  → Response should include the newly created resource
  → Response status code: 201 Created
  → NOT idempotent (each call creates a new resource)
```

### PUT - Replace Entire Resource

```
Purpose: COMPLETELY REPLACE a resource with the provided data.

The client MUST send ALL fields - even unchanged ones.
Any missing fields are treated as "set to null/default."

Example:
  PUT /api/v1/users/123
  Body: {
    "name": "Alice Updated",
    "email": "alice@new-email.com",
    "bio": "New bio",
    "phone": "+1-555-9999"
  }

  → The entire user 123 is REPLACED with this data.
  → If you forgot to include "phone", it becomes null!

Rules:
  → Send the COMPLETE representation
  → Missing fields = intentionally removed/reset
  → Idempotent
```

### PATCH - Partial Update

```
Purpose: Update SPECIFIC FIELDS of a resource without touching others.

Only send the fields you want to change.
Unmentioned fields stay as they are.

Example:
  PATCH /api/v1/users/123
  Body: {
    "name": "Alice Updated"
  }

  → Only "name" is updated
  → email, bio, phone stay UNCHANGED
  → Much safer than PUT for partial updates

Rules:
  → Send ONLY the fields you want to change
  → Unincluded fields remain untouched
  → Idempotent
```

### PUT vs PATCH - When to Use Which

```
PUT:
  → "Here is the COMPLETE new version of this resource"
  → You're REPLACING the entire entity
  → Missing fields = set to null
  → Like overwriting a file completely

PATCH:
  → "Here are the SPECIFIC CHANGES I want to make"
  → You're UPDATING specific fields
  → Missing fields = don't touch
  → Like editing specific lines in a file

In practice:
  → PATCH is used ~90% of the time for updates
  → PUT is used when you genuinely need full replacement
  → Most "update" features are partial (change name, change bio, etc.)
  → Using PUT when you mean PATCH forces the client to send ALL fields
    (wasteful and error-prone)
```

**Real-world example:**

```
Spotify - Update Playlist:
  PATCH /v1/playlists/{playlist_id}
  Body: { "name": "My New Playlist Name" }
  → Only changes the name. Description, songs, etc. stay the same.

If they used PUT:
  PUT /v1/playlists/{playlist_id}
  Body: { 
    "name": "My New Playlist Name",
    "description": "...",
    "public": true,
    "collaborative": false,
    "songs": [... ALL songs must be listed again ...]
  }
  → Must send EVERYTHING, even if only name changed. Impractical.
```

### DELETE - Remove a Resource

```
Purpose: Remove a resource from the server.

Examples:
  DELETE /api/v1/users/123    → Delete user 123
  DELETE /api/v1/books/456    → Delete book 456

Rules:
  → Usually no request body needed (ID is in the URL)
  → Response status: 200 OK or 204 No Content
  → Idempotent (deleting something already deleted = same state)
```

## Custom Actions with POST

### The Problem

Not every operation fits neatly into CRUD (Create, Read, Update, Delete). Some operations are **custom actions** - they don't create, read, update, or delete a resource.

```
Examples of custom actions:
  → Send an email
  → Generate a report
  → Run a batch job
  → Lock an account
  → Verify an email address
  → Process a payment
  → Export data to CSV
```

### The Solution: POST for Custom Actions

The REST specification makes POST **open-ended** - it can be used for any action that doesn't fit other methods:

```
Send an email:
  POST /api/v1/emails/send
  Body: { "to": "alice@example.com", "subject": "Hello", "body": "..." }

  → Not a fetch (GET). Not an update (PATCH). Not a delete.
  → It's an ACTION → use POST.

Lock an account:
  POST /api/v1/users/123/lock
  Body: { "reason": "Suspicious activity" }

  → You could argue this is an "update" (updating is_locked to true)
  → But semantically, "locking" is an ACTION, not a field update
  → POST is more descriptive here.

Generate a report:
  POST /api/v1/reports/generate
  Body: { "type": "monthly", "month": "March", "format": "pdf" }

  → This creates a report, but it's more of an ACTION than a CRUD create.

Process a refund:
  POST /api/v1/orders/456/refund
  Body: { "amount": 29.99, "reason": "Item damaged" }

  → A refund is a business ACTION, not a simple CRUD operation.
```

### Naming Convention for Custom Actions

```
Pattern: POST /api/v1/{resource}/{action}
         POST /api/v1/{resource}/{id}/{action}

Examples:
  POST /api/v1/emails/send
  POST /api/v1/users/123/lock
  POST /api/v1/users/123/verify
  POST /api/v1/orders/456/refund
  POST /api/v1/reports/generate
  POST /api/v1/accounts/123/deactivate
```

> **Rule:** When you have an operation that doesn't fit GET, PUT, PATCH, or DELETE - use POST. The path should clearly describe the action (e.g., `/send`, `/lock`, `/refund`).

## Why Follow Standards?

### The Cost of Not Following Standards

```
Imagine you're building a public API and you DON'T follow standards:

Your API:
  DELETE /api/v1/users         → Actually fetches all users (!)
  GET /api/v1/users/create     → Actually creates a user (!)
  POST /api/v1/users/123       → Actually deletes user 123 (!)

What happens:
  1. Consumer reads your docs: "GET /users/create to create a user?"
  2. They assume it's a typo and send POST instead → gets a 404
  3. They try DELETE /users → expects deletion → gets a list of users
  4. They waste hours debugging
  5. They schedule a meeting with your team to understand the API
  6. Every new consumer goes through the same pain
  7. Your API gets a reputation for being terrible to integrate with
```

### The Benefit of Following Standards

```
If you follow REST standards:

  GET /api/v1/users           → "I know this fetches users"
  POST /api/v1/users          → "I know this creates a user"
  PATCH /api/v1/users/123     → "I know this updates user 123"
  DELETE /api/v1/users/123    → "I know this deletes user 123"

A developer who has NEVER seen your API can:
  → Guess the correct endpoints
  → Predict the response format
  → Handle errors correctly
  → Integrate in minutes instead of hours

Zero guesswork. Zero assumptions. Zero confusion.
```

**Real-world example:** A developer who has integrated Stripe's payment API can integrate Square's payment API much faster because both follow REST conventions. The methods, URL patterns, response formats, and error handling are all similar.

### Step 1: Understand the Requirements (Wireframes)

```
Start from the UI DESIGN - Figma, wireframes, mockups.

Why? Because the UI shows you:

  → What forms users fill out (determines POST/PATCH APIs)
  → What actions users can perform (determines custom action APIs)
  → How data flows through the application

The flow:
  Users → Frontend → YOUR API → Database

When you study the wireframes, you understand:
  → How users at the FIRST level relate to data at the LAST level
  → What resources need to exist
  → What operations are needed on those resources
```

```
From the requirements, extract NOUNS - these are your resources.

Example: Project Management Platform (like Jira/Linear)

From wireframes and requirements, we find these nouns:

  → Organizations  (companies/teams)
  → Projects       (work containers)
  → Tasks          (work items within projects)
  → Tags           (labels for organizing tasks)
  → Comments       (discussions on tasks)

Each noun = one resource = one API group
```

```
E-commerce platform (like Amazon):
  Products, Categories, Orders, Users, Reviews, Cart, Payments, Shipping

  Users, Posts, Comments, Likes, Followers, Messages, Notifications
```

After identifying resources, design the database tables. (Covered in depth in the Database with PostgreSQL guide.)

### Step 4: Design the API Interface

Now design how clients will interact with each resource - methods, URLs, payloads, responses.

```
The workflow:

  Wireframes → Resources → DB Schema → API Interface Design

  1. Wireframes: What does the user see?

  3. DB Schema: How is data stored?
  4. API Interface: How do clients access and modify data?
```

> **Critical point:** Design the API interface BEFORE writing any code. The interface should be intuitive, consistent, and follow REST standards. Then implement the business logic behind it.

## Identifying Resources

### Resources = Nouns

```
A resource is any NOUN that can be identified in your system.

Resources are NOT actions. Resources are THINGS.

✅ Resources (nouns):      ❌ Not resources (verbs):
  users                      login
  books                      send-email
  orders                     calculate
  products                   search
  organizations              validate

Actions/verbs are handled by HTTP methods and custom endpoints,
NOT by resource naming.
```

### Resource Hierarchy

Resources often have natural parent-child relationships:

```
Organization
  └── Projects (belong to an organization)
       └── Tasks (belong to a project)
            └── Comments (belong to a task)

This hierarchy maps directly to URL structure:

/api/v1/organizations
/api/v1/organizations/123/projects
/api/v1/organizations/123/projects/456/tasks
/api/v1/organizations/123/projects/456/tasks/789/comments
```

### How Deep Should Nesting Go?

```
Rule of thumb: Maximum 2-3 levels of nesting.

✅ GOOD (1-2 levels):
  /api/v1/projects/123/tasks
  /api/v1/users/123/orders

⚠️ ACCEPTABLE (3 levels):
  /api/v1/organizations/123/projects/456/tasks

❌ TOO DEEP (4+ levels):
  /api/v1/organizations/123/projects/456/tasks/789/comments/012/replies

Better alternative for deep nesting:
  /api/v1/comments?task_id=789     → Use query parameters instead
  /api/v1/replies?comment_id=012   → Flatten the hierarchy
```

Let's design APIs for a **Project Management Platform** with three resources: Organizations, Projects, and Tasks.

```
Resource: organizations
Database table: organizations (id, name, status, description, created_at, updated_at)
```

#### 1. List All Organizations

```
GET /api/v1/organizations

Purpose: Fetch all organizations

Request:
  Method: GET
  URL: /api/v1/organizations
  Body: None (GET requests don't have bodies)
  Query params (optional): ?page=1&limit=20&sort=name

Response (200 OK):
{
  "data": [
    {
      "id": "org-001",
      "name": "Acme Corp",
      "status": "active",
      "description": "A technology company",
      "created_at": "2024-03-15T14:30:00Z",
      "updated_at": "2024-03-15T14:30:00Z"
    },
    {
      "id": "org-002",
      "name": "Globex Inc",
      "status": "active",
      "description": "A consulting firm",
      "created_at": "2024-03-14T10:00:00Z",
      "updated_at": "2024-03-14T10:00:00Z"
    }
  ],
  "pagination": {
    "page": 1,
    "limit": 20,
    "total": 2,
    "total_pages": 1
  }
}
```

**Key design decisions:**
- Method: **GET** (we're fetching/reading data)
- URL: `/organizations` plural (always plural)
- Response wrapped in `"data"` key (consistent structure)
- Pagination metadata included (for list APIs)

#### 2. Create Organization

```
POST /api/v1/organizations

Purpose: Create a new organization

Request:
  Method: POST
  URL: /api/v1/organizations
  Headers: Content-Type: application/json
  Body:
  {
    "name": "Acme Corp",
    "status": "active",
    "description": "A technology company"
  }

Response (201 Created):
{
  "data": {
    "id": "org-001",
    "name": "Acme Corp",
    "status": "active",
    "description": "A technology company",
    "created_at": "2024-03-15T14:30:00Z",
    "updated_at": "2024-03-15T14:30:00Z"
  }
}
```

**Key design decisions:**
- Method: **POST** (we're creating a new resource)
- Same URL as list API - server differentiates by HTTP method
- Body contains only client-provided fields (`id`, `created_at`, `updated_at` are server-generated)
- Response status: **201 Created** (not 200!)
- Response includes the newly created resource with its generated `id`

```
Notice: POST and GET use the SAME URL:

  GET  /api/v1/organizations    → List all (server routes to list handler)
  POST /api/v1/organizations    → Create one (server routes to create handler)

The HTTP METHOD differentiates the action, not the URL.
This is a core REST principle.
```

#### 3. Get Single Organization

```
GET /api/v1/organizations/:id

Purpose: Fetch a specific organization by its ID

Request:
  Method: GET
  URL: /api/v1/organizations/org-001
  Body: None

Response (200 OK):
{
  "data": {
    "id": "org-001",

    "status": "active",
    "description": "A technology company",
    "created_at": "2024-03-15T14:30:00Z",
    "updated_at": "2024-03-15T14:30:00Z"
  }
}

If not found - Response (404 Not Found):
{
  "error": {
    "code": "RESOURCE_NOT_FOUND",
    "message": "Organization with id 'org-999' not found"
  }
}
```

#### 4. Update Organization

```
PATCH /api/v1/organizations/:id

Purpose: Update specific fields of an organization

Request:
  Method: PATCH
  URL: /api/v1/organizations/org-001
  Body:
  {
    "name": "Acme Corporation",
    "description": "Updated description"
  }

Response (200 OK):
{
  "data": {
    "id": "org-001",
    "name": "Acme Corporation",
```

```
Request:
```

## Request Payloads & Response Design

### Request Body - What to Include

```
Rule: The client sends ONLY the fields they control.
Server-generated fields are NEVER in the request body.

Client-provided (include in body):
  ✅ name, email, description, status, title, etc.

Server-generated (NEVER in body):
  ❌ id (generated by database)
  ❌ created_at (set by server/database)
  ❌ updated_at (set by server/trigger)
  ❌ computed fields (totals, counts, etc.)
```

### Response Design - Consistent Structure

```
Every API response should follow a consistent structure:

Success response:
{
  "data": { ... }    ← The actual resource or list of resources
}

Success list response:
{
  "data": [ ... ],        ← Array of resources
  "pagination": { ... }   ← Pagination metadata
}

Error response:
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Email is required",
    "details": [ ... ]    ← Optional: field-level errors
  }
}
```

**Why wrap in `"data"`?**

```
Without wrapper:
  Response: { "id": 1, "name": "Alice" }
  → If you later need to add metadata, you break the structure

With wrapper:
  Response: { "data": { "id": 1, "name": "Alice" } }
  → Adding metadata is easy: { "data": {...}, "meta": {...} }
  → The structure is EXTENSIBLE without breaking clients
```

## HTTP Status Codes - Complete Guide

### Status Code Categories

```
1xx - Informational (rarely used directly)
2xx - Success ✅
3xx - Redirection ↪️
4xx - Client Error ❌ (the CLIENT did something wrong)
5xx - Server Error 💥 (the SERVER did something wrong)
```

### Success Codes (2xx)

| Code | Name | When to Use |
|------|------|-------------|
| **200** | OK | General success - GET, PATCH, PUT, DELETE |
| **201** | Created | Resource was created - POST |
| **204** | No Content | Success but no body to return - DELETE |

```
200 OK - Most common. Use for:
  → Successful GET (returning data)
  → Successful PATCH/PUT (returning updated resource)
  → Successful DELETE (returning deleted resource info)

201 Created - Specifically for POST:
  → A new resource was created
  → Response body contains the new resource
  → This tells the client "your create operation succeeded"

204 No Content - For operations with no response body:
  → Successful DELETE when you don't need to return the deleted resource
  → "I did what you asked, there's nothing to show you"
```

### Client Error Codes (4xx)

| Code | Name | When to Use |
|------|------|-------------|
| **400** | Bad Request | Invalid syntax, malformed JSON, missing required fields |
| **401** | Unauthorized | Not authenticated (no token, expired token) |
| **403** | Forbidden | Authenticated but not authorized (no permission) |
| **404** | Not Found | Resource doesn't exist |
| **405** | Method Not Allowed | HTTP method not supported for this endpoint |
| **409** | Conflict | Resource conflict (duplicate email, version conflict) |
| **422** | Unprocessable Entity | Valid syntax but semantically invalid (age = -5) |
| **429** | Too Many Requests | Rate limit exceeded |

```
400 Bad Request:
  → Malformed JSON: { name: Alice }  (missing quotes)
  → Missing Content-Type header
  → Request body can't be parsed

401 Unauthorized:
  → No authentication token provided
  → Token is expired or invalid
  → "I don't know WHO you are"

403 Forbidden:
  → Valid token, but user lacks permission
  → Regular user trying to access admin endpoint
  → "I know who you are, but you CAN'T do this"

404 Not Found:
  → GET /api/v1/users/non-existent-id → 404
  → The resource doesn't exist in the database

409 Conflict:
  → POST /api/v1/users with email that already exists
  → Unique constraint violation → 409 Conflict

422 Unprocessable Entity:
  → JSON is valid, but data makes no sense
  → { "age": -5 } or { "email": "not-an-email" }
  → Validation errors
```

**Real-world example - 401 vs 403:**

```
401 (Unauthorized):
  You walk into a building without a badge.
  Security guard: "Who are you? Show me your badge."
  → You're NOT IDENTIFIED.

403 (Forbidden):
  You have a badge, but it's for Floor 1 only.
  You try to enter Floor 5.
  Security guard: "I see your badge, but you can't go to Floor 5."
  → You're IDENTIFIED but NOT ALLOWED.
```

### Server Error Codes (5xx)

| Code | Name | When to Use |
|------|------|-------------|
| **500** | Internal Server Error | Unhandled exception, bug in server code |
| **502** | Bad Gateway | Upstream server returned an invalid response |
| **503** | Service Unavailable | Server is temporarily down (maintenance, overload) |
| **504** | Gateway Timeout | Upstream server didn't respond in time |

```
500 - "Something broke on OUR end"
  → Database connection failed
  → Unhandled null pointer exception
  → The client did nothing wrong, it's our bug

503 - "We're temporarily unavailable"
  → Server is under maintenance
  → Server is overloaded
  → Try again later
```

## Query Parameters - Filtering, Sorting & Pagination

### Filtering

```
Filter by specific field values:

GET /api/v1/tasks?status=pending
  → Only tasks with status "pending"

GET /api/v1/users?role=admin
  → Only users with role "admin"

GET /api/v1/orders?status=shipped&min_amount=100
  → Orders that are shipped AND have amount >= 100

GET /api/v1/books?author=rowling&genre=fantasy
  → Books by Rowling in the fantasy genre
```

### Sorting

```
Sort by a specific field:

GET /api/v1/users?sort_by=created_at&sort_order=desc
  → Users sorted by newest first

GET /api/v1/products?sort_by=price&sort_order=asc
  → Products sorted cheapest first

GET /api/v1/tasks?sort_by=priority&sort_order=desc
  → Tasks sorted by highest priority first

Common convention (alternative):
GET /api/v1/users?sort=-created_at
  → "-" prefix = descending
  → No prefix = ascending
```

### Pagination

```
Control which "page" of results to return:

GET /api/v1/users?page=1&limit=20
  → First 20 users

GET /api/v1/users?page=3&limit=10
  → Users 21-30

Response includes pagination metadata:
{
  "data": [...],
  "pagination": {
    "page": 3,
    "limit": 10,
    "total": 150,
    "total_pages": 15,
    "has_next": true,
    "has_previous": true
  }
}
```

### Searching

```
Free-text search:

GET /api/v1/books?search=harry+potter
  → Books matching "harry potter" in title, description, or author

GET /api/v1/users?search=alice
  → Users matching "alice" in name or email
```

### Combined Example

```
GET /api/v1/tasks?status=pending&sort_by=priority&sort_order=desc&page=1&limit=20&search=database

Translation:
  → From tasks collection
  → Only pending tasks (filter)
  → Containing "database" in title/description (search)
  → Sorted by priority, highest first (sort)
  → First page, 20 items per page (pagination)
```

## API Versioning

### Why Version?

```
Without versioning:
  → You change the response format
  → ALL existing clients break immediately
  → No migration path for consumers

With versioning:
  → v1 still works for old clients
  → v2 has the new format for new clients
  → Old clients migrate at their own pace
```

### Versioning Strategies

```
1. URL Path (Most Common) - RECOMMENDED
   /api/v1/users
   /api/v2/users

2. Header-based
   GET /api/users
   Header: Accept: application/vnd.api.v1+json

3. Query parameter
   GET /api/users?version=1

URL path versioning is the simplest and most widely used.
```

## Pagination - Deep Dive with Practical Demo

### Why Do We Need Pagination?

```
Imagine you have 1,000 organizations in your database.
The client calls GET /api/v1/organizations.

Without pagination:
  → Server fetches ALL 1,000 organizations
  → Server serializes 1,000 objects into JSON
  → JSON serialization/deserialization is a HEAVY operation
  → Sends a massive payload across the network
  → Client receives 1,000 items
  → On the UI, the user only SEES 10-20 at a time
  → 980 organizations were fetched for nothing
  → Wasted: server CPU, network bandwidth, client memory
  → Result: 3-4 second delay. User notices. Bad experience.

With pagination:
  → Server fetches only 20 organizations (page 1)
  → Small JSON payload, fast serialization
  → Tiny network transfer
  → Client renders 20 items instantly
  → User wants more? Load page 2 (next 20)
  → Fast. Efficient. Great experience. ✅
```

> **Rule:** Whenever you are implementing a List API (an API that returns a collection of resources), you should almost always implement pagination.

### Query Parameters for Pagination

Pagination is controlled by the **client** through query parameters (since list APIs use GET, and GET requests send data via query params):

```
Two parameters:

1. limit  → How many items per page (page size)
2. page   → Which page/portion of data to return

GET /api/v1/organizations?limit=2&page=1
  → Return 2 organizations, from the 1st portion

GET /api/v1/organizations?limit=2&page=2
  → Return 2 organizations, from the 2nd portion
```

### Pagination Metadata - What the Server Returns

```json
{
  "data": [
    { "id": "org-005", "name": "Org Five", "status": "active" },
    { "id": "org-004", "name": "Org Four", "status": "active" }
  ],
  "total": 5,
  "page": 1,
  "total_pages": 3
}
```

```
What each field means:

data:
  → The actual portion of data for THIS page
  → Only 2 items (because limit = 2)

total:
  → The TOTAL count of ALL organizations in the database
  → Independent of pagination - always shows the full count
  → UI uses this: "Showing 2 of 5 organizations"

page:
  → Which page/portion the server is returning
  → "This response is for page 1"

total_pages:
  → How many pages exist in total
  → If limit = 2 and total = 5, then total_pages = 3
    Page 1: items 1-2
    Page 2: items 3-4
    Page 3: item 5
  → Frontend uses this to:
    - Show page numbers (1, 2, 3)
    - Know when to stop fetching (infinite scroll)
    - Disable "Next" button on last page
```

### Live Demo Walkthrough

```
We have 5 organizations in the database.
Default sort: created_at DESC (newest first)

So the order is: Org 5, Org 4, Org 3, Org 2, Org 1

Request: GET /organizations?limit=2&page=1
Response:
  data: [Org 5, Org 4]    ← First 2 items
  total: 5
  page: 1
  total_pages: 3

Request: GET /organizations?limit=2&page=2
Response:
  data: [Org 3, Org 2]    ← Next 2 items
  total: 5
  page: 2
  total_pages: 3

Request: GET /organizations?limit=2&page=3
Response:
  data: [Org 1]            ← Last page, only 1 item remaining
  total: 5
  page: 3
  total_pages: 3
```

### Default Values - Server Must Set Them

```
If the client sends NO pagination parameters:

  GET /api/v1/organizations
  (no limit, no page)

The server should NOT:
  ❌ Return a validation error
  ❌ Return ALL items without pagination
  ❌ Crash or behave unpredictably

The server SHOULD:
  ✅ Set default page = 1
  ✅ Set default limit = 10 or 20
  ✅ Return the first page with sensible defaults

This is what we call SANE DEFAULTS - the API works reasonably
even if the client sends minimal information.
```

## Sorting in List APIs - Deep Dive

### Why Sort By Default?

```
Critical point most beginners miss:

  If you don't sort your results, the database returns rows
  in NO GUARANTEED ORDER.

  That means:
  Call 1: GET /organizations → [Org 3, Org 1, Org 5, Org 2, Org 4]
  Call 2: GET /organizations → [Org 5, Org 2, Org 1, Org 4, Org 3]  ← DIFFERENT!

  The order changes randomly between API calls.
  This is confusing for users and breaks pagination.

  Solution: ALWAYS apply a default sort.
  Default sort = ORDER BY created_at DESC
  → Always return newest first (natural, expected behavior)
```

### Sort Parameters

```
Two query parameters for sorting:

1. sort_by     → Which field to sort by (name, status, created_at, etc.)
2. sort_order  → Ascending (ASC) or Descending (DESC)

GET /api/v1/organizations?sort_by=name&sort_order=asc
  → Org 1, Org 2, Org 3, Org 4, Org 5 (alphabetical A→Z)

GET /api/v1/organizations?sort_by=name&sort_order=desc
  → Org 5, Org 4, Org 3, Org 2, Org 1 (alphabetical Z→A)
```

### Sort Defaults

```
If client sends NO sort parameters:
  → sort_by defaults to: created_at
  → sort_order defaults to: desc

If client sends sort_by but NOT sort_order:
  → sort_order defaults to: desc

Why descending by default?
  → Newest items first is the most natural expectation
  → "Show me the latest organizations" is the default behavior
  → Users almost always want to see newest first
```

### How Filtering Works

```
Filtering = return ONLY items that match specific criteria

GET /api/v1/organizations?status=active
  → Only organizations with status "active"

GET /api/v1/organizations?status=archived
  → Only organizations with status "archived"

GET /api/v1/organizations?status=active&name=Acme
```

```
Translation:

  2. FILTER: only "active" ones
  3. SORT: by name, alphabetically A→Z
  4. PAGINATE: first 10 results (page 1)

This is what a production-grade list API looks like.
```

## The 404 Rule - When to Use and When NOT to Use

This is one of the most important rules in API design that many engineers get wrong:

### Rule: NEVER Return 404 for List APIs

```
SINGLE resource APIs → 404 when not found:

  GET /api/v1/organizations/non-existent-id
  Response: 404 Not Found
  {
    "error": {
      "code": "RESOURCE_NOT_FOUND",
      "message": "Organization not found"
    }
  }
  → The client asked for a SPECIFIC resource
  → That resource doesn't exist
  → 404 is correct ✅

LIST APIs → NEVER 404, return empty array:

  GET /api/v1/organizations?status=some_random_value
  Response: 200 OK
  {
    "data": [],
    "total": 0,
    "page": 1,
    "total_pages": 0
  }
  → The client asked for a LIST of resources
  → No resources match the filter
  → BUT the endpoint exists and worked correctly
  → Return 200 with empty data ✅
  → NOT 404 ❌
```

### Why This Rule Exists

```
Think about it semantically:

GET /organizations/xyz-123
  → "Give me the SPECIFIC organization with ID xyz-123"
  → If it doesn't exist → "Resource not found" → 404

GET /organizations?status=archived
  → "Give me ALL organizations that match this filter"
  → No matches → "Here's your list. It's empty." → 200 + []
  → The LIST endpoint itself was found and worked fine
  → The result just happens to be empty

Analogy:
  → "Show me Book #12345" → "That book doesn't exist" → 404
  → "Show me all sci-fi books" → "Here's your shelf. It's empty." → 200 + []
  → The shelf exists, there's just nothing on it.
```

```
Summary:

┌──────────────────────┬────────────────────────────────────┐
│ Scenario             │ Response                           │
├──────────────────────┼────────────────────────────────────┤
│ GET /orgs/:id        │                                    │
│   Found              │ 200 + organization data            │
│   Not found          │ 404 + error message                │
│                      │                                    │
│ GET /orgs            │                                    │
│   Has results        │ 200 + [org1, org2, ...]            │
│   No results         │ 200 + [] (EMPTY ARRAY, NOT 404!)   │
│   Filter no match    │ 200 + [] (EMPTY ARRAY, NOT 404!)   │
└──────────────────────┴────────────────────────────────────┘
```

## Custom Actions - Practical Demo

### Example 1: Archive Organization

**Scenario:** We want to archive an organization. This involves:
- Changing the organization's status from "active" to "archived"
- Removing or archiving all projects under that organization
- Deleting or archiving all tasks under those projects
- Sending notifications/emails to users in that organization

```
Why NOT use PATCH?

  You might think: "Archiving = updating status to 'archived'.
  So just use PATCH /organizations/:id with { "status": "archived" }"

  BUT archiving is NOT just a field update:
    → Projects must be removed/archived
    → Tasks must be deleted/archived
    → Users must be notified
    → Emails must be sent
    → Audit logs must be created
    → Background jobs might be triggered

  This is a BUSINESS ACTION, not a simple field update.
  The side effects go far beyond changing one field.
  That's why it's a CUSTOM ACTION.
```

```
The API:

  POST /api/v1/organizations/:id/archive

  Method: POST (custom actions always use POST)
  URL breakdown:
    /organizations     → resource collection (plural)
    /:id               → specific organization
    /archive           → the ACTION to perform

  This follows a clear hierarchical relationship:
    Collection → Specific Item → Action on that Item

  Request:
    POST /api/v1/organizations/org-005/archive
    Body: None (or optional: { "reason": "No longer active" })

  Response (200 OK):
  {
    "data": {
      "id": "org-005",
      "name": "Org Five",
      "status": "archived",
      ...
    }
  }
```

**Important:** Even though this is a POST call, the response is **200** (not 201) because no new resource was created. The status code depends on what the server actually DID, not on which HTTP method was used.

```
Don't blindly assume:
  POST → always 201 ❌

Reality:
  POST (create)        → 201 Created ✅
  POST (custom action) → 200 OK ✅ (if no new resource created)
  POST (clone/copy)    → 201 Created ✅ (if a new resource WAS created)
```

### Example 2: Clone Project

**Scenario:** The client wants to duplicate a project - all its values, and possibly all its tasks, should be cloned into a new project with a new ID.

```
Why NOT use POST /projects (create)?

  You might think: "Just copy all values and call the create API"

  BUT cloning is NOT just creating:
    → All tasks inside the project must also be cloned
    → Task assignees might need to be preserved or reset
    → The project owner gets a notification
    → An audit log entry is created
    → Maybe labels and tags are also duplicated
    → The server decides what "clone" means, not the client

  The client shouldn't have to know all the internal steps.
  The client just says: "Clone this project."
  The server handles everything else.
```

```
The API:

  POST /api/v1/projects/:id/clone

  Request:
    POST /api/v1/projects/proj-002/clone
    Body: None (or optional overrides: { "name": "Project 2 Copy" })

  Response (201 Created):  ← 201 because a NEW project was created
  {
    "data": {
      "id": "proj-new-uuid",
      "name": "Project 2 - Clone",
      "organization_id": "org-001",
      "status": "planned",
      "description": "Cloned from Project 2",
      "created_at": "2024-03-16T10:00:00Z"
    }
  }
```

### Custom Action Route Pattern

```
Pattern:
  POST /api/v1/{resource}/{id}/{action}

Examples:
  POST /organizations/:id/archive    → Archive an organization
  POST /projects/:id/clone           → Clone a project
  POST /users/:id/deactivate         → Deactivate a user
  POST /orders/:id/refund            → Refund an order
  POST /tasks/:id/reassign           → Reassign a task
  POST /invoices/:id/send            → Send an invoice
  POST /accounts/:id/freeze          → Freeze an account

The hierarchical path is always clear:
  /collection → /specific-item → /action

This reads like a sentence:
  "In the PROJECTS collection, take PROJECT xyz, and CLONE it."
```

## JSON Payload Consistency - Critical Principle

### The Rule: Same Fields = Same Names Everywhere

```
If "description" is called "description" in one API,
it MUST be called "description" in EVERY API.

Create Organization:
  POST /organizations
  { "name": "Acme", "description": "A tech company" }

Create Project:
  POST /projects
  { "name": "Project One", "description": "A new project" }
                            ↑ SAME field name ✅

If you change it to "desc" in one API:
  POST /projects
  { "name": "Project One", "desc": "A new project" }
                            ↑ INCONSISTENT ❌

The frontend engineer integrated the organizations API first.
They saw "description" and assumed projects will use the same key.
They send "description" → server doesn't recognize it → validation error.
Time wasted. Frustration caused. Bug raised. Sync meeting scheduled.

ALL because you used "desc" instead of "description."
```

### camelCase for JSON Fields

```
Standard: JSON fields ALWAYS follow camelCase.

✅ CORRECT:
  {
    "organizationId": "org-001",
    "fullName": "Alice Johnson",
    "createdAt": "2024-03-15T14:30:00Z",
    "sortOrder": "desc"
  }

❌ WRONG:
  {
    "organization_id": "org-001",   ← snake_case
    "FullName": "Alice Johnson",     ← PascalCase
    "created-at": "...",             ← kebab-case
    "sort_order": "desc"             ← snake_case
  }

Why camelCase?
  → It is the established JSON standard
  → JavaScript (the most common consumer) uses camelCase natively
  → Consistency across the entire industry
  → Clients don't have to guess
```

> **Note:** Your DATABASE uses `snake_case` (organization_id, created_at). Your API uses `camelCase` (organizationId, createdAt). Your backend code transforms between the two. This is a well-established convention.

### Consistency Across Resources

```
Once you establish a pattern in ONE API, ALL APIs must follow it:

Pattern established in Organizations:
  Route: /organizations (plural, lowercase)
  Body fields: name, description, status (full words, camelCase)
  Response: { "data": { ... } } wrapper
  Pagination: total, page, total_pages
  Status codes: 200 (success), 201 (create), 204 (delete), 404 (not found)

Now Projects MUST follow the SAME patterns:
  Route: /projects (plural, lowercase) ← NOT /project or /Project
  Body fields: name, description, status ← NOT desc, nm, sts
  Response: { "data": { ... } } ← SAME wrapper
  Pagination: SAME metadata fields
  Status codes: SAME conventions

If someone integrates your organizations API,
they should be able to predict EXACTLY how your projects API works.
ZERO guesswork. ZERO surprises. ZERO sync calls.
```

## Sane Defaults - Design Philosophy

### The Principle

> **Sane defaults** = The API works reasonably even if the client sends minimal information. Don't force clients to provide obvious values.

### Sane Defaults for List APIs (GET)

```
If the client sends:
  GET /api/v1/organizations
  (no query parameters at all)

The server should apply:
  page      = 1           (start from the beginning)
  limit     = 10 or 20    (reasonable page size)
  sort_by   = created_at  (most natural default)
  sort_order = desc        (newest first)

Result: Client gets the first page of 10-20 latest organizations.
No parameters needed. It just works.
```

### Sane Defaults for Create APIs (POST)

```
Organizations have a "status" field: active or archived.

When creating a new organization:
  → Does it make sense to require the client to send status?
  → A newly created organization is OBVIOUSLY "active"
  → The client shouldn't have to state the obvious

Sane default: status = "active" if not provided.

So this should work:
  POST /organizations
  { "name": "Acme" }
  → Server sets status = "active" because it's obvious
  → Server sets description = null because it's optional
  → Organization is created successfully with just a name ✅

NOT this:
  POST /organizations
  { "name": "Acme" }
  → 400 Error: "status is required" ❌
  → "description is required" ❌
  → Forcing clients to state the obvious = bad design
```

### What to Require vs Default

```
REQUIRE only what you ABSOLUTELY need:
  ✅ name (can't create an org without a name)
  ✅ email (can't create a user without an email)
  ✅ password (can't create an account without a password)

DEFAULT everything else:
  ✅ status → "active"
  ✅ role → "member"
  ✅ visibility → "private"
  ✅ page → 1
  ✅ limit → 10
  ✅ sort_by → created_at
  ✅ sort_order → desc

Real-world examples:
  GitHub: New repo defaults to "public", README = false
  Stripe: New customer defaults to no description, no metadata
  Slack: New channel defaults to not archived, purpose empty
```

## Avoid Abbreviations - Readability Over Brevity

```
The amount of context YOU have while creating APIs is NOT the same
as the context the CONSUMER will have while integrating your APIs.

You know "desc" means "description" because you wrote the code.
The consumer sees "desc" and thinks:
  → descending?
  → description?
  → descriptor?

❌ BAD - Abbreviated:
  {
    "nm": "Acme Corp",
    "desc": "A tech company",
    "org_id": "...",
    "dt": "2024-03-15",
    "sts": "active"
  }

✅ GOOD - Full words:
  {
    "name": "Acme Corp",
    "description": "A tech company",
    "organizationId": "...",
    "date": "2024-03-15",
    "status": "active"
  }

Rule: Always use full, descriptive words for:
  → JSON field names
  → Query parameter names
  → URL path segments
  → Error codes
  → Response metadata
```

## API Design Best Practices - Summary

### 1. Interactive Documentation (Swagger/OpenAPI)

```
One of the MOST IMPORTANT things that sets apart a good backend engineer:

→ Provide an interactive API playground (Swagger UI)
→ Consumers can TRY your APIs without writing code
→ Documentation stays up-to-date with the actual implementation
→ Consumers can see request/response formats
→ Reduces the need for "how does this API work?" meetings

Use Swagger/OpenAPI specification to:
  → Document every endpoint
  → Show request bodies and parameters
  → Show response formats and status codes
  → Provide example values
  → Enable "Try it out" functionality

Start integrating Swagger from the INITIAL PLANNING phase,
not as an afterthought.
```

### 2. Make APIs Intuitive and Consistent

```
Every route, every payload, every response should follow
the SAME patterns across your entire project.

Consistency means:
  → Same URL naming convention everywhere (plural, lowercase, hyphens)
  → Same JSON field naming everywhere (camelCase, full words)
  → Same response wrapper everywhere ({ "data": ... })
  → Same error format everywhere ({ "error": { "code": ..., "message": ... } })
  → Same pagination format everywhere (data, total, page, total_pages)
  → Same status code conventions everywhere

If you follow global REST standards → great.
If for some reason you can't → at least follow YOUR OWN standard.
But NEVER mix different styles across different resources.
```

### 3. Provide Sane Defaults

```
Minimize the information clients MUST send:
  → page, limit, sort → have defaults
  → status on create → default to logical value
  → optional fields → don't require them
  → Only require what is ABSOLUTELY necessary
```

### 4. Design BEFORE You Code

```
Critical mindset shift:

  ❌ "Let me write the code first, I'll figure out the API later"
  ✅ "Let me design the interface first, then implement it"

Steps:
  1. Open Insomnia/Postman/Swagger
  2. Design every endpoint (URL, method, body, response)
  3. Review: Is it intuitive? Consistent? Complete?
  4. Get feedback from the frontend team / API consumers
  5. THEN start coding

Why?
  → Designing first reveals edge cases early
  → You think about the API from the consumer's perspective
  → You catch inconsistencies before they're coded
  → Changing a design document is FREE
  → Changing code after integration is EXPENSIVE
```

### 5. The Golden Rule

```
Your API should be so intuitive that someone who has
integrated ONE of your resources can predict EXACTLY
how all other resources work - without reading any docs.

If they integrated /organizations:
  → They know /projects follows the same patterns
  → They know /tasks follows the same patterns
  → They know the response format, status codes, pagination
  → They can integrate 10 more resources with zero confusion

THAT is the mark of well-designed APIs.
```

## API Design Checklist

```
Before coding any API, verify:

URL Design:
  ✅ Resource names are PLURAL (/users, /books, /orders)
  ✅ Lowercase with hyphens (/user-profiles, NOT /userProfiles)
  ✅ No verbs in URLs (/users, NOT /getUsers)
  ✅ Nesting is max 2-3 levels deep
  ✅ API versioning is in the URL (/api/v1/...)

HTTP Methods:
  ✅ GET for reading (no side effects)
  ✅ POST for creating (returns 201)
  ✅ POST for custom actions (returns 200 or 201 depending on outcome)
  ✅ PATCH for partial updates
  ✅ PUT for full replacement
  ✅ DELETE for removal (returns 204 No Content)

Responses:
  ✅ Consistent response structure (data wrapper)
  ✅ Correct status codes (200, 201, 204, 400, 401, 403, 404, 409, 500)
  ✅ Pagination for ALL list APIs (data, total, page, total_pages)
  ✅ Default sort (created_at DESC) for all list APIs
  ✅ Empty array for list APIs with no results (NOT 404)
  ✅ 404 ONLY for single resource not found
  ✅ Meaningful error messages with error codes

Payloads:
  ✅ Request body only contains client-provided fields
  ✅ Server-generated fields (id, created_at) are NOT in request body
  ✅ JSON format with camelCase field names
  ✅ Same field names across all resources (description, NOT desc)
  ✅ Full words, no abbreviations

Defaults:
  ✅ page = 1 if not provided
  ✅ limit = 10 or 20 if not provided
  ✅ sort_by = created_at if not provided
  ✅ sort_order = desc if not provided
  ✅ status = "active" (or logical default) on create if not provided

Security:
  ✅ Parameterized queries (SQL injection prevention)
  ✅ Input validation on all endpoints
  ✅ Rate limiting on sensitive endpoints
  ✅ Authentication required where needed

Documentation:
  ✅ Swagger/OpenAPI interactive playground
  ✅ Every endpoint documented (URL, method, body, response, status codes)
  ✅ Example requests and responses provided
```
