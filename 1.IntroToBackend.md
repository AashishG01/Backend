# Introduction to Backend Engineering - From First Principles

> **Core idea:** A backend is a computer listening for requests through an open port, processing them, and sending responses. But more fundamentally, backend engineering exists because of **data** - the need to fetch it, receive it, persist it, and perform actions on it securely from a centralized place. Learning backend from first principles (not framework-specific tutorials) makes you a language-agnostic engineer who can navigate any codebase.

## What Exactly Is a Backend?

### The Traditional Definition

```
A backend is a COMPUTER that:
  → Listens for requests (HTTP, WebSocket, gRPC, etc.)
  → Through an open PORT (80, 443, 3000, etc.)
  → That is accessible over the internet
  → So that clients (browsers, apps) can connect to it
  → Send data TO it or receive data FROM it

We call it a SERVER because it SERVES content:
  → Static files (images, JavaScript, HTML, CSS)
  → Dynamic data (JSON responses from APIs)
  → And it also ACCEPTS data (when clients send POST requests, etc.)
```

### If You Strip It Down to One Word

```
The responsibility of a backend, condensed to a single word:

                          DATA

  → The need to FETCH data
  → The need to RECEIVE data
  → The need to PERSIST (save) data
  → Any kind of ACTION that deals with data

That is the fundamental reason backends exist.
```

## How a Backend Works - The Full Picture

Let's trace an actual request from start to finish. Not theoretically but using a real deployed server.

### The Full Request Journey

```
You type: https://backend-demo.senus.do/users
                 ↓
What actually happens (every single hop):

 ① BROWSER: You hit Enter
    → Browser sees the domain: backend-demo.senus.do
    → Browser needs to find the IP address for this domain
                 ↓

 ② DNS SERVER: Domain → IP resolution
    → DNS has records that map domain names to IP addresses
    → A Record: backend-demo → 52.xx.xx.xx (an AWS IP)
    → Browser now knows WHERE to send the request
                 ↓

 ③ AWS EC2 INSTANCE: The actual computer
    → Request arrives at the EC2 instance via its public IP
    → But first...
                 ↓

 ④ FIREWALL (Security Group):
    → AWS checks: "Is this port allowed?"
    → Security group rules define which ports are open:
      - Port 22:  SSH (terminal access for developers)
      - Port 80:  HTTP traffic ✅
      - Port 443: HTTPS traffic ✅
    → If the port is NOT allowed → AWS blocks it right here
    → If allowed → request enters the server
                 ↓

 ⑤ REVERSE PROXY (Nginx):
    → A server that sits IN FRONT of your actual application servers
    → Manages routing, SSL, and redirects from one central place
    → Nginx config says:
      "If request comes to backend-demo.senus.do →
       forward it to localhost:3001"
    → Nginx handles SSL (HTTPS) via Certbot certificates
    → Nginx redirects port 80 (HTTP) to port 443 (HTTPS)
                 ↓

 ⑥ APPLICATION SERVER (Node.js on port 3001):
    → The actual backend code runs here
    → Processes the request, queries database, runs logic
    → Returns JSON response: [{ id: 1, name: "..." }, ...]
                 ↓

 ⑦ RESPONSE travels the same path in reverse:
    Node.js → Nginx → AWS firewall → Internet → Browser
```

### Visualization

```
BROWSER  →  DNS  →  AWS FIREWALL  →  EC2 INSTANCE  →  NGINX  →  LOCALHOST:3001
   ↑                                                                    │
   └────────────────────── RESPONSE ────────────────────────────────────┘
```

### Key Components Explained

**DNS (Domain Name System):**
```
DNS is like a phone book for the internet.

You type: backend-demo.senus.do
DNS resolves: 52.xx.xx.xx (the server's IP)

DNS record types:
  A Record:     Maps domain → IP address
  CNAME Record: Maps domain → another domain name

Without DNS, you'd have to type IP addresses in your browser.
```

**Firewall (Security Group):**
```
A firewall decides which traffic is allowed into your server.

Think of it as a security guard at a building entrance:
  → "Do you have authorization to enter this port?"
  → Port 443 (HTTPS): "Yes, come in." ✅
  → Port 5432 (PostgreSQL): "No, blocked." ❌

If you forget to allow port 443 in your security group:
  → HTTPS requests can't reach your server
  → Users see "Connection timed out"
  → Your server is fine - the firewall is blocking traffic before it arrives

This is the first layer of security for any deployed backend.
```

**Reverse Proxy (Nginx):**
```
A reverse proxy sits in FRONT of your application servers.
It receives ALL incoming requests and decides where to forward them.

Why use it?
  → One server can host MULTIPLE applications on different ports:
    backend-demo.senus.do  → localhost:3001 (backend API)
    frontend-demo.senus.do → localhost:3000 (frontend app)
    
  → SSL termination: Nginx handles HTTPS certificates
    (your Node.js app doesn't need to deal with SSL)
    
  → Load balancing: Nginx can distribute requests across
    multiple instances of your app
    
  → Centralized config: Change routing rules in one place
    instead of modifying each application
```

### Local Development vs Production

```
In LOCAL DEVELOPMENT:
  Browser → localhost:3000/users → Your server responds directly
  No DNS, no firewall, no Nginx. Just you and your server.

In PRODUCTION:
  Browser → DNS → Firewall → Nginx → localhost:3001 → Your server
  Multiple hops, multiple layers of security and routing.

But the SERVER CODE is exactly the same.
Your Node.js app doesn't know or care if it's being called
from localhost or from the other side of the world.
It just listens on a port and responds to requests.
```

## Why Do We Need Backends?

### The Instagram Example

```
You open Instagram. You see your friend's post. You click "Like."

What happens between YOU clicking "Like" and YOUR FRIEND
getting a notification?

  1. You tap the like button
  2. The app sends a request to Instagram's server
  3. The server receives the request
  4. The server identifies WHO you are (authentication)
  5. The server finds WHICH post you liked
  6. The server SAVES the like in a database (persistence)
  7. The server checks WHO posted it (your friend)
  8. The server TRIGGERS a notification to your friend
  9. Your friend receives a push notification: "Aashish liked your post"

ALL of this happened on a CENTRALIZED SERVER.

Why centralized?
  → The server has ALL users' data
  → Your app only has YOUR data
  → Your friend's phone only has THEIR data
  → Someone has to know about BOTH of you → the server
  → Someone has to PERSIST the like → the server
  → Someone has to SEND the notification → the server
```

## How Frontends Work - The Contrast

Let's trace how a frontend works to understand the fundamental difference:

```
You visit: https://frontend-demo.senus.do

What happens:

  1. Browser sends GET request to the domain
  2. DNS resolves → AWS EC2 IP
  3. Firewall allows port 443
  4. Nginx forwards to localhost:3000 (Next.js server)
  5. Next.js serves: HTML file + JavaScript files + CSS files + fonts + images

  6. Browser receives the HTML file FIRST
  7. Browser reads the HTML and finds references to JS, CSS, fonts
  8. Browser fetches each resource in separate requests
  9. Once CSS is loaded → browser PAINTS the page (colors, fonts, layout)
  10. Once JS is loaded → browser HYDRATES the page (event listeners, interactions)
  11. Now buttons work, navigation works, the app is interactive
```

### The Key Difference

```
BACKEND:
  → Client sends a request
  → SERVER processes the request (runs the code)
  → Server sends back the RESULT
  → Processing happens ON THE SERVER

FRONTEND:
  → Client requests the code (HTML, JS, CSS)
  → Server sends the CODE ITSELF to the client
  → BROWSER runs the code on the user's machine
  → Processing happens ON THE CLIENT

Backend: Server runs the code, sends the result.
Frontend: Server sends the code, browser runs it.

This is a FUNDAMENTAL difference.
```

## Why Can't We Write Backend Logic in the Frontend?

There are four major reasons:

### 1. Security Restrictions (Sandboxing)

```
Browsers are SANDBOX environments.

What that means:
  → The code running in a browser is ISOLATED from the operating system
  → It CANNOT access the file system (can't read/write files)
  → It CANNOT access environment variables
  → It CANNOT run system commands
  → It CANNOT open arbitrary network connections

Why?
  → The browser is downloading and executing CODE FROM THE INTERNET
  → If that code could access your file system:
    - A malicious website could read your personal files
    - It could copy your SSH keys, passwords, documents
    - It could send everything to their servers
    - This is a horrifying scenario

  → So browsers RESTRICT what JavaScript can do:
    ✅ Can access: DOM, localStorage, cookies, browser APIs
    ❌ Cannot access: file system, OS processes, environment variables
    ❌ Cannot access: external APIs freely (restricted by CORS policy)

Backend servers need to:
  → Read/write log files
  → Access environment variables (DB passwords, API keys)
  → Execute system commands
  → None of this is possible in a browser
```

### 2. CORS Restrictions (Can't Call External APIs Freely)

```
CORS (Cross-Origin Resource Sharing) is a browser security policy.

Rule: JavaScript running on domain A cannot freely call APIs on domain B.

Example:
  Your frontend: https://myapp.com
  External API: https://payment-gateway.com/charge

  Browser says: "These are different origins. BLOCKED."
  Unless payment-gateway.com sends specific CORS headers allowing it.

Problem for backends:
  → Backends often need to call 5-10 different external services
  → Payment APIs, email services, SMS providers, other microservices
  → You have ZERO control over whether those APIs send CORS headers
  → In a browser, most of these calls would be blocked

  → Backend servers don't have CORS restrictions
  → They can freely call any API, any service, any database
```

### 3. Database Connections

```
Backend servers connect to databases using NATIVE DRIVERS:
  → pg (PostgreSQL driver for Node.js)
  → mongodb (MongoDB driver)
  → mysql2 (MySQL driver)

These drivers:
  → Open TCP socket connections to the database server
  → Maintain a CONNECTION POOL (a reusable set of connections)
  → Handle binary protocol communication
  → Manage persistent connections efficiently

Why browsers CAN'T do this:
  → Browsers cannot open raw TCP sockets
  → Browsers cannot maintain persistent database connections
  → Even IF they could:
    - Each user would need their own database connection
    - 10,000 users = 10,000 connections
    - Most databases crash beyond 100-200 connections
    - No connection pooling possible from browsers

Backend servers solve this:
  → ONE server maintains 20-50 pooled connections
  → Serves thousands of concurrent users through those 20-50 connections
  → Connection is reused, not created/destroyed per request
  → This is called CONNECTION POOLING - fundamental backend concept
```

### 4. Computing Power

```
Frontend devices vary wildly in power:
  → A ₹5,000 smartphone with 2GB RAM and a single-core processor
  → A 10-year-old laptop with 256MB free RAM
  → A high-end desktop with 32GB RAM

You cannot rely on the client having enough power
to run heavy business logic:
  → Image processing
  → PDF generation
  → Complex database queries
  → Machine learning inference
  → Report generation

If you run this on a weak phone → the app freezes, crashes, lags.
Users uninstall. Business lost.

Backend servers:
  → YOU control the hardware
  → Need more CPU? Upgrade the instance.
  → Need more RAM? Add it.
  → Need more throughput? Add more servers (horizontal scaling).
  → Consistent performance for ALL users regardless of their device.
```

```

  → How do you get up to speed without wasting hours on library docs?

ANSWER: First principles.

The ability to break down complex systems into their most basic,
UNIVERSAL components gives you a massive edge.

```

Without first principles:
  → Overwhelmed by folder structure
  → Confused by unfamiliar libraries
  → Can't tell what's important vs. what's over-engineered
  → Spend days just understanding the structure

With first principles:
  → Mentally separate the system into known components:
    "OK, this folder is routing. This is the service layer.

  → Identify core logic vs. boilerplate
  → Filter out the noise
  → Start making changes or fixing bugs with CONFIDENCE

This is what senior engineers and CTOs do instinctively.
They pick up PATTERNS across codebases because the patterns
are the same everywhere. You can develop this skill deliberately
in 6-12 months instead of waiting for years of experience.

  Without principles: "What is FastAPI? How does auth work in Python?

### Exercise 2 â€” Build a Minimal Server (No Framework)
```javascript
// Using ONLY Node.js built-in http module:
const http = require('http');

const server = http.createServer((req, res) => {
  if (req.url === '/users' && req.method === 'GET') {
    res.writeHead(200, { 'Content-Type': 'application/json' });
    res.end(JSON.stringify([{ id: 1, name: 'Aashish' }]));
  } else if (req.url === '/users' && req.method === 'POST') {
    let body = '';
    req.on('data', chunk => body += chunk);
    req.on('end', () => {
      res.writeHead(201, { 'Content-Type': 'application/json' });
      res.end(body);  // echo back
    });
  } else {
    res.writeHead(404);
    res.end('Not Found');
  }
});

server.listen(3000, () => console.log('Running on port 3000'));

// Tasks:
//   -> Run this and hit localhost:3000/users from browser
//   -> Test POST with cURL or Postman
//   -> Add a /products endpoint
//   -> You just built a backend from scratch!
```
