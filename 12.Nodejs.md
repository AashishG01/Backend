> **Core idea:** Node.js is a JavaScript runtime built on Chrome's V8 engine that lets you run JavaScript OUTSIDE the browser. It uses an event-driven, non-blocking I/O model that makes it perfect for building scalable, high-performance backend servers. Understanding Node.js deeply means understanding the event loop, V8, libuv, and how asynchronous code actually works under the hood.

## What is Node.js?

```
Node.js is NOT:
  x A programming language (JavaScript is the language)
  x A framework (Express, Koa, Fastify are frameworks)
  x A library

Node.js IS:
  -> A RUNTIME ENVIRONMENT for executing JavaScript
  -> Built on Google Chrome's V8 JavaScript engine
  -> Extended with C++ bindings for system-level operations
  -> Comes with built-in modules for file I/O, networking, crypto, etc.

Before Node.js (2009):
  JavaScript could ONLY run inside a browser.
  Browser gave JS access to: DOM, window, fetch, localStorage
  JS could NOT: read files, start servers, access databases, run system commands

After Node.js:
  JavaScript can run on the SERVER.
  Node gives JS access to: file system, TCP/HTTP, child processes, OS info
  JS can NOW: read/write files, create web servers, connect to databases
  
  Ryan Dahl created Node.js in 2009.
  He took V8 (Chrome's JS engine) and wrapped it with C++ libraries
  that give JavaScript access to the operating system.
```

## Why Node.js for Backend?

```
ADVANTAGE 1: Non-Blocking I/O
  
  Traditional servers (Java, PHP, Ruby):
    Request 1 -> Start DB query -> WAIT... -> Get result -> Respond
    Request 2 -> BLOCKED until Request 1 finishes (or new thread needed)
    Request 3 -> BLOCKED

  Node.js:
    Request 1 -> Start DB query -> DON'T WAIT, handle other requests
    Request 2 -> Start DB query -> DON'T WAIT
    Request 3 -> Start DB query -> DON'T WAIT
    DB result for Request 1 arrives -> Process and respond
    DB result for Request 3 arrives -> Process and respond
    DB result for Request 2 arrives -> Process and respond

  Node doesn't create a new THREAD for each request.
  It uses a SINGLE thread with an event loop.
  While waiting for I/O (DB, file, network) -> it handles OTHER requests.


ADVANTAGE 2: Same language front and back
  -> Frontend: JavaScript
  -> Backend: JavaScript
  -> One language for the entire stack
  -> Share code between client and server (validation, types, utils)
  -> Team doesn't need to context-switch between languages


ADVANTAGE 3: Massive ecosystem (npm)
  -> 2+ million packages on npm
  -> Package for almost anything you need
  -> Largest open-source ecosystem in the world


ADVANTAGE 4: Real-time applications
  -> WebSockets are native and performant
  -> Perfect for: chat apps, live dashboards, gaming, collaboration tools
  -> Event-driven architecture = naturally suited for real-time


WHEN NOT TO USE Node.js:
  -> CPU-intensive tasks (video encoding, ML training, heavy computation)
  -> Why? Single-threaded. CPU work BLOCKS the event loop.
  -> A 5-second calculation blocks ALL other requests for 5 seconds.
  -> For CPU work: use Worker Threads, or a different language (Go, Rust, C++).
```

## The V8 Engine

V8 is the JavaScript engine that powers both Google Chrome and Node.js. It was written by Google in C++ and is open-source. The key thing to understand is that V8 **compiles** JavaScript into machine code rather than interpreting it line by line. This is why JavaScript (and by extension Node.js) is so fast compared to traditional interpreted languages like PHP or Ruby. V8 uses a technique called **JIT (Just-In-Time) compilation** where it first interprets code quickly, then optimizes frequently-run ("hot") code paths into highly efficient machine code. V8 also handles **memory management** automatically through garbage collection, freeing developers from manual memory allocation.

```
V8 is Google's open-source JavaScript engine written in C++.
It powers both Chrome browser AND Node.js.

What V8 does:
  1. Takes your JavaScript source code
  2. Parses it into an AST (Abstract Syntax Tree)
  3. Compiles it to machine code (NOT interpreted line by line)
  4. Executes the machine code directly on the CPU

The compilation pipeline:

  JavaScript Source Code
        |
        v
  +-- Parser --+
  |  Tokenize  |  -> Break code into tokens: const, x, =, 5, +, 3
  |  Parse     |  -> Build AST (tree structure of the program)
  +------------+
        |
        v
  +-- Ignition (Interpreter) --+
  |  Generates BYTECODE        |  -> Quick to produce, slower to execute
  |  Starts executing immediately
  +----------------------------+
        |
        v (if a function runs many times - "hot code")
  +-- TurboFan (Optimizing Compiler) --+
  |  Generates OPTIMIZED MACHINE CODE  |  -> Slower to produce, FAST to execute
  |  Uses type feedback from Ignition  |
  +------------------------------------+

This is called JIT (Just-In-Time) compilation:
  -> First run: interpreted (fast startup)
  -> After many runs: compiled to optimized machine code (fast execution)
  -> If assumptions are wrong: "deoptimize" back to bytecode

Why this matters for you:
  -> V8 is FAST. Node.js inherits this speed.
  -> Write consistent types (don't mix strings and numbers in same variable)
  -> Consistent types = V8 optimizes better = faster code
```

### V8 Memory Management

```
V8 Memory (Heap):

  +-------------------------------------------+
  |              New Space (Young Gen)         |
  |   Small, short-lived objects               |
  |   Collected frequently (Scavenger GC)      |
  |   Two semi-spaces: From-space, To-space    |
  +-------------------------------------------+
  |              Old Space (Old Gen)           |
  |   Long-lived objects (survived 2+ GCs)     |
  |   Collected less frequently (Mark-Sweep)   |
  +-------------------------------------------+

  Scavenger (New Space):
    -> Very fast (~1-2ms)
    -> Copies surviving objects to the other semi-space
    -> After 2 survivals -> promoted to Old Space

  Mark-Sweep-Compact (Old Space):
    -> Slower (~50-100ms for large heaps)
    -> Marks all reachable objects
    -> Sweeps (frees) unreachable objects
    -> Compacts memory to reduce fragmentation

  Default memory limits:
    -> ~1.5 GB on 64-bit systems
    -> Can increase: node --max-old-space-size=4096 app.js (4GB)

  Memory leak patterns to avoid:
    -> Global variables that grow forever
    -> Event listeners that are never removed
    -> Closures holding references to large objects
    -> Caching without size limits
```

## The Event Loop - Heart of Node.js

The event loop is the single most important concept to understand in Node.js. It is the mechanism that allows Node.js to perform non-blocking I/O operations despite JavaScript being single-threaded. Think of it like a waiter in a restaurant - instead of standing at one table waiting for the food to be cooked (blocking), the waiter takes orders from multiple tables, passes them to the kitchen, and serves whichever table's food is ready first. The event loop does the same thing: it registers async operations, continues executing other code, and processes the results (via callbacks) when they're ready. The event loop runs in **phases**, each responsible for a specific type of callback, and it cycles through these phases repeatedly until there is nothing left to process.

```
The event loop is THE most important concept in Node.js.
If you understand the event loop, you understand Node.js.

What is it?
  -> A LOOP that continuously checks for pending tasks
  -> When there's nothing to do, it waits for events
  -> When an event arrives (I/O complete, timer fires), it runs the callback
  -> It's what makes Node.js non-blocking despite being single-threaded

The big picture:
  1. Your code runs (synchronous, top to bottom)
  2. Node registers all async operations (setTimeout, fs.readFile, etc.)
  3. Your code finishes
  4. The event loop starts checking: "Any callbacks ready to run?"
  5. It runs ready callbacks one at a time
  6. Repeats until there's nothing left to do
  7. Process exits
```

### Event Loop Phases (In Order)

```
The event loop has 6 phases. Each phase has a queue of callbacks.
It goes through them IN ORDER, repeatedly:

   +--------------------------+
   |    timers                |  -> setTimeout, setInterval callbacks
   +--------------------------+
              |
   +--------------------------+
   |    pending callbacks     |  -> system-level callbacks (TCP errors, etc.)
   +--------------------------+
              |
   +--------------------------+
   |    idle, prepare         |  -> internal Node.js use only
   +--------------------------+
              |
   +--------------------------+
   |    poll                  |  -> I/O callbacks (fs, network, etc.)
   |                          |  -> This is where Node WAITS for new events
   +--------------------------+
              |
   +--------------------------+
   |    check                 |  -> setImmediate() callbacks
   +--------------------------+
              |
   +--------------------------+
   |    close callbacks       |  -> socket.on('close'), etc.
   +--------------------------+
              |
              v
   (loop back to timers)


SPECIAL QUEUES (run between phases):
  -> process.nextTick() queue  -> runs BEFORE the next phase
  -> Promise microtask queue   -> runs after nextTick, before next phase
  
  Priority order:
    1. process.nextTick()     (highest priority)
    2. Promise callbacks      (microtask queue)
    3. Next event loop phase  (timers, poll, check, etc.)
```

### Event Loop in Action - Execution Order

```javascript
console.log('1 - Start');                    // SYNC - runs immediately

setTimeout(() => console.log('2 - setTimeout'), 0);  // timers phase

setImmediate(() => console.log('3 - setImmediate'));  // check phase

Promise.resolve().then(() => console.log('4 - Promise')); // microtask

process.nextTick(() => console.log('5 - nextTick'));  // nextTick queue

console.log('6 - End');                      // SYNC - runs immediately

// OUTPUT:
// 1 - Start
// 6 - End
// 5 - nextTick       <- nextTick runs first (highest priority)
// 4 - Promise         <- then promises (microtask queue)
// 2 - setTimeout      <- then timers phase
// 3 - setImmediate    <- then check phase

// Why this order?
// 1. Synchronous code runs first: 1, 6
// 2. Between phases: nextTick queue (5), then microtask queue (4)
// 3. Timer phase: setTimeout (2)
// 4. Check phase: setImmediate (3)
```

### Event Loop - The Blocking Trap

```javascript
// CRITICAL: The event loop is SINGLE-THREADED.
// If you block it with CPU work, EVERYTHING stops.

// BAD - blocking the event loop:
app.get('/fibonacci', (req, res) => {
  const n = parseInt(req.query.n);
  const result = fibonacci(n);  // CPU-intensive, blocks for seconds!
  res.json({ result });
  // While this runs, NO other request can be handled!
  // 1000 users waiting -> all blocked by ONE computation
});

// GOOD - offload CPU work:
const { Worker } = require('worker_threads');

app.get('/fibonacci', (req, res) => {
  const worker = new Worker('./fib-worker.js', {
    workerData: { n: parseInt(req.query.n) }
  });
  worker.on('message', (result) => res.json({ result }));
  worker.on('error', (err) => res.status(500).json({ error: err.message }));
  // Worker runs in a SEPARATE thread. Event loop is FREE.
});
```

## Non-Blocking I/O and libuv

**I/O** (Input/Output) refers to any operation that interacts with the outside world: reading a file from disk, querying a database, making an HTTP request, etc. These operations are **slow** compared to CPU operations (a disk read takes millions of CPU cycles). In traditional blocking I/O, the entire thread stops and waits for the operation to complete. Node.js uses **non-blocking I/O** — when an I/O operation starts, Node.js hands it off and continues running other code. When the result is ready, a callback is triggered. This is made possible by **libuv**, a C library written specifically for Node.js. libuv provides the event loop, manages a thread pool for operations that can't be truly async at the OS level (like file reads), and uses OS-level async mechanisms (epoll on Linux, kqueue on macOS, IOCP on Windows) for network operations.

```
libuv is the C library that gives Node.js its async superpowers.
It provides:
  -> Event loop implementation
  -> Async I/O for files, DNS, network
  -> Thread pool for operations that CAN'T be async at OS level
  -> Cross-platform support (Linux, Windows, macOS)

How non-blocking I/O works:

  SYNCHRONOUS (blocking):
    const data = fs.readFileSync('file.txt');  // Thread STOPS here
    console.log(data);                          // Runs after file is read
    doOtherWork();                              // Must wait

  ASYNCHRONOUS (non-blocking):
    fs.readFile('file.txt', (err, data) => {
      console.log(data);                        // Runs when file is done
    });
    doOtherWork();  // Runs IMMEDIATELY, doesn't wait for file


  What happens under the hood with async:
    1. Node tells libuv: "Read this file"
    2. libuv delegates to a worker thread (from its thread pool)
    3. Main thread continues executing your code
    4. Worker thread finishes reading the file
    5. libuv puts the callback in the poll queue
    6. Event loop picks it up and runs your callback


The libuv Thread Pool:
  -> Default size: 4 threads
  -> Used for: file I/O, DNS lookup, crypto, zlib
  -> NOT used for: network I/O (uses OS-level async: epoll/kqueue/IOCP)
  -> Can increase: UV_THREADPOOL_SIZE=8 node app.js (max 1024)

  Network I/O (TCP, HTTP) does NOT use the thread pool:
    -> Linux: epoll (kernel-level event notification)
    -> macOS: kqueue
    -> Windows: IOCP
    -> These are truly async at the OS level, no threads needed
```

## Modules System

Modules are how you organize code in Node.js. Instead of putting everything in one giant file, you split your code into separate files (modules), each responsible for a specific piece of functionality. A module can **export** functions, objects, or values, and other modules can **import** them. Node.js has two module systems: **CommonJS (CJS)** which was the original system using `require()` and `module.exports`, and **ES Modules (ESM)** which is the newer JavaScript standard using `import` and `export`. Most existing npm packages still use CommonJS, but new projects are increasingly adopting ES Modules. Understanding both is essential because you will encounter both in real-world codebases.

### CommonJS (CJS) - The Original

```javascript
// CommonJS is the DEFAULT module system in Node.js.
// Uses require() and module.exports.

// math.js - EXPORTING
function add(a, b) { return a + b; }
function multiply(a, b) { return a * b; }

module.exports = { add, multiply };
// OR: module.exports.add = add;
// OR: exports.add = add;  (shorthand, but has gotchas)


// app.js - IMPORTING
const { add, multiply } = require('./math');  // .js extension optional
console.log(add(2, 3));      // 5
console.log(multiply(4, 5)); // 20


// How require() works internally:
// 1. Resolve: Find the file (./math -> ./math.js -> ./math/index.js)
// 2. Load: Read the file contents
// 3. Wrap: Wraps in a function (that's why you have module, exports, __dirname)
// 4. Execute: Runs the wrapped function
// 5. Cache: Stores the result. Next require() returns cached version.

// The wrapping looks like:
(function(exports, require, module, __filename, __dirname) {
  // YOUR CODE runs here
  // That's why __dirname and __filename exist in every file!
});


// IMPORTANT: require() is SYNCHRONOUS
// It blocks until the file is read and executed.
// That's fine at startup, NOT fine in request handlers.
```

### ES Modules (ESM) - The Modern Way

```javascript
// ES Modules use import/export syntax (same as browser JavaScript).
// To use ESM in Node.js:
//   Option 1: Use .mjs file extension
//   Option 2: Add "type": "module" in package.json

// math.mjs - EXPORTING
export function add(a, b) { return a + b; }
export function multiply(a, b) { return a * b; }
export default function subtract(a, b) { return a - b; }


// app.mjs - IMPORTING
import subtract, { add, multiply } from './math.mjs';

console.log(add(2, 3));
console.log(subtract(10, 4));


// KEY DIFFERENCES from CommonJS:
// CJS: require() is synchronous, can be called anywhere
// ESM: import is async, must be at the top of the file
// CJS: module.exports can be anything (object, function, class)
// ESM: named exports + one default export
// CJS: require() can be conditional: if(x) require('./a')
// ESM: import is ALWAYS static (enables tree-shaking)
// CJS: variables are copies of exported values
// ESM: variables are LIVE bindings (updates reflect immediately)
```

### CommonJS vs ES Modules Comparison

```
+--------------------+---------------------------+---------------------------+
| Feature            | CommonJS (CJS)            | ES Modules (ESM)          |
+--------------------+---------------------------+---------------------------+
| Syntax             | require() / module.exports| import / export           |
| Loading            | Synchronous               | Asynchronous              |
| When resolved      | At runtime                | At parse time (static)    |
| Conditional import | Yes                       | No (use dynamic import()) |
| Top-level await    | No                        | Yes                       |
| File extension     | .js (default)             | .mjs (or "type":"module") |
| Used by            | Most npm packages (still) | Modern projects, browsers |
| __dirname          | Available                 | Not available (use import.meta)|
+--------------------+---------------------------+---------------------------+

Recommendation:
  -> New projects: use ESM ("type": "module" in package.json)
  -> Existing codebases: mostly CJS (slowly migrating)
  -> At interviews: know BOTH systems deeply
```

## npm and package.json

**npm** (Node Package Manager) is the default package manager for Node.js and the world's largest software registry with over 2 million packages. It lets you install third-party libraries, manage project dependencies, and run scripts. Every Node.js project has a **package.json** file at its root — this is the project's manifest. It lists the project's name, version, scripts (commands you can run), dependencies (libraries needed in production), and devDependencies (libraries needed only during development, like testing tools). The **package-lock.json** file records the exact version of every installed package and its sub-dependencies, ensuring every developer and CI server gets identical installations.

### package.json Explained

```json
{
  "name": "my-api",              // project name (lowercase, no spaces)
  "version": "1.0.0",            // semantic versioning
  "description": "My awesome API",
  "main": "src/index.js",        // entry point for CJS
  "type": "module",              // enable ESM (import/export)
  "scripts": {
    "start": "node src/index.js",
    "dev": "nodemon src/index.js",      // auto-restart on changes
    "test": "jest",
    "lint": "eslint src/"
  },
  "dependencies": {
    "express": "^4.18.2",        // production dependency
    "jsonwebtoken": "^9.0.0",
    "pg": "^8.11.0"
  },
  "devDependencies": {
    "nodemon": "^3.0.0",         // dev-only dependency
    "jest": "^29.0.0",
    "eslint": "^8.0.0"
  }
}
```

### Semantic Versioning (SemVer)

```
Version format: MAJOR.MINOR.PATCH

  MAJOR (1.x.x -> 2.x.x):
    -> Breaking changes (your code might break)
    -> API changed in incompatible way

  MINOR (1.1.x -> 1.2.x):
    -> New features added
    -> Backward compatible (existing code still works)

  PATCH (1.1.1 -> 1.1.2):
    -> Bug fixes
    -> No new features, no breaking changes

Version ranges in package.json:
  "express": "4.18.2"    -> EXACT version (risky: no bug fixes)
  "express": "^4.18.2"   -> CARET: >=4.18.2, <5.0.0 (most common)
  "express": "~4.18.2"   -> TILDE: >=4.18.2, <4.19.0
  "express": "*"          -> ANY version (dangerous, never do this)

  ^ (caret) is the DEFAULT and the recommended approach.
  It allows minor and patch updates but not major (breaking) updates.
```

## Core Modules Deep Dive

Node.js comes with a rich set of **built-in modules** that you can use without installing anything. These modules provide fundamental capabilities like creating HTTP servers (`http`), reading/writing files (`fs`), manipulating file paths (`path`), getting OS information (`os`), and working with cryptography (`crypto`). Understanding these core modules is essential because they are the foundation that all frameworks (Express, Fastify, etc.) are built upon. When you use Express to create a server, under the hood it's using Node's `http` module. When you read a config file, it's using `fs`. Knowing the raw modules helps you understand what your framework is doing and debug issues effectively.

### http - Building Servers from Scratch

```javascript
const http = require('http');

const server = http.createServer((req, res) => {
  // req = IncomingMessage (what the client sent)
  // res = ServerResponse (what you send back)

  const { method, url, headers } = req;
  console.log(`${method} ${url}`);

  // Routing (the hard way, without a framework):
  if (method === 'GET' && url === '/') {
    res.writeHead(200, { 'Content-Type': 'text/plain' });
    res.end('Hello World');
  }
  else if (method === 'GET' && url === '/api/users') {
    res.writeHead(200, { 'Content-Type': 'application/json' });
    res.end(JSON.stringify([{ id: 1, name: 'Aashish' }]));
  }
  else if (method === 'POST' && url === '/api/users') {
    let body = '';
    req.on('data', chunk => body += chunk);  // collect body chunks
    req.on('end', () => {
      const user = JSON.parse(body);
      res.writeHead(201, { 'Content-Type': 'application/json' });
      res.end(JSON.stringify({ id: 2, ...user }));
    });
  }
  else {
    res.writeHead(404, { 'Content-Type': 'application/json' });
    res.end(JSON.stringify({ error: 'Not Found' }));
  }
});

server.listen(3000, () => {
  console.log('Server running on http://localhost:3000');
});

// This is what Express does under the hood, but with 
// cleaner syntax, routing, middleware, etc.
```

### fs - File System

```javascript
const fs = require('fs');
const fsPromises = require('fs/promises');   // Promise-based (modern)

// SYNC (blocks event loop - use ONLY at startup):
const data = fs.readFileSync('config.json', 'utf-8');

// ASYNC with callbacks (old way):
fs.readFile('data.json', 'utf-8', (err, data) => {
  if (err) throw err;
  console.log(JSON.parse(data));
});

// ASYNC with Promises (modern, recommended):
const data = await fsPromises.readFile('data.json', 'utf-8');
console.log(JSON.parse(data));

// Write file:
await fsPromises.writeFile('output.json', JSON.stringify({ key: 'value' }));

// Append to file:
await fsPromises.appendFile('log.txt', 'New log entry\n');

// Check if file exists:
try {
  await fsPromises.access('file.txt');
  console.log('File exists');
} catch {
  console.log('File does not exist');
}

// Read directory:
const files = await fsPromises.readdir('./uploads');

// Create directory:
await fsPromises.mkdir('./uploads', { recursive: true });

// Delete file:
await fsPromises.unlink('temp.txt');

// File stats:
const stats = await fsPromises.stat('file.txt');
console.log(stats.size);          // size in bytes
console.log(stats.isDirectory()); // false
console.log(stats.isFile());      // true
```

### path - Path Manipulation

```javascript
const path = require('path');

// NEVER do string concatenation for paths:
// BAD:  const file = __dirname + '/uploads/' + filename;
// GOOD: const file = path.join(__dirname, 'uploads', filename);

path.join('/users', 'aashish', 'docs', 'file.txt');
// -> '/users/aashish/docs/file.txt' (OS-appropriate separators)

// -> 'C:\Users\Hp\project\src\index.js' (absolute path)

path.parse('/home/user/file.txt');
// { root: '/', dir: '/home/user', base: 'file.txt', ext: '.txt', name: 'file' }
```

### os and process

```javascript
const os = require('os');

os.cpus().length;   // Number of CPU cores (useful for clustering)
os.totalmem();      // Total RAM in bytes
os.freemem();       // Free RAM in bytes
os.platform();      // 'win32', 'linux', 'darwin'
os.homedir();       // User's home directory

// process (global, no require needed):
process.env.PORT              // environment variable
process.env.NODE_ENV          // 'development' | 'production'
process.cwd()                 // current working directory
process.pid                   // process ID
process.memoryUsage()         // { rss, heapTotal, heapUsed, external }
process.uptime()              // seconds since process started

// Graceful shutdown:
process.on('SIGTERM', () => {
  console.log('Received SIGTERM. Shutting down gracefully...');
  server.close(() => {
    db.disconnect();
    process.exit(0);
  });
});

process.on('uncaughtException', (err) => {
  console.error('Uncaught Exception:', err);
  process.exit(1);   // MUST exit after uncaught exception
});

process.on('unhandledRejection', (reason) => {
  console.error('Unhandled Rejection:', reason);
  // In Node 15+, this causes process exit by default
});
```

## Buffers and Streams

In the world of backend development, you frequently deal with raw binary data — file contents, network packets, images, encrypted data. JavaScript was originally designed for text manipulation in browsers, not binary data. **Buffers** solve this by providing a way to work with raw binary data directly in memory. A Buffer is essentially a fixed-size chunk of memory allocated outside the V8 heap, storing raw bytes.

**Streams** are equally important. Imagine you need to process a 5GB log file — you can't load it all into memory at once (your server would crash). Streams let you process data **piece by piece** (in chunks), so memory usage stays constant regardless of data size. Streams are everywhere in Node.js: HTTP request/response bodies are streams, file reads can be streams, TCP sockets are streams. The ability to **pipe** data from one stream to another (like piping a file read directly to an HTTP response) is one of Node.js's most powerful patterns.

### Buffers

```javascript
// Buffer = raw binary data in memory.
// Used when dealing with: files, network packets, images, encryption.
// JavaScript strings are Unicode. Buffers are raw bytes.

// Create buffers:
const buf1 = Buffer.from('Hello');                  // from string
const buf2 = Buffer.from([72, 101, 108, 108, 111]); // from byte array
const buf3 = Buffer.alloc(10);                       // 10 zero-filled bytes
const buf4 = Buffer.allocUnsafe(10);                 // 10 uninitialized bytes (faster)

// Buffer operations:
console.log(buf1.toString());        // 'Hello'
console.log(buf1.toString('hex'));   // '48656c6c6f'
console.log(buf1.length);            // 5 (bytes, not characters)
console.log(buf1[0]);                // 72 (byte value of 'H')

// Why allocUnsafe?
// alloc: fills with zeros first (safe but slower)
// allocUnsafe: doesn't fill (may contain old memory data, faster)
// -> Use allocUnsafe ONLY if you'll overwrite ALL bytes immediately
```

### Streams

```
Streams are one of Node.js's most POWERFUL features.

The problem without streams:
  Reading a 2GB file:
    const data = fs.readFileSync('huge.csv');  // LOADS 2GB INTO RAM
    // If 10 users do this -> 20GB RAM needed -> server crashes

  Sending a large response:
    const data = await fs.promises.readFile('movie.mp4');
    res.send(data);  // 800MB loaded into RAM, THEN sent

With streams:
  const stream = fs.createReadStream('huge.csv');
  // Reads in small CHUNKS (default 64KB). Never more in RAM.
  // If 10 users do this -> maybe 640KB RAM total. Works fine.

Types of streams:
  1. Readable   - source of data (reading a file, HTTP request body)
  2. Writable   - destination of data (writing a file, HTTP response)
  3. Duplex     - both readable and writable (TCP socket)
  4. Transform  - modify data as it passes through (gzip compression)
```

```javascript
// Reading a file with streams:
const readStream = fs.createReadStream('large-file.csv', {
  encoding: 'utf-8',
  highWaterMark: 64 * 1024,  // chunk size: 64KB
});

readStream.on('data', (chunk) => {
  console.log(`Received ${chunk.length} bytes`);
  // Process each chunk independently
});
readStream.on('end', () => console.log('Done reading'));
readStream.on('error', (err) => console.error(err));


// Writing with streams:
const writeStream = fs.createWriteStream('output.txt');
writeStream.write('Line 1\n');
writeStream.write('Line 2\n');
writeStream.end('Final line\n');  // end() = last write + close


// PIPING - the most common stream pattern:
// Connect a readable stream to a writable stream.
const readStream = fs.createReadStream('input.txt');
const writeStream = fs.createWriteStream('output.txt');

readStream.pipe(writeStream);
// That's it! Data flows chunk by chunk from input to output.
// Backpressure is handled automatically.


// Real-world: serve a large file over HTTP:
app.get('/video', (req, res) => {
  const stream = fs.createReadStream('./movie.mp4');
  res.setHeader('Content-Type', 'video/mp4');
  stream.pipe(res);   // Stream directly to client. No RAM issues.
});


// Piping through a transform (compress on the fly):
const zlib = require('zlib');

app.get('/data', (req, res) => {
  const fileStream = fs.createReadStream('data.json');
  const gzip = zlib.createGzip();    // transform stream

  res.setHeader('Content-Encoding', 'gzip');
  fileStream.pipe(gzip).pipe(res);
  // Read file -> compress each chunk -> send compressed chunk to client
  // Memory usage: ~128KB regardless of file size!
});
```

## Event Emitters

The **EventEmitter** is a fundamental design pattern in Node.js, and nearly every core module is built on top of it. Streams are EventEmitters (they emit `data`, `end`, `error` events). HTTP servers are EventEmitters (they emit `request` events). The pattern is simple: an object can **emit** named events, and other parts of your code can **listen** for those events and react. This is the **Observer pattern** — the emitter doesn't know or care who is listening, it just announces that something happened. This creates loose coupling between components, making your code easier to extend and maintain. You can also create your own EventEmitters for custom business logic.

```javascript
// Node.js is built on events. Almost everything is an EventEmitter.
// Streams, HTTP servers, child processes - all emit events.

const EventEmitter = require('events');

// Create your own event emitter:
class OrderService extends EventEmitter {
  placeOrder(order) {
    // Process the order...
    this.emit('order:placed', order);    // fire the event
  }

  cancelOrder(orderId) {
    // Cancel logic...
    this.emit('order:cancelled', { orderId });
  }
}

const orderService = new OrderService();

// Register listeners:
orderService.on('order:placed', (order) => {
  console.log('Send confirmation email for order:', order.id);
});

orderService.on('order:placed', (order) => {
  console.log('Update inventory for order:', order.id);
});

orderService.on('order:cancelled', ({ orderId }) => {
  console.log('Process refund for order:', orderId);
});

// Trigger:
orderService.placeOrder({ id: 1, product: 'Laptop', qty: 1 });
// -> Send confirmation email for order: 1
// -> Update inventory for order: 1

// This is the Observer pattern.
// orderService doesn't KNOW who is listening.
// It just emits events. Listeners react independently.
// Easy to add new behaviors without modifying OrderService.


// Important methods:
emitter.on('event', callback);        // listen (persistent)
emitter.once('event', callback);      // listen ONCE then auto-remove
emitter.off('event', callback);       // remove a specific listener
emitter.removeAllListeners('event');   // remove all listeners
emitter.listenerCount('event');        // how many listeners
emitter.setMaxListeners(20);           // default max is 10 (memory leak warning)
```

## Error Handling Patterns

Error handling is one of the most critical aspects of building reliable backend systems. In Node.js, errors can occur in three different contexts: **synchronous code** (caught with try/catch), **callback-based async code** (handled via the error-first callback pattern where the first argument is always the error), and **Promise-based async code** (handled with `.catch()` or try/catch with async/await). Getting error handling wrong in Node.js can crash your entire server since it runs on a single thread — one unhandled exception takes down everything. In production, you must handle `uncaughtException` and `unhandledRejection` events at the process level as a safety net.

### Synchronous Errors

```javascript
// Use try/catch for synchronous code:
try {
  const data = JSON.parse(invalidJson);
} catch (err) {
  console.error('Parse error:', err.message);
}
```

### Callback Errors (Error-First Pattern)

```javascript
// Node.js convention: callback(error, result)
// ALWAYS check error FIRST.

fs.readFile('file.txt', 'utf-8', (err, data) => {
  if (err) {
    console.error('Failed to read:', err.message);
    return;  // MUST return to avoid running success code
  }
  console.log(data);
});
```

### Promise Errors

```javascript
// .catch() for promises:
fetchUser(1)
  .then(user => sendEmail(user.email))
  .then(result => console.log('Email sent'))
  .catch(err => console.error('Failed:', err.message));

// try/catch for async/await:
async function handleRequest() {
  try {
    const user = await fetchUser(1);
    const email = await sendEmail(user.email);
    return { success: true };
  } catch (err) {
    console.error('Error:', err.message);
    return { success: false, error: err.message };
  }
}
```

### Express Error Handling

```javascript
// Async errors in Express MUST be caught and passed to next():

// Without wrapper (verbose, error-prone):
app.get('/users/:id', async (req, res, next) => {
  try {
    const user = await db.findUser(req.params.id);
    if (!user) return res.status(404).json({ error: 'Not found' });
    res.json(user);
  } catch (err) {
    next(err);   // pass to global error handler
  }
});

// With asyncHandler wrapper (clean):
const asyncHandler = (fn) => (req, res, next) =>
  Promise.resolve(fn(req, res, next)).catch(next);

app.get('/users/:id', asyncHandler(async (req, res) => {
  const user = await db.findUser(req.params.id);
  if (!user) return res.status(404).json({ error: 'Not found' });
  res.json(user);
}));
// Errors automatically forwarded to global handler. No try/catch needed.

// Global error handler (MUST have 4 parameters):
app.use((err, req, res, next) => {
  console.error(err.stack);
  res.status(err.statusCode || 500).json({
    error: {
      message: err.message || 'Internal Server Error',
      ...(process.env.NODE_ENV === 'development' && { stack: err.stack }),
    }
  });
});
```

## Async Patterns

Asynchronous programming is at the heart of everything in Node.js. Since Node is single-threaded, it cannot afford to wait around for slow operations (file reads, DB queries, API calls). Instead, it starts the operation and moves on, handling the result later when it's ready. Over the years, JavaScript has evolved through three major approaches to writing async code: **Callbacks** (the original way, prone to "callback hell"), **Promises** (introduced in ES6, chainable with `.then()`), and **async/await** (introduced in ES2017, makes async code look synchronous and readable). Today, async/await is the standard and what you should use in all new code. Understanding all three is important because you'll encounter legacy codebases using callbacks and promise chains.

### The Evolution

```javascript
// STAGE 1: Callbacks (2009 - the original way)
fs.readFile('a.txt', (err, a) => {
  if (err) throw err;
  fs.readFile('b.txt', (err, b) => {
    if (err) throw err;
    fs.readFile('c.txt', (err, c) => {
      if (err) throw err;
      console.log(a + b + c);   // CALLBACK HELL!
    });
  });
});


// STAGE 2: Promises (2015 - ES6)
fsPromises.readFile('a.txt')
  .then(a => fsPromises.readFile('b.txt').then(b => [a, b]))
  .then(([a, b]) => fsPromises.readFile('c.txt').then(c => [a, b, c]))
  .then(([a, b, c]) => console.log(a + b + c))
  .catch(err => console.error(err));


// STAGE 3: async/await (2017 - ES2017) - THE STANDARD TODAY
async function readAll() {
  try {
    const a = await fsPromises.readFile('a.txt', 'utf-8');
    const b = await fsPromises.readFile('b.txt', 'utf-8');
    const c = await fsPromises.readFile('c.txt', 'utf-8');
    console.log(a + b + c);
  } catch (err) {
    console.error(err);
  }
}


// PARALLEL execution (when operations are independent):
async function readAllParallel() {
  const [a, b, c] = await Promise.all([
    fsPromises.readFile('a.txt', 'utf-8'),
    fsPromises.readFile('b.txt', 'utf-8'),
    fsPromises.readFile('c.txt', 'utf-8'),
  ]);
  console.log(a + b + c);
  // All three files read SIMULTANEOUSLY. Much faster!
}


// Promise.allSettled (don't fail if one fails):
const results = await Promise.allSettled([
  fetchUser(1),
  fetchUser(2),
  fetchUser(999),  // this might fail
]);
// results: [
//   { status: 'fulfilled', value: { name: 'Aashish' } },
//   { status: 'fulfilled', value: { name: 'Someone' } },
//   { status: 'rejected', reason: Error('Not found') }
// ]
// All results available! Failed ones don't kill the others.
```

## Interview Questions - Part 1

### Q1: What is the event loop? Explain its phases.
```
The event loop is Node.js's mechanism for handling asynchronous operations
on a single thread. It continuously checks for pending tasks and executes
their callbacks when ready.

Phases (in order):
  1. Timers: setTimeout, setInterval callbacks
  2. Pending callbacks: system-level callbacks (TCP errors)
  3. Idle/prepare: internal use
  4. Poll: I/O callbacks, waits for new events
  5. Check: setImmediate() callbacks
  6. Close callbacks: socket.on('close'), etc.

Between phases: process.nextTick() and Promise microtasks run.
```

### Q2: Is Node.js single-threaded?
```
Yes and No.

YES: Your JavaScript code runs on a SINGLE thread (the main event loop thread).
NO: Node.js itself uses MULTIPLE threads behind the scenes.
  -> libuv has a thread pool (default 4 threads) for:
     file I/O, DNS lookups, crypto operations
  -> Worker Threads API lets you create JS threads for CPU-intensive work

The event loop (your code) is single-threaded.
The RUNTIME (libuv, V8, C++ bindings) is multi-threaded.
```

### Q3: What is the difference between process.nextTick() and setImmediate()?
```
process.nextTick():
  -> Runs BEFORE the next event loop phase
  -> Highest priority among async callbacks
  -> Can starve I/O if called recursively (dangerous)
  -> Use for: critical operations that MUST run before any I/O

setImmediate():
  -> Runs in the CHECK phase (after I/O poll phase)
  -> Lower priority than nextTick
  -> Won't starve I/O
  -> Use for: running code after I/O events are processed

General rule: Prefer setImmediate() over process.nextTick()
unless you have a specific reason to run before I/O.
```

### Q4: What is callback hell and how do you avoid it?
```
Callback hell = deeply nested callbacks that become unreadable.

fs.readFile('a.txt', (err, a) => {
  fs.readFile('b.txt', (err, b) => {
    db.save(data, (err, result) => {
      sendEmail(result, (err, ok) => {
        // 4 levels deep... unreadable, unmaintainable
      });
    });
  });
});

Solutions:
  1. Promises: .then().then().catch()
  2. async/await: sequential, readable (the modern standard)
  3. Named functions: extract each callback into a named function
```

### Q5: What is the difference between require() and import?
```
require() = CommonJS (CJS):
  -> Synchronous loading
  -> Can be called conditionally: if(x) require('./a')
  -> Returns a copy of the exported value
  -> __dirname, __filename available
  -> Default in Node.js

import = ES Modules (ESM):
  -> Asynchronous loading
  -> Must be at the top of the file (static analysis)
  -> Live bindings (updates reflect)
  -> Uses import.meta.url instead of __dirname
  -> Requires "type": "module" or .mjs extension
```

### Q6: What is the purpose of package-lock.json?
```
package-lock.json records the EXACT version of every installed package
and all its sub-dependencies.

Without it: npm install might install different versions on different
machines (because ^4.18.2 allows any 4.x.x).

With it: everyone gets the EXACT same dependency tree.

Rules:
  -> Always commit package-lock.json to git
  -> Never edit it manually
  -> Use npm ci (not npm install) in CI/CD for exact reproducibility
```

### Q7: Explain streams in Node.js. Why are they important?
```
Streams process data PIECE BY PIECE instead of loading everything
into memory at once.

Without streams: A 2GB file needs 2GB of RAM to process.
With streams: Same file needs ~64KB (one chunk at a time).

4 types:
  Readable: source of data (fs.createReadStream)
  Writable: destination (fs.createWriteStream)
  Duplex: both (TCP socket)
  Transform: modify in transit (zlib gzip)

Pipe pattern: readable.pipe(transform).pipe(writable)
This is how Node.js handles large files, video streaming,
HTTP request/response bodies, and data compression.
```

### Q8: How does V8 garbage collection work?
```
V8 divides memory into two generations:

Young Generation (New Space):
  -> Small, short-lived objects
  -> Scavenger GC: fast, ~1-2ms
  -> Two semi-spaces; surviving objects copied between them
  -> After 2 survivals -> promoted to Old Space

Old Generation (Old Space):
  -> Long-lived objects
  -> Mark-Sweep-Compact: slower, ~50-100ms
  -> Marks reachable objects, sweeps unreachable, compacts memory

Common memory leak causes:
  -> Global variables that accumulate data
  -> Event listeners never removed
  -> Closures referencing large objects
  -> Unbounded caches
```

### Q9: What is backpressure in streams?
```
Backpressure happens when the writable stream can't keep up with
the readable stream. Data comes in faster than it can be written.

Without backpressure handling:
  -> Data piles up in memory -> memory bloat -> crash

With proper handling (pipe does this automatically):
  -> Writable says "I'm full, pause reading"
  -> Readable pauses until writable is ready
  -> Writable says "I'm ready" -> readable resumes

readable.pipe(writable) handles backpressure automatically.
If you manually use .on('data'), YOU must handle it yourself.
```

### Q10: What happens when you require() a module?
```
5 steps:

1. Resolving: Find the file path
   -> './math' -> tries ./math.js, ./math/index.js, ./math.json
   -> 'express' -> looks in node_modules/ (walks up directory tree)

2. Loading: Read the file contents from disk

3. Wrapping: Node wraps the code in a function:
   (function(exports, require, module, __filename, __dirname) {
     // your module code here
   });

4. Execution: The wrapper function is called

5. Caching: The module.exports object is cached.
   Future require() calls return the CACHED version.
   Module code runs ONLY ONCE.
```
