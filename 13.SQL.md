## What is SQL?

**SQL = Structured Query Language**

It's the **standard language** used to communicate with relational database management systems (RDBMS) like PostgreSQL, MySQL, SQLite, SQL Server, and Oracle.

```
┌─────────────────────────────────────────────────────────────────┐
│                         SQL IN A NUTSHELL                        │
│                                                                  │
│  • Developed by IBM in the 1970s (originally called SEQUEL)     │
│  • Standardized by ANSI (1986) and ISO (1987)                   │
│  • Declarative language — you tell WHAT you want, not HOW       │
│  • Used to Create, Read, Update, Delete data (CRUD)             │
│  • Also manages database structure, permissions, transactions   │
│                                                                  │
│  "SQL is to databases what HTTP is to the web"                  │
└─────────────────────────────────────────────────────────────────┘
```

### Declarative vs Imperative

```
Imperative (JavaScript):
  → "Open the file, loop through rows, check if age > 25, collect matches, return"
  → You specify EVERY STEP.

Declarative (SQL):
  → SELECT * FROM users WHERE age > 25;
  → You specify WHAT you want. The database figures out HOW.

The database's query optimizer decides the most efficient execution plan.
You focus on the result, not the algorithm.
```

> **Intuition:** Think of SQL like ordering food at a restaurant. You say "I want a pizza with mushrooms" (declarative). You don't tell the chef which oven to use, what temperature, or when to add cheese. That's the database engine's job.

## SQL vs NoSQL — Quick Recap

| Aspect | SQL (Relational) | NoSQL (Non-Relational) |
|--------|------------------|----------------------|
| **Structure** | Tables (rows & columns) | Documents, Key-Value, Graph, Wide-Column |
| **Schema** | Fixed, predefined | Dynamic, flexible |
| **Language** | SQL (standardized) | Varies per database |
| **Scaling** | Vertical (scale up) | Horizontal (scale out) |
| **ACID** | ✅ Full support | Varies (eventual consistency often) |
| **Joins** | ✅ Native support | ❌ Typically no native joins |
| **Best for** | Structured data, complex queries | Unstructured data, massive scale |
| **Examples** | PostgreSQL, MySQL, Oracle | MongoDB, Redis, Cassandra |

> **Key takeaway:** SQL databases prioritize **data integrity and relationships**. NoSQL databases prioritize **flexibility and horizontal scalability**.

## SQL Sub-Languages

SQL is not just one thing — it's divided into **five sub-languages**, each responsible for different operations:

```
┌─────────────────────────────────────────────────────────────────┐
│                     SQL SUB-LANGUAGES                            │
│                                                                  │
│  DDL (Data Definition Language)                                 │
│    → Define/modify database STRUCTURE                           │
│    → CREATE, ALTER, DROP, TRUNCATE, RENAME                      │
│                                                                  │
│  DML (Data Manipulation Language)                               │
│    → Manipulate the actual DATA                                 │
│    → INSERT, UPDATE, DELETE, MERGE                              │
│                                                                  │
│  DQL (Data Query Language)                                      │
│    → Query/retrieve data                                        │
│    → SELECT                                                     │
│                                                                  │
│  DCL (Data Control Language)                                    │
│    → Control ACCESS and permissions                             │
│    → GRANT, REVOKE                                              │
│                                                                  │
│  TCL (Transaction Control Language)                             │
│    → Manage TRANSACTIONS                                        │
│    → COMMIT, ROLLBACK, SAVEPOINT, SET TRANSACTION               │
└─────────────────────────────────────────────────────────────────┘
```

> **Interview tip:** Knowing these categories shows you understand SQL at a structural level, not just syntax.

## Data Definition Language (DDL)

DDL commands define the **structure** of your database — tables, columns, data types, constraints.

### CREATE — Build New Structures

```sql
-- Create a database
CREATE DATABASE company_db;

-- Create a table
CREATE TABLE employees (
    emp_id      INT PRIMARY KEY AUTO_INCREMENT,  -- MySQL
    -- emp_id   SERIAL PRIMARY KEY,              -- PostgreSQL equivalent
    first_name  VARCHAR(50) NOT NULL,
    last_name   VARCHAR(50) NOT NULL,
    email       VARCHAR(100) UNIQUE NOT NULL,
    department  VARCHAR(50) DEFAULT 'General',
    salary      DECIMAL(10, 2) CHECK (salary > 0),
    hire_date   DATE NOT NULL,
    is_active   BOOLEAN DEFAULT TRUE
);
```

```
Anatomy of a CREATE TABLE:
┌─────────────────────────────────────────────────────────────┐
│  Column Name  │  Data Type      │  Constraints              │
├───────────────┼─────────────────┼───────────────────────────┤
│  emp_id       │  INT            │  PRIMARY KEY, AUTO_INCR   │
│  first_name   │  VARCHAR(50)    │  NOT NULL                 │
│  email        │  VARCHAR(100)   │  UNIQUE, NOT NULL         │
│  salary       │  DECIMAL(10,2)  │  CHECK (salary > 0)       │
│  department   │  VARCHAR(50)    │  DEFAULT 'General'        │
└─────────────────────────────────────────────────────────────┘
```

### ALTER — Modify Existing Structures

```sql
-- Add a new column
ALTER TABLE employees ADD COLUMN phone VARCHAR(15);

-- Modify column data type
ALTER TABLE employees MODIFY COLUMN phone VARCHAR(20);    -- MySQL
-- ALTER TABLE employees ALTER COLUMN phone TYPE VARCHAR(20); -- PostgreSQL

-- Drop a column
ALTER TABLE employees DROP COLUMN phone;

-- Rename a column
ALTER TABLE employees RENAME COLUMN department TO dept;

-- Add a constraint
ALTER TABLE employees ADD CONSTRAINT chk_salary CHECK (salary >= 15000);

-- Rename a table
ALTER TABLE employees RENAME TO staff;
```

### DROP — Permanently Delete Structures

```sql
-- Drop a table (GONE FOREVER — no undo!)
DROP TABLE employees;

-- Drop only if it exists (prevents errors)
DROP TABLE IF EXISTS employees;

-- Drop a database
DROP DATABASE company_db;
```

> **⚠️ Warning:** `DROP` is **irreversible** in production. Always have backups. Always double-check.

### TRUNCATE — Remove All Data, Keep Structure

```sql
-- Removes ALL rows but keeps the table structure intact
TRUNCATE TABLE employees;
```

**TRUNCATE vs DELETE:**

| Aspect | TRUNCATE | DELETE |
|--------|----------|--------|
| **Removes** | All rows | Specific rows (with WHERE) or all |
| **Speed** | Very fast (deallocates pages) | Slower (row-by-row) |
| **Logging** | Minimal logging | Full logging (each row logged) |
| **Rollback** | ❌ Cannot rollback (most RDBMS) | ✅ Can rollback |
| **Triggers** | ❌ Does NOT fire triggers | ✅ Fires DELETE triggers |
| **Identity reset** | ✅ Resets auto-increment | ❌ Does not reset |
| **WHERE clause** | ❌ Not supported | ✅ Supported |

> **Intuition:** `TRUNCATE` is like erasing a whiteboard clean. `DELETE` is like erasing specific lines one by one.

## Data Manipulation Language (DML)

DML commands work with the **data** inside tables.

### INSERT — Add New Data

```sql
-- Insert a single row
INSERT INTO employees (first_name, last_name, email, salary, hire_date)
VALUES ('Alice', 'Smith', 'alice@company.com', 75000.00, '2024-01-15');

-- Insert multiple rows at once (batch insert)
INSERT INTO employees (first_name, last_name, email, salary, hire_date)
VALUES
    ('Bob', 'Jones', 'bob@company.com', 82000.00, '2024-02-01'),
    ('Charlie', 'Brown', 'charlie@company.com', 68000.00, '2024-03-10'),
    ('Diana', 'Prince', 'diana@company.com', 95000.00, '2024-01-20');

-- Insert from another table
INSERT INTO archived_employees (first_name, last_name, email)
SELECT first_name, last_name, email
FROM employees
WHERE is_active = FALSE;
```

### UPDATE — Modify Existing Data

```sql
-- Update specific rows
UPDATE employees
SET salary = 80000.00, department = 'Engineering'
WHERE emp_id = 1;

-- Update with calculation
UPDATE employees
SET salary = salary * 1.10   -- 10% raise
WHERE department = 'Engineering';

-- Update all rows (DANGEROUS without WHERE!)
UPDATE employees SET is_active = TRUE;
```

> **⚠️ Golden Rule:** NEVER run `UPDATE` or `DELETE` without a `WHERE` clause in production. Always test with `SELECT` first.

```sql
-- SAFE approach: First SELECT to verify which rows will be affected
SELECT * FROM employees WHERE department = 'Sales';

-- Then UPDATE only if the SELECT returned the expected rows
UPDATE employees SET salary = salary * 1.05 WHERE department = 'Sales';
```

### DELETE — Remove Data

```sql
-- Delete specific rows
DELETE FROM employees WHERE emp_id = 5;

-- Delete with condition
DELETE FROM employees WHERE is_active = FALSE AND hire_date < '2020-01-01';

-- Delete ALL rows (use TRUNCATE instead for better performance)
DELETE FROM employees;
```

## Data Query Language (DQL) — SELECT Deep Dive

`SELECT` is the **most used SQL command**. Most of your work as a backend engineer involves reading data.

### Basic Syntax

```sql
SELECT column1, column2, ...
FROM table_name
WHERE condition
ORDER BY column
LIMIT n;
```

### SELECT Variations

```sql
-- Select all columns
SELECT * FROM employees;

-- Select specific columns
SELECT first_name, last_name, salary FROM employees;

-- Aliases — rename columns in output
SELECT
    first_name AS "First Name",
    last_name AS "Last Name",
    salary AS "Annual Salary"
FROM employees;

-- Calculated columns
SELECT
    first_name,
    salary,
    salary * 12 AS annual_salary,
    salary * 0.30 AS tax_deduction
FROM employees;

-- DISTINCT — remove duplicates
SELECT DISTINCT department FROM employees;

-- Count distinct values
SELECT COUNT(DISTINCT department) AS dept_count FROM employees;
```

### SQL Execution Order (CRITICAL for Interviews)

```
Written Order:          Execution Order:
─────────────           ─────────────────
SELECT                  1. FROM / JOIN
FROM                    2. WHERE
WHERE                   3. GROUP BY
GROUP BY                4. HAVING
HAVING                  5. SELECT
ORDER BY                6. DISTINCT
LIMIT                   7. ORDER BY
                        8. LIMIT / OFFSET

Why does this matter?
→ You CANNOT use a column alias (defined in SELECT) inside WHERE
   because WHERE executes BEFORE SELECT!

-- ❌ This FAILS:
SELECT salary * 12 AS annual FROM employees WHERE annual > 500000;
-- Error: "annual" doesn't exist yet when WHERE runs!

-- ✅ This WORKS:
SELECT salary * 12 AS annual FROM employees WHERE salary * 12 > 500000;
-- Or use a subquery / CTE
```

> **Interview gold:** Understanding SQL execution order separates beginners from intermediate developers. Always remember: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT.

## Filtering & Sorting

### WHERE Clause — The Filter

```sql
-- Comparison operators
SELECT * FROM employees WHERE salary > 70000;
SELECT * FROM employees WHERE department = 'Engineering';
SELECT * FROM employees WHERE hire_date >= '2024-01-01';

-- AND, OR, NOT
SELECT * FROM employees
WHERE department = 'Engineering' AND salary > 80000;

SELECT * FROM employees
WHERE department = 'Engineering' OR department = 'Sales';

SELECT * FROM employees
WHERE NOT department = 'HR';

-- IN — match against a list
SELECT * FROM employees
WHERE department IN ('Engineering', 'Sales', 'Marketing');

-- BETWEEN — range (inclusive on both ends)
SELECT * FROM employees
WHERE salary BETWEEN 60000 AND 90000;
-- Equivalent to: salary >= 60000 AND salary <= 90000

-- LIKE — pattern matching
SELECT * FROM employees WHERE first_name LIKE 'A%';      -- starts with A
SELECT * FROM employees WHERE last_name LIKE '%son';      -- ends with 'son'
SELECT * FROM employees WHERE email LIKE '%@gmail%';      -- contains @gmail
SELECT * FROM employees WHERE first_name LIKE '_o%';      -- second char is 'o'
-- % = any number of characters, _ = exactly one character

-- IS NULL / IS NOT NULL
SELECT * FROM employees WHERE phone IS NULL;
SELECT * FROM employees WHERE phone IS NOT NULL;
-- Note: NULL != empty string. NULL means "unknown/missing value"
-- You CANNOT use = NULL, you MUST use IS NULL
```

### ORDER BY — Sorting Results

```sql
-- Ascending (default)
SELECT * FROM employees ORDER BY salary ASC;

-- Descending
SELECT * FROM employees ORDER BY salary DESC;

-- Multiple columns
SELECT * FROM employees
ORDER BY department ASC, salary DESC;
-- First sorts by department A→Z, then within each department sorts salary high→low

-- Order by column position (not recommended but valid)
SELECT first_name, salary FROM employees ORDER BY 2 DESC;
-- Orders by the 2nd selected column (salary)
```

### LIMIT & OFFSET — Pagination

```sql
-- Get first 10 rows
SELECT * FROM employees ORDER BY hire_date DESC LIMIT 10;

-- Pagination: Page 1 (rows 1-10)
SELECT * FROM employees ORDER BY emp_id LIMIT 10 OFFSET 0;

-- Page 2 (rows 11-20)
SELECT * FROM employees ORDER BY emp_id LIMIT 10 OFFSET 10;

-- Page 3 (rows 21-30)
SELECT * FROM employees ORDER BY emp_id LIMIT 10 OFFSET 20;

-- Formula: OFFSET = (page_number - 1) * page_size
```

## SQL Functions — Aggregate & Scalar

### Aggregate Functions — Operate on Multiple Rows

These functions take a **set of rows** and return a **single value**.

```sql
-- COUNT — number of rows
SELECT COUNT(*) AS total_employees FROM employees;
SELECT COUNT(phone) AS employees_with_phone FROM employees;
-- COUNT(*) counts all rows, COUNT(column) ignores NULLs

-- SUM — total of a numeric column
SELECT SUM(salary) AS total_payroll FROM employees;

-- AVG — average value
SELECT AVG(salary) AS avg_salary FROM employees;

-- MIN / MAX — smallest / largest
SELECT MIN(salary) AS lowest_salary FROM employees;
SELECT MAX(salary) AS highest_salary FROM employees;

-- Combining aggregates
SELECT
    COUNT(*) AS total,
    SUM(salary) AS total_payroll,
    AVG(salary) AS avg_salary,
    MIN(salary) AS min_salary,
    MAX(salary) AS max_salary
FROM employees
WHERE is_active = TRUE;
```

### Scalar Functions — Operate on Individual Values

```sql
-- String functions
SELECT UPPER(first_name) FROM employees;                  -- ALICE
SELECT LOWER(email) FROM employees;                       -- alice@company.com
SELECT LENGTH(first_name) FROM employees;                 -- 5
SELECT CONCAT(first_name, ' ', last_name) AS full_name FROM employees;
SELECT SUBSTRING(email, 1, 5) FROM employees;             -- alice
SELECT TRIM('  hello  ');                                  -- 'hello'
SELECT REPLACE('Hello World', 'World', 'SQL');            -- 'Hello SQL'

-- Numeric functions
SELECT ROUND(salary / 12, 2) AS monthly_salary FROM employees;  -- 6250.00
SELECT CEIL(4.2);     -- 5
SELECT FLOOR(4.8);    -- 4
SELECT ABS(-100);     -- 100
SELECT MOD(10, 3);    -- 1 (remainder)

-- Date functions
SELECT CURRENT_DATE;                                       -- 2024-12-15
SELECT CURRENT_TIMESTAMP;                                  -- 2024-12-15 10:30:00
SELECT EXTRACT(YEAR FROM hire_date) FROM employees;       -- 2024
SELECT AGE(CURRENT_DATE, hire_date) FROM employees;       -- PostgreSQL: interval
SELECT DATE_PART('month', hire_date) FROM employees;      -- 1, 2, 3, etc.

-- Conditional functions
SELECT
    first_name,
    salary,
    CASE
        WHEN salary >= 90000 THEN 'Senior'
        WHEN salary >= 70000 THEN 'Mid-Level'
        ELSE 'Junior'
    END AS level
FROM employees;

-- COALESCE — returns first non-NULL value
SELECT COALESCE(phone, 'No phone on file') AS contact FROM employees;

-- NULLIF — returns NULL if two values are equal
SELECT NULLIF(salary, 0);  -- returns NULL if salary is 0, prevents division by zero
```

## GROUP BY & HAVING

### GROUP BY — Group Rows for Aggregation

`GROUP BY` divides rows into groups, then applies aggregate functions **per group**.

```sql
-- Count employees per department
SELECT department, COUNT(*) AS emp_count
FROM employees
GROUP BY department;
-- Output:
-- Engineering | 15
-- Sales       | 10
-- HR          | 5

-- Average salary per department
SELECT department, AVG(salary) AS avg_salary
FROM employees
GROUP BY department
ORDER BY avg_salary DESC;

-- Multiple grouping columns
SELECT department, is_active, COUNT(*) AS count
FROM employees
GROUP BY department, is_active;
```

### The GROUP BY Rule

```
┌─────────────────────────────────────────────────────────────────┐
│  CRITICAL RULE:                                                  │
│                                                                  │
│  Every column in SELECT must either be:                         │
│    1. Inside an aggregate function (COUNT, SUM, AVG, etc.)      │
│    2. Listed in the GROUP BY clause                             │
│                                                                  │
│  ❌ INVALID:                                                    │
│  SELECT department, first_name, COUNT(*)                        │
│  FROM employees                                                 │
│  GROUP BY department;                                           │
│  -- Error! first_name is not aggregated or in GROUP BY          │
│                                                                  │
│  ✅ VALID:                                                      │
│  SELECT department, COUNT(*), MAX(first_name)                   │
│  FROM employees                                                 │
│  GROUP BY department;                                           │
└─────────────────────────────────────────────────────────────────┘
```

### HAVING — Filter Groups (Not Rows)

`HAVING` is like `WHERE`, but for **groups** created by `GROUP BY`.

```sql
-- Departments with more than 5 employees
SELECT department, COUNT(*) AS emp_count
FROM employees
GROUP BY department
HAVING COUNT(*) > 5;

-- Departments where average salary exceeds 80k
SELECT department, AVG(salary) AS avg_salary
FROM employees
GROUP BY department
HAVING AVG(salary) > 80000
ORDER BY avg_salary DESC;
```

**WHERE vs HAVING:**

| Aspect | WHERE | HAVING |
|--------|-------|--------|
| **Filters** | Individual rows | Groups |
| **Executes** | Before GROUP BY | After GROUP BY |
| **Can use aggregates?** | ❌ No | ✅ Yes |
| **Example** | `WHERE salary > 50000` | `HAVING AVG(salary) > 50000` |

```sql
-- Combined example: Both WHERE and HAVING
SELECT department, AVG(salary) AS avg_salary
FROM employees
WHERE is_active = TRUE              -- 1st: filter OUT inactive employees
GROUP BY department                  -- 2nd: group remaining by department
HAVING AVG(salary) > 70000          -- 3rd: keep only high-salary departments
ORDER BY avg_salary DESC;            -- 4th: sort results
```

## Joins — The Heart of SQL

Joins combine rows from **two or more tables** based on a related column. This is what makes relational databases powerful.

### Setup — Example Tables

```sql
-- employees table
CREATE TABLE employees (
    emp_id INT PRIMARY KEY,
    name VARCHAR(50),
    dept_id INT,
    manager_id INT
);

INSERT INTO employees VALUES
(1, 'Alice',   101, NULL),
(2, 'Bob',     102, 1),
(3, 'Charlie', 101, 1),
(4, 'Diana',   103, 2),
(5, 'Eve',     NULL, 2);    -- Eve has NO department

-- departments table
CREATE TABLE departments (
    dept_id INT PRIMARY KEY,
    dept_name VARCHAR(50)
);

INSERT INTO departments VALUES
(101, 'Engineering'),
(102, 'Sales'),
(103, 'HR'),
(104, 'Marketing');          -- Marketing has NO employees
```

### INNER JOIN — Only Matching Rows

Returns rows that have **matching values in BOTH tables**.

```sql
SELECT e.name, d.dept_name
FROM employees e
INNER JOIN departments d ON e.dept_id = d.dept_id;

-- Result:
-- Alice   | Engineering
-- Bob     | Sales
-- Charlie | Engineering
-- Diana   | HR

-- ❌ Eve is excluded (no matching dept_id — she has NULL)
-- ❌ Marketing is excluded (no employees in it)
```

```
┌──────────────┐   ┌──────────────┐
│  employees   │   │ departments  │
│              │   │              │
│  ●  Alice    │───│ Engineering  │  ✅ Match
│  ●  Bob      │───│ Sales        │  ✅ Match
│  ●  Charlie  │───│ Engineering  │  ✅ Match
│  ●  Diana    │───│ HR           │  ✅ Match
│  ●  Eve      │   │ Marketing    │
│  (no dept)   │   │ (no emps)    │
│  ❌ excluded │   │ ❌ excluded  │
└──────────────┘   └──────────────┘
```

### LEFT JOIN (LEFT OUTER JOIN) — All from Left + Matches from Right

Returns **all rows from the left table**, and matching rows from the right. Non-matching right-side columns get `NULL`.

```sql
SELECT e.name, d.dept_name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.dept_id;

-- Result:
-- Alice   | Engineering
-- Bob     | Sales
-- Charlie | Engineering
-- Diana   | HR
-- Eve     | NULL          ✅ Eve included (her dept_name is NULL)

-- Marketing still excluded (it's in the RIGHT table)
```

### RIGHT JOIN (RIGHT OUTER JOIN) — All from Right + Matches from Left

Returns **all rows from the right table**, and matching rows from the left.

```sql
SELECT e.name, d.dept_name
FROM employees e
RIGHT JOIN departments d ON e.dept_id = d.dept_id;

-- Result:
-- Alice   | Engineering
-- Bob     | Sales
-- Charlie | Engineering
-- Diana   | HR
-- NULL    | Marketing    ✅ Marketing included (no employee name)

-- Eve excluded (she's in the LEFT table with no match)
```

### FULL OUTER JOIN — All from Both Tables

Returns **all rows from both tables**. Non-matching columns get `NULL`.

```sql
SELECT e.name, d.dept_name
FROM employees e
FULL OUTER JOIN departments d ON e.dept_id = d.dept_id;

-- Result:
-- Alice   | Engineering
-- Bob     | Sales
-- Charlie | Engineering
-- Diana   | HR
-- Eve     | NULL          ✅ From left, no match in right
-- NULL    | Marketing     ✅ From right, no match in left
```

### CROSS JOIN — Cartesian Product

Returns **every combination** of rows from both tables. No `ON` condition.

```sql
SELECT e.name, d.dept_name
FROM employees e
CROSS JOIN departments d;

-- If employees has 5 rows and departments has 4 rows:
-- Result = 5 × 4 = 20 rows (every possible combination)
```

> **Use case:** Generating all possible combinations (e.g., all products × all sizes × all colors).

### SELF JOIN — Table Joined to Itself

Used when a table has a **self-referential relationship** (e.g., employee → manager).

```sql
-- Find each employee and their manager's name
SELECT
    e.name AS employee,
    m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.emp_id;

-- Result:
-- Alice   | NULL       (Alice has no manager)
-- Bob     | Alice      (Bob's manager is Alice)
-- Charlie | Alice      (Charlie's manager is Alice)
-- Diana   | Bob        (Diana's manager is Bob)
-- Eve     | Bob        (Eve's manager is Bob)
```

### Join Summary Diagram

```
┌──────────────────────────────────────────────────────────────┐
│                      JOIN TYPES SUMMARY                       │
│                                                               │
│  INNER JOIN     →  Only matching rows from BOTH tables       │
│  LEFT JOIN      →  ALL left + matching right (NULL if none)  │
│  RIGHT JOIN     →  ALL right + matching left (NULL if none)  │
│  FULL OUTER     →  ALL from both (NULL where no match)       │
│  CROSS JOIN     →  Every combination (Cartesian product)     │
│  SELF JOIN      →  Table joined with itself                  │
│                                                               │
│  Most used: INNER JOIN and LEFT JOIN (90% of real queries)   │
└──────────────────────────────────────────────────────────────┘
```

## Subqueries & Nested Queries

A subquery is a **query inside another query**. It can return a single value, a list, or a table.

### Single-Value Subquery (Scalar)

```sql
-- Find employees earning more than the average salary
SELECT name, salary
FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);

-- The inner query returns a single value (e.g., 75000)
-- The outer query uses that value: WHERE salary > 75000
```

### Multi-Value Subquery (List)

```sql
-- Find employees in departments that are in New York
SELECT name FROM employees
WHERE dept_id IN (
    SELECT dept_id FROM departments WHERE location = 'New York'
);
```

### Subquery in FROM (Derived Table)

```sql
-- Average of department averages
SELECT AVG(dept_avg) AS overall_avg
FROM (
    SELECT department, AVG(salary) AS dept_avg
    FROM employees
    GROUP BY department
) AS dept_averages;
```

### Correlated Subquery

A correlated subquery **references the outer query** — it runs **once per row** of the outer query.

```sql
-- Find employees earning more than the average in THEIR department
SELECT name, salary, department
FROM employees e1
WHERE salary > (
    SELECT AVG(salary)
    FROM employees e2
    WHERE e2.department = e1.department   -- References outer query!
);
```

### EXISTS — Check for Existence

```sql
-- Find departments that have at least one employee
SELECT dept_name FROM departments d
WHERE EXISTS (
    SELECT 1 FROM employees e WHERE e.dept_id = d.dept_id
);

-- EXISTS is more efficient than IN for large datasets
-- It stops searching as soon as it finds ONE match
```

**IN vs EXISTS:**

| Aspect | IN | EXISTS |
|--------|-----|--------|
| **Best for** | Small subquery result set | Large subquery, small outer query |
| **NULL handling** | Problems with NULLs | Handles NULLs correctly |
| **Performance** | Evaluates entire subquery | Short-circuits on first match |

## Set Operations

Combine results from **multiple SELECT statements**.

```sql
-- UNION — combine and REMOVE duplicates
SELECT name FROM employees
UNION
SELECT name FROM contractors;

-- UNION ALL — combine and KEEP duplicates (faster)
SELECT name FROM employees
UNION ALL
SELECT name FROM contractors;

-- INTERSECT — only rows present in BOTH queries
SELECT dept_id FROM employees
INTERSECT
SELECT dept_id FROM departments;

-- EXCEPT (MINUS in Oracle) — rows in first query but NOT in second
SELECT dept_id FROM departments
EXCEPT
SELECT DISTINCT dept_id FROM employees;
-- Returns departments with NO employees
```

**Rules for Set Operations:**
1. Both queries must have the **same number of columns**
2. Corresponding columns must have **compatible data types**
3. Column names come from the **first** query

## Views

A view is a **virtual table** based on a SQL query. It doesn't store data itself — it's a saved query.

```sql
-- Create a view
CREATE VIEW active_employees AS
SELECT emp_id, first_name, last_name, department, salary
FROM employees
WHERE is_active = TRUE;

-- Use it like a regular table
SELECT * FROM active_employees WHERE salary > 70000;

-- Update a view
CREATE OR REPLACE VIEW active_employees AS
SELECT emp_id, first_name, last_name, department, salary, email
FROM employees
WHERE is_active = TRUE;

-- Drop a view
DROP VIEW active_employees;
```

**Why use views?**
- **Security** — Expose only specific columns/rows to users
- **Simplicity** — Hide complex JOINs behind a simple name
- **Consistency** — Everyone uses the same query logic
- **Abstraction** — Change underlying table structure without breaking applications

### Materialized View (PostgreSQL)

Unlike regular views, materialized views **store the result physically** for faster reads.

```sql
-- Create a materialized view
CREATE MATERIALIZED VIEW monthly_sales_summary AS
SELECT DATE_TRUNC('month', order_date) AS month, SUM(amount) AS total
FROM orders
GROUP BY month;

-- Refresh when underlying data changes
REFRESH MATERIALIZED VIEW monthly_sales_summary;
```

## Indexes

An index is a **data structure** (usually B-tree) that speeds up data retrieval at the cost of extra storage and slower writes.

```sql
-- Create an index on a frequently queried column
CREATE INDEX idx_emp_email ON employees(email);

-- Composite index (multiple columns)
CREATE INDEX idx_emp_dept_salary ON employees(department, salary);

-- Unique index
CREATE UNIQUE INDEX idx_unique_email ON employees(email);

-- Drop an index
DROP INDEX idx_emp_email;
```

```
┌─────────────────────────────────────────────────────────────────┐
│                         INDEX TRADEOFFS                          │
│                                                                  │
│  ✅ Pros:                                                       │
│    → Dramatically speeds up SELECT / WHERE queries              │
│    → Speeds up JOIN operations                                  │
│    → Speeds up ORDER BY and GROUP BY                            │
│                                                                  │
│  ❌ Cons:                                                       │
│    → Extra storage space                                        │
│    → Slows down INSERT, UPDATE, DELETE (index must be updated)  │
│    → Too many indexes = diminishing returns                     │
│                                                                  │
│  When to create indexes:                                        │
│    → Columns in WHERE clauses (frequently filtered)             │
│    → Columns in JOIN conditions (foreign keys)                  │
│    → Columns in ORDER BY (frequently sorted)                    │
│    → High cardinality columns (many unique values)              │
│                                                                  │
│  When NOT to index:                                             │
│    → Small tables (full scan is faster anyway)                  │
│    → Columns with low cardinality (e.g., boolean, gender)       │
│    → Tables with heavy write operations                         │
└─────────────────────────────────────────────────────────────────┘
```

> **Intuition:** An index is like a book's index — instead of reading every page to find "SQL", you check the index at the back: "SQL → page 142". Without it, you'd scan every page (full table scan).

## Transactions & ACID Properties

A **transaction** is a group of SQL operations that are treated as a **single unit of work**. Either ALL succeed, or NONE do.

### Basic Transaction Syntax

```sql
-- Start a transaction
BEGIN;  -- or START TRANSACTION

-- Perform operations
UPDATE accounts SET balance = balance - 500 WHERE account_id = 1;
UPDATE accounts SET balance = balance + 500 WHERE account_id = 2;

-- If everything is correct → make it permanent
COMMIT;

-- If something went wrong → undo everything
ROLLBACK;
```

### ACID Properties

```
┌─────────────────────────────────────────────────────────────────┐
│                    ACID PROPERTIES                               │
│                                                                  │
│  A — Atomicity                                                  │
│    → "All or Nothing"                                           │
│    → Either ALL operations in a transaction succeed,            │
│      or NONE of them do.                                        │
│    → If transferring money: both debit AND credit happen,       │
│      or neither happens.                                        │
│                                                                  │
│  C — Consistency                                                │
│    → Database moves from one VALID state to another.            │
│    → All constraints, rules, and relationships are maintained.  │
│    → If a constraint says balance >= 0, the transaction         │
│      fails if it would violate this.                            │
│                                                                  │
│  I — Isolation                                                  │
│    → Concurrent transactions don't interfere with each other.   │
│    → Each transaction behaves as if it's the ONLY one running.  │
│    → Prevents dirty reads, phantom reads, etc.                  │
│                                                                  │
│  D — Durability                                                 │
│    → Once COMMITTED, data survives crashes, power failures.     │
│    → Written to disk, not just RAM.                             │
│    → Even if the server crashes 1 second after COMMIT,          │
│      the data persists.                                         │
└─────────────────────────────────────────────────────────────────┘
```

### SAVEPOINT — Partial Rollback

```sql
BEGIN;
INSERT INTO orders (customer_id, amount) VALUES (1, 100);
SAVEPOINT sp1;

INSERT INTO orders (customer_id, amount) VALUES (2, 200);
-- Oops, this was wrong! Rollback ONLY to the savepoint
ROLLBACK TO SAVEPOINT sp1;

-- The first INSERT is still intact
COMMIT;
```

### Isolation Levels

| Level | Dirty Read | Non-Repeatable Read | Phantom Read | Performance |
|-------|-----------|-------------------|-------------|-------------|
| **READ UNCOMMITTED** | ✅ Possible | ✅ Possible | ✅ Possible | Fastest |
| **READ COMMITTED** | ❌ Prevented | ✅ Possible | ✅ Possible | Fast |
| **REPEATABLE READ** | ❌ Prevented | ❌ Prevented | ✅ Possible | Medium |
| **SERIALIZABLE** | ❌ Prevented | ❌ Prevented | ❌ Prevented | Slowest |

```sql
-- Set isolation level
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
BEGIN;
-- Operations here are fully isolated
COMMIT;
```

> **Default in PostgreSQL:** READ COMMITTED. **Default in MySQL:** REPEATABLE READ.

## Constraints

Constraints enforce **rules** on data to maintain integrity.

```sql
CREATE TABLE products (
    product_id   INT PRIMARY KEY,                    -- PRIMARY KEY
    product_name VARCHAR(100) NOT NULL,              -- NOT NULL
    sku          VARCHAR(50) UNIQUE,                 -- UNIQUE
    price        DECIMAL(10,2) CHECK (price > 0),    -- CHECK
    category_id  INT DEFAULT 1,                      -- DEFAULT
    supplier_id  INT REFERENCES suppliers(id)        -- FOREIGN KEY
);
```

### Constraint Summary

| Constraint | Purpose | Example |
|-----------|---------|---------|
| **PRIMARY KEY** | Uniquely identifies each row. NOT NULL + UNIQUE | `emp_id INT PRIMARY KEY` |
| **FOREIGN KEY** | Links to another table's primary key | `REFERENCES departments(dept_id)` |
| **NOT NULL** | Column cannot be NULL | `name VARCHAR(50) NOT NULL` |
| **UNIQUE** | All values must be different | `email VARCHAR(100) UNIQUE` |
| **CHECK** | Custom validation rule | `CHECK (age >= 18)` |
| **DEFAULT** | Auto-assign value if none provided | `status VARCHAR(20) DEFAULT 'active'` |

### Foreign Key Actions

```sql
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    FOREIGN KEY (customer_id) REFERENCES customers(id)
        ON DELETE CASCADE       -- Delete order when customer is deleted
        ON UPDATE SET NULL      -- Set to NULL when customer ID changes
);
```

| Action | On DELETE | On UPDATE |
|--------|----------|----------|
| **CASCADE** | Delete child rows | Update child foreign key |
| **SET NULL** | Set FK to NULL | Set FK to NULL |
| **SET DEFAULT** | Set FK to default | Set FK to default |
| **RESTRICT** | Block if children exist | Block if referenced |
| **NO ACTION** | Same as RESTRICT (default) | Same as RESTRICT |

## Normalization

Normalization is the process of organizing database tables to **reduce redundancy and improve data integrity**.

### Why Normalize?

```
Without normalization:
┌────┬──────────┬─────────────┬───────────┬──────────────────┐
│ id │ student  │ course      │ teacher   │ teacher_email    │
├────┼──────────┼─────────────┼───────────┼──────────────────┤
│ 1  │ Alice    │ Math        │ Dr. Smith │ smith@uni.com    │
│ 2  │ Bob      │ Math        │ Dr. Smith │ smith@uni.com    │  ← Redundant!
│ 3  │ Alice    │ Physics     │ Dr. Jones │ jones@uni.com    │
│ 4  │ Charlie  │ Math        │ Dr. Smith │ smith@uni.com    │  ← Redundant!
└────┴──────────┴─────────────┴───────────┴──────────────────┘

Problems:
  → "Dr. Smith" and "smith@uni.com" repeated for EVERY Math student
  → If Dr. Smith changes email, you must update EVERY row (update anomaly)
  → If all Math students drop out, you lose Dr. Smith's info (delete anomaly)
  → Inserting a new teacher with no students is impossible (insert anomaly)
```

### Normal Forms

#### 1NF (First Normal Form)
- Each column contains **atomic (indivisible) values**
- No repeating groups or arrays in a single column

```
❌ Violates 1NF:
│ student │ courses              │
│ Alice   │ Math, Physics, Chem  │   ← Multiple values in one cell!

✅ 1NF:
│ student │ course   │
│ Alice   │ Math     │
│ Alice   │ Physics  │
│ Alice   │ Chem     │
```

#### 2NF (Second Normal Form)
- Must be in 1NF
- No **partial dependency** — every non-key column depends on the **entire** primary key

```
❌ Violates 2NF (composite key: student_id + course_id):
│ student_id │ course_id │ student_name │ course_name │ grade │
│ 1          │ 101       │ Alice        │ Math        │ A     │

student_name depends ONLY on student_id (not the full composite key)
course_name depends ONLY on course_id (not the full composite key)
grade depends on BOTH (student_id + course_id) ✅

✅ 2NF: Split into three tables:
students(student_id, student_name)
courses(course_id, course_name)
enrollments(student_id, course_id, grade)
```

#### 3NF (Third Normal Form)
- Must be in 2NF
- No **transitive dependency** — non-key columns depend ONLY on the primary key, not on other non-key columns

```
❌ Violates 3NF:
│ emp_id │ dept_id │ dept_name    │
│ 1      │ 101     │ Engineering  │

dept_name depends on dept_id, NOT on emp_id
emp_id → dept_id → dept_name (transitive!)

✅ 3NF: Split into two tables:
employees(emp_id, dept_id)
departments(dept_id, dept_name)
```

#### BCNF (Boyce-Codd Normal Form)
- Stricter version of 3NF
- Every determinant must be a **candidate key**
- Resolves edge cases that 3NF misses

> **Practical rule:** Most production databases aim for **3NF**. Over-normalizing (4NF, 5NF) can hurt performance with too many JOINs. **Denormalization** is intentionally breaking normal forms for read performance (common in analytics/reporting).

## Keys in SQL

```
┌─────────────────────────────────────────────────────────────────┐
│                      TYPES OF KEYS                               │
│                                                                  │
│  Primary Key                                                    │
│    → Uniquely identifies each row                               │
│    → NOT NULL + UNIQUE (always)                                 │
│    → Only ONE per table                                         │
│    → Example: emp_id, user_id, order_id                         │
│                                                                  │
│  Foreign Key                                                    │
│    → References a primary key in ANOTHER table                  │
│    → Creates relationships between tables                       │
│    → CAN be NULL (optional relationship)                        │
│    → CAN have duplicates                                        │
│                                                                  │
│  Candidate Key                                                  │
│    → Any column(s) that COULD be a primary key                 │
│    → Unique + Not NULL                                          │
│    → Example: email, SSN, phone_number (all unique per person)  │
│    → The one you CHOOSE becomes the primary key                 │
│    → The rest become alternate keys                             │
│                                                                  │

│    → Example: (student_id, course_id) in enrollment table       │
│                                                                  │
│  Super Key                                                      │
│    → Any set of columns that uniquely identifies a row          │
│    → Superset of candidate keys                                 │
│    → Example: {emp_id}, {emp_id, name}, {email}, {email, phone} │
│                                                                  │
│  Unique Key                                                     │
│    → Like primary key but ALLOWS one NULL                       │
│    → Multiple unique keys per table allowed                     │
│    → Example: email (unique but not the primary key)            │
└─────────────────────────────────────────────────────────────────┘
```

## Stored Procedures & Functions

### Stored Procedure — Reusable SQL Logic

```sql
-- Create a stored procedure (MySQL syntax)
DELIMITER //
CREATE PROCEDURE give_raise(IN dept VARCHAR(50), IN raise_pct DECIMAL(5,2))
BEGIN
    UPDATE employees
    SET salary = salary * (1 + raise_pct / 100)
    WHERE department = dept;
END //
DELIMITER ;

-- Call the procedure
CALL give_raise('Engineering', 10);  -- 10% raise for Engineering
```

```sql
-- PostgreSQL syntax
CREATE OR REPLACE FUNCTION give_raise(dept TEXT, raise_pct NUMERIC)
RETURNS VOID AS $$
BEGIN
    UPDATE employees
    SET salary = salary * (1 + raise_pct / 100)
    WHERE department = dept;
END;
$$ LANGUAGE plpgsql;

-- Call it
SELECT give_raise('Engineering', 10);
```

### Function vs Procedure

| Aspect | Function | Procedure |
|--------|----------|-----------|
| **Returns** | A value (scalar/table) | Nothing (void) or OUT params |
| **Used in** | SELECT, WHERE, etc. | Called with CALL/EXECUTE |
| **Transaction** | Cannot manage transactions | Can COMMIT/ROLLBACK |
| **Use case** | Calculations, transformations | Business logic, batch operations |

## Triggers

A trigger is a **stored procedure that runs automatically** when a specific event occurs (INSERT, UPDATE, DELETE).

```sql
-- Create an audit log trigger (PostgreSQL)
CREATE TABLE employee_audit (
    audit_id SERIAL PRIMARY KEY,
    emp_id INT,
    action VARCHAR(10),
    old_salary DECIMAL(10,2),
    new_salary DECIMAL(10,2),
    changed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE OR REPLACE FUNCTION log_salary_change()
RETURNS TRIGGER AS $$
BEGIN
    INSERT INTO employee_audit (emp_id, action, old_salary, new_salary)
    VALUES (OLD.emp_id, 'UPDATE', OLD.salary, NEW.salary);
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER salary_change_trigger
AFTER UPDATE OF salary ON employees
FOR EACH ROW
EXECUTE FUNCTION log_salary_change();

-- Now every salary change is automatically logged!
UPDATE employees SET salary = 90000 WHERE emp_id = 1;
-- The audit table automatically gets a new row
```

### Trigger Types

| Type | When It Fires |
|------|--------------|
| **BEFORE INSERT** | Before a new row is inserted |
| **AFTER INSERT** | After a new row is inserted |
| **BEFORE UPDATE** | Before a row is modified |
| **AFTER UPDATE** | After a row is modified |
| **BEFORE DELETE** | Before a row is removed |
| **AFTER DELETE** | After a row is removed |
| **INSTEAD OF** | Replaces the triggering action (for views) |

> **Use cases:** Audit logging, enforcing complex business rules, auto-updating related tables, sending notifications.

## Window Functions

Window functions perform calculations across a **set of rows related to the current row** — without collapsing them into a single output like GROUP BY does.

```sql
-- ROW_NUMBER — unique sequential number for each row
SELECT
    name, department, salary,
    ROW_NUMBER() OVER (ORDER BY salary DESC) AS rank
FROM employees;

-- RANK — same rank for ties, skips next rank
-- DENSE_RANK — same rank for ties, does NOT skip
SELECT
    name, department, salary,
    RANK() OVER (ORDER BY salary DESC) AS rank,
    DENSE_RANK() OVER (ORDER BY salary DESC) AS dense_rank
FROM employees;

-- Example output:
-- Alice   | 95000 | rank: 1 | dense_rank: 1
-- Bob     | 90000 | rank: 2 | dense_rank: 2
-- Charlie | 90000 | rank: 2 | dense_rank: 2  ← TIE
-- Diana   | 85000 | rank: 4 | dense_rank: 3  ← RANK skips 3, DENSE doesn't
```

### PARTITION BY — Window Within Groups

```sql
-- Rank employees WITHIN each department
SELECT
    name, department, salary,
    RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS dept_rank
FROM employees;

-- Each department gets its own ranking (1, 2, 3, ...)
```

### Aggregate Window Functions

```sql
SELECT
    name, department, salary,
    SUM(salary) OVER (PARTITION BY department) AS dept_total,
    AVG(salary) OVER (PARTITION BY department) AS dept_avg,
    COUNT(*) OVER (PARTITION BY department) AS dept_count,
    salary - AVG(salary) OVER (PARTITION BY department) AS diff_from_avg
FROM employees;

-- Each row keeps its individual data AND gets aggregate info
-- Unlike GROUP BY, rows are NOT collapsed!
```

### Running Totals & Moving Averages

```sql
-- Running total of sales
SELECT
    order_date, amount,
    SUM(amount) OVER (ORDER BY order_date) AS running_total
FROM orders;

-- Moving average (last 3 orders)
SELECT
    order_date, amount,
    AVG(amount) OVER (ORDER BY order_date ROWS BETWEEN 2 PRECEDING AND CURRENT ROW) AS moving_avg
FROM orders;

-- LAG / LEAD — access previous/next row
SELECT
    name, salary,
    LAG(salary, 1) OVER (ORDER BY salary) AS prev_salary,
    LEAD(salary, 1) OVER (ORDER BY salary) AS next_salary,
    salary - LAG(salary, 1) OVER (ORDER BY salary) AS salary_gap
FROM employees;
```

### NTILE — Divide into Buckets

```sql
-- Divide employees into 4 salary quartiles
SELECT
    name, salary,
    NTILE(4) OVER (ORDER BY salary) AS quartile
FROM employees;
```

## Common Table Expressions (CTEs)

A CTE is a **temporary named result set** that exists only during query execution. Think of it as a cleaner alternative to subqueries.

```sql
-- Basic CTE
WITH high_earners AS (
    SELECT name, salary, department
    FROM employees
    WHERE salary > 80000
)
SELECT department, COUNT(*) AS high_earner_count
FROM high_earners
GROUP BY department;
```

### Multiple CTEs

```sql
WITH
dept_stats AS (
    SELECT department, AVG(salary) AS avg_salary, COUNT(*) AS emp_count
    FROM employees
    GROUP BY department
),
expensive_depts AS (
    SELECT department, avg_salary
    FROM dept_stats
    WHERE avg_salary > 75000
)
SELECT * FROM expensive_depts ORDER BY avg_salary DESC;
```

### Recursive CTE — Hierarchical Data

```sql
-- Find all employees in the management chain under Alice (emp_id = 1)
WITH RECURSIVE org_chart AS (
    -- Base case: start with Alice
    SELECT emp_id, name, manager_id, 1 AS level
    FROM employees
    WHERE emp_id = 1

    UNION ALL

    -- Recursive case: find employees managed by current level
    SELECT e.emp_id, e.name, e.manager_id, oc.level + 1
    FROM employees e
    INNER JOIN org_chart oc ON e.manager_id = oc.emp_id
)
SELECT * FROM org_chart ORDER BY level;

-- Result:
-- 1 | Alice   | NULL | 1   (top level)
-- 2 | Bob     | 1    | 2   (reports to Alice)
-- 3 | Charlie | 1    | 2   (reports to Alice)
-- 4 | Diana   | 2    | 3   (reports to Bob)
-- 5 | Eve     | 2    | 3   (reports to Bob)
```

> **CTE vs Subquery:** CTEs are more readable, reusable within the same query, and support recursion. Use CTEs for complex queries; use subqueries for simple, one-off filters.

## 50+ Interview Questions & Answers

### Fundamentals (Q1–Q10)

---

**Q1. What is SQL?**

SQL (Structured Query Language) is a standardized declarative language used to manage and manipulate relational databases. It allows you to create, read, update, and delete data (CRUD), define database structures, control access permissions, and manage transactions.

---

**Q2. What are the different types of SQL sub-languages?**

| Sub-Language | Purpose | Commands |
|-------------|---------|----------|
| **DDL** (Data Definition) | Define structure | CREATE, ALTER, DROP, TRUNCATE |
| **DML** (Data Manipulation) | Manipulate data | INSERT, UPDATE, DELETE |
| **DQL** (Data Query) | Query data | SELECT |
| **DCL** (Data Control) | Permissions | GRANT, REVOKE |
| **TCL** (Transaction Control) | Transactions | COMMIT, ROLLBACK, SAVEPOINT |

---

**Q3. What is the difference between DELETE, TRUNCATE, and DROP?**

| Feature | DELETE | TRUNCATE | DROP |
|---------|--------|----------|------|
| **What it does** | Removes specific rows | Removes all rows | Removes entire table |
| **WHERE clause** | ✅ Supported | ❌ Not supported | N/A |
| **Rollback** | ✅ Yes | ❌ No (most RDBMS) | ❌ No |
| **Triggers** | ✅ Fires | ❌ Doesn't fire | N/A |
| **Speed** | Slower (row-by-row) | Faster (page dealloc) | Instant |
| **Auto-increment** | Keeps counter | Resets counter | N/A |
| **Structure** | Keeps table structure | Keeps table structure | Removes everything |

---

**Q4. What is a Primary Key?**

A primary key is a column (or combination of columns) that **uniquely identifies each row** in a table. It is always **NOT NULL** and **UNIQUE**. A table can have only **one** primary key.

```sql
CREATE TABLE users (
    user_id INT PRIMARY KEY,
    email VARCHAR(100) UNIQUE NOT NULL
);
```

---

**Q5. What is a Foreign Key?**

A foreign key is a column that **references the primary key of another table**, creating a relationship between the two tables. It enforces **referential integrity** — you can't insert a value that doesn't exist in the referenced table.

```sql
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT REFERENCES customers(customer_id)
);
```

---

**Q6. What is the difference between WHERE and HAVING?**

- **WHERE** filters **individual rows** BEFORE grouping (executes before GROUP BY)
- **HAVING** filters **groups** AFTER grouping (executes after GROUP BY)
- WHERE **cannot** use aggregate functions; HAVING **can**

```sql
-- WHERE: filter rows
SELECT * FROM employees WHERE salary > 50000;

-- HAVING: filter groups
SELECT department, AVG(salary)
FROM employees
GROUP BY department
HAVING AVG(salary) > 50000;
```

---

**Q7. What is the difference between CHAR and VARCHAR?**

| Feature | CHAR(n) | VARCHAR(n) |
|---------|---------|-----------|
| **Storage** | Fixed-length, padded with spaces | Variable-length, no padding |
| **Space** | Always uses n bytes | Uses only what's needed |
| **Performance** | Slightly faster for fixed-size data | Better for variable-size data |
| **Use case** | Country codes (IN, US), status codes | Names, emails, addresses |

---

**Q8. What is NULL in SQL?**

NULL represents a **missing or unknown** value. It is NOT zero, NOT an empty string, NOT false. NULL cannot be compared using `=` or `!=` — you must use `IS NULL` or `IS NOT NULL`.

```sql
-- ❌ Wrong
SELECT * FROM users WHERE phone = NULL;

-- ✅ Correct
SELECT * FROM users WHERE phone IS NULL;
```

**Key behavior:** Any arithmetic operation with NULL returns NULL. `5 + NULL = NULL`.

---

**Q9. What is the difference between UNION and UNION ALL?**

- **UNION** combines results and **removes duplicates** (slower — requires sorting/hashing)
- **UNION ALL** combines results and **keeps duplicates** (faster — no dedup overhead)

Both require the same number of columns with compatible data types.

---

**Q10. What is the SQL execution order?**

```
1. FROM / JOIN       → Identify tables and joins
2. WHERE             → Filter individual rows
3. GROUP BY          → Group rows
4. HAVING            → Filter groups
5. SELECT            → Choose columns, calculate expressions
6. DISTINCT          → Remove duplicates
7. ORDER BY          → Sort results
8. LIMIT / OFFSET    → Restrict output rows
```

This is why you **cannot use a SELECT alias in WHERE** — SELECT runs after WHERE.

### Joins & Relationships (Q11–Q20)

---

**Q11. What are the different types of JOINs?**

| Join | Returns |
|------|---------|
| **INNER JOIN** | Only matching rows from both tables |
| **LEFT JOIN** | All rows from left + matching from right (NULL if no match) |
| **RIGHT JOIN** | All rows from right + matching from left (NULL if no match) |
| **FULL OUTER JOIN** | All rows from both (NULL where no match) |
| **CROSS JOIN** | Cartesian product — every combination |
| **SELF JOIN** | Table joined with itself |

---

**Q12. What is a Self Join? Give an example.**

A self join is when a table is joined **with itself**. Common use case: hierarchical data like employee-manager relationships.

```sql
SELECT e.name AS employee, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.emp_id;
```

---

**Q13. What is a Cross Join?**

A cross join returns the **Cartesian product** — every row from table A paired with every row from table B. If A has 5 rows and B has 4 rows, result = 20 rows. No `ON` condition is used.

**Use case:** Generating all combinations (products × sizes × colors).

---

**Q14. What is the difference between INNER JOIN and LEFT JOIN?**

- **INNER JOIN** returns only rows where there is a match in **both** tables
- **LEFT JOIN** returns **all** rows from the left table, even if there's no match in the right table (right-side columns get NULL)

---

**Q15. Can you join more than two tables? How?**

Yes. Chain multiple JOINs:

```sql
SELECT o.order_id, c.name, p.product_name
FROM orders o
INNER JOIN customers c ON o.customer_id = c.customer_id
INNER JOIN products p ON o.product_id = p.product_id;
```

---

**Q16. What is referential integrity?**

Referential integrity ensures that a foreign key value in one table **always matches** a valid primary key in the referenced table. You cannot insert an `order` with a `customer_id` that doesn't exist in the `customers` table.

---

**Q17. What are the different types of relationships in databases?**

| Relationship | Description | Example |
|-------------|-------------|---------|
| **One-to-One** | One row in A links to exactly one in B | User → Profile |
| **One-to-Many** | One row in A links to many in B | Customer → Orders |
| **Many-to-Many** | Many in A link to many in B (via junction table) | Students ↔ Courses |

---

**Q18. How do you implement a Many-to-Many relationship?**

Using a **junction (bridge) table** that holds foreign keys to both tables:

```sql
CREATE TABLE students (student_id INT PRIMARY KEY, name TEXT);
CREATE TABLE courses (course_id INT PRIMARY KEY, title TEXT);

-- Junction table
CREATE TABLE enrollments (
    student_id INT REFERENCES students(student_id),
    course_id INT REFERENCES courses(course_id),
    grade CHAR(2),
    PRIMARY KEY (student_id, course_id)   -- Composite key
);
```

---

**Q19. What is a Natural Join?**

A natural join **automatically** joins tables on columns with the **same name and data type**. It's generally discouraged in production because it's implicit and fragile — if column names change, the join breaks silently.

```sql
-- Natural join (implicit)
SELECT * FROM employees NATURAL JOIN departments;

-- Equivalent explicit join (preferred)
SELECT * FROM employees e JOIN departments d ON e.dept_id = d.dept_id;
```

---

**Q20. What is the difference between JOIN and Subquery? When to use which?**

| Aspect | JOIN | Subquery |
|--------|------|----------|
| **Readability** | Better for multi-table data | Better for single comparisons |
| **Performance** | Generally faster | Can be slower (especially correlated) |
| **Use case** | Combining columns from multiple tables | Filtering with computed values |

**Rule of thumb:** Use JOINs when you need columns from multiple tables. Use subqueries when you need to filter based on an aggregate or derived value.

### Advanced Concepts (Q21–Q35)

---

**Q21. What are ACID properties?**

| Property | Meaning |
|----------|---------|
| **Atomicity** | All operations succeed or all fail (all-or-nothing) |
| **Consistency** | Database moves from one valid state to another |
| **Isolation** | Concurrent transactions don't interfere |
| **Durability** | Committed data survives crashes |

---

**Q22. What is a Transaction?**

A transaction is a **logical unit of work** that groups multiple SQL operations. Either all operations succeed (COMMIT) or all are undone (ROLLBACK).

```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;
```

---

**Q23. What are Isolation Levels?**

From least to most restrictive: READ UNCOMMITTED → READ COMMITTED → REPEATABLE READ → SERIALIZABLE. Higher levels prevent more anomalies (dirty reads, non-repeatable reads, phantom reads) but reduce concurrency.

---

**Q24. What is a Dirty Read?**

A dirty read occurs when a transaction reads data that has been **modified but not yet committed** by another transaction. If the other transaction rolls back, the read data was never valid.

---

**Q25. What is Normalization? Explain up to 3NF.**

Normalization organizes tables to reduce redundancy:
- **1NF:** Atomic values only (no arrays/lists in cells)
- **2NF:** 1NF + no partial dependencies (non-key columns depend on the ENTIRE primary key)
- **3NF:** 2NF + no transitive dependencies (non-key columns depend ONLY on the primary key)

---

**Q26. What is Denormalization? Why would you use it?**

Denormalization is intentionally **adding redundancy** back into a normalized database to **improve read performance**. It reduces the need for JOINs by storing related data together. Common in read-heavy systems like analytics dashboards and reporting.

**Tradeoff:** Faster reads, but more complex writes and potential data inconsistency.

---

**Q27. What is an Index? How does it work?**

An index is a data structure (typically a B-tree) that provides a **fast lookup path** to rows, similar to a book's index. Instead of scanning every row (full table scan), the database uses the index to jump directly to matching rows.

**Tradeoff:** Faster reads, but slower writes (the index must be maintained on every INSERT/UPDATE/DELETE).

---

**Q28. What is a Clustered vs Non-Clustered Index?**

| Feature | Clustered | Non-Clustered |
|---------|-----------|---------------|
| **Physical order** | Data rows stored in index order | Separate structure pointing to rows |
| **Per table** | Only ONE allowed | Multiple allowed |
| **Speed** | Faster for range queries | Faster for specific lookups |
| **Example** | Primary key (default) | Index on email, index on name |

---

**Q29. What is a View? What is a Materialized View?**

- **View:** A virtual table defined by a query. Data is NOT stored — the query re-runs each time. Good for security and abstraction.
- **Materialized View:** A view where the result **IS stored physically**. Must be manually refreshed. Good for expensive aggregate queries.

---

**Q30. What is a Stored Procedure?**

A stored procedure is a **precompiled set of SQL statements** stored in the database. Benefits: reusability, reduced network traffic (send one call instead of multiple queries), security (users call procedures without seeing the underlying tables).

---

**Q31. What is a Trigger? Give a use case.**

A trigger is a SQL procedure that **automatically executes** when a specific event (INSERT, UPDATE, DELETE) occurs on a table. Use case: automatically logging changes to an audit table whenever an employee's salary is updated.

---

**Q32. What are Window Functions?**

Window functions perform calculations across a **set of rows (window) related to the current row** without collapsing the result. Unlike GROUP BY, every row is preserved. Key functions: ROW_NUMBER(), RANK(), DENSE_RANK(), LEAD(), LAG(), SUM() OVER, AVG() OVER.

---

**Q33. What is the difference between RANK(), DENSE_RANK(), and ROW_NUMBER()?**

| Function | Ties | Gaps |
|----------|------|------|
| **ROW_NUMBER()** | Each row gets unique number (arbitrary for ties) | No gaps |
| **RANK()** | Same rank for ties | Gaps after ties (1, 2, 2, **4**) |
| **DENSE_RANK()** | Same rank for ties | No gaps (1, 2, 2, **3**) |

---

**Q34. What is a CTE (Common Table Expression)?**

A CTE is a **temporary named result set** defined with `WITH` that exists only during query execution. It improves readability and can reference itself (recursive CTE) for hierarchical queries.

```sql
WITH dept_avg AS (
    SELECT dept_id, AVG(salary) AS avg_salary FROM employees GROUP BY dept_id
)
SELECT * FROM dept_avg WHERE avg_salary > 80000;
```

---

**Q35. What is a Correlated Subquery?**

A correlated subquery **references the outer query** and executes **once per row** of the outer query. It's slower than a non-correlated subquery but solves problems that require row-by-row comparison.

```sql
-- Employees earning more than their department average
SELECT name, salary FROM employees e1
WHERE salary > (
    SELECT AVG(salary) FROM employees e2
    WHERE e2.dept_id = e1.dept_id  -- References outer query
);
```

**Q41. What is the difference between IN and EXISTS?**

- **IN:** Evaluates the entire subquery first, then checks membership. Best for small subquery results.
- **EXISTS:** Stops at the first match (short-circuits). Best for large subqueries with small outer queries.

`EXISTS` also handles NULLs correctly while `IN` can produce unexpected results with NULL values.

---

**Q42. What is a Composite Key?**

A composite key is a primary key made up of **two or more columns** together. Neither column alone is unique, but the combination is. Common in junction/bridge tables.

```sql
PRIMARY KEY (student_id, course_id)
```

---

**Q43. What is the difference between a Candidate Key and a Primary Key?**

- **Candidate Key:** Any column or combination that CAN uniquely identify rows (there can be multiple)
- **Primary Key:** The candidate key you CHOOSE as the main identifier (only one per table)

All primary keys are candidate keys, but not all candidate keys are primary keys.

---

**Q44. What is a Schema?**

A schema is the **logical structure** of the database — the blueprint that defines tables, columns, data types, relationships, constraints, views, and indexes. It's the "design" of your database.

---

**Q45. What is the difference between COALESCE and ISNULL/IFNULL?**

- **COALESCE:** Part of SQL standard, works on multiple arguments, returns the first non-NULL value. `COALESCE(a, b, c, d)`.
- **ISNULL/IFNULL:** Database-specific (SQL Server / MySQL), takes only 2 arguments.

Always prefer `COALESCE` for portability.

---

**Q46. How do you swap column values of two rows without using a temp table?**

```sql
-- Swap salaries of emp_id 1 and emp_id 2
UPDATE employees
SET salary = CASE
    WHEN emp_id = 1 THEN (SELECT salary FROM employees WHERE emp_id = 2)
    WHEN emp_id = 2 THEN (SELECT salary FROM employees WHERE emp_id = 1)
END
WHERE emp_id IN (1, 2);
```

---

**Q47. What is the difference between EXCEPT and NOT IN?**

- **EXCEPT** returns rows in the first query that are NOT in the second query. Handles NULLs properly.
- **NOT IN** can return unexpected results if the subquery contains NULLs (entire result can become empty).

**Best practice:** Use `NOT EXISTS` or `EXCEPT` instead of `NOT IN` to avoid NULL pitfalls.

---

**Q48. What is a Phantom Read?**

A phantom read occurs when a transaction reads a set of rows matching a condition, but a second read within the same transaction returns **different rows** because another transaction inserted/deleted rows that match the condition. Prevented by SERIALIZABLE isolation level.

---

**Q49. What is a Deadlock?**

A deadlock occurs when **two or more transactions are waiting for each other** to release locks, creating a circular dependency. Neither can proceed. The DBMS detects this and **kills one transaction** (the "victim") to resolve it.

```
Transaction A: Locks Row 1, wants Row 2
Transaction B: Locks Row 2, wants Row 1
→ Both waiting forever → DEADLOCK!
```

**Prevention:** Always access tables/rows in the same order across transactions.

**Q50. What is the difference between Horizontal and Vertical Partitioning?**

|------|-------------|---------|
| **Horizontal (Sharding)** | Split rows across tables/servers | Orders 2023 in Table A, Orders 2024 in Table B |
| **Vertical** | Split columns across tables | Frequently accessed columns in one table, rarely accessed in another |

```sql

```

Use this to diagnose slow queries and verify if indexes are being used.

**Q61. How do you remove duplicate rows from a table WITHOUT using a temporary table? (The Classic Tricky Question)**

This is the hardest because you have to delete the duplicates while keeping *exactly one* copy. 

**Approach 1: Using CTE & `ROW_NUMBER()` (Best Practice for Modern SQL)**
We assign a unique row number to each group of duplicates, and delete any row that got a number > 1.
```sql
WITH CTE AS (
    SELECT *, ROW_NUMBER() OVER(
        PARTITION BY email, first_name   -- the columns that make the row duplicate
        ORDER BY id                      -- determines WHICH one to keep (e.g., lowest ID)
    ) as rn
    FROM employees
)
DELETE FROM CTE WHERE rn > 1;  -- Note: In PostgreSQL/MySQL, you might need to use the CTE to fetch IDs and then delete.
```
*PostgreSQL/MySQL safe version:*
```sql
DELETE FROM employees WHERE id IN (
    SELECT id FROM (
        SELECT id, ROW_NUMBER() OVER(PARTITION BY email ORDER BY id) as rn
        FROM employees
    ) temp WHERE rn > 1
);
```

**Approach 2: Using Self-Join (Old School / No Window Functions)**
Delete rows where there is another row with the same data but a smaller ID.
```sql
DELETE e1 FROM employees e1
INNER JOIN employees e2 
WHERE e1.email = e2.email AND e1.id > e2.id;
```

**Q62. If Table A has 5 rows containing only the value `1`, and Table B has 4 rows containing only the value `1`. How many rows will be returned by INNER JOIN, LEFT JOIN, RIGHT JOIN, and FULL OUTER JOIN?**

*This is a mind-bender. Most people guess 5 or 4.*
- **INNER JOIN:** Since EVERY row in A matches EVERY row in B, it creates a Cartesian product for that value. Output = $5 \times 4 = \textbf{20 rows}$.
- **LEFT JOIN:** Same as Inner Join because all rows match. Output = **20 rows**.
- **RIGHT JOIN:** Same reason. Output = **20 rows**.
- **FULL OUTER JOIN:** Same reason. Output = **20 rows**.

*(If Table B had ZERO rows of `1`, Inner Join = 0, Left Join = 5.)*

**Q66. Find the Median of a column (e.g., Salary).**

Median is notoriously tricky in SQL because there is no built-in `MEDIAN()` function in most RDBMS (unlike AVG/SUM).

*Using Window Functions (Standard Approach):*
```sql
WITH Ranked As (
   SELECT salary,
          PERCENT_RANK() OVER (ORDER BY salary) as pct_rank
   FROM employees
)
SELECT AVG(salary) AS median_salary
FROM Ranked
WHERE pct_rank = 0.5;
```

*Math trick method (If no Window Functions):*
```sql
SELECT AVG(salary) 
FROM (
    SELECT e1.salary
    FROM employees e1, employees e2
    GROUP BY e1.salary
    HAVING SUM(CASE WHEN e2.salary <= e1.salary THEN 1 ELSE 0 END) >= COUNT(*)/2.0
       AND SUM(CASE WHEN e2.salary >= e1.salary THEN 1 ELSE 0 END) >= COUNT(*)/2.0
) as temp;
```
