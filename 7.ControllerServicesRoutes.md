# Handlers, Services, Repositories, Middleware & Request Context

> **Core idea:** A backend request travels through many layers between arriving at the server and returning a response. Each layer has a single, clear responsibility. Understanding these layers and how they cooperate is fundamental to writing scalable, maintainable backend code.

## The Request Life Cycle Inside the Server

When a client sends an HTTP request to a server, many things happen between the moment the request arrives and the moment the response is sent back. This entire journey is called the **request life cycle**.

### High-Level View

```
CLIENT
  │
  │  HTTP Request
  ▼
SERVER (Entry Point)
  │
  ▼
Middleware 1 (CORS)
  │
  ▼
Middleware 2 (Logging)
  │
  ▼
Routing Algorithm (which URL? which handler?)
  │
  ▼
Middleware 3 (Authentication)
  │
  ▼
Handler / Controller
  │
  ▼
Service Layer
  │
  ▼
Repository Layer
  │
  ▼
Database
  │  (result comes back up through layers)
  ▼
Handler / Controller (sends response)
  │
  ▼
Middleware (Global Error Handler)
  │
  ▼
CLIENT ← HTTP Response
```

### What Happens at Each Stage

```
1. Operating System receives the TCP packet on port 3000
2. OS forwards it to your server application (entry point)
3. Server runs it through middleware functions
4. Routing matches the URL to a specific handler function
5. Handler extracts request data (body, params, query)
6. Handler validates and transforms the data
7. Handler calls the service layer with clean data
8. Service layer runs business logic
9. Service calls repository for database operations
10. Repository builds and executes the SQL query
11. Database returns results
12. Results travel back: Repository → Service → Handler
13. Handler sends appropriate HTTP response to client
```

## Why Three Layers?

You **can** put all your logic in a single function. Nothing stops you. But as your application grows, this becomes a nightmare:

```
❌ Without separation (all in one handler):

function getBooks(req, res) {
  const sort = req.query.sort ?? "date"    // data extraction
  const userId = req.query.userId          // more extraction
  
  if (!userId) {                           // validation
    return res.status(400).json({ error: "userId required" })
  }
  
  // business logic
  const hasPermission = checkPermissions(userId, "books:read")
  if (!hasPermission) return res.status(403).json({ ... })
  
  // database logic
  const query = `SELECT * FROM books WHERE ... ORDER BY ${sort}`
  const books = db.query(query)
  
  // send email notification
  sendNotification(userId, "Books fetched")
  
  // format response
  res.status(200).json({ data: books })
}

Problems with this:
  → Function does too many things (God function)
  → Cannot reuse the business logic elsewhere
  → Cannot unit test individual pieces
  → Swapping databases requires rewriting the handler
  → Adding caching requires modifying the handler
  → Code is hard to read after 200+ lines
```

```
✅ With separation (three layers):

Handler:    Only extracts data + validates + sends response
Service:    Only runs business logic (agnostic to HTTP)
Repository: Only executes database queries

Benefits:
  → Each layer can be changed without affecting others
  → Each layer can be unit tested in isolation
  → Business logic is reusable across different contexts
  → Database can be swapped by only changing the repository
  → Code is DRY (Don't Repeat Yourself)
  → Easy to debug (know exactly which layer caused a bug)
```

**Real-world analogy:**

```
Think of a restaurant:

Waiter (Handler/Controller):
  → Takes the customer's order
  → Validates it ("Sorry, kitchen is out of pasta")
  → Sends the order to the kitchen
  → Brings the food back and presents it

Chef (Service):
  → Receives the order from the waiter
  → Runs the recipe logic (business logic)
  → Doesn't directly interact with the customer
  → Can use multiple ingredients (datasets)

Ingredient Supplier (Repository):
  → Only job is to provide raw ingredients to the chef
  → Doesn't know what dish is being made
  → Just: "Give me tomatoes" → returns tomatoes

Each role is separate. The Chef doesn't take orders
directly from customers. The Waiter doesn't cook.
The result is clean, efficient, maintainable.
```

## Layer 1 - Handler / Controller

### What It Is

The **handler** (also called **controller**) is the entry point for a specific API route inside your server. It's the function that gets invoked when the routing algorithm matches an incoming request to a URL pattern.

```
Route definition (Express.js example):
  app.get("/api/v1/books", getBooksHandler)

When a request hits GET /api/v1/books:
  → Routing matches this URL to getBooksHandler
  → getBooksHandler is called with (request, response)
```

### What the Handler Receives

In virtually every backend language/framework, a handler receives **two objects** by default:

```
REQUEST OBJECT (req):
  Contains everything the client sent:
  → URL path:        req.params.id (dynamic segments)
  → Query params:    req.query.sort, req.query.page
  → Request body:    req.body (for POST, PATCH, PUT)
  → Headers:         req.headers.authorization, req.headers["content-type"]
  → Cookies:         req.cookies.session_id
  → IP address:      req.ip
  → HTTP method:     req.method (GET, POST, etc.)

RESPONSE OBJECT (res):
  Lets you control what is sent back to the client:
  → Set status code: res.status(200)
  → Send body:       res.json({ data: [...] })
  → Set headers:     res.setHeader("X-Request-Id", "abc123")
  → End response:    res.end()

These two objects are GIVEN to you by the framework.
You don't create them. The runtime creates them per request.
```

### Responsibility 1 - Data Extraction + Deserialization (Binding)

The first thing a Handler does is **extract data from the request object** and **convert it into the native language format** (deserialization):

```
Client sends:
  POST /api/v1/books
  Content-Type: application/json
  Body: { "title": "Dune", "author": "Frank Herbert", "year": 1965 }

The body arrives as a RAW STRING (JSON text):
  '{ "title": "Dune", "author": "Frank Herbert", "year": 1965 }'

Handler must DESERIALIZE (convert from string to native format):
  → In Node.js: Express body parser does this automatically
               req.body.title  → "Dune"  (JavaScript object)
  → In Go:     json.Unmarshal(req.Body, &bookInput)
               bookInput.Title → "Dune"  (Go struct)
  → In Python: json.loads(request.body)
               book_input["title"] → "Dune" (Python dict)

If deserialization FAILS (malformed JSON, wrong Content-Type):
  → Return 400 Bad Request immediately
  → Do NOT continue processing
```

**Real-world example:**

```
Stripe's payment API receives:
  { "amount": 1000, "currency": "usd", "customer": "cust_123" }

If your frontend accidentally sends:
  { amount: 1000, currency: usd }   ← Invalid JSON (missing quotes)

Stripe's handler deserialization fails → returns:
  {
    "error": {
      "type": "invalid_request_error",
      "message": "Invalid JSON in request body"
    }
  }
  Status: 400 Bad Request
```

### Responsibility 2 - Validation and Transformation

After the data is in native format, the handler **validates** and optionally **transforms** it:

#### Validation

```
Validation = checking that the data is correct before processing.

Things to validate:
  → Required fields are present
  → Fields are the correct data type
  → Values are within expected ranges/formats
  → No malicious content (XSS, injection attempts)

Example:
  POST /api/v1/users
  Body: { "email": "not-an-email", "age": -5, "name": "" }

Validation errors:
  → email: not a valid email format
  → age: must be a positive number
  → name: cannot be empty

Response: 400 Bad Request
  {
    "error": {
      "code": "VALIDATION_ERROR",
      "details": [
        { "field": "email", "message": "Invalid email format" },
        { "field": "age",   "message": "Age must be positive" },
        { "field": "name",  "message": "Name is required" }
      ]
    }
  }
```

**Golden Rule:** Validate EVERYTHING coming from outside your server. Never trust client input.

```
Validate:
  ✅ req.params.id       → Is this a valid UUID format?
  ✅ req.query.page      → Is this a positive integer?
  ✅ req.query.sort_by   → Is this in the allowed list?
  ✅ req.body.email      → Is this a valid email?
  ✅ req.body.status     → Is this one of "active" | "archived"?
  ✅ req.headers.content-type → Is this "application/json"?
```

#### Transformation

```
Transformation = modifying the validated data to make it convenient
for downstream layers (Service and Repository).

Example: Fetching books with optional sort parameter

Client may or may not send:
  GET /api/v1/books?sort=name   OR
  GET /api/v1/books             (no sort param)

Without transformation:
  Service receives sort = undefined
  Service must then check: if (sort) { ... } else { ... }
  Every downstream layer must handle the "missing" case

With transformation in Handler:
  if (!sort) sort = "created_at"  // set sane default
  Service always receives a valid sort value
  Service code is clean, no null checks needed

Other transformation examples:
  → Trim whitespace: "  Alice  " → "Alice"
  → Lowercase email: "ALICE@GMAIL.COM" → "alice@gmail.com"
  → Parse numbers: "42" (string from query param) → 42 (integer)
  → Set defaults: page not sent → page = 1, limit not sent → limit = 20
```

### Responsibility 3 - Call the Service Layer

After extraction, validation, and transformation, the handler has **clean, validated, typed data**. It passes this to the service layer:

```javascript
// Node.js / Express example
async function getBooksHandler(req, res) {
  // Step 1: Extract
  const sort     = req.query.sort_by
  const order    = req.query.sort_order
  const page     = parseInt(req.query.page)
  const limit    = parseInt(req.query.limit)
  
  // Step 2: Transform (set defaults)
  const finalSort  = sort  || "created_at"
  const finalOrder = order || "desc"
  const finalPage  = isNaN(page)  ? 1  : page
  const finalLimit = isNaN(limit) ? 10 : limit
  
  // Step 3: Call service (pass clean data)
  const result = await bookService.getBooks({
    sort:  finalSort,
    order: finalOrder,
    page:  finalPage,
    limit: finalLimit,
    userId: req.context.userId  // from auth middleware
  })
  
  // Step 4: Send response
  res.status(200).json(result)
}
```

### Responsibility 4 - Send the Response

After the service returns data, the handler sends an appropriate HTTP response:

```
Success cases:
  → GET (fetched data)     → 200 OK + data
  → POST (created)         → 201 Created + new entity
  → PATCH/PUT (updated)    → 200 OK + updated entity
  → DELETE (removed)       → 204 No Content (empty body)
  → Custom action          → 200 OK + result

Failure cases:
  → Validation failed      → 400 Bad Request
  → Not authenticated      → 401 Unauthorized
  → Not authorized         → 403 Forbidden
  → Resource not found     → 404 Not Found
  → Conflict (duplicate)   → 409 Conflict
  → Server error           → 500 Internal Server Error

The CONTROLLER decides the appropriate status code and format.
The SERVICE does not know about status codes.
```

### Handler Responsibility Summary

```
Handler / Controller:
  1. ✅ Extract data from request (params, query, body, headers)
  2. ✅ Deserialize JSON to native format (binding)
  3. ✅ Validate the data → 400 if invalid
  4. ✅ Transform the data (defaults, type conversion, trim)
  5. ✅ Call service layer with clean data
  6. ✅ Receive result from service
  7. ✅ Send appropriate HTTP response (correct status code + body)

Handler does NOT:
  ❌ Execute business logic
  ❌ Directly write SQL queries
  ❌ Know how data is stored
  ❌ Send emails or push notifications (that's service's job)
```

## Layer 2 - Service

### What It Is

The **service layer** contains all the **business logic** of your application. It is the brain - it decides what to do, in what order, and how to combine data from multiple sources.

> **Key Principle:** The service layer should be completely **agnostic to HTTP**. If you look at a service function, you should NOT be able to tell it's part of an API. It's just a function that takes data, processes it, and returns data.

### Why Agnostic to HTTP?

```
Today: Your business logic runs as a REST API.
Tomorrow: You might want to expose the same logic via GraphQL.
Next year: You might want to call it from a CLI tool or a cron job.

If your business logic is inside the handler (mixed with HTTP code):
  → You CANNOT reuse it for GraphQL
  → You CANNOT reuse it for CLI tools
  → You CANNOT unit test it without mocking HTTP

If your business logic is in a pure service function:
  → REST API handler calls it
  → GraphQL resolver calls it
  → CLI tool calls it
  → Cron job calls it
  → Unit tests call it directly
  → ZERO code duplication ✅
```

**Real-world example:**

```
Twitter/X has:
  → A REST API (for developers)
  → A web app (browser)
  → A mobile app (iOS/Android)
  → A GraphQL API (internal)
  → Background workers (cron jobs for notifications)

They ALL share the same service layer:
  → "Create a tweet" service is used by REST API, web, mobile
  → "Get feed" service is used by web and mobile
  → "Send notification" service is used by API and cron jobs

Zero business logic duplication.
Only the delivery mechanism (handler vs resolver vs job) changes.
```

### What the Service Layer Does

```
Service layer responsibilities:
  1. Business logic execution (the actual work)
  2. Orchestration (combining multiple data sources)
  3. External API calls (sending emails, SMS, push notifications)
  4. Background job triggering
  5. Calling multiple repository methods and merging results
  6. Transaction management (all-or-nothing database operations)
```

### Example - Create Organization Service

```javascript
// Service function - NO HTTP knowledge, pure business logic
async function createOrganization({ name, description, ownerId }) {
  
  // 1. Check if organization with same name already exists for this owner
  const existing = await orgRepository.findByNameAndOwner(name, ownerId)
  if (existing) {
    throw new ConflictError("Organization with this name already exists")
  }
  
  // 2. Create the organization
  const org = await orgRepository.create({
    name,
    description: description || null,
    status: "active",        // default
    ownerId,
    createdAt: new Date(),
  })
  
  // 3. Create default settings for the org
  await settingsRepository.createDefaults(org.id)
  
  // 4. Send welcome email to owner
  const owner = await userRepository.findById(ownerId)
  await emailService.sendOrgWelcomeEmail(owner.email, org.name)
  
  // 5. Log the action for audit trail
  await auditRepository.log({
    action: "org:created",
    entityId: org.id,
    performedBy: ownerId,
  })
  
  // 6. Return the created organization
  return org
}
```

Notice:
- No `req`, no `res`, no status codes
- Uses multiple repository methods (org, settings, user, audit)
- Calls an external service (email)
- Pure data in, data out
- Can be called from anywhere

### Example - Archive Organization Service

```javascript
// This demonstrates orchestration - one service method coordinates many operations
async function archiveOrganization({ orgId, requestedBy }) {
  
  // 1. Check if organization exists
  const org = await orgRepository.findById(orgId)
  if (!org) throw new NotFoundError("Organization not found")
  
  // 2. Check permissions
  if (org.ownerId !== requestedBy) {
    throw new ForbiddenError("Only the owner can archive an organization")
  }
  
  // 3. Get all projects under this org
  const projects = await projectRepository.findAllByOrgId(orgId)
  
  // 4. Archive all tasks under each project
  for (const project of projects) {
    await taskRepository.archiveAllByProjectId(project.id)
  }
  
  // 5. Archive all projects
  await projectRepository.archiveAllByOrgId(orgId)
  
  // 6. Get all members of the org
  const members = await memberRepository.findAllByOrgId(orgId)
  
  // 7. Send notifications to all members
  for (const member of members) {
    await emailService.sendOrgArchivedNotification(member.email, org.name)
  }
  
  // 8. Finally archive the organization
  const archived = await orgRepository.update(orgId, { status: "archived" })
  
  return archived
}
```

This is why archiving is a **custom action** (`POST /organizations/:id/archive`) and NOT a simple PATCH - the service triggers a cascade of operations.

## Layer 3 - Repository

### What It Is

The **repository layer** is the only layer that talks to the database. Its sole responsibility is:
- **Taking data** (filters, values to insert, IDs to update/delete)
- **Building a database query**
- **Executing the query**
- **Returning the raw result**

That's it. The repository is completely **agnostic to business logic** - it doesn't know WHY the data is needed, it just fetches or stores it.

### Single Responsibility Per Method

```
One repository method = One database operation = One type of result

✅ CORRECT:
  findAll(filters)       → Returns list of books
  findById(id)           → Returns ONE book (or null)
  create(data)           → Inserts and returns new book
  updateById(id, data)   → Updates and returns updated book
  deleteById(id)         → Deletes and returns boolean

❌ WRONG:
  findBooks(id?)         → If id passed, return one book.
                           If not, return all books.

Why wrong?
  → Mixed responsibility = unpredictable behavior
  → Caller doesn't know what type they'll get back
  → Hard to test (need to test two behaviors in one function)
  → When things go wrong, harder to debug
```

### Example - Book Repository

```javascript
// repository/bookRepository.js

const bookRepository = {
  
  // Get all books with filtering, sorting, pagination
  async findAll({ status, sort = "created_at", order = "desc", page = 1, limit = 10 }) {
    const offset = (page - 1) * limit
    
    // Build parameterized query
    let query = `
      SELECT *, COUNT(*) OVER() AS total_count
      FROM books
      WHERE ($1::text IS NULL OR status = $1)
      ORDER BY ${allowedSortFields[sort]} ${order === "asc" ? "ASC" : "DESC"}
      LIMIT $2
      OFFSET $3
    `
    const result = await db.query(query, [status || null, limit, offset])
    return {
      books: result.rows,
      total: result.rows[0]?.total_count ?? 0
    }
  },
  
  // Get ONE book by ID
  async findById(id) {
    const result = await db.query(
      "SELECT * FROM books WHERE id = $1",
      [id]
    )
    return result.rows[0] ?? null  // null if not found
  },
  
  // Create a new book
  async create({ title, authorId, year, status }) {
    const result = await db.query(`
      INSERT INTO books (title, author_id, year, status, created_at, updated_at)
      VALUES ($1, $2, $3, $4, NOW(), NOW())
      RETURNING *
    `, [title, authorId, year, status])
    return result.rows[0]
  },
  
  // Update specific fields of a book
  async updateById(id, { title, status }) {
    const result = await db.query(`
      UPDATE books
      SET title = COALESCE($2, title),
          status = COALESCE($3, status),
          updated_at = NOW()
      WHERE id = $1
      RETURNING *
    `, [id, title, status])
    return result.rows[0] ?? null
  },
  
  // Delete a book
  async deleteById(id) {
    const result = await db.query(
      "DELETE FROM books WHERE id = $1 RETURNING id",
      [id]
    )
    return result.rowCount > 0  // true if deleted
  }
}
```

### Why Isolate the Repository?

```
Scenario: Your company migrates from PostgreSQL to MongoDB.

Without repository layer:
  → SQL queries are scattered across handlers and services
  → Every handler and service must be rewritten
  → 200+ files to touch
  → High risk of bugs during migration
  → Weeks of work

With repository layer:
  → Only the repository files need to change
  → Handlers and services are untouched (same function signatures)
  → 10-15 files to change at most
  → Handlers/services don't even know the database changed
  → Days of work (not weeks)
```

**Real-world example:**

```
GitHub migrated from MySQL to a custom database system.
Because they had repository abstractions, they could:
  → Swap the storage layer without changing application logic
  → Run old MySQL and new storage in parallel during migration
  → Roll back if something went wrong
  → The "create repository" service method signature stayed the same
```

## How All Three Layers Cooperate

### Complete Example - GET /api/v1/books

```
CLIENT: GET /api/v1/books?sort_by=title&page=2&limit=5
  │
  ▼
HANDLER (BooksController.getAll):
  1. Extract query params:
     sort_by = "title", page = 2, limit = 5
  2. Parse types:
     page = parseInt("2") = 2
     limit = parseInt("5") = 5
  3. Validate:
     sort_by "title" ∈ allowed ["title", "year", "created_at"] ✅
     page = 2 > 0 ✅
     limit = 5 ≤ 100 ✅
  4. Call service:
     books = await bookService.getBooks({ sort: "title", page: 2, limit: 5 })
  
  │
  ▼
SERVICE (bookService.getBooks):
  1. No additional business logic here (simple read)
  2. Just calls repository:
     result = await bookRepository.findAll({ sort: "title", page: 2, limit: 5 })
  3. Returns the result to handler
  
  │
  ▼
REPOSITORY (bookRepository.findAll):
  1. Calculate offset = (2 - 1) * 5 = 5
  2. Build SQL:
     SELECT *, COUNT(*) OVER() as total
     FROM books
     ORDER BY title ASC
     LIMIT 5 OFFSET 5
  3. Execute query against PostgreSQL
  4. Return: { books: [...5 books...], total: 42 }
  
  │  (result travels back up)
  ▼
SERVICE: Returns { books: [...], total: 42 }
  │
  ▼
HANDLER: Formats response:
  {
    "data": [...5 books...],
    "pagination": {
      "page": 2,
      "limit": 5,
      "total": 42,
      "total_pages": 9
    }
  }
  Status: 200 OK
  │
  ▼
CLIENT receives the response ✅
```

## Middleware

### What Is Middleware?

**Middleware** is a function that runs **in the middle** of the request life cycle - between the request arriving and the handler processing it (or between the handler and the response going back).

Middleware functions are **optional**, **reusable**, and execute in a **chain**.

```
Without middleware (no chain):
  Request → Routing → Handler → Response

With middleware (chain of functions):
  Request →
    CORS middleware →
      Logging middleware →
        Routing →
          Auth middleware →
            Handler → Service → Repository →
          Handler sends response →
        Error handling middleware →
      Response
```

### The Three Parameters of Middleware

In virtually every backend framework, a middleware receives **three things**:

```javascript
function myMiddleware(req, res, next) {
  //                  ▲    ▲    ▲
  //                  │    │    └── "next" - a function to continue to the next middleware
  //                  │    └─────── response object (same as in handlers)
  //                  └──────────── request object (same as in handlers)
}
```

```
req:  Same request object as handlers. Read request data, headers, cookies.
res:  Same response object as handlers. CAN send a response back at any point.
next: A function that passes execution to the NEXT middleware in the chain.

If you DON'T call next():
  → Request stops here
  → Whatever you sent via res is returned to the client
  → Next middlewares and handlers are NEVER executed

If you DO call next():
  → Execution passes to the next middleware or handler
  → Current middleware's work is done, it can still do work after next() in some cases

If you call next(error):
  → Error is passed to the error handling middleware
  → Skips all regular middlewares
```

### Middleware vs Handler - What's Different?

```
Handler:    receives (req, res)        - 2 params
Middleware: receives (req, res, next)  - 3 params

Handler:    route-specific. Only runs for one route.
Middleware: can run for ALL routes, or specific groups of routes.

Handler:    must always send a response (it's the final destination).
Middleware: usually just modifies req/res and calls next() to continue.
```

### The Chain - Execution Order

```
Each middleware calls next() to pass to the next one.
Think of it as a relay race:

Middleware A → does work → calls next()
  └── Middleware B → does work → calls next()
        └── Middleware C → does work → calls next()
              └── Handler → does work → sends response
```

```javascript
// Express.js example - chain of middleware

app.use(corsMiddleware)       // runs for ALL routes, calls next()
app.use(loggingMiddleware)    // runs for ALL routes, calls next()
app.use(authMiddleware)       // runs for ALL routes, calls next() or sends 401

app.get("/api/v1/books", getBooksHandler)  // runs only for this route
```

## Common Middleware Examples

### 1. CORS Middleware

**CORS** (Cross-Origin Resource Sharing) is a browser security mechanism that blocks requests from different domains unless the server explicitly allows them.

```
The problem:
  Your frontend: https://myapp.com
  Your API:      https://api.myapp.com (different subdomain = different origin)

Without CORS headers:
  Browser blocks the request from myapp.com to api.myapp.com
  User sees: "Network Error" or blank page

With CORS middleware:
  Server adds headers telling the browser "this origin is allowed"
  Browser allows the response
  Everything works ✅
```

```javascript
// CORS middleware implementation (simplified)
function corsMiddleware(req, res, next) {
  const allowedOrigins = [
    "https://myapp.com",
    "https://admin.myapp.com",
    "http://localhost:3001"  // local development
  ]
  
  const origin = req.headers.origin
  
  if (allowedOrigins.includes(origin)) {
    // Tell the browser this origin is allowed
    res.setHeader("Access-Control-Allow-Origin", origin)
    res.setHeader("Access-Control-Allow-Methods", "GET, POST, PATCH, DELETE, OPTIONS")
    res.setHeader("Access-Control-Allow-Headers", "Content-Type, Authorization")
    res.setHeader("Access-Control-Allow-Credentials", "true")
  }
  
  // Preflight request (browser sends OPTIONS first to check if allowed)
  if (req.method === "OPTIONS") {
    return res.status(204).end()  // No content, just headers
  }
  
  next()  // ← Pass to next middleware
}
```

**Why middleware and not handler?**
- Must run for **every single request** (all routes)
- Must access and modify both request (read origin) and response (set headers)
- That's the perfect middleware use case

**Real-world:** Every major API (Stripe, GitHub, Spotify) has CORS middleware configured to allow only their own frontend domains.

// Logging middleware - logs every request for debugging and audit
function loggingMiddleware(req, res, next) {

### 3. Authentication Middleware

```javascript
// Authentication middleware - verifies token, attaches user to request
async function authMiddleware(req, res, next) {
  
  // 1. Extract token from Authorization header
  const authHeader = req.headers.authorization
  if (!authHeader || !authHeader.startsWith("Bearer ")) {
    return res.status(401).json({
      error: { code: "UNAUTHORIZED", message: "No token provided" }
    })
    // ← Does NOT call next(). Request stops here.
  }
  
  const token = authHeader.split(" ")[1]  // "Bearer <token>"
  
  // 2. Verify token
  let payload
  try {
    payload = jwt.verify(token, process.env.JWT_SECRET)
  } catch (err) {
    return res.status(401).json({
      error: { code: "TOKEN_INVALID", message: "Token is invalid or expired" }
    })
    // ← Does NOT call next(). Request stops here.
  }
  
  // 3. Attach user info to request context (for downstream handlers)
  req.context = {
    userId: payload.userId,
    role:   payload.role,    // "admin", "user", "editor"
    email:  payload.email
  }
  
  next()  // ← Token valid, pass to next middleware/handler
}
```

```
Two outcomes:
  1. Token missing or invalid → 401 Unauthorized → request STOPS
     Handler is NEVER called. No wasted resources.

  2. Token valid → user info attached to req.context → next() called
     Handler receives req.context.userId, req.context.role
     No need to re-verify token in the handler.

Why middleware for authentication?
  → Every protected route needs it
  → Without middleware: copy-paste token verification in every handler
  → With middleware: write once, apply to all routes → DRY ✅
```

**Real-world:**

```
GitHub's API:
  Every request to api.github.com checks the Authorization header.
  If valid token → req.user is set with GitHub user info.
  If invalid → immediately returns 401 Unauthorized.
  All their 200+ API endpoints benefit from ONE middleware.
```

### 4. Rate Limiting Middleware

```javascript
// Rate limiting middleware - prevent abuse
const requestCounts = new Map()  // IP → { count, windowStart }

function rateLimitMiddleware(req, res, next) {
  const ip      = req.ip
  const now     = Date.now()
  const WINDOW  = 60 * 1000   // 1 minute window
  const MAX_REQ = 100         // max 100 requests per minute
  
  const current = requestCounts.get(ip)
  
  if (!current || now - current.windowStart > WINDOW) {
    // New client or time window reset
    requestCounts.set(ip, { count: 1, windowStart: now })
    return next()
  }
  
  if (current.count >= MAX_REQ) {
    // Too many requests from this IP
    return res.status(429).json({
      error: {
        code: "RATE_LIMIT_EXCEEDED",
        message: "Too many requests. Please slow down.",
        retryAfter: Math.ceil((current.windowStart + WINDOW - now) / 1000) + " seconds"
      }
    })
  }
  
  current.count++
  next()  // Under the limit, proceed
}
```

```
Without rate limiting:
  → A malicious actor sends 10,000 requests/second
  → Your database gets overwhelmed
  → Legitimate users can't use the service
  → DDoS attack succeeds

With rate limiting:
  → After 100 requests in one minute from the same IP
  → They get 429 Too Many Requests
  → Your server is protected even if one client goes crazy
```

**Real-world:**

```
GitHub API: 5,000 requests/hour for authenticated users
Twitter API: 300 requests/15 minutes for timeline
Stripe API: 100 read operations/second
OpenAI API: 500 requests/minute (varies by plan)

All of these are implemented via rate limiting middleware.
```

### 5. Global Error Handling Middleware

```javascript
// This is a SPECIAL Express middleware - 4 parameters signals it's an error handler
function globalErrorHandler(err, req, res, next) {
                          // ▲ NOTE: 4 params, first is the error
  
  // Log the error for debugging
  console.error({
    requestId: req.requestId,
    error: err.message,
    stack: err.stack,
  })
  
  // Determine response based on error type
  if (err.name === "ValidationError" || err.statusCode === 400) {
    return res.status(400).json({
      error: {
        code: "VALIDATION_ERROR",
        message: err.message,
        details: err.details
      }
    })
  }
  
  if (err.name === "NotFoundError" || err.statusCode === 404) {
    return res.status(404).json({
      error: { code: "NOT_FOUND", message: err.message }
    })
  }
  
  if (err.name === "UnauthorizedError" || err.statusCode === 401) {
    return res.status(401).json({
      error: { code: "UNAUTHORIZED", message: "Authentication required" }
    })
  }
  
  // Default: unknown server error
  return res.status(500).json({
    error: {
      code: "INTERNAL_SERVER_ERROR",
      message: "Something went wrong on our end"
    }
  })
}

// Register it LAST, after all other middleware and routes
app.use(globalErrorHandler)
```

```
Without global error handler:
  An uncaught error in an async function:
    → Server crashes OR
    → Request hangs forever (client never gets a response) OR
    → Raw stack trace is sent to client (HUGE security risk!)

With global error handler:
  → Any error from any layer is caught
  → Properly formatted error is sent to client
  → Client never sees internal stack traces
  → Server keeps running (doesn't crash)
```

**Why must it be LAST?**

```
Execution flow for a request:

Request →
  CORS middleware ────────────────────────────────────────────┐
    Logging middleware ────────────────────────────────────┐   │
      Auth middleware → 401 if fails ─────────────────┐   │   │
        Handler (error thrown here!) ──────────────┐   │   │   │
          Global Error Handler ◄────────────────────┘   │   │   │
        Response sent to client ──────────────────────────┘   │   │
      End ────────────────────────────────────────────────────┘   │
    End ──────────────────────────────────────────────────────────┘
  End

The Global Error Handler must be at the END so it can catch errors
from earlier in the chain. If placed before Handler, it would
never catch Handler errors (they haven't happened yet).
```

## Middleware Ordering - Why It Matters

The order you register middlewares is **critical**. Middlewares run in registration order.

### Recommended Order for a Typical Production App

```
1. CORS
   → Do this FIRST. Pointless to do any work for a blocked origin.

2. Request Logger
   → Early so we capture all request info immediately.

3. Body Parser / Data Binding
   → Parse JSON body before any middleware that reads the body.

4. Security Headers
   → Add security headers early.

5. Rate Limiting
   → Before authentication, so even unauthenticated requests are
     rate limited (protection against brute force).

6. Authentication
   → Verify JWT (or session cookie). Attach user to req.context.

7. Application-specific middleware
   → Route-level authorization checks, feature flags, etc.

8. Routes + Handlers
   → The actual business logic runs here.

9. Global Error Handler (ALWAYS LAST)
   → Must be after routes to catch errors from handlers.
```

```
Bad ordering example:
  app.use(authMiddleware)     ← Auth runs first
  app.use(loggingMiddleware)  ← Logging runs AFTER auth

Problem: If auth fails and returns 401, the logging middleware
never runs. You have no record in your logs that this request
even happened. Debugging authentication issues becomes impossible.

Good ordering:
  app.use(loggingMiddleware)  ← Always log first
  app.use(authMiddleware)     ← Then authenticate
```

## Request Context

### What Is Request Context?

**Request context** is a **shared state** attached to a single HTTP request. It's accessible by all middleware and handlers that process that request - but it's isolated from every other request.

```
Think of it like a "clipboard" for each request:
  → Auth middleware writes user info to the clipboard
  → Rate limit middleware writes the request ID to the clipboard
  → Handler reads from the clipboard to get the user ID

One clipboard per request.
Clipboard is destroyed when the response is sent.
Clipboard is NOT shared between different requests.
```

### Why Do We Need It?

```
Without request context (passing data manually):

  authMiddleware → passes userId as function argument
    → route handler receives userId
      → service receives userId (must be passed explicitly)
        → repository receives userId (must be passed all the way down)

Every function needs to accept userId as a parameter.
Every call site must pass userId.
10 layers deep → userId in 10 function signatures.
Change userId type → change 10 function signatures.
Extremely tightly coupled.

With request context:
  authMiddleware → stores userId in req.context

  Later, anywhere in the chain:
    → handler reads req.context.userId
    → or middleware reads req.context.userId
  
  Decoupled. Clean function signatures. Easy to add new data.
```

### How Request Context Works in Practice

```javascript
// Middleware 1: Generate request ID and attach it
app.use((req, res, next) => {
  req.context = {}  // initialize context
  req.context.requestId = generateUUID()
  next()
})

// Middleware 2: Authentication - attach user info
app.use(async (req, res, next) => {
  const token = req.headers.authorization?.split(" ")[1]
  
  if (!token) {
    return res.status(401).json({ error: "No token" })
  }
  
  const payload = jwt.verify(token, JWT_SECRET)
  
  // Store in context - accessible everywhere downstream
  req.context.userId   = payload.userId
  req.context.role     = payload.role
  req.context.email    = payload.email
  req.context.isAdmin  = payload.role === "admin"
  
  next()
})

// Handler - reads from context, never re-verifies token
app.post("/api/v1/books", async (req, res) => {
  const { userId, role } = req.context  // ← read what auth middleware stored
  
  if (role !== "admin") {
    return res.status(403).json({ error: "Admins only" })
  }
  
  const book = await bookService.createBook({
    ...req.body,
    createdBy: userId  // ← using userId from context, NOT from request body!
    // WHY? If we used req.body.userId, a malicious user could send
    // someone else's userId and spoof requests!
  })
  
  res.status(201).json({ data: book })
})
```

### Security Importance of Request Context

```
❌ INSECURE - Taking user ID from client:
  POST /api/v1/books
  Body: { "title": "...", "userId": "other-users-id" }

  handler:
    const createdBy = req.body.userId  // NEVER DO THIS!
    await bookService.createBook({ ..., createdBy })
  
  Attacker sends: { "userId": "admin-user-id" }
  → Creates a book attributed to the admin!
  → Data integrity violated!

✅ SECURE - Taking user ID from context (set by auth middleware):
  handler:
    const createdBy = req.context.userId  // ← from verified JWT token
    await bookService.createBook({ ..., createdBy })
  
  Attacker can't change this - it comes from their JWT token which
  was signed by the server. They can't forge the server's signature.
```

### What Else Gets Stored in Request Context?

```
req.context = {
  requestId:   "abc-123-def",      // unique ID for this request (for logging)
  userId:      "user-uuid-here",   // authenticated user's ID
  role:        "admin",            // user's role
  email:       "alice@gmail.com",  // user's email
  isAdmin:     true,               // computed permission
  orgId:       "org-uuid-here",    // current organization context
  permissions: ["books:read", "books:write"],  // granular permissions
  startTime:   1234567890          // for measuring request duration
}
```

### Request Context for Cancellation and Deadlines

A critical use case in production backends:

```
Scenario:
  Client sends a request.
  Your handler makes a database query that takes 30 seconds.
  At 5 seconds, the client gives up and closes the connection.

Without context deadline:
  → Your server keeps running the 30-second query uselessly
  → Database resources are wasted
  → If this happens 1000 times/minute → database is overloaded

With context deadline (especially common in Go):
  → Request context has a 10-second deadline
  → If client disconnects, context is cancelled
  → Database query is ABORTED immediately
  → Resources are freed

// Go example:
  ctx, cancel := context.WithTimeout(req.Context(), 10*time.Second)
  defer cancel()
  
  rows, err := db.QueryContext(ctx, "SELECT * FROM books")
  // If ctx is cancelled (timeout or client disconnect), query stops.
```

**Real-world:** Google's internal framework (used for all their services) uses context cancellation extensively. When you close a Google Search tab mid-search, the server stops the database queries immediately - saving compute resources.

## Real-World Examples

### Example 1 - GitHub API Request Life Cycle

```
YOU: GET https://api.github.com/repos/torvalds/linux

Request hits GitHub's servers:
  
  1. CORS Middleware:
     Origin: null (you're using curl or a non-browser tool)
     → Sets public headers (GitHub API is public)
  
  2. Rate Limit Middleware:
     Check your IP: 60 requests/hour without auth
     You've made 5 requests → still under limit
     → Adds X-RateLimit-Remaining: 55 header
     → Calls next()
  
  3. Authentication Middleware:
     No Authorization header sent
     → req.context.authenticated = false
     → req.context.rateLimit = 60 (unauthenticated limit)
     → Calls next() (this endpoint is public)
  
  4. Routing:
     Matches /repos/:owner/:repo → getRepoHandler
  
  5. Handler (getRepoHandler):
     Extract: owner = "torvalds", repo = "linux"
     Validate: both strings present ✅
     Call: await repoService.getRepo("torvalds", "linux")
  
  6. Service (repoService.getRepo):
     Check cache: is this repo cached? (it's famous, probably yes)
     If cached: return cached data
     If not: call repoRepository.findByOwnerAndName(owner, name)
  
  7. Repository (repoRepository.findByOwnerAndName):
     SELECT * FROM repositories WHERE owner = $1 AND name = $2
     Returns repo row
  
  8. Handler: Format response
     → { id, name: "linux", owner: { login: "torvalds" }, stars: ..., forks: ... }
     → Status: 200 OK
  
  9. Response reaches you.
```

### Example 2 - Stripe Payment Request Life Cycle

```
YOU: POST https://api.stripe.com/v1/charges
     Header: Authorization: Bearer sk_live_...
     Body: { "amount": 2000, "currency": "usd", "customer": "cust_xxxx" }

  1. CORS: Stripe's API is called server-to-server (not browser), passes through.
  
  2. Auth Middleware:
     Extracts API key from "Authorization: Bearer sk_live_..."
     Looks up key in database → found, valid
     req.context.accountId = "acct_stripe_merchant_id"
     req.context.permissions = ["charges:write"]
     Calls next()
  
  3. Rate Limit Middleware:
     acct_stripe_merchant_id has made 3 requests this second
     Limit is 100/second → passes
     Calls next()
  
  4. Idempotency Middleware:
     Check Idempotency-Key header → not present
     (If it were, Stripe would return cached response, preventing double-charge)
     Calls next()
  
  5. Body Parsing Middleware:
     Parses application/x-www-form-urlencoded body (Stripe uses form encoding)
     Calls next()
  
  6. Handler (chargeHandler):
     Extract: amount = 2000, currency = "usd", customer = "cust_xxxx"
     Validate: amount is positive integer ✅, currency is valid ✅
     Call: chargeService.create({ amount, currency, customerId, accountId })
  
  7. Service (chargeService.create):
     Look up customer → customerRepository.findById("cust_xxxx")
     Get customer's payment method → paymentMethodRepository.findByCustomer(...)
     Call external payment network (Visa/Mastercard) → chargePaymentNetwork(...)
     If approved: create charge record → chargeRepository.create(...)
     Send receipt email → emailService.sendReceipt(...)
     Log audit trail → auditRepository.log(...)
     Return charge object
  
  8. Handler: Return 201 Created + charge object
```

### Example 3 - Middleware Without Authentication (Public Route)

```
What happens when a protected route has auth middleware
but an attacker calls it without a token?

Attacker: DELETE https://api.example.com/api/v1/users/admin-id

  1. CORS Middleware: passes (valid public origin maybe)
  
  2. Logging Middleware: logs the request (useful for detecting attacks)
  
  3. Auth Middleware:
     Checks Authorization header → MISSING
     Returns: 401 Unauthorized immediately
     { "error": { "code": "UNAUTHORIZED", "message": "No token provided" } }
     Does NOT call next()
  
  ← Request STOPS here. Handler is NEVER reached.
     Admin user is NOT deleted.
     
This is why auth middleware is crucial. No matter how many endpoints
you add, they are ALL protected by this single middleware function.
```

## Summary - The Three Layers at a Glance

```
┌─────────────────────────────────────────────────────────────┐
│ LAYER          │ RESPONSIBILITY              │ KNOWS ABOUT   │
├─────────────────────────────────────────────────────────────┤
│ Handler /      │ Extract, validate,          │ HTTP (req/    │
│ Controller     │ transform, call service,    │ res, status   │
│                │ send response               │ codes)        │
├─────────────────────────────────────────────────────────────┤
│ Service        │ Business logic,             │ Data          │
│                │ orchestration, external     │ structures,   │
│                │ API calls                   │ business rules│
├─────────────────────────────────────────────────────────────┤
│ Repository     │ Build and execute database  │ SQL/Database  │
│                │ queries, return raw results │ queries only  │
└─────────────────────────────────────────────────────────────┘

MIDDLEWARE      │ Reusable functions that run for EVERY request
                │ or a group of routes. Access req and res.
                │ Must call next() to continue chain or res.send()
                │ to stop the chain.

REQUEST CONTEXT │ Shared state per request. Set by middlewares
                │ (auth, logging, etc.), read by handlers.
                │ Prevents passing data explicitly through every layer.
```
