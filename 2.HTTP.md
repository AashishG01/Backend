# HTTP Protocol - Complete Guide

> **Core idea:** HTTP (HyperText Transfer Protocol) is the medium through which browsers talk to servers - to send data or receive data. It is **stateless** (each request is independent, the server forgets everything after responding) and follows a **client-server model** (communication is always initiated by the client). Understanding HTTP deeply means understanding headers, methods, status codes, CORS, caching, compression, and content negotiation - the building blocks of every web interaction.

## The Two Core Ideas of HTTP

Everything in HTTP is built on two foundational ideas:

### 1. Statelessness

```
Stateless = the server has NO MEMORY of past interactions.

Each HTTP request:
  → Carries ALL the information the server needs to process it
  → Headers, URL, method, body - everything is in ONE request
  → After the server responds, it FORGETS about the request completely
  → If the client makes another request, the server treats it as
    a completely new, unrelated event

It's like talking to someone with amnesia:
  → Every conversation starts from scratch
  → They don't remember what you said 5 minutes ago
  → You must re-introduce yourself every time
```

**Why is statelessness a benefit?**

```
1. SIMPLICITY
   → Server doesn't need to store session info
   → No memory management for tracking who's talking to it
   → Less complexity, fewer bugs

2. SCALABILITY
   → Requests can go to ANY server (no "sticky" sessions needed)
   → If you have 10 servers, any one can handle any request
   → Just add more servers when traffic increases

3. RELIABILITY
   → If a server crashes, nothing is lost
   → There's no "session state" that disappears
   → Client just sends the next request to a different server
```

**But wait - what about logins and shopping carts?**

```
HTTP is stateless, but APPLICATIONS need state (who's logged in, etc.)

Solution: State management techniques built ON TOP of HTTP:
  → Cookies:  Server sends a small token, browser stores and resends it
  → Sessions: Server-side storage keyed by a session ID in a cookie
  → Tokens:   JWT tokens sent in Authorization header

These don't break HTTP's statelessness - they work WITHIN it.
Every request still carries ALL its info (the token/cookie is part of the request).
The server doesn't "remember" - it just reads the token each time.
```

### 2. Client-Server Model

```
In HTTP, there are always exactly TWO roles:

CLIENT:
  → Initiates the communication (always the one to send first)
  → Provides all information the server needs (URL, headers, body)
  → Examples: web browsers, mobile apps, Postman, curl, another server

SERVER:
  → Waits for incoming requests (never initiates contact with client)
  → Processes the request
  → Sends back a response (web page, JSON, error, file, etc.)

Key rule: Communication is ALWAYS initiated by the client.
The server never randomly sends data to a client unprompted.
(WebSockets break this rule, but that's a different protocol.)
```

> **Note:** Throughout this guide, HTTP and HTTPS are interchangeable. HTTPS is simply HTTP with encryption (TLS). The underlying protocol principles are identical - HTTPS just adds security.

## HTTP and TCP - The Transport Layer

Before HTTP messages can be exchanged, client and server need a **connection**. HTTP uses **TCP** (Transmission Control Protocol) for this.

```
Why TCP and not UDP?

TCP = reliable delivery (guarantees all data arrives, in order)
UDP = fast but unreliable (data might get lost or arrive out of order)

HTTP needs reliability - if half your web page doesn't arrive,
that's a broken experience. So HTTP uses TCP.

How TCP works (simplified):
  1. Client says: "I want to connect" (SYN)
  2. Server says: "OK, acknowledged" (SYN-ACK)
  3. Client says: "Great, connection open" (ACK)
  → This is called the TCP 3-way handshake

After the handshake, HTTP messages can flow freely.
```

```
OSI Model - Where HTTP fits:

Layer 7: Application  ← HTTP lives here (what WE work with)
Layer 6: Presentation
Layer 5: Session
Layer 4: Transport    ← TCP lives here
Layer 3: Network      ← IP lives here
Layer 2: Data Link
Layer 1: Physical     ← Electrical signals / radio waves

As backend engineers, we work at Layer 7 (HTTP).
TCP handshakes happen at Layer 4 automatically.
We don't need to manage TCP directly - our framework handles it.
```

## Evolution of HTTP

HTTP has evolved significantly over the years:

```
┌──────────┬──────────────────────────────────────────────────────────────┐
│ Version  │ Key Changes                                                  │
├──────────┼──────────────────────────────────────────────────────────────┤
│ HTTP/1.0 │ Each request opens a NEW TCP connection                      │
│          │ Connection closed after every response                       │
│          │ Very inefficient - creating connections is expensive          │
├──────────┼──────────────────────────────────────────────────────────────┤
│ HTTP/1.1 │ Persistent connections (reuse same TCP connection)           │
│ (most    │ Multiple requests over one connection                        │
│  used)   │ Chunked transfer encoding (stream large responses)           │
│          │ Better caching mechanisms                                    │
│          │ Keep-alive is DEFAULT (no need to re-open connections)       │
├──────────┼──────────────────────────────────────────────────────────────┤
│ HTTP/2.0 │ Multiplexing (many requests/responses over ONE connection)   │
│          │ Binary framing instead of text (faster parsing)              │
│          │ Header compression (HPACK - smaller headers)                 │
│          │ Server push (server sends resources before client asks)      │
├──────────┼──────────────────────────────────────────────────────────────┤
│ HTTP/3.0 │ Built on QUIC protocol (uses UDP instead of TCP)             │
│          │ Faster connection establishment                              │
│          │ Reduced latency                                              │
│          │ Better packet loss handling                                   │
│          │ No head-of-line blocking (an HTTP/2 problem)                 │
└──────────┴──────────────────────────────────────────────────────────────┘

For our purposes: HTTP/1.1 is the most widely used and the one
we interact with daily. The core concepts (methods, headers,
status codes) remain the same across all versions.
```

## HTTP Messages - Request and Response

Every HTTP interaction involves exactly two messages: a **request** (from client) and a **response** (from server).

### Request Message Structure

```
POST /api/v1/users HTTP/1.1          ← Request line: METHOD + URL + VERSION
Host: api.example.com                ← Headers start
Content-Type: application/json
Authorization: Bearer eyJhbGci...
Accept: application/json
User-Agent: Mozilla/5.0
                                     ← Blank line (separates headers from body)
{                                    ← Request body starts
  "name": "Aashish",
  "email": "aashish@example.com"
}

Breaking it down:
  POST                    → HTTP method (the WHAT - create something)
  /api/v1/users           → Resource URL (the WHERE - on what resource)
  HTTP/1.1                → Protocol version
  Host: api.example.com   → The domain we're talking to
  Content-Type            → "My body is JSON"
  Authorization           → "Here's my auth token"
  Accept                  → "I want JSON back"
  User-Agent              → "I'm a browser (not curl, not Postman)"
  Body                    → The actual data being sent
```

### Response Message Structure

```
HTTP/1.1 200 OK                      ← Status line: VERSION + CODE + TEXT
Content-Type: application/json       ← Response headers
Cache-Control: max-age=3600
X-Request-Id: abc-123-def
                                     ← Blank line (separates headers from body)
{                                    ← Response body
  "data": {
    "id": 1,
    "name": "Aashish",
    "email": "aashish@example.com"
  }
}

Breaking it down:
  HTTP/1.1            → Protocol version
  200                 → Status code (success!)
  OK                  → Human-readable status text
  Content-Type        → "My response body is JSON"
  Cache-Control       → "Cache this for 1 hour"
  X-Request-Id        → Custom header for tracking
  Body                → The actual response data
```

## HTTP Headers - Deep Dive

### What Are Headers?

Headers are **key-value pairs** of metadata sent with every request and response. They carry information ABOUT the message, not the message content itself.

### The Parcel Analogy

```
Think of sending a courier package:

  PACKAGE INSIDE:  The actual item (= request/response body)
  
  LABEL ON TOP:    Recipient name, address, phone number,
                   "FRAGILE", weight, tracking number
                   (= HTTP headers)

Why put the label ON TOP?
  → The delivery person needs to read it WITHOUT opening the package
  → Multiple handlers along the route need this info
  → It's metadata ABOUT the package, not the package itself

Same with HTTP headers:
  → Intermediaries (proxies, CDNs, load balancers) read headers
    without opening the body
  → Headers tell everyone HOW to handle the message
  → Body tells the final recipient WHAT the message contains
```

### Categories of Headers

#### 1. Request Headers (Client → Server)

```
These tell the server about the CLIENT and its preferences:

User-Agent: Mozilla/5.0 (Windows NT 10.0; ...)
  → "I am a Chrome browser on Windows 10"
  → Server can customize response for different clients

Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
  → "Here are my credentials (JWT token)"
  → Server uses this to identify and authenticate the user

Accept: application/json
  → "I prefer the response in JSON format"
  → Server tries to respond in that format

Accept-Language: en-US
  → "I prefer English content"
  → Server can return localized content

Accept-Encoding: gzip, deflate, br
  → "I support these compression formats"
  → Server can compress the response to save bandwidth
```

#### 2. Response Headers (Server → Client)

```
These tell the CLIENT about the response and the server:

Content-Type: application/json
  → "My response body is JSON"

Content-Length: 256
  → "My response body is 256 bytes"

Set-Cookie: session_id=abc123; HttpOnly; Secure
  → "Store this cookie and send it back on future requests"

Access-Control-Allow-Origin: https://myapp.com
  → "I allow requests from myapp.com" (CORS)
```

#### 3. General Headers (Both Directions)

```
Used in both requests AND responses:

Date: Tue, 15 Mar 2024 14:30:00 GMT
  → Timestamp of the message

Cache-Control: max-age=3600
  → Caching instructions (cache for 1 hour)

Connection: keep-alive
  → Keep the TCP connection open for more requests

Connection: close
  → Close the TCP connection after this response
```

#### 4. Representation Headers (About the Body)

```
Describe the format and encoding of the body:

Content-Type: application/json
  → Body format is JSON

Content-Type: text/html; charset=utf-8
  → Body format is HTML with UTF-8 encoding

Content-Length: 1024
  → Body is 1024 bytes

Content-Encoding: gzip
  → Body is compressed with gzip

ETag: "abc123hash"
  → Unique hash of the response body (for caching)
```

#### 5. Security Headers

```
Enhance security of the communication:

Strict-Transport-Security: max-age=31536000
  → "Only communicate with me over HTTPS, never HTTP"
  → Prevents protocol downgrade attacks

Content-Security-Policy: default-src 'self'
  → "Only load JS/CSS/images from the same domain"
  → Prevents XSS (cross-site scripting) attacks

X-Frame-Options: DENY
  → "Don't allow my page to be embedded in an iframe"
  → Prevents clickjacking attacks

X-Content-Type-Options: nosniff
  → "Don't try to guess the content type, trust my header"
  → Prevents MIME type sniffing attacks

Set-Cookie: token=abc; HttpOnly; Secure
  → HttpOnly: JavaScript CANNOT access this cookie (prevents XSS theft)
  → Secure: Cookie is only sent over HTTPS (prevents interception)
```

### Headers Enable Two Powerful Ideas

**1. Extensibility:**
```
HTTP is highly extensible BECAUSE of headers.
You can add new headers without changing the protocol itself.

Custom headers: X-Custom-Header: myvalue
Security headers added years after HTTP was invented
CORS headers added to solve cross-origin problems
Caching headers evolved over time

All without changing HTTP's core. Just new headers.
```

**2. Remote Control:**
```
Headers act like a "remote control" - the client sends
instructions that change how the server responds:

Accept: application/xml     → Server switches response to XML
Accept-Language: es          → Server switches to Spanish
Accept-Encoding: gzip       → Server compresses the response
Authorization: Bearer xyz   → Server authenticates and changes access level
Cache-Control: no-cache     → Server sends fresh response, no caching

The same endpoint can behave differently based on headers.
Headers are the control mechanism of HTTP.
```

## HTTP Methods

HTTP methods express the **intent** of a request - what action the client wants to perform.

```
┌──────────┬──────────────────────────────────────────────────────────────┐
│ Method   │ Purpose                                                      │
├──────────┼──────────────────────────────────────────────────────────────┤
│ GET      │ Fetch / read a resource. Should NOT modify anything.         │
│          │ Has NO request body. Data sent via query params.             │
│          │ Example: GET /api/users → returns list of users              │
├──────────┼──────────────────────────────────────────────────────────────┤
│ POST     │ Create a new resource. HAS a request body.                   │
│          │ Example: POST /api/users → creates a new user               │
│          │ Body: { "name": "Aashish", "email": "a@b.com" }            │
├──────────┼──────────────────────────────────────────────────────────────┤
│ PATCH    │ Update SPECIFIC FIELDS of a resource. HAS a body.            │
│          │ Selective/partial update - only changes specified fields.     │
│          │ Example: PATCH /api/users/123                                │
│          │ Body: { "name": "New Name" } ← only updates name            │
├──────────┼──────────────────────────────────────────────────────────────┤
│ PUT      │ COMPLETELY REPLACE a resource. HAS a body.                   │
│          │ Full replacement - entire resource is overwritten.            │
│          │ Example: PUT /api/users/123                                  │
│          │ Body: { "name": "New", "email": "new@b.com", "age": 25 }   │
│          │ ← Every field must be provided, old data is REPLACED         │
├──────────┼──────────────────────────────────────────────────────────────┤
│ DELETE   │ Remove a resource from the server.                           │
│          │ Example: DELETE /api/users/123 → deletes user 123            │
├──────────┼──────────────────────────────────────────────────────────────┤
│ OPTIONS  │ Ask the server about its capabilities.                       │
│          │ Used in CORS pre-flight requests (browser sends this).       │
│          │ You rarely use this directly as a developer.                 │
├──────────┼──────────────────────────────────────────────────────────────┤
│ HEAD     │ Same as GET but returns ONLY headers (no body).              │
│          │ Used to check if a resource exists without downloading it.   │
└──────────┴──────────────────────────────────────────────────────────────┘

Thumb rule: Always use PATCH unless you have a specific use case for PUT.
Most real-world updates are partial (changing a name, updating a status).
```

### PATCH vs PUT - The Key Difference

```
User in database:
  { id: 1, name: "Aashish", email: "a@b.com", age: 22 }

PATCH /api/users/1
Body: { "name": "New Name" }
Result: { id: 1, name: "New Name", email: "a@b.com", age: 22 }
  → Only "name" changed. Everything else stays the same.
  → PATCH = selective update (append/modify specific fields)

PUT /api/users/1
Body: { "name": "New Name" }
Result: { id: 1, name: "New Name", email: null, age: null }
  → ENTIRE resource was replaced with what you sent
  → Fields you didn't include are gone/null
  → PUT = complete replacement
```

## Idempotent vs Non-Idempotent Methods

**Idempotent** = calling the method multiple times produces the **same result** as calling it once.

```
IDEMPOTENT METHODS:

  GET:    Fetch user 123 once → { name: "Aashish" }
          Fetch user 123 again → { name: "Aashish" }  (same result)
          No matter how many times you GET → same data, no side effects.

  PUT:    Replace user 123 with { name: "New" }
          Replace user 123 with { name: "New" } again
          → Same result. Replacing the same data with the same data = same.

  DELETE: Delete user 123 → user is gone
          Delete user 123 again → user is already gone (same end state)
          → Whether you delete once or 10 times, the result is the same:
            user 123 doesn't exist.


NON-IDEMPOTENT METHODS:

  POST:   Create a note → note #1 created
          Create a note again → note #2 created (DIFFERENT result!)
          Each POST creates a new resource. The result is different every time.
          → That's why POST is non-idempotent.
```

```
Why does idempotency matter?

Network failures:
  → Client sends a request, network drops, client doesn't get response
  → Client retries. Was the first request processed?

With idempotent method (PUT):
  → Doesn't matter. Sending the same PUT twice → same result. Safe to retry.

With non-idempotent method (POST):
  → Dangerous! If first request went through, retrying creates a DUPLICATE.
  → That's why we have idempotency keys for payment APIs (Stripe, etc.)
```

## CORS - Cross-Origin Resource Sharing

### The Problem

```
Same-Origin Policy:
  Browsers block requests from one domain to a DIFFERENT domain by default.

  Your frontend:  https://myapp.com
  Your API:       https://api.myapp.com

  These are DIFFERENT ORIGINS (different subdomains).
  Browser says: "BLOCKED. You can only fetch from myapp.com."

Why?
  → Security. Without this, any website could call your bank's API
    using your cookies and steal your money.
  → The browser protects users by restricting cross-origin calls.

CORS is the mechanism that lets servers EXPLICITLY ALLOW
specific origins to make cross-origin requests.
```

### Simple Request Flow

```
When a cross-origin request meets ALL these conditions:
  → Method is GET, POST, or HEAD
  → Only standard headers (no Authorization, no custom headers)
  → Content-Type is form-urlencoded, multipart, or text/plain

It's a SIMPLE REQUEST - browser sends it directly.

Flow:

CLIENT (https://myapp.com)                   SERVER (https://api.myapp.com)
  │                                              │
  │  GET /api/users                              │
  │  Origin: https://myapp.com    ────────────►  │
  │                                              │ Server checks: "Is myapp.com allowed?"
  │                                              │
  │  ◄──────────────────────────────────────────  │
  │  200 OK                                      │
  │  Access-Control-Allow-Origin: https://myapp.com
  │  Body: [{ ... users ... }]                   │
  │                                              │

Browser checks the response:
  → Does Access-Control-Allow-Origin header exist? YES
  → Does it match our origin (myapp.com) or "*"? YES
  → Response is allowed through ✅

If the header is MISSING or doesn't match:
  → Browser BLOCKS the response
  → JavaScript gets a CORS error
  → The response data is thrown away (even if server sent it)
```

### Pre-flight Request Flow

```
When a cross-origin request has ANY of these:
  → Method is PUT, DELETE, PATCH (not GET/POST/HEAD)
  → Has non-standard headers (Authorization, custom headers)
  → Content-Type is application/json (not simple types)

Browser sends a PRE-FLIGHT request FIRST (before the real one).

The pre-flight is an OPTIONS request asking:
  "Hey server, CAN I make this kind of request?"
```

```
STEP 1: Browser sends OPTIONS (pre-flight)

OPTIONS /api/users HTTP/1.1
Host: api.myapp.com
Origin: https://myapp.com
Access-Control-Request-Method: PUT          ← "Can I use PUT?"
Access-Control-Request-Headers: Authorization, Content-Type  ← "Can I send these?"

  → This has NO body. It's just an inquiry.


STEP 2: Server responds to pre-flight

HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://myapp.com    ← "Yes, myapp.com is allowed"
Access-Control-Allow-Methods: GET, POST, PUT, DELETE  ← "Yes, PUT is allowed"
Access-Control-Allow-Headers: Authorization, Content-Type  ← "Yes, those headers are fine"
Access-Control-Max-Age: 86400                     ← "Don't pre-flight again for 24 hours"

  → 204 = success, no body, just headers with permissions.


STEP 3: Browser checks pre-flight response

  ✅ Origin allowed? Yes (myapp.com matches)
  ✅ Method allowed? Yes (PUT is in the allowed list)
  ✅ Headers allowed? Yes (Authorization and Content-Type are approved)
  → All checks pass. Browser sends the REAL request.


STEP 4: Browser sends the original request

PUT /api/users/123 HTTP/1.1
Host: api.myapp.com
Origin: https://myapp.com
Authorization: Bearer eyJhbGci...
Content-Type: application/json

{ "name": "Updated Name" }


STEP 5: Server responds normally

HTTP/1.1 200 OK
Access-Control-Allow-Origin: https://myapp.com
Content-Type: application/json

{ "id": 123, "name": "Updated Name" }
```

### Pre-flight Trigger Conditions Summary

```
A pre-flight request is triggered when ANY of these are true:

  1. Method is NOT GET, POST, or HEAD
     → PUT, DELETE, PATCH all trigger pre-flight

  2. Request includes non-simple headers
     → Authorization header
     → Custom headers (X-Custom-Header)

  3. Content-Type is NOT one of the "simple" types
     → application/json triggers pre-flight ← most common trigger!
     → (simple types: text/plain, multipart/form-data,
        application/x-www-form-urlencoded)

Since most modern APIs use JSON (application/json),
most real-world cross-origin requests are pre-flighted.
```

### Access-Control-Max-Age

```
Without max-age:
  Every single cross-origin request sends TWO requests:
    1. OPTIONS (pre-flight)
    2. The actual request
  → Double the network traffic. Slow.

With Access-Control-Max-Age: 86400:
  First request: OPTIONS + actual (2 requests)
  Next 24 hours: Only actual request (1 request)
  → Browser caches the pre-flight result
  → Saves bandwidth for both client and server
```

## HTTP Response Status Codes

Status codes are three-digit numbers that communicate the **result of a request** in a standardized way.

```
Why status codes exist:

Without them:
  → Client has to GUESS if the request worked by inspecting the body
  → "If body has 'error', it failed?" → Inconsistent, unreliable

With them:
  → Client looks at the status code FIRST → immediately knows the outcome
  → 200? Success. 401? Need to re-login. 500? Server broke.
  → Universal standard, same across every language and framework
```

### 1xx - Informational

```
100 Continue
  → Server says: "I received your headers, go ahead and send the body"
  → Used for large uploads - client sends headers first,
    waits for server approval before sending the file

101 Switching Protocols
  → Server is switching from HTTP to a different protocol
  → Common use: upgrading from HTTP to WebSocket
```

### 2xx - Success

```
200 OK
  → The most common success code
  → Request was successful, server is returning the data
  → Used for: successful GET, PATCH, PUT requests
  → Example: GET /users → 200 + list of users

201 Created
  → A new resource was successfully created
  → Used for: successful POST requests
  → Example: POST /users → 201 + the newly created user
  → Server should return the created entity in the body

204 No Content
  → Request was successful, but there's nothing to return
  → Used for: DELETE requests, CORS pre-flight responses
  → Example: DELETE /users/123 → 204 (user deleted, nothing to show)
  → Response body is EMPTY
```

### 3xx - Redirection

```
301 Moved Permanently
  → Resource has permanently moved to a new URL
  → Future requests should use the new URL
  → Example: /users route permanently changed to /people
    → 301 redirect from /users to /people
    → Maintains backward compatibility for old clients

302 Found (Temporary Redirect)
  → Resource is temporarily at a different URL
  → Future requests should still use the original URL
  → Example: During a marketing campaign, /home temporarily
    redirects to /promo-page, but reverts after the campaign

304 Not Modified
  → Resource hasn't changed since last request
  → Client should use its cached version
  → Used with caching headers (ETag, If-None-Match)
  → Saves bandwidth - no body is sent
```

### 4xx - Client Errors

```
400 Bad Request
  → Client sent invalid or malformed data
  → Example: Expected a number, received a string
  → Example: Required fields missing in the body
  → Tells client: "Fix your request and try again"

401 Unauthorized
  → Client is NOT authenticated (no token, or token expired/invalid)
  → Example: Request without Authorization header to a protected endpoint
  → Example: JWT token has expired
  → Tells client: "You need to log in (again)"

403 Forbidden
  → Client IS authenticated but does NOT have permission
  → Example: Regular user trying to delete admin's resource
  → Tells client: "I know who you are, but you can't do this"

  Key difference:
    401 = "WHO are you?" (identity unknown)
    403 = "I know who you are, but NO" (identity known, permission denied)

404 Not Found
  → The requested resource doesn't exist
  → Example: GET /users/999 when user 999 doesn't exist
  → Example: Typo in URL - /api/usr instead of /api/users

405 Method Not Allowed
  → Correct URL, wrong HTTP method
  → Example: Sending PUT to an endpoint that only accepts GET
  → Usually a typo or client-side bug

409 Conflict
  → Request conflicts with current state of the resource
  → Example: Creating a folder with a name that already exists
  → Example: Two users trying to update the same resource simultaneously
  → Tells client: "This conflicts with existing data, try something different"

429 Too Many Requests
  → Client has exceeded the rate limit
  → Example: Server allows 60 requests/minute, client sent 61
  → Used for rate limiting to protect server from abuse
  → Response often includes Retry-After header
```

### 5xx - Server Errors

```
500 Internal Server Error
  → Something unexpected broke on the server
  → Unhandled exception, bug in code, unexpected null pointer
  → Generic "something went wrong" error
  → NEVER expose internal details to the client (security risk)
  → Log the full error internally, send generic message to client

501 Not Implemented
  → Server doesn't support this functionality YET
  → Example: Endpoint is planned but not built yet
  → Tells client: "This feature is coming, but not ready"

502 Bad Gateway
  → Server (acting as proxy) received an invalid response from upstream
  → Common with Nginx: the backend app crashed or returned garbage
  → Nginx → your app → app crashes → Nginx returns 502

503 Service Unavailable
  → Server is temporarily unable to handle requests
  → Example: During maintenance, under heavy load, or restarting
  → Tells client: "Try again later"

504 Gateway Timeout
  → Server (acting as proxy) didn't get a response in time
  → Common with Nginx: the backend app is too slow
  → Nginx waits 60 seconds → no response → returns 504
```

## HTTP Caching

Caching stores copies of responses for reuse, reducing repeated requests to the server. This improves load time, reduces bandwidth, and decreases server load.

### How Caching Works - ETag and Last-Modified

```
FIRST REQUEST (no cache exists yet):

Client: GET /api/resource
Server responds:
  200 OK
  Cache-Control: max-age=10        ← "Cache this for 10 seconds"
  ETag: "hash-3141"                ← Unique hash of this response body
  Last-Modified: Tue, 15 Mar 2024  ← When this resource was last changed
  Body: { ... actual data ... }

Client stores:
  → The response body (cached data)
  → The ETag ("hash-3141")
  → The Last-Modified date
```

```
SECOND REQUEST (within cache window):

If within max-age (10 seconds):
  → Browser uses cached version directly
  → No request sent to server at all

If max-age expired:
  Client sends: GET /api/resource
  If-None-Match: "hash-3141"          ← "Is my cached ETag still valid?"
  If-Modified-Since: Tue, 15 Mar 2024 ← "Has it changed since this date?"

Server checks:
  → Is the current ETag still "hash-3141"? YES
  → Has it been modified since Mar 15? NO

Server responds:
  304 Not Modified                    ← "Your cache is still good"
  (NO body sent - saves bandwidth!)

Client:
  → Uses its local cached version ✅
  → No large response downloaded
```

```
THIRD REQUEST (after resource was updated):

Someone updated the resource. ETag changed to "hash-2943".

Client sends: GET /api/resource
If-None-Match: "hash-3141"          ← Client's old ETag

Server checks:
  → Is the current ETag "hash-3141"? NO, it's "hash-2943" now
  → Resource HAS been modified

Server responds:
  200 OK                             ← "Here's the NEW version"
  ETag: "hash-2943"                  ← New ETag
  Last-Modified: Wed, 16 Mar 2024
  Body: { ... updated data ... }

Client:
  → Stores new response, new ETag, new Last-Modified
  → Uses fresh data ✅
```

### Caching Flow Diagram

```
Client Request
     │
     ├── Cache exists and max-age NOT expired?
     │     └── YES → Use cached version. No server request. ✅
     │
     └── Cache expired or no cache?
           └── Send request with If-None-Match (ETag) header
                 │
                 ├── Server: ETag matches → 304 Not Modified
                 │     └── Use cached version. No body downloaded. ✅
                 │
                 └── Server: ETag doesn't match → 200 OK + new data
                       └── Update cache with new data and ETag. ✅
```

> **Modern note:** Client-side caching libraries (like React Query, SWR) often handle caching more elegantly than HTTP-based caching. They give the client full control over when to refetch, stale times, and cache invalidation. But HTTP caching is still valuable for simple use cases and CDN caching.

## Content Negotiation

Content negotiation is the mechanism by which client and server **agree on the best format** to exchange data.

### Three Types of Negotiation

```
1. MEDIA TYPE NEGOTIATION (Accept header)
   Client: Accept: application/json    → "I want JSON"
   Server: Content-Type: application/json + JSON body

   Client: Accept: application/xml     → "I want XML"
   Server: Content-Type: application/xml + XML body

   Client: Accept: text/html           → "I want HTML"
   Server: Content-Type: text/html + HTML page


2. LANGUAGE NEGOTIATION (Accept-Language header)
   Client: Accept-Language: en         → "I want English"
   Server: { "message": "Hello, welcome!" }

   Client: Accept-Language: es         → "I want Spanish"
   Server: { "message": "¡Hola, bienvenido!" }

   Client: Accept-Language: hi         → "I want Hindi"
   Server: { "message": "नमस्ते, स्वागत है!" }


3. ENCODING NEGOTIATION (Accept-Encoding header)
   Client: Accept-Encoding: gzip, deflate, br
     → "I support these compression formats"
   Server: Content-Encoding: gzip
     → "I've compressed the response with gzip"
```

### How It Works in Practice

```
Client sends:
  GET /api/data
  Accept: application/json        ← preferred format
  Accept-Language: es              ← preferred language
  Accept-Encoding: gzip           ← supported compression

Server reads these headers and:
  → Serializes response as JSON (because Accept says JSON)
  → Returns content in Spanish (because Accept-Language says es)
  → Compresses with gzip (because Accept-Encoding includes gzip)

Response:
  200 OK
  Content-Type: application/json
  Content-Language: es
  Content-Encoding: gzip

  { "mensaje": "¡Hola desde el servidor!" }
```

## HTTP Compression

Compression reduces the size of HTTP responses, saving bandwidth and improving load times.

```
WITHOUT compression:
  Original JSON response: 26 MB (11,000 entries)
  Every client downloads: 26 MB

WITH gzip compression:
  Compressed response: 3.8 MB
  Every client downloads: 3.8 MB
  Savings: ~85% bandwidth reduction!

How it works:
  1. Client sends: Accept-Encoding: gzip, deflate, br
     → "I can handle these compression formats"

  2. Server compresses the response body with gzip

  3. Server sends:
     Content-Encoding: gzip
     → "This body is gzip compressed"
     + compressed body bytes

  4. Browser automatically decompresses
     → Client gets the original full response
     → Decompression is automatic, no code needed

The client and server both win:
  → Client downloads less data (faster load)
  → Server sends less data (lower bandwidth cost)
  → Browser handles decompression automatically
```

## Persistent Connections and Keep-Alive

### The Evolution of Connection Handling

```
HTTP/1.0 (OLD - each request = new connection):

  Request 1: Open TCP → Send request → Get response → Close TCP
  Request 2: Open TCP → Send request → Get response → Close TCP
  Request 3: Open TCP → Send request → Get response → Close TCP

  Problem: TCP handshake (3 messages) happens for EVERY request.
  A page with 20 resources = 20 TCP handshakes = very slow.


HTTP/1.1 (CURRENT - persistent connections):

  Open TCP connection ONCE
    → Send request 1 → Get response 1
    → Send request 2 → Get response 2
    → Send request 3 → Get response 3
    → ... many more requests ...
  Close TCP connection when done

  ONE handshake, many requests. Much faster.
```

### The Keep-Alive Header

```
Connection: keep-alive
  → "Keep this TCP connection open for more requests"
  → DEFAULT in HTTP/1.1 (you don't need to set this explicitly)

Connection: close
  → "Close this TCP connection after the response"
  → Used when you explicitly want to close
  → Was the DEFAULT in HTTP/1.0

Keep-Alive options:
  Keep-Alive: timeout=5, max=100
  → timeout=5: Close connection after 5 seconds of inactivity
  → max=100: Allow max 100 requests before closing

In practice:
  → HTTP/1.1 handles this automatically
  → You rarely need to configure keep-alive manually
  → The default values work fine for most applications
```

## Handling Large Data - Multipart and Chunked Transfer

### Multipart Requests (Client → Server: File Uploads)

When clients need to send **large files** (images, videos, documents), they use **multipart/form-data**:

```
Normal JSON request:
  Content-Type: application/json
  Body: { "name": "Aashish" }    ← just text

Multipart request (file upload):
  Content-Type: multipart/form-data; boundary=----abc123
  Body:
    ------abc123
    Content-Disposition: form-data; name="file"; filename="photo.jpg"
    Content-Type: image/jpeg
    
    [binary data of the image... thousands of bytes]
    ------abc123--

What's happening:
  → The file's binary data is sent in PARTS
  → Each part is separated by a BOUNDARY string (------abc123)
  → The boundary acts as a delimiter between parts
  → Server reads each part, reassembles the file
  → Multiple files can be sent in one request (each between boundaries)
```

### Chunked Transfer / Server-Sent Events (Server → Client: Streaming)

When servers need to send **large responses**, they can stream data in **chunks**:

```
Normal response:
  Server waits until ALL data is ready
  Server sends entire response at once
  Client gets everything or nothing

Chunked/streamed response:
  Server sends data in small pieces as they become available
  Client receives and processes each chunk immediately
  Connection stays open until all chunks are sent

Headers for streamed response:
  Content-Type: text/event-stream    ← "I'll stream data to you"
  Connection: keep-alive             ← "Keep the connection open"

Use cases:
  → Streaming a large file (server sends chunks as it reads from disk)
  → Real-time updates (server pushes events as they happen)
  → ChatGPT-style responses (words appear one at a time)
  → Live logs (tailing a log file from the browser)
```

## SSL, TLS, and HTTPS

```
SSL (Secure Sockets Layer):
  → The ORIGINAL encryption protocol for HTTP
  → Encrypted data between client and server
  → NOW OUTDATED - has known security vulnerabilities
  → No longer used in modern systems

TLS (Transport Layer Security):
  → The MODERN replacement for SSL
  → Encrypts data in transit between client and server
  → Uses certificates to authenticate the server
  → Prevents eavesdropping (no one can read the data in transit)
  → Prevents tampering (no one can modify the data in transit)
  → Current recommended version: TLS 1.3

HTTPS (HTTP Secure):
  → Simply HTTP + TLS encryption
  → When you visit https://... → your browser uses TLS under the hood
  → All data between browser and server is encrypted
  → Without HTTPS: passwords, credit cards sent in plain text (anyone on
    the network can read them)
  → With HTTPS: everything is encrypted (only client and server can decrypt)

In practice:
  → You set up TLS certificates on your server (or reverse proxy like Nginx)
  → Tools like Certbot automate certificate generation (Let's Encrypt)
  → Your application code doesn't change - encryption is handled by the
    transport layer underneath
  → HTTPS is now expected everywhere - browsers mark HTTP sites as "Not Secure"
```
