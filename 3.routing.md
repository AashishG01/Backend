Routing -> It is the process of defining how an application responds to a client request to a particular endpoint, which is a URI (or path) and a specific HTTP method (GET, POST, etc.). Each route can have one or more handler functions, which are executed when the route is matched. Routing allows developers to create dynamic web applications by defining different endpoints for different functionalities.

Static Routes 
example 
GET /api/books
POST /api/books

This are called static routes because the path is fixed and does not change. No variable part is there and it is a fixed string.
static routes are straightforward to implement and are used when the resource being accessed is known and does not require any dynamic parameters.

Dynamic Routes
example
GET /api/users/123
here:
/api/users -> static part 
123 -> dynamic part (user id)

in server code it usually looks like this
GET /api/users/:id

This :id means that it is a placeholder for any value that can be passed in that position. When a request is made to this route, the server will extract the value of :id and make it available to the handler function. 
Dynamic routes are useful when you want to access resources that are identified by a unique identifier, such as a user ID, product ID, etc. They allow you to create more flexible and reusable routes that can handle a wide range of requests without having to define a separate route for each possible value.

id = "123" this is called path parameter/route parameter

Path parameters vs Query parameters

Path parameters
structure: /api/users/:id
example: /api/users/123
used for : identifying a specific resource
semantics: /users/123 means fetch the user with id 123

Qury parameters
structure: /search?query=hello

general format : ?key=value
multiple : /search?query=hello&sort=asc
used for : filtering, sorting, pagination, etc.
semantics: /search?query=hello means search for items that match the query "hello"

Why not use path for everything? 
Using path parameters for everything can lead to a cluttered and less intuitive URL structure. Path parameters are best suited for identifying specific resources, while query parameters are more appropriate for filtering, sorting, and other operations that do not necessarily identify a single resource.
For example, if you have a route like /search/hello, it may not be clear whether "hello" is a search term or an identifier for a specific resource. Using query parameters allows for a clearer distinction between the two and makes the URL more intuitive for users.

Why GET Uses Query Params
GET requests:
Should not have body (REST principle)
So how do we send extra data?
Answer:
→ Query parameters.
Example:
GET /books?page=2&limit=10
Server understands:
page = 2
limit = 10
This allows clients to specify additional information about the request without violating the REST principle of not having a body in GET requests. Query parameters are a convenient way to pass data that is not part of the resource identifier, such as pagination, filtering, or sorting options.

Nested Routes(Resource Hierarchy)
This is about expressing relationships.

Example:
GET /api/users/123/posts

Meaning:
Fetch all posts of user 123.

Even deeper:
GET /api/users/123/posts/456

Meaning:
Fetch post 456 that belongs to user 123.

Structure:
/users/:userId/posts/:postId
This is called nested routing.

Why Nested?

Because REST APIs try to represent:
Data hierarchy using URL structure.

Hierarchy example:

Users -> Posts -> Comments
So route mirrors database relationship.

Route Versioning 
As APIs evolve, we may need to make changes that are not backward compatible. To manage this, we can use route versioning to maintain multiple versions of the API simultaneously.
Example:
GET /api/v1/books
GET /api/v2/books
In this example, we have two versions of the books endpoint. The v1 version may return a different response format or include different fields compared to the v2 version. This allows clients to choose which version of the API they want to use and ensures that existing clients are not broken when new changes are introduced. Route versioning is an important aspect of API design that helps maintain stability and compatibility as the API evolves over time.
