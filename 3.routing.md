# Routing - Complete Guide

> **Core idea:** Routing is the bridge between *a request arriving* and *business logic running*. It is the process by which the server looks at an incoming HTTP request (method + URL) and decides which function should handle it. Without routing, a server cannot distinguish between "give me all books" and "delete this user".

## What Are HTTP Methods?

Before understanding routing, you need to understand **HTTP methods** - because routing is built on top of them.

An HTTP method tells the server **WHAT action** the client wants to perform on a resource. The server uses both the method AND the URL path together to route the request to the correct handler.

```
┌──────────┬──────────────────────────────────────────────────┐
│ Method   │ Purpose / Semantic Meaning                       │
├──────────┼──────────────────────────────────────────────────┤
│ GET      │ Read/fetch a resource. No side effects.          │
│ POST     │ Create a new resource.                           │
│ PUT      │ Replace an entire resource.                      │
│ PATCH    │ Update specific fields of a resource.            │
│ DELETE   │ Remove a resource.                               │
│ OPTIONS  │ Ask the server what methods are allowed.         │
│ HEAD     │ Like GET, but only returns headers (no body).    │
└──────────┴──────────────────────────────────────────────────┘
```

HTTP methods express the **intention** of the request:

```
GET    → "I want to READ something"
POST   → "I want to CREATE something"
PATCH  → "I want to UPDATE something"
DELETE → "I want to REMOVE something"
```

> **Real-world analogy:**
> Think of a library. The URL is the address of the shelf (`/books/harry-potter`).
> The HTTP method is your intent at that shelf:
> - **GET** = "I want to READ this book"
> - **POST** = "I want to ADD a new book here"
> - **PATCH** = "I want to EDIT the description of this book"
> - **DELETE** = "I want to REMOVE this book from the shelf"

## What Is Routing?

**Routing** is the mechanism by which a server maps an incoming request - a combination of **HTTP method + URL path** - to a specific handler function that processes that request.

```
Short definition:
  Routing = mapping (method + path) → handler function

Longer definition:
  Routing is the process of defining how an application responds
  to a client request to a particular endpoint (URI/path) and
  HTTP method. Each route has one or more handler functions that
  are executed when the route is matched.
```

### The Bridge Analogy

```
REQUEST ARRIVES AT SERVER
        │
        ▼
┌───────────────────┐
│    ROUTING LAYER  │  ← "Which function handles this?"
└───────────────────┘
        │
        ▼
CORRECT HANDLER RUNS
(business logic, DB queries, etc.)
        │
        ▼
RESPONSE SENT BACK TO CLIENT
```

Without routing, the server has no way to distinguish between:
- "List all books" → GET /books
- "Create a book" → POST /books
- "Delete a book" → DELETE /books/123

They all go to the same server. Routing decides which function handles which request.

## Routing = WHAT + WHERE

Every route is made up of exactly two parts:

```
HTTP Method  →  expresses the WHAT (the action to be performed)
URL Path     →  expresses the WHERE (the resource to act upon)

Together → METHOD + PATH = a UNIQUE KEY in the routing table
```

### Example

```
GET   /api/books   →  WHERE: books resource,  WHAT: fetch them
POST  /api/books   →  WHERE: books resource,  WHAT: create one
PATCH /api/books/5 →  WHERE: book with id 5,  WHAT: update it
```

Same URL path, different method = **completely different behavior**:

```
GET  /api/books  →  returns a list of all books
POST /api/books  →  creates a new book and returns it

Same path "/api/books"
Different HTTP methods
Different handler functions
Different behavior
Different response
```

This is why you need **both pieces** together. The path alone doesn't tell you what to do. The method alone doesn't tell you where.

> **Real-world analogy:**
> At a post office:
> - **WHERE**: Counter 3 (the URL)
> - **WHAT**: Send a package / Track a package / Pick up a package (the method)
>
> The counter number alone doesn't tell you what you're doing. Your intent (what you want) combined with the location (where you go) together forms the unique action.

## How Routing Works Internally

Routing is conceptually a series of `if` checks. Let's strip away all the framework magic and see what's actually happening:

```javascript
// Pseudo-code - what a router actually does internally

function route(method, path) {
  
  if (method === "GET" && path === "/api/books") {
    return getBooksHandler()          // → returns all books
  }
  
  if (method === "POST" && path === "/api/books") {
    return createBookHandler()        // → creates a new book
  }
  
  if (method === "GET" && path.match("/api/books/:id")) {
    const id = extractId(path)        // extracts "123" from "/api/books/123"
    return getBookByIdHandler(id)     // → returns one book
  }
  
  if (method === "PATCH" && path.match("/api/books/:id")) {
    const id = extractId(path)
    return updateBookHandler(id)      // → updates the book
  }
  
  if (method === "DELETE" && path.match("/api/books/:id")) {
    const id = extractId(path)
    return deleteBookHandler(id)      // → deletes the book
  }
  
  // No match found
  return send404("Route not found")
}
```

Modern frameworks (Express, FastAPI, Gin, Spring) just do this same logic more elegantly under the hood.

### The Routing Algorithm Steps

```
When a request arrives:

Step 1: Read the METHOD from the request (GET, POST, etc.)
Step 2: Read the PATH from the URL (/api/books/123)
Step 3: Try to match METHOD + PATH against each registered route pattern
Step 4: First match wins → extract parameters (path params, query params)
Step 5: Call the associated handler function
Step 6: If no match → call 404 handler
```

## Static Routes

A **static route** is one where the path is completely fixed. No variables, no placeholders - the path is always exactly the same string.

```
GET  /api/books        ← Static
POST /api/books        ← Static
GET  /api/health       ← Static
GET  /api/users        ← Static
POST /api/auth/login   ← Static
POST /api/auth/logout  ← Static
```

The path never changes. Every client calling `GET /api/books` always hits exactly the same route.

```javascript
// Express.js example of static routes
app.get("/api/books", getBooksHandler)
app.post("/api/books", createBookHandler)
app.get("/api/health", healthCheckHandler)
app.post("/api/auth/login", loginHandler)
```

### When to Use Static Routes

```
Static routes are best for:
  ✅ Endpoints that operate on a whole collection (list all, create new)
  ✅ Actions that don't target a specific item
  ✅ Utility endpoints (health check, status page)
  ✅ Auth endpoints (login, logout, register)

Examples:
  GET  /api/books         → List all books
  POST /api/books         → Create a new book
  GET  /api/stats         → Server statistics
  POST /api/auth/login    → Login
```

## Dynamic Routes and Path Parameters

A **dynamic route** is one where part of the URL path is a **variable** - it changes for different requests. These variables are called **path parameters** (also called route parameters).

### The Problem Static Routes Solve

```
Imagine you want to fetch one specific book.

One approach - make a static route for each book:
  GET /api/books/harry-potter
  GET /api/books/dune
  GET /api/books/foundation
  ... (10,000 more routes for 10,000 books)

This is obviously impossible. You can't define a new route
for every book that ever gets added to your database.

Solution: Dynamic routes.
  GET /api/books/:id

The :id is a placeholder. It matches ANY value at that position.
```

### How Dynamic Routes Work

```
Route definition:   GET /api/books/:id

Client sends:       GET /api/books/123
Router sees:        :id = "123"   ← extracted!

Client sends:       GET /api/books/456
Router sees:        :id = "456"   ← extracted!

Client sends:       GET /api/books/harry-potter-slug
Router sees:        :id = "harry-potter-slug"   ← extracted!

Same route pattern. Different actual values every time.
The router extracts the value and makes it available to the handler.
```

### In Code

```javascript
// Express.js
app.get("/api/books/:id", (req, res) => {
  const bookId = req.params.id  // ← the extracted value
  // fetch book from database using bookId
  // return the book
})

// Multiple path parameters
app.get("/api/users/:userId/posts/:postId", (req, res) => {
  const userId = req.params.userId   // e.g. "123"
  const postId = req.params.postId   // e.g. "456"
})
```

### Path Parameters Express Identity

The semantic meaning of a path parameter is **identity** - it identifies a specific resource:

```
GET /users/123        → "Give me the user whose identity is 123"
GET /books/dune       → "Give me the book whose identity is 'dune'"
GET /orders/order-99  → "Give me the order whose identity is order-99"

Path params say: "THIS specific one, not just any"
```

> **Real-world analogy:**
> A path parameter is like a seat number on a train ticket:
> - `/trains/express-delhi` = the Express Delhi train (static, identifies the train)
> - `/trains/express-delhi/coach/A/seat/15` = Coach A, seat 15 (dynamic, identifies your specific seat)
>
> You can't book a train without referring to the specific coach and seat number.

### Common Naming Conventions

```
:id        → generic numeric or UUID identifier
:userId    → user's ID
:bookId    → book's ID
:postId    → post's ID
:slug      → human-readable identifier (e.g. "harry-potter")
:username  → user's unique username

Examples:
  /api/users/:userId
  /api/products/:productId
  /api/articles/:slug
  /api/organizations/:orgId/projects/:projectId
```

## Query Parameters

**Query parameters** are key-value pairs appended to the URL after a `?` symbol. They are used to pass optional or supplementary data to the server - especially for filtering, sorting, searching, and pagination.

### Anatomy of Query Parameters

```
URL with query parameters:

  /api/books?sort=title&order=asc&page=2&limit=10
            ▲
            │ the "?" separates the path from query params
  
  Breaking it down:
  /api/books          → the path (resource)
  ?                   → query string starts
  sort=title          → key "sort", value "title"
  &                   → separator between pairs
  order=asc           → key "order", value "asc"
  &
  page=2              → key "page", value "2"
  &
  limit=10            → key "limit", value "10"
```

### How to Access Query Params in Code

```javascript
// Express.js
app.get("/api/books", (req, res) => {
  const sort  = req.query.sort    // "title"
  const order = req.query.order   // "asc"
  const page  = req.query.page    // "2" (always string from URL)
  const limit = req.query.limit   // "10"
  
  // Convert numbers (query params are always strings)
  const pageNum  = parseInt(page)  || 1
  const limitNum = parseInt(limit) || 10
  
  // Build database query using these params
})
```

### Query Param Examples

```
Filtering:
  GET /api/books?status=published
  GET /api/users?role=admin
  GET /api/orders?status=pending&city=mumbai

Sorting:
  GET /api/books?sort_by=title&sort_order=asc
  GET /api/products?sort_by=price&sort_order=desc

Pagination:
  GET /api/books?page=1&limit=20  → first 20 books
  GET /api/books?page=2&limit=20  → next 20 books

Searching:
  GET /api/books?search=harry+potter
  GET /api/users?search=alice

Combined (real-world style):
  GET /api/products?category=electronics&sort_by=price&sort_order=asc&page=1&limit=20
  → electronics, sorted by price cheapest first, first 20 results
```

> **Real-world example - Google Search:**
> When you search on Google:
> `https://www.google.com/search?q=routing+in+express&hl=en&num=10`
>
> - `q=routing+in+express` → search query
> - `hl=en` → language preference
> - `num=10` → number of results per page
>
> All of these are optional filters/preferences. The route `/search` is static. The query params carry the search details.

## Path Parameters vs Query Parameters

This is one of the most commonly confused topics. Here is the definitive breakdown:

### Comparison Table

```
┌──────────────────────┬─────────────────────────┬────────────────────────────────────┐
│ Feature              │ Path Parameter           │ Query Parameter                    │
├──────────────────────┼─────────────────────────┼────────────────────────────────────┤
│ Location in URL      │ Embedded in the path     │ After the ? symbol                 │
│                      │ /users/123               │ /users?role=admin                  │
├──────────────────────┼─────────────────────────┼────────────────────────────────────┤
│ Format               │ /resource/:param         │ ?key=value&key2=value2             │
├──────────────────────┼─────────────────────────┼────────────────────────────────────┤
│ Purpose              │ Identify a specific      │ Filter, sort, search, paginate     │
│                      │ resource / express       │ a collection. Modify the result,   │
│                      │ identity                 │ not the resource being accessed.   │
├──────────────────────┼─────────────────────────┼────────────────────────────────────┤
│ Required?            │ Usually required         │ Almost always optional             │
├──────────────────────┼─────────────────────────┼────────────────────────────────────┤
│ Semantic meaning     │ "This specific item"     │ "Give me these items, filtered     │
│                      │                          │  and sorted like this"             │
├──────────────────────┼─────────────────────────┼────────────────────────────────────┤
│ Example              │ GET /users/123           │ GET /users?role=admin&page=2       │
│                      │ → User #123 specifically │ → All admin users, page 2          │
└──────────────────────┴─────────────────────────┴────────────────────────────────────┘
```

### Decision Rule - Which One to Use?

```
Ask yourself: "Am I identifying ONE specific resource?"

YES → Use path parameter
  GET /users/123            → user 123 specifically
  GET /books/dune           → the book "Dune" specifically
  DELETE /orders/order-99   → delete that specific order

NO → Use query parameter
  GET /users?role=admin     → not one user, filtering a list
  GET /books?genre=sci-fi   → not one book, filtering a collection
  GET /orders?status=pending → all pending orders
```

### Why NOT Use Path for Everything?

```
❌ BAD - using path params for filtering:
  /books/sci-fi/published/asc/page/2/limit/20

Problems:
  → Ugly, unreadable URL
  → Every filter requires a new route definition
  → Optional filters are impossible (you always need all segments)
  → Adding a new filter breaks every existing URL

✅ GOOD - using query params for filtering:
  /books?genre=sci-fi&status=published&sort_order=asc&page=2&limit=20

Benefits:
  → Clean, readable URL
  → Any filter can be optional (just omit it)
  → Adding a new query param never breaks existing URLs
  → Industry standard, immediately understood by any developer
```

### Why NOT Use Query Params for Identity?

```
❌ BAD - using query params to identify resources:
  GET /books?id=123
  DELETE /books?id=123
  PATCH /books?id=123

Problems:
  → Semantically wrong - identity should be in the path
  → Non-standard - breaks REST conventions
  → Ambiguous - is "id" a filter or an identifier?
  → Breaks caching (browsers don't cache query-param URLs well)

✅ GOOD - using path params to identify resources:
  GET /books/123
  DELETE /books/123
  PATCH /books/123
```

## Why GET Uses Query Params (Not Body)

**GET requests must not have a request body.** This is a fundamental REST (and HTTP spec) principle. So how does a GET request pass data (like filters, page number, search term) to the server?

**Answer: Query parameters.**

```
GET request = should be a "pure read" with no side effects.
  → No body (REST principle)
  → Body would make GET behave like POST (mutation risk)
  → Many servers, proxies, caches IGNORE the body on GET requests

So to pass data to a GET endpoint:
  → All data goes in the URL query string
  → ?key=value&key2=value2

Example:
  GET /api/books?page=2&limit=10&sort_by=title&search=harry
  
  Server reads:
    page    = 2
    limit   = 10
    sort_by = "title"
    search  = "harry"
  
  Everything is in the URL. No body needed.
```

> **Why does this make sense?**
> GET requests are **safe and idempotent** - they read data, they don't change anything. By putting all parameters in the URL, the entire request is self-contained and shareable. You can copy-paste the URL and the other person makes the exact same query. Try doing that with a request body!

```
Real-world example - Amazon product search:
https://www.amazon.in/s?k=mechanical+keyboard&rh=p_36%3A500-3000&s=price-asc-rank&page=2

Breaking it down:
  k=mechanical+keyboard   → search keyword
  rh=...                  → price range filter (₹500-₹3000)
  s=price-asc-rank        → sort by price ascending
  page=2                  → page 2

This URL is SHAREABLE. Send it to a friend → they see the same results.
```

## Nested Routes - Resource Hierarchy

**Nested routes** express **relationships between resources**. When one resource belongs to another, the URL structure reflects that parent-child relationship.

### The Concept

```
Data has hierarchy:
  User → has many → Posts → have many → Comments

URL mirrors that hierarchy:
  /users/:userId                           → the user
  /users/:userId/posts                     → all posts of that user
  /users/:userId/posts/:postId             → one post of that user
  /users/:userId/posts/:postId/comments    → comments on that post
```

### Examples

```
Organization → Projects → Tasks:

  GET /organizations                              → All organizations
  GET /organizations/:orgId                       → One organization
  GET /organizations/:orgId/projects              → All projects of an org
  GET /organizations/:orgId/projects/:projectId   → One project
  GET /organizations/:orgId/projects/:projectId/tasks          → All tasks
  GET /organizations/:orgId/projects/:projectId/tasks/:taskId  → One task

GitHub-style URLs:
  GET /repos/torvalds/linux              → linux repo by torvalds
  GET /repos/torvalds/linux/commits      → commits in that repo
  GET /repos/torvalds/linux/issues/123   → issue #123 in that repo

Blog URLs:
  GET /users/alice/posts              → all of Alice's posts
  GET /users/alice/posts/hello-world  → her specific post
  GET /users/alice/posts/hello-world/comments   → comments on it
```

### Why Nested Routes?

```
They communicate OWNERSHIP and CONTEXT:

  GET /posts               → give me ALL posts from ALL users
  GET /users/123/posts     → give me ONLY posts belonging to USER 123

The nested route adds CONTEXT - it scopes the resource.
The second query is faster (smaller result set)
and clearer in intent (I want THIS user's posts, not everyone's).
```

### How Deep Should Nesting Go?

```
Rule of thumb: Maximum 2-3 levels of nesting for readability.

✅ 1 level:   /organizations/:orgId/projects
✅ 2 levels:  /organizations/:orgId/projects/:projectId/tasks
⚠️  3 levels:  /organizations/:orgId/projects/:projectId/tasks/:taskId/comments
❌ 4+ levels: /a/:aId/b/:bId/c/:cId/d/:dId - confusing and hard to maintain

For deep nesting, flatten with query params:
  /comments?taskId=task-123   (instead of /orgs/:id/projects/:id/tasks/:id/comments)
```

## Route Versioning

As your backend evolves over time, you will sometimes need to make **breaking changes** - changes that would break existing clients if applied to the same endpoint. **Route versioning** solves this by running multiple versions of the API simultaneously.

### Why APIs Break

```
Scenario:
  You have: GET /api/products
  Current response:
    { "id": 1, "name": "Laptop", "price": 999 }

  Your frontend and 50 third-party apps depend on the "name" field.

  New requirement: rename "name" to "title" for consistency.
  New response:    { "id": 1, "title": "Laptop", "price": 999 }

  If you change the existing /api/products:
    → Every frontend component reading ".name" breaks → shows undefined
    → All 50 third-party integrations break instantly
    → Emergency calls, rollbacks, angry clients

  This is called a BREAKING CHANGE.
```

### How Versioning Solves It

```
Instead of modifying the existing route, you create a NEW version:

  /api/v1/products → old format: { id, name, price }    ← still works!
  /api/v2/products → new format: { id, title, price }   ← new format!

Now:
  Old clients continue using v1 (nothing breaks)
  New clients / updated apps use v2
  Both versions coexist peacefully
```

### URL Versioning Format

```
Most common approach - version in the URL path:

  /api/v1/books
  /api/v2/books
  /api/v3/books

Real-world examples:
  Stripe:  https://api.stripe.com/v1/charges
  GitHub:  https://api.github.com/v3/repos
  Twitter: https://api.twitter.com/2/tweets
  Spotify: https://api.spotify.com/v1/me/tracks
```

### The Full Versioning Lifecycle

```
Phase 1 - Initial API
  /api/v1/products    → live, used by everyone

Phase 2 - Breaking change needed
  /api/v1/products    → still live (unchanged)
  /api/v2/products    → new version with breaking change

Phase 3 - Deprecation
  /api/v1/products    → still works, but sends deprecation warning:
                         Response header: Deprecation: true
                         Response header: Sunset: 2025-06-01
                         "This endpoint will be removed on June 1, 2025.
                          Please migrate to /api/v2/products"

Phase 4 - Migration window
  Clients have time to update their code to use v2.
  Frontend team updates the app.
  Third-party partners update their integrations.

Phase 5 - Shutdown
  /api/v1/products    → REMOVED. Returns 410 Gone.
  /api/v2/products    → the new standard
```

### Why Versioning Is powerful

```
✅ Allows breaking changes without breaking existing clients
✅ Gives clients a migration window (don't have to update immediately)
✅ Maintains backward compatibility (old code keeps working)
✅ Enables cleaner API evolution over time
✅ Professional, industry-standard practice
✅ Service Level Agreement (SLA) compliance

Real-world: Stripe announces version changes 6-12 months in advance.
They never silently break existing integrations.
This is why developers trust Stripe's API.
```

## Catch-All Routes (404 Handling)

A **catch-all route** is a special route that matches any request that didn't match any other defined route. It is your safety net.

### The Problem Without a Catch-All

```
Your server has these routes:
  GET  /api/books
  POST /api/books
  GET  /api/books/:id

Client sends: GET /api/v3/products  (a route you haven't defined)

Without catch-all:
  → Server might crash (unhandled request)
  → Server might return an empty body with no status code
  → Client is confused - no useful error message
  → Debugging is hard
```

### The Catch-All Solution

```javascript
// Express.js - register this LAST after all other routes

// All specific routes first
app.get("/api/books", getBooksHandler)
app.post("/api/books", createBookHandler)
app.get("/api/books/:id", getBookByIdHandler)

// Catch-ALL - matches anything that didn't match above
app.use("*", (req, res) => {
  res.status(404).json({
    error: {
      code: "ROUTE_NOT_FOUND",
      message: `Cannot ${req.method} ${req.path}`
    }
  })
})
```

```
Client sends: GET /api/v3/products

Server checks all routes:
  GET /api/books?  → No
  POST /api/books? → No
  GET /api/books/:id? → No (pattern doesn't match)
  Catch-all *?     → YES ✅

Response:
  {
    "error": {
      "code": "ROUTE_NOT_FOUND",
      "message": "Cannot GET /api/v3/products"
    }
  }
  Status: 404 Not Found
```

### Order Matters - Catch-All Must Be Last

```
❌ WRONG - catch-all first:
  app.use("*", notFoundHandler)   ← catches EVERYTHING!
  app.get("/api/books", handler)  ← never reached!

✅ CORRECT - catch-all last:
  app.get("/api/books", handler)  ← checked first
  app.post("/api/books", handler) ← checked second
  app.use("*", notFoundHandler)   ← only runs if nothing matched
```

## The Routing Table - Mental Model

Think of a server's routing system as a **lookup table**. When a request arrives, the server checks each entry from top to bottom until a match is found.

```
Routing Table:

┌─────────────────────────────────┬──────────────────────────┐
│ METHOD + PATH PATTERN           │ HANDLER FUNCTION         │
├─────────────────────────────────┼──────────────────────────┤
│ GET    /api/books               │ getBooksHandler          │
│ POST   /api/books               │ createBookHandler        │
│ GET    /api/books/:id           │ getBookByIdHandler       │
│ PATCH  /api/books/:id           │ updateBookHandler        │
│ DELETE /api/books/:id           │ deleteBookHandler        │
│ GET    /api/users               │ getUsersHandler          │
│ POST   /api/users               │ createUserHandler        │
│ GET    /api/users/:id           │ getUserByIdHandler       │
│ GET    /api/users/:id/posts     │ getUserPostsHandler      │
│ *      *                        │ notFoundHandler (404)    │
└─────────────────────────────────┴──────────────────────────┘

Incoming request: GET /api/users/123/posts

Router checks:
  GET /api/books           → no
  POST /api/books          → no (wrong method)
  GET /api/books/:id       → no (path structure doesn't match)
  PATCH /api/books/:id     → no
  DELETE /api/books/:id    → no
  GET /api/users           → no (path structure doesn't match)
  POST /api/users          → no
  GET /api/users/:id       → no (path has more segments than :id covers)
  GET /api/users/:id/posts → YES ✅ → extract userId = "123"
                                    → call getUserPostsHandler(userId)
```

> **Traffic control analogy:**
> Routing is like a traffic officer at a busy intersection:
> - Every vehicle (request) arrives with a `METHOD` (intent) + `PATH` (destination)
> - The officer (router) checks their map (routing table)
> - The officer directs the vehicle to the correct lane (handler)
> - If the destination doesn't exist - the officer says "wrong address, turn around" (404)

## Why Routing Matters

If you deeply understand routing, you can:

```
✅ Design clean, intuitive, REST-compliant APIs
✅ Debug 404 errors instantly (wrong method? wrong path? wrong version?)
✅ Avoid bad patterns (verbs in URLs, mixing path/query param concerns)
✅ Handle nested resource relationships cleanly
✅ Implement versioning without breaking existing clients
✅ Understand large codebases faster (routes tell you the API's capabilities)
✅ Onboard on any project quickly (reading routes = reading the API)
```

Routing is the **entry gate** to everything else:

```
HTTP Request
     │
     ▼
 ROUTING  ←── Understanding this is the key to understanding everything else
     │
     ├──► Controllers (handle the request data)
     │
     ├──► Services (run business logic)
     │
     ├──► Middleware (authentication, logging, rate limiting)
     │
     └──► Database Repositories (data fetching and storing)
```

> **The simple truth:**
> Every feature in your backend exists because of a route.
> If there's no route, no code runs, no feature works.
> Routing is where it all begins.

## Summary - Key Concepts at a Glance

```
┌──────────────────────┬───────────────────────────────────────────────────┐
│ Concept              │ One-line summary                                  │
├──────────────────────┼───────────────────────────────────────────────────┤
│ HTTP Method          │ The WHAT - GET reads, POST creates, etc.          │
│ URL Path             │ The WHERE - which resource to act on              │
│ Routing              │ METHOD + PATH → handler function                  │
│ Static Route         │ Fixed path: /api/books                            │
│ Dynamic Route        │ Variable path: /api/books/:id                     │
│ Path Parameter       │ Identity: /users/123 → user 123 specifically      │
│ Query Parameter      │ Options: /books?page=2&sort=price (filter/sort)   │
│ Nested Route         │ Hierarchy: /users/:id/posts → posts of that user  │
│ Route Versioning     │ /v1/ and /v2/ coexist - no breaking changes       │
│ Catch-All Route      │ Safety net: unmatched → clean 404 response        │
└──────────────────────┴───────────────────────────────────────────────────┘
```
