## Why Do We Need Databases?

### The Core Problem: Persistence

At its core, a database solves **one fundamental problem** — **persistence**.

**Persistence** = storing data in a way so that it survives even after the program that created it has been stopped.

Think of a to-do list app:

```
1. You open the app
2. You add tasks: "Buy groceries", "Study databases", "Call mom"
3. You check off "Call mom" ✅
4. You close the app
5. You open it again tomorrow

Expected behavior: All your tasks are STILL there, in the SAME state you left them.
```

This is persistence. The data **persists** across:
- Different sessions (closing and reopening the app)
- Different points in time (coming back after days)
- Different physical locations (accessing from another device)

**Without persistence:**
- Every time you open the app, you start from scratch
- All your previous data is gone
- The app is essentially useless

> **Intuition:** Every time you sign into Netflix and your watchlist is still there, that's persistence. Every time you open WhatsApp and your messages haven't disappeared, that's persistence. Databases are the invisible backbone behind all of this.

## What Is a Database?

The term "database" is **surprisingly broad**. At its simplest:

> **Any kind of structured storage** that lets you create, read, update, and delete data can be considered a database.

### Examples You Already Use

| Example | Why It's a Database |
|---------|-------------------|
| **Phone contact list** | Stores names, numbers — you can add, search, edit, delete contacts |
| **Browser localStorage** | Key-value storage — websites store settings and preferences here |
| **Browser sessionStorage** | Same as localStorage, but cleared when the tab closes |
| **Browser cookies** | Small data pieces — sites use them for sessions, preferences |
| **A plain text file** | You jot down notes, refer back later — basic persistence |
| **An Excel spreadsheet** | Rows, columns, structured data — very much a database |

### The Pattern

Every database, regardless of type, provides:

```
┌──────────────────────────────────────┐
│         DATABASE = Any system that:  │
│                                      │
│  1. PERSISTS data (survives restarts)│
│  2. Allows CREATE (add new data)     │
│  3. Allows READ (retrieve data)      │
│  4. Allows UPDATE (modify data)      │
│  5. Allows DELETE (remove data)      │
│                                      │
│  These 4 operations = CRUD           │
└──────────────────────────────────────┘
```

### Important Distinction

In **general context**: A database is any persistence layer with CRUD.

In **backend systems context**: When developers say "database," they specifically mean **disk-based database systems** — specialized software like PostgreSQL, MySQL, MongoDB that are designed for structured, efficient, large-scale data storage.

## Disk-Based Storage vs RAM

### Why Disk-Based?

When we talk about databases in the backend context, we mean **disk-based databases** — systems that store data on hard drives (HDD) or solid-state drives (SSD).

But why disk? Why not RAM (Random Access Memory)?

### The Memory Hierarchy

```
┌─────────────────────────────────────────────────────────────────┐
│                    STORAGE HIERARCHY                             │
│                                                                  │
│  CPU Registers ──→ Fastest, tiniest (bytes)                     │
│       ↓                                                          │
│  CPU Cache (L1/L2/L3) ──→ Very fast, small (KB-MB)             │
│       ↓                                                          │
│  RAM (Primary Memory) ──→ Fast, moderate (8-128 GB)             │
│       ↓                                                          │
│  Disk (Secondary Memory) ──→ Slow, massive (512 GB - 4 TB)     │
│       ↓                                                          │
│  Network Storage ──→ Slowest, unlimited (cloud)                 │
│                                                                  │
│  Rule: As you go DOWN → speed decreases, capacity increases,    │
│                          cost per GB decreases                   │
└─────────────────────────────────────────────────────────────────┘
```

### RAM vs Disk — The Tradeoff

| Aspect | RAM (Primary Memory) | Disk (Secondary Memory) |
|--------|---------------------|------------------------|
| **Speed** | Extremely fast (~10 nanoseconds) | Relatively slow (~10 milliseconds for HDD) |
| **Capacity** | Limited (8-128 GB typical) | Massive (512 GB - 4 TB typical) |
| **Cost per GB** | Expensive (~$3-5/GB) | Cheap (~$0.02-0.10/GB) |
| **Persistence** | ❌ Volatile — data lost on power-off | ✅ Non-volatile — data survives power-off |
| **Typical size** | 8, 16, 32, 64, 128 GB | 512 GB, 1 TB, 2 TB |

### The Key Insight

```
For DATABASES, what matters most:
  → We need LOTS of space (potentially hundreds of GBs)
  → Data must SURVIVE restarts (persistence)
  → We can tolerate SOME speed tradeoff

  ✅ Disk storage = perfect fit
  ❌ RAM = too expensive, too small, not persistent

For CACHING (Redis, in-memory caches):
  → Speed is CRITICAL (sub-millisecond lookups)
  → We only store frequently accessed data (small subset)
  → Data can be rebuilt if lost

  ✅ RAM = perfect fit
  ❌ Disk = too slow for cache purposes
```

### Where Each Technology Stores Data

| Technology | Storage Location | Why |
|-----------|-----------------|-----|
| **PostgreSQL** | Disk (SSD/HDD) | Needs space, needs persistence |
| **MySQL** | Disk (SSD/HDD) | Same reasons |
| **MongoDB** | Disk (SSD/HDD) | Same reasons |
| **Redis** | RAM (primary memory) | Speed is priority, used for caching |
| **Memcached** | RAM (primary memory) | Same — caching layer |

> **Intuition:** Think of disk storage as a warehouse — massive, cheap, but takes time to find things. RAM is like your desk — fast to grab things from, but limited space. Databases are warehouses. Caches are desks.

## What Is a DBMS?

**DBMS = Database Management System**

Just storing data on disk isn't enough. We need a **software system** that efficiently manages all operations on that data. That's what a DBMS is.

### Responsibilities of a DBMS

```
┌────────────────────────────────────────────────────────────────┐
│                    DBMS RESPONSIBILITIES                        │
│                                                                  │
│  1. DATA ORGANIZATION                                           │
│     → Efficiently structure data for fast retrieval              │
│     → Indexing, partitioning, storage optimization              │
│                                                                  │
│  2. DATA ACCESS (CRUD)                                          │
│     → Create, Read, Update, Delete operations                   │
│     → Efficient query execution                                 │
│     → Query optimization                                        │
│                                                                  │
│  3. DATA INTEGRITY                                              │
│     → Ensure data accuracy and validity                         │
│     → Enforce data types, constraints, relationships            │
│     → Prevent corrupt/invalid data from entering                │
│                                                                  │
│  4. SECURITY                                                    │
│     → Protect from unauthorized access                          │
│     → Users, roles, permissions                                 │
│     → Encryption, authentication                                │
│                                                                  │
│  5. CONCURRENCY CONTROL                                         │
│     → Handle multiple users accessing data simultaneously       │
│     → Prevent conflicts and inconsistencies                     │
│                                                                  │
│  6. SCALABILITY & RELIABILITY                                   │
│     → Handle growing data volumes                               │
│     → Backup, recovery, replication                             │
└────────────────────────────────────────────────────────────────┘
```

### Data Integrity — What Does It Mean?

Data Integrity = **the data in your database is accurate, valid, and consistent**.

**Example:** You have an e-commerce platform with an `orders` table. Each order has a `payment_amount` field.

```
orders table:
┌────┬──────────────┬────────────────┐
│ id │ customer     │ payment_amount │
├────┼──────────────┼────────────────┤
│ 1  │ Alice        │ 299.99         │ ← ✅ Valid (number)
│ 2  │ Bob          │ 149.50         │ ← ✅ Valid (number)
│ 3  │ Charlie      │ "hello"        │ ← ❌ INVALID! (string in a number field)
└────┴──────────────┴────────────────┘

The DBMS must REJECT the third row.
If payment_amount is defined as a numeric type,
the DBMS will throw an error when someone tries to insert "hello".

That's data integrity — the DATABASE enforces the rules,
not your application code (though you should validate there too).
```

> **Intuition:** Think of the DBMS as a gatekeeper with strict rules. Even if your application code has a bug and tries to insert garbage data, the DBMS stands guard and says "No, this doesn't match the rules." That's integrity.

## Why Not Just Use Text Files?

Before DBMS software existed, people literally stored data in text files. Here's why that doesn't scale:

### Problem 1: Parsing Is Slow and Error-Prone

```
Imagine a text file storing customer data:

customers.txt:
---
Alice|alice@email.com|29|New York
Bob|bob@email.com|35|San Francisco
Charlie|charlie@email.com|22|Chicago
... (100,000 more lines)
```

To find Charlie's email:
1. Open the file
2. Read ALL lines (even if Charlie is the last one)
3. Split each line by `|`
4. Compare the first field with "Charlie"
5. Return the second field

```javascript
// This is what you'd have to write in your application code:
const fs = require('fs');
const data = fs.readFileSync('customers.txt', 'utf8');
const lines = data.split('\n');

for (const line of lines) {
  const fields = line.split('|');
  if (fields[0] === 'Charlie') {
    console.log(fields[1]); // charlie@email.com
    break;
  }
}
// Problems:
// 1. Reads ENTIRE file into memory (what if it's 10 GB?)
// 2. Linear search — O(n) for every lookup
// 3. If file format changes, all parsing code breaks
// 4. No indexes — can't speed up lookups
// 5. Very error-prone — one malformed line crashes everything
```

**With a DBMS:**
```sql
SELECT email FROM customers WHERE name = 'Charlie';
-- Done. The DBMS handles indexing, optimization, memory management.
-- With an index on name, this is O(log n) — instant even with millions of rows.
```

### Problem 2: No Structure

```
Text files have no formal structure. Anyone can write anything:

customers.txt:
Alice|alice@email.com|29|New York
Bob|35|bob@email.com|San Francisco    ← Oops! Age and email are swapped!
Charlie|charlie@email.com|twenty-two  ← "twenty-two" instead of 22, missing city!
NOTE: Remember to update Alice's email  ← Random note mixed in with data!
```

There's **no schema** to enforce:
- Which fields exist
- What order they're in
- What data types they must be
- Whether a field is required

> A DBMS enforces schema at the database level. You define a table with specific columns, types, and constraints. Data that doesn't match gets rejected before it enters.

### Problem 3: Concurrency — The Race Condition

This is the most dangerous problem. What happens when two users modify the file at the same time?

```
Initial state:
  account_balance.txt → amount: 40

User A wants to ADD 20 (deposit)
User B wants to SUBTRACT 20 (withdrawal)

Timeline:
┌─────────────────────────────────────────────┐
│ Time 1: User A reads file → sees 40         │
│ Time 1: User B reads file → sees 40         │
│                                              │
│ Time 2: User A calculates 40 + 20 = 60      │
│ Time 2: User B calculates 40 - 20 = 20      │
│                                              │
│ Time 3: User A writes 60 to file            │
│ Time 4: User B writes 20 to file ← OVERWRITES!│
│                                              │
│ Final result: 20 ❌                          │
│ Expected result: 40 (original) + 20 - 20 = 40│
└─────────────────────────────────────────────┘
```

User A's deposit of 20 was **completely lost** because User B's write overwrote it. This is called a **race condition**.

**A DBMS solves this with:**
- **Transactions** — Group operations into atomic units
- **Locking** — Prevent simultaneous conflicting modifications
- **ACID properties** — Atomicity, Consistency, Isolation, Durability

```sql
-- In PostgreSQL, this is handled correctly:
BEGIN;
UPDATE accounts SET balance = balance + 20 WHERE user_id = 'A';
COMMIT;

-- Even if two transactions run simultaneously, PostgreSQL ensures
-- they execute in a way that produces correct results.
-- No data is lost. No race conditions.
```

### Summary: Text Files vs DBMS

| Problem | Text Files | DBMS |
|---------|-----------|------|
| **Parsing** | Manual, slow, error-prone | Optimized query engine with indexes |
| **Structure** | None — any data goes anywhere | Strict schema enforcement |
| **Concurrency** | Race conditions, data loss | Transactions, locking, ACID |
| **Integrity** | No validation | Constraints, types, foreign keys |
| **Scalability** | Falls apart at thousands of rows | Handles millions/billions of rows |
| **Security** | File system permissions only | Users, roles, row-level security |

## Relational vs Non-Relational Databases

At a high level, there are **two major categories** of database systems:

### Relational Databases (SQL)

**How it works:** Data is organized in **tables** with **rows** and **columns**. Relationships between tables are defined using **foreign keys**.

```
┌─────────────────────────────────────────────┐
│                  RELATIONAL                  │
│                                              │
│  Data → Tables (rows × columns)              │
│  Schema → Predefined, strict                 │
│  Query Language → SQL                        │
│  Relationships → Foreign keys                │
│  Examples → PostgreSQL, MySQL, SQLite,       │
│             SQL Server, Oracle               │
└─────────────────────────────────────────────┘
```

**Key characteristics:**
- **Predefined schema** — You MUST define all columns and their data types BEFORE inserting data
- **Strict data types** — Each column enforces a specific type (number, text, date, etc.)
- **Relationships** — Tables can reference each other using foreign keys
- **Data integrity** — The database guarantees data is valid and consistent
- **SQL** — Structured Query Language for all operations

```sql
-- Example: You MUST define the schema first
CREATE TABLE users (
  id UUID PRIMARY KEY,
  email TEXT NOT NULL UNIQUE,
  full_name TEXT NOT NULL,
  age INTEGER CHECK (age > 0)
);

-- Then you can insert data (must match the schema)
INSERT INTO users (id, email, full_name, age)
VALUES (gen_random_uuid(), 'alice@email.com', 'Alice Smith', 29);

-- This FAILS — age can't be a string:
INSERT INTO users (id, email, full_name, age)
VALUES (gen_random_uuid(), 'bob@email.com', 'Bob Jones', 'twenty-nine');
-- ERROR: invalid input syntax for type integer
```

### Non-Relational Databases (NoSQL)

**How it works:** Data is stored in flexible formats — documents (JSON), key-value pairs, graphs, or wide columns. No predefined schema required.

```
┌─────────────────────────────────────────────┐
│               NON-RELATIONAL                 │
│                                              │
│  Data → Documents, key-value, graph, etc.    │
│  Schema → Flexible, schema-less              │
│  Query Language → Varies (not SQL)           │
│  Relationships → Embedded documents or refs  │
│  Examples → MongoDB, Redis, Cassandra,       │
│             DynamoDB, Neo4j                  │
└─────────────────────────────────────────────┘
```

**Key characteristics:**
- **Flexible schema** — Each document can have different fields
- **No predefined schema** — Insert any JSON structure, no table creation needed
- **Fast for prototyping** — Just dump data and go
- **Horizontal scaling** — Designed for distributed systems

```javascript
// MongoDB example — no schema needed
// Document 1:
{
  name: "Alice",
  email: "alice@email.com",
  age: 29,
  address: { city: "New York", state: "NY" }
}

// Document 2 (same collection, DIFFERENT structure!):
{
  name: "Bob",
  email: "bob@email.com",
  phone: "+1-555-0123",         // Alice doesn't have this field
  hobbies: ["reading", "coding"] // Alice doesn't have this field either
  // No age field! No address! And that's FINE in MongoDB.
}
```

### Terminology Mapping

| Concept | Relational (SQL) | Non-Relational (MongoDB) |
|---------|-----------------|-------------------------|
| Container for data | **Table** | **Collection** |
| Single data entry | **Row** | **Document** |
| Data field | **Column** | **Field** (key-value) |
| Schema | **Strict, predefined** | **Flexible, dynamic** |

### When to Use Which?

| Use Case | Best Choice | Why |
|----------|-------------|-----|
| **CRM system** (customer data) | ✅ Relational (PostgreSQL) | Need strict data integrity, complex queries, relationships between customers/sales/contacts |
| **E-commerce** (orders, inventory) | ✅ Relational (PostgreSQL) | Financial data needs accuracy, relationships between orders/products/customers |
| **Banking system** | ✅ Relational (PostgreSQL) | ACID compliance is critical, data integrity is non-negotiable |
| **CMS** (content management) | ✅ Non-relational (MongoDB) | Content structure varies widely (articles, images, embeds, code blocks) |
| **Real-time analytics** | ✅ Non-relational (Cassandra) | Massive write throughput, horizontal scaling |
| **Caching layer** | ✅ Non-relational (Redis) | Key-value lookups, sub-millisecond speed |
| **Social network** (graph data) | ✅ Non-relational (Neo4j) | Relationships ARE the data (friends, followers, connections) |
| **General SaaS product** | ✅ Relational (PostgreSQL) | Best default choice — covers 90% of use cases |

### The Flexibility vs Integrity Tradeoff

```
Relational (PostgreSQL):
  ✅ Data integrity guaranteed
  ✅ Complex queries with JOINs
  ✅ ACID transactions
  ❌ Schema changes need migrations
  ❌ Less flexible for unstructured data

Non-Relational (MongoDB):
  ✅ Flexible schema — move fast
  ✅ Great for prototyping
  ✅ Natural JSON format
  ❌ No guaranteed data integrity at DB level
  ❌ Integrity must be enforced in application code (more bugs)
  ❌ Easier to introduce inconsistencies
```

> **Key insight:** When you choose MongoDB for flexibility, you're **moving the responsibility of data integrity from the database to your application code**. This means more code to write, more bugs to encounter, and more things to maintain. The database no longer protects you from bad data — your code has to.

## Why PostgreSQL?

Out of all the relational databases (MySQL, SQLite, SQL Server, Oracle), **PostgreSQL stands out** for several important reasons:

### 1. Open Source and Free

PostgreSQL is **completely open source**. You can:
- Host it on your own servers
- Inspect the source code
- No licensing fees, ever
- Companies love open source — no vendor lock-in

### 2. SQL Standard Compliance

PostgreSQL **strictly follows the SQL standard**. This means:

```sql
-- Any standard SQL query works in PostgreSQL:
SELECT * FROM users WHERE email = 'alice@email.com';
INSERT INTO orders (user_id, amount) VALUES (1, 299.99);

-- If you ever need to migrate to MySQL or another SQL database,
-- your queries will mostly work without changes.
```

**Why this matters:** If you learn SQL through PostgreSQL, you can work with ANY relational database. The knowledge transfers directly.

### 3. Extremely Extensible

PostgreSQL's documentation is ~3,400 pages. It has features for virtually every use case:
- Full-text search
- Geospatial queries (PostGIS)
- JSON querying and indexing
- Custom types, operators, and functions
- Window functions, CTEs, recursive queries

### 4. Reliable and Scalable

- Used by companies like Instagram, Spotify, Netflix, Apple
- Handles millions of concurrent connections
- Built-in replication and high availability
- Battle-tested for decades (since 1996)

### 5. Native JSON Support (The Game Changer)

This is what makes PostgreSQL unique among relational databases:

```
The argument for MongoDB:
  "We need flexible data! Our content doesn't have a fixed schema!"

The PostgreSQL counter:
  "We have JSONB columns. Store any JSON you want, AND get indexing,
   querying, and all the relational features on top."
```

```sql
-- You can store any JSON in a JSONB column:
CREATE TABLE products (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  metadata JSONB  -- Store ANY JSON here
);

-- Insert products with completely different metadata structures:
INSERT INTO products (name, metadata) VALUES
  ('Laptop', '{"brand": "Dell", "ram": "16GB", "screen": "15.6 inch"}'),
  ('T-Shirt', '{"size": "M", "color": "blue", "material": "cotton"}'),
  ('Book', '{"author": "Martin", "pages": 464, "isbn": "978-0134757599"}');

-- Query JSON fields:
SELECT name FROM products WHERE metadata->>'brand' = 'Dell';
-- Returns: Laptop

-- You can even INDEX JSON fields for fast lookups:
CREATE INDEX idx_brand ON products USING GIN (metadata);
```

**The bottom line:** With PostgreSQL's JSONB support, you get:
- ✅ Relational features (strict schema where needed)
- ✅ Flexible JSON storage (dynamic data where needed)
- ✅ Both in ONE database

There's almost no reason to use MongoDB "just for flexible data" when PostgreSQL offers the same flexibility with better data integrity.

### PostgreSQL = Default Choice

```
Decision flowchart:

Building a new project?
  → Use PostgreSQL.

Need dynamic/flexible data?
  → Use PostgreSQL with JSONB columns.

Need caching?
  → Use Redis alongside PostgreSQL.

Need full-text search?
  → PostgreSQL has it built-in. Or add Elasticsearch.

Need to scale massively?
  → PostgreSQL supports read replicas, partitioning, sharding.

Need a graph database?
  → THEN consider Neo4j. But for most graph needs, PostgreSQL recursive
    CTEs are sufficient.
```

> **Practical rule:** Start with PostgreSQL. Add specialized databases only when PostgreSQL genuinely can't handle a specific requirement. For 90% of projects, PostgreSQL is all you need.

## PostgreSQL Data Types — Complete Guide

Understanding data types is crucial because choosing the right type affects:
- **Storage efficiency** — Wrong type = wasted disk space
- **Query performance** — Right type = faster lookups
- **Data integrity** — Type enforcement prevents bad data

### Integer Types

```sql
CREATE TABLE data_types_demo (
  id            SERIAL PRIMARY KEY,       -- Auto-incrementing integer
  small_number  SMALLINT,                 -- -32,768 to 32,767
  normal_number INTEGER,                  -- -2,147,483,648 to 2,147,483,647
  big_number    BIGINT                    -- -9.2 quintillion to 9.2 quintillion
);
```

| Type | Size | Range | Use Case |
|------|------|-------|----------|
| **SMALLINT** | 2 bytes | -32,768 to 32,767 | Age, quantity, small counts |
| **INTEGER** | 4 bytes | ~-2.1 billion to ~2.1 billion | Most general-purpose integers |
| **BIGINT** | 8 bytes | ~-9.2 quintillion to ~9.2 quintillion | IDs in systems with billions of rows |
| **SERIAL** | 4 bytes | 1 to ~2.1 billion | Auto-incrementing IDs |
| **BIGSERIAL** | 8 bytes | 1 to ~9.2 quintillion | Auto-incrementing IDs (production) |

#### SERIAL vs BIGSERIAL

```sql
-- SERIAL: Auto-incrementing integer (range up to ~2.1 billion)
id SERIAL PRIMARY KEY
-- First row: id = 1
-- Second row: id = 2
-- Third row: id = 3
-- You DON'T provide the ID — PostgreSQL generates it automatically.

-- BIGSERIAL: Same, but much larger range (for production systems)
id BIGSERIAL PRIMARY KEY
-- Use this when you expect billions of rows over time.
-- Most production systems use BIGSERIAL for safety.
```

> **Rule of thumb:** Use `BIGSERIAL` for primary keys in production. Use `INTEGER` for regular number fields. Use `SMALLINT` only when you're certain the range is sufficient.

### Decimal / Numeric Types

This is where most developers get confused. There are **two categories**:

#### Exact Numbers: DECIMAL / NUMERIC

```sql
price DECIMAL(10, 2)
-- 10 = total number of digits across the ENTIRE number
-- 2 = number of digits AFTER the decimal point

-- Valid values:
-- 12345678.99  (8 digits + 2 decimals = 10 total ✅)
-- 99.99        (2 digits + 2 decimals = 4 total ✅)
-- 1234567890.12 ❌ (12 total digits — exceeds 10!)
```

**DECIMAL and NUMERIC are identical in PostgreSQL.** Use either one.

#### Approximate Numbers: REAL / DOUBLE PRECISION / FLOAT

```sql
measurement REAL,             -- 4 bytes, ~6 decimal digits precision
big_measurement DOUBLE PRECISION,  -- 8 bytes, ~15 decimal digits precision
```

#### When to Use Which?

```
┌─────────────────────────────────────────────────────────────────┐
│  DECIMAL vs FLOAT — THE DECISION                                │
│                                                                  │
│  Is ACCURACY critical?                                          │
│    → Money, prices, financial calculations                      │
│    → Invoice amounts, account balances                          │
│    → ALWAYS use DECIMAL                                         │
│                                                                  │
│  Is SPEED more important than exact precision?                  │
│    → Scientific measurements                                    │
│    → Area calculations, GPS coordinates                         │
│    → Sensor readings                                            │
│    → Use REAL or DOUBLE PRECISION                               │
│                                                                  │
│  Why not always use DECIMAL if it's more accurate?              │
│    → Float operations are FASTER (hardware-optimized)           │
│    → Floats use less storage                                    │
│    → For scientific/engineering domains, speed > exact precision │
└─────────────────────────────────────────────────────────────────┘
```

**The floating-point trap:**

```
-- Floats can have tiny inaccuracies:
SELECT 0.1 + 0.2;
-- In floating point: 0.30000000000000004 (not exactly 0.3!)

-- For money, this is UNACCEPTABLE.
-- For area measurements, this tiny difference is negligible.
```

> **Rule:** If the field involves money or anything financial → `DECIMAL`. For everything else where small inaccuracies don't matter → `REAL` or `DOUBLE PRECISION`.

### String Types

PostgreSQL has three string types:

#### CHAR(n) — Fixed Length

```sql
day_code CHAR(2)

-- Stores: 'MO' → stored as 'MO' (2 bytes)
-- Stores: 'W'  → stored as 'W ' (padded with space to reach 2!)
```

The database **always pads** the value with spaces to reach the defined length. This wastes storage if values vary in length.

#### VARCHAR(n) — Variable Length with Limit

```sql
name VARCHAR(255)

-- Stores: 'AB'     → stored as 'AB' (2 bytes, no padding ✅)
-- Stores: 'Alice'  → stored as 'Alice' (5 bytes ✅)
-- Stores: a 300-character string → ERROR! Exceeds 255 limit ❌
```

Variable length — only uses as much space as the actual value. But has a maximum limit.

#### TEXT — Unlimited Variable Length

```sql
description TEXT

-- Stores: 'AB'     → stored as 'AB' (2 bytes ✅)
-- Stores: 'Alice'  → stored as 'Alice' (5 bytes ✅)
-- Stores: a 10,000-character essay → works fine ✅
-- No length limit (practical limit is ~1 GB)
```

#### Which String Type Should You Use?

```
┌──────────────────────────────────────────────────────────────┐
│  STRING TYPE DECISION (PostgreSQL-specific advice)            │
│                                                               │
│  ❌ CHAR(n) — Almost never use.                              │
│     Only if EVERY value is EXACTLY the same length.           │
│     Example: country codes (US, IN, UK) → CHAR(2)            │
│     Otherwise, it wastes space by padding.                    │
│                                                               │
│  ⚠️ VARCHAR(n) — Avoid in PostgreSQL.                        │
│     The "255" convention comes from MySQL, where it had       │
│     meaning. In PostgreSQL, 255 is just an arbitrary number   │
│     with NO special significance.                             │
│                                                               │
│  ✅ TEXT — Always use this in PostgreSQL.                     │
│     PostgreSQL documentation RECOMMENDS TEXT over VARCHAR.    │
│     There is NO performance difference between them.          │
│     TEXT can be indexed just like VARCHAR.                     │
│     Enforce length limits in application code instead.        │
└──────────────────────────────────────────────────────────────┘
```

#### Why Not VARCHAR(255)?

```
The problems with VARCHAR(255) in PostgreSQL:

1. The number 255 has NO MEANING in PostgreSQL.
   → It's a MySQL convention (255 is the max for a single-byte length prefix).
   → People copy it blindly without understanding why.

2. If you need to increase the limit later:
   → You need a DATABASE MIGRATION (risky, affects all data).
   → With TEXT, you enforce limits in application code.
   → Changing application code is MUCH safer than database migrations.

3. It misleads new team members:
   → They see VARCHAR(255) and think "255 has some meaning!"
   → They research why, waste time, find nothing (because there IS no reason).
   → With TEXT, it's clear: "this is a text field. Period."

4. No performance benefit:
   → PostgreSQL treats VARCHAR and TEXT identically internally.
   → Both are stored the same way, indexed the same way.
```

> **PostgreSQL best practice:** Use `TEXT` for all string fields. Validate length in your application code (e.g., using Zod, Joi, or custom validation). Keep your database schema simple and your application code responsible for business logic validation.

### Boolean

```sql
is_active BOOLEAN DEFAULT true

-- Values: true, false (or NULL if not NOT NULL constrained)
-- PostgreSQL also accepts: 'yes', 'no', 1, 0, 't', 'f'
```

### Date and Time Types

```sql
-- Date only (no time)
birth_date DATE
-- Stores: '2024-03-15'

-- Time only (no date)
start_time TIME
-- Stores: '14:30:00' (2:30 PM)

-- Date + Time (no timezone)
event_at TIMESTAMP
-- Stores: '2024-03-15 14:30:00'

-- Date + Time + Timezone (RECOMMENDED for most use cases)
created_at TIMESTAMPTZ
-- Stores: '2024-03-15 14:30:00+05:30'
-- TIMESTAMPTZ = TIMESTAMP WITH TIME ZONE

-- Duration / Interval
duration INTERVAL
-- Stores: '10 days', '3 hours', '1 year 2 months'
```

> **Rule:** Almost always use `TIMESTAMPTZ` (timestamp with time zone) for any datetime field. Without timezone info, you can't correctly display times for users in different time zones.

### UUID

```sql
id UUID PRIMARY KEY DEFAULT gen_random_uuid()

-- Stores: 'a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11'
-- 128-bit universally unique identifier
-- Practically impossible to have collisions
```

**Why UUIDs over auto-incrementing integers?**

| Aspect | SERIAL (1, 2, 3...) | UUID |
|--------|-------------------|------|
| **Guessable?** | Yes — if user 5 exists, try user 6 | No — random and unpredictable |
| **Security** | Exposes count, easy to enumerate | Doesn't leak information |
| **Distributed systems** | Need centralized counter | Can generate independently anywhere |
| **URL-safe** | `/users/5` — easy to scrape | `/users/a0eebc99-...` — can't guess |

> **Production preference:** Use `UUID` for primary keys that are exposed in APIs/URLs. Use `BIGSERIAL` for internal-only sequential IDs.

### JSON and JSONB

```sql
-- JSON: Stored as plain text, no processing
metadata JSON

-- JSONB: Stored in a binary format, optimized for queries
metadata JSONB
```

| Feature | JSON | JSONB |
|---------|------|-------|
| **Storage** | Plain text (as-is) | Binary format (pre-processed) |
| **Insert speed** | Faster (no processing) | Slower (must convert to binary) |
| **Query speed** | Slower (must parse text) | Much faster (already parsed) |
| **Indexing** | ❌ Cannot be indexed | ✅ Can be indexed (GIN, GiST) |
| **Duplicate keys** | Preserved | Last value wins |
| **Key order** | Preserved | Not guaranteed |

> **Rule:** Almost always use `JSONB`. The insert speed difference is negligible, but the query and indexing advantages are massive.

```sql
-- Inserting JSONB data:
INSERT INTO products (name, metadata) VALUES
  ('Laptop', '{"brand": "Dell", "specs": {"ram": "16GB", "storage": "512GB"}}');

-- Querying JSONB:
SELECT name FROM products WHERE metadata->>'brand' = 'Dell';
SELECT name FROM products WHERE metadata->'specs'->>'ram' = '16GB';

-- Operators:
-- ->  : Get JSON object field (returns JSON)
-- ->> : Get JSON object field as TEXT
-- @>  : Contains (does left contain right?)
-- ?   : Does key exist?
```

### Array Types

```sql
-- Array of integers:
tags INTEGER[]
-- Store: '{1, 2, 3, 4}'

-- Array of text:
categories TEXT[]
-- Store: '{"tech", "science", "education"}'

-- Querying arrays:
SELECT * FROM posts WHERE 'tech' = ANY(categories);
```

### Other Specialized Types

```sql
-- Network addresses:
ip_address INET        -- '192.168.1.1' or '::1'
mac       MACADDR      -- '08:00:2b:01:02:03'

-- Geometric:
location  POINT        -- '(40.7128, -74.0060)' (latitude, longitude)

-- XML:
config    XML          -- '<config><key>value</key></config>'
```

### Complete Data Type Cheat Sheet

| Category | Type | When to Use |
|----------|------|-------------|
| **IDs** | `UUID` | Primary keys (production) |
| **IDs** | `BIGSERIAL` | Auto-increment IDs (internal) |
| **Numbers** | `INTEGER` | General-purpose numbers |
| **Numbers** | `BIGINT` | Very large numbers |
| **Money** | `DECIMAL(p,s)` | Financial values (exact) |
| **Measurements** | `DOUBLE PRECISION` | Scientific values (approximate) |
| **Text** | `TEXT` | All strings (PostgreSQL best practice) |
| **Boolean** | `BOOLEAN` | True/false flags |
| **Datetime** | `TIMESTAMPTZ` | Any date+time field |
| **Duration** | `INTERVAL` | Time spans |
| **Flexible data** | `JSONB` | Semi-structured / dynamic data |
| **Lists** | `TEXT[]`, `INT[]` | Simple arrays |
| **Fixed choices** | Custom `ENUM` | Status fields, role types |

## Database Migrations

### What Are Migrations?

In production systems, you **never** just open a GUI tool (like TablePlus or pgAdmin) and type SQL directly. That's dangerous because:

- No record of what changes were made
- No way to review changes before applying
- No way to roll back if something breaks
- No version control
- Team members can't see what changed or when

**Migrations** are the solution: **version-controlled SQL files** that track every change to your database schema over time.

### How Migrations Work

```
Your project structure:
project/
├── src/
│   └── (application code)
├── db/
│   └── migrations/
│       ├── 001_create_users_table.sql
│       ├── 002_create_projects_table.sql
│       ├── 003_add_email_index.sql
│       ├── 004_create_tasks_table.sql
│       └── 005_add_status_column.sql
├── .env
└── package.json
```

Each migration file:
1. Has a **sequential number** or **timestamp** (for ordering)
2. Contains SQL statements
3. Is committed to **version control (git)**
4. Is applied to the database using a **migration tool**

### Migration Tools

| Tool | Language | Command |
|------|----------|---------|
| **dbmate** | Go (works with any language) | `dbmate up`, `dbmate down` |
| **golang-migrate** | Go | `migrate up`, `migrate down` |
| **Prisma Migrate** | JavaScript/TypeScript | `npx prisma migrate dev` |
| **Knex.js** | JavaScript | `npx knex migrate:latest` |
| **Sequelize** | JavaScript | `npx sequelize db:migrate` |
| **Flyway** | Java | `flyway migrate` |

### Up Migrations vs Down Migrations

Each migration file has two sections:

```sql
-- +migrate Up
-- (Changes to APPLY)
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email TEXT NOT NULL UNIQUE,
  full_name TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- +migrate Down
-- (Changes to REVERT — undo everything from "Up")
DROP TABLE IF EXISTS users;
```

**Up migration:** The forward change. Creates tables, adds columns, creates indexes.

**Down migration:** The reverse. Undoes everything the up migration did. Used for rollbacks.

```
The workflow:

Apply changes:
  dbmate up → Executes "Up" sections of all pending files

Something breaks:
  dbmate down → Executes "Down" section of the LAST applied file
  → Database returns to its previous state
  → Fix the issue, update the migration, try again
```

### How the Tool Tracks State

The migration tool creates a special table in your database (usually called `schema_migrations`):

```sql
-- This table is automatically managed by the tool:
schema_migrations:
┌───────────────────────────────────────────────┐
│ version                                        │
├───────────────────────────────────────────────┤
│ 001_create_users_table                        │
│ 002_create_projects_table                     │
│ 003_add_email_index                           │
│ 004_create_tasks_table                        │  ← Current state
└───────────────────────────────────────────────┘

--  If you add "005_add_status_column.sql" and run `dbmate up`:
-- → Tool checks schema_migrations
-- → Sees 001-004 already applied
-- → Only applies 005 (the new one)
-- → Adds "005_add_status_column" to schema_migrations
```

### Why Migrations Matter

```
Without migrations:
  1. Developer A opens pgAdmin and adds a column
  2. Developer B doesn't know about it
  3. Their code expects the old schema → crashes
  4. No one knows what the database looked like 3 months ago
  5. Rolling back to a previous state is impossible

With migrations:
  1. Developer A writes a migration file and commits to git
  2. Developer B pulls the code, runs `dbmate up` → same schema
  3. Git shows the full history of every schema change
  4. Rolling back = `dbmate down`
  5. New team members run `dbmate up` → exact same database
```

### Migration Best Practices

```
1. NEVER modify a migration that has already been applied
   → If you need changes, create a NEW migration file

2. Always write Down migrations
   → Your future self will thank you when something breaks

3. Make migrations SMALL and FOCUSED
   → One migration per logical change
   → "create_users_table" — not "create_everything"

4. Test migrations on a staging/dev database FIRST
   → Never apply untested migrations to production

5. Use timestamps in filenames (not sequential numbers)
   → Prevents merge conflicts when multiple developers
     create migrations simultaneously
```

## Database Modeling — Project Management Platform

Let's model a real database for a project management platform (like Jira or Asana). This will demonstrate all the concepts we've discussed.

### Creating Your First Migration

Using **dbmate** as our migration tool:

```bash
# Create a new migration file
dbmate new create_users_table

# This creates:
# db/migrations/20240315143000_create_users_table.sql
```

The tool creates the folder structure (`db/migrations/`) and a timestamped file automatically.

### Step 1: Define Custom Types (Enums)

Before creating tables, we define **enum types** — predefined sets of allowed values.

```sql
-- +migrate Up

-- Custom enum types for strict value enforcement
CREATE TYPE project_status AS ENUM ('active', 'completed', 'archived');
CREATE TYPE task_status AS ENUM ('pending', 'in_progress', 'completed', 'cancelled');
CREATE TYPE member_role AS ENUM ('owner', 'admin', 'member');
```

#### Why Use Enums?

**Reason 1: Data Integrity**

```sql
-- WITHOUT enum (using TEXT):
INSERT INTO projects (name, status) VALUES ('My Project', 'actve');
-- Typo "actve" is stored as-is. No error. ❌
-- Database doesn't know it's wrong. Your data is now inconsistent.

-- WITH enum:
INSERT INTO projects (name, status) VALUES ('My Project', 'actve');
-- ERROR: invalid input value for enum project_status: "actve" ❌
-- Database REJECTS the invalid value. Data integrity maintained. ✅
```

The database itself becomes a gatekeeper — it won't accept invalid values, regardless of whether your application code has a bug.

**Reason 2: Self-Documenting Schema**

```sql
-- WITHOUT enum — seeing the table definition:
CREATE TABLE projects (
  status TEXT  -- What values are allowed? 🤷 No idea! Check application code.
);

-- WITH enum — seeing the table definition:
CREATE TYPE project_status AS ENUM ('active', 'completed', 'archived');
CREATE TABLE projects (
  status project_status  -- Allowed values are RIGHT HERE. Crystal clear. ✅
);
```

When a new team member reads the migration files, they can instantly see what values are allowed for each field. No need to dig through application code.

> **The documentation benefit is MORE important than the integrity benefit**, because integrity can also be enforced in application code. But documentation at the schema level is uniquely valuable — it's the single source of truth.

### Step 2: Create the Users Table

```sql
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email TEXT NOT NULL UNIQUE,
  full_name TEXT NOT NULL,
  password_hash TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

Let's break down every field and WHY it's designed this way:

#### `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`

```
id        → Column name (convention: always called "id")
UUID      → Data type (universally unique identifier)
PRIMARY KEY → This field uniquely identifies each row
              Implicitly adds: NOT NULL + UNIQUE
DEFAULT gen_random_uuid() → If not provided during INSERT,
              PostgreSQL auto-generates a random UUID
```

**Why UUID over SERIAL?**
- UUIDs are unguessable (security — can't enumerate users)
- Can be generated independently (no centralized counter needed)
- Safe for distributed systems

#### `email TEXT NOT NULL UNIQUE`

```
email     → Column name
TEXT      → Data type (string of any length)
NOT NULL  → This field CANNOT be empty. INSERT fails if email is missing.
UNIQUE    → No two rows can have the same email. Prevents duplicate accounts.
```

**About NOT NULL:**

By default, PostgreSQL allows `NULL` for any column. This is **dangerous**.

```sql
-- WITHOUT NOT NULL:
INSERT INTO users (id, full_name, password_hash) VALUES (...);
-- email is NULL — user has NO email! How do you send them a reset link? 💀

-- WITH NOT NULL:
INSERT INTO users (id, full_name, password_hash) VALUES (...);
-- ERROR: null value in column "email" violates not-null constraint ✅
```

> **Rule:** Add `NOT NULL` to every column unless you have a **specific reason** for that field to be nullable. More than 70% of your columns should have `NOT NULL`.

#### `password_hash TEXT NOT NULL`

```
password_hash → Named "hash" to clarify we NEVER store plain passwords
TEXT          → Hashed passwords are strings (bcrypt produces ~60 chars)
NOT NULL      → Every user must have a password
```

#### `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`

```
created_at   → When was this row first inserted?
TIMESTAMPTZ  → Timestamp WITH timezone (always use this variant)
NOT NULL     → Must always have a creation timestamp
DEFAULT NOW() → If not provided, use the current server time automatically
```

**Why track `created_at` and `updated_at`?**

```
Use cases for created_at:
  → "Show me all users who signed up this month"
  → "Sort users by registration date"
  → Analytics and reporting

Use cases for updated_at:
  → "When was this user's profile last modified?"
  → "Show recently updated records first"
  → Cache invalidation (check if data changed since last fetch)
  → Debugging ("when did this data change?")
```

### Step 3: Create Related Tables

```sql
-- User profiles (1-to-1 relationship with users)
CREATE TABLE user_profiles (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL UNIQUE REFERENCES users(id) ON DELETE CASCADE,
  bio TEXT,
  avatar_url TEXT,
  phone TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Projects
CREATE TABLE projects (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  description TEXT,
  status project_status NOT NULL DEFAULT 'active',
  owner_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Project members (many-to-many: users ↔ projects)
CREATE TABLE project_members (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  role member_role NOT NULL DEFAULT 'member',
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(project_id, user_id)  -- A user can only be a member of a project ONCE
);

-- Tasks
CREATE TABLE tasks (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  title TEXT NOT NULL,
  description TEXT,
  status task_status NOT NULL DEFAULT 'pending',
  priority INTEGER NOT NULL DEFAULT 0,
  project_id UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  assignee_id UUID REFERENCES users(id) ON DELETE SET NULL,
  due_date TIMESTAMPTZ,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### Understanding REFERENCES (Foreign Keys)

```sql
user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE
```

This is a **foreign key** — it creates a relationship between tables.

```
What this says:
  → user_id must contain a UUID that EXISTS in the users table's id column
  → You CANNOT insert a user_id that doesn't exist in users → Error!
  → ON DELETE CASCADE: If the user is deleted, this row is also deleted

Visual:
users table:                  user_profiles table:
┌──────────────────┐         ┌──────────────────────────┐
│ id (PK)          │ ←────── │ user_id (FK → users.id)  │
│ email            │         │ bio                      │
│ full_name        │         │ avatar_url               │
└──────────────────┘         └──────────────────────────┘

If you try: INSERT INTO user_profiles (user_id, ...) VALUES ('non-existent-id', ...)
→ ERROR: insert or update on table "user_profiles" violates foreign key constraint
→ The referenced user doesn't exist. Data integrity preserved. ✅
```

### ON DELETE Options

| Option | What Happens When Referenced Row Is Deleted |
|--------|-------------------------------------------|
| `CASCADE` | Delete this row too (most common) |
| `SET NULL` | Set the foreign key to NULL |
| `SET DEFAULT` | Set the foreign key to its default value |
| `RESTRICT` | Prevent deletion of the referenced row |
| `NO ACTION` | Same as RESTRICT (default) |

```
When to use which:

CASCADE:  "If a user is deleted, delete their profile too"
          user_profiles.user_id → ON DELETE CASCADE

SET NULL: "If the assignee is deleted, keep the task but unassign it"
          tasks.assignee_id → ON DELETE SET NULL

RESTRICT: "Don't allow deleting a user who owns projects"
          projects.owner_id → ON DELETE RESTRICT
```

### Step 4: Write Down Migrations

```sql
-- +migrate Down

DROP TABLE IF EXISTS tasks;
DROP TABLE IF EXISTS project_members;
DROP TABLE IF EXISTS projects;
DROP TABLE IF EXISTS user_profiles;
DROP TABLE IF EXISTS users;

DROP TYPE IF EXISTS member_role;
DROP TYPE IF EXISTS task_status;
DROP TYPE IF EXISTS project_status;
```

**Order matters in down migrations!** You must drop tables in **reverse order** of creation because of foreign key dependencies:
- `tasks` references `projects` and `users` → drop first
- `projects` references `users` → drop before users
- `users` → drop last

## Constraints & Data Integrity

Constraints are **rules enforced at the database level** that protect your data from becoming invalid or inconsistent.

### Types of Constraints

#### 1. PRIMARY KEY

```sql
id UUID PRIMARY KEY
```
- Uniquely identifies each row
- Implicitly applies: `NOT NULL` + `UNIQUE`
- Each table should have exactly one primary key

#### 2. NOT NULL

```sql
email TEXT NOT NULL
```
- Field cannot contain NULL
- INSERT/UPDATE fails if value is missing
- **Use this on almost every column** (default is nullable!)

#### 3. UNIQUE

```sql
email TEXT NOT NULL UNIQUE
```
- No two rows can have the same value in this column
- Duplicate insert → database error
- Common for: email, username, slug, phone number

#### 4. FOREIGN KEY (REFERENCES)

```sql
user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE
```
- Value must exist in the referenced table
- Maintains referential integrity across tables
- INSERT of non-existent reference → database error

#### 5. CHECK

```sql
age INTEGER CHECK (age > 0 AND age < 150)
price DECIMAL(10,2) CHECK (price >= 0)
```
- Custom validation rule for a column
- Evaluated on INSERT and UPDATE

#### 6. DEFAULT

```sql
status project_status NOT NULL DEFAULT 'active'
created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
```
- Auto-fills value if not provided during INSERT
- NOT a constraint per se, but a schema feature for convenience

#### 7. UNIQUE (Composite)

```sql
UNIQUE(project_id, user_id)
-- The COMBINATION of project_id + user_id must be unique
-- Same user can be in different projects
-- Same project can have different users
-- But the SAME user in the SAME project? Only once!
```

### Applying Migrations

```bash
# Set up your database URL in .env:
# DATABASE_URL=postgres://user:password@localhost:5432/project_management

# Apply all pending migrations:
dbmate up

# Roll back the last migration:
dbmate down

# Check migration status:
dbmate status
```

## Naming Conventions & Best Practices

### Table Names

```sql
-- ✅ GOOD: Plural, lowercase, snake_case
CREATE TABLE users (...);
CREATE TABLE user_profiles (...);
CREATE TABLE project_members (...);

-- ❌ BAD: Singular
CREATE TABLE user (...);

-- ❌ BAD: CamelCase
CREATE TABLE UserProfiles (...);

-- ❌ BAD: Mixed case
CREATE TABLE User_Profiles (...);
```

**Use plural form** because a table is a collection of multiple rows. The `users` table contains multiple users.

### Column Names

```sql
-- ✅ GOOD: Lowercase, snake_case
full_name TEXT
created_at TIMESTAMPTZ
password_hash TEXT
project_id UUID

-- ❌ BAD: CamelCase
fullName TEXT        -- PostgreSQL treats this as "fullname"
createdAt TIMESTAMPTZ  -- PostgreSQL treats this as "createdat"
```

### Why CamelCase Breaks in PostgreSQL

PostgreSQL is **case-insensitive by default**. It converts all identifiers to lowercase:

```sql
-- You write:
CREATE TABLE UserProfiles (
  fullName TEXT,
  createdAt TIMESTAMPTZ
);

-- PostgreSQL stores it as:
CREATE TABLE userprofiles (
  fullname TEXT,
  createdat TIMESTAMPTZ
);

-- To use actual CamelCase, you MUST quote it:
CREATE TABLE "UserProfiles" (
  "fullName" TEXT,
  "createdAt" TIMESTAMPTZ
);

-- Now EVERY query must use quotes:
SELECT "fullName" FROM "UserProfiles";  -- Tedious!
SELECT fullName FROM UserProfiles;      -- ERROR: column "fullname" not found
```

> **Rule:** Never use CamelCase in PostgreSQL. Always use `snake_case`. You'll avoid quoting headaches and follow the PostgreSQL community convention.

### Standard Fields Every Table Should Have

```sql
-- Every table should have these:
CREATE TABLE any_table (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  -- ... your business fields ...
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

| Field | Purpose |
|-------|---------|
| `id` | Primary key — uniquely identifies each row |
| `created_at` | When was this row first created? (audit trail) |
| `updated_at` | When was this row last modified? (cache invalidation, sorting) |

## Relationship Types — Deep Dive

Understanding relationships is THE core skill of relational database modeling. There are three types:

### 1. One-to-One (1:1) Relationship

**Example:** Each user has exactly ONE profile. Each profile belongs to exactly ONE user.

#### Why Separate Tables?

A common question: "Why not put bio, avatar_url, phone directly in the users table?"

```
The answer comes from thinking about HOW the data changes over time:

users table (core identity):
  → email, password_hash, full_name
  → Changes RARELY (maybe email change once a year)
  → Used in EVERY authentication check
  → Should stay SMALL and FAST

user_profiles table (extended info):
  → bio, avatar_url, phone, social_links, website, skills, location...
  → Changes FREQUENTLY (user updates their profile often)
  → Only fetched when viewing a profile page
  → Can grow over time (add new fields like timezone, language, etc.)
```

**Practical reasons to separate:**

```
1. SEPARATION OF CONCERNS
   → Core auth data (email, password) stays in users
   → Display/profile data stays in user_profiles
   → Different concerns, different tables

2. MIGRATION SAFETY
   → Adding a new profile field = migration to user_profiles (low risk)
   → If it were all in users, every profile change touches the auth table (high risk)
   → The users table is THE most critical table — minimize changes to it

3. PERFORMANCE
   → Auth checks only query the users table (small, fast)
   → Profile views JOIN with user_profiles (only when needed)
   → No wasted columns in auth queries

4. SCALABILITY
   → Profile data can grow to 20+ fields without bloating the users table
   → Each table can be optimized independently
```

#### How to Implement 1:1

The key technique: **make the foreign key ALSO the primary key** in the child table.

```sql
CREATE TABLE user_profiles (
  -- user_id is BOTH the primary key AND the foreign key
  user_id UUID PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
  bio TEXT,
  avatar_url TEXT,
  phone TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

```
Why user_id is PRIMARY KEY (not a separate id):
  → PRIMARY KEY = NOT NULL + UNIQUE
  → UNIQUE means only ONE profile per user (enforces 1:1)
  → No need for a separate id column — user_id uniquely identifies the row
  → Saves storage and simplifies queries

Visual:
users:                        user_profiles:
┌──────────────┐              ┌──────────────────┐
│ id (PK)      │◄─────────── │ user_id (PK + FK)│
│ email        │    1 : 1    │ bio              │
│ full_name    │              │ avatar_url       │
│ password_hash│              │ phone            │
└──────────────┘              └──────────────────┘

One user → one profile. One profile → one user.
```

**Notice:** Fields like `bio`, `avatar_url`, `phone` do NOT have `NOT NULL`. These are optional — a user can choose not to fill them. Only add `NOT NULL` when the field is genuinely required.

### 2. One-to-Many (1:N) Relationship

**Example:** One project has MANY tasks. But each task belongs to exactly ONE project.

```
projects:                     tasks:
┌──────────────┐              ┌─────────────────────┐
│ id: 1        │◄──────┐     │ id: 10              │
│ name: "App"  │       ├─── │ project_id: 1 (FK)  │
└──────────────┘       │     │ title: "Setup DB"   │
                       │     ├─────────────────────┤
                       ├─── │ id: 11              │
                       │     │ project_id: 1 (FK)  │
                       │     │ title: "Build API"  │
                       │     ├─────────────────────┤
                       └─── │ id: 12              │
                             │ project_id: 1 (FK)  │
                             │ title: "Write Tests"│
                             └─────────────────────┘

One project (id: 1) → three tasks (all have project_id: 1)
```

#### How to Implement 1:N

Put the foreign key in the "many" side (tasks), referencing the "one" side (projects):

```sql
CREATE TABLE tasks (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  title TEXT NOT NULL,
  description TEXT,
  status task_status NOT NULL DEFAULT 'pending',
  priority INTEGER NOT NULL DEFAULT 1,

  -- Foreign key to projects (the "one" side)
  project_id UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,

  -- Foreign key to users (who is this task assigned to?)
  assigned_to UUID REFERENCES users(id) ON DELETE SET NULL,

  due_date DATE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),

  -- CHECK constraint: priority must be between 1 and 5
  CHECK (priority >= 1 AND priority <= 5)
);
```

**Key differences from 1:1:**
- The foreign key (`project_id`) is **NOT** the primary key
- The table has its **own separate** `id` as primary key
- `project_id` is NOT unique — many tasks can have the same `project_id` (that's the "many" part)

#### The CHECK Constraint

```sql
CHECK (priority >= 1 AND priority <= 5)
```

This is a custom validation rule at the database level:

```
What it does:
  → When inserting or updating, PostgreSQL checks: is priority between 1 and 5?
  → If YES → operation succeeds
  → If NO → database error, operation rejected

Without CHECK:
  INSERT INTO tasks (priority, ...) VALUES (55, ...);
  -- Succeeds! Now you have a nonsensical priority of 55. ❌

With CHECK:
  INSERT INTO tasks (priority, ...) VALUES (55, ...);
  -- ERROR: new row violates check constraint ✅
  -- Database protects data integrity.
```

> **When to use CHECK:** Whenever a field has a natural range or valid set of values that can be expressed as a condition. Priority (1-5), age (0-150), quantity (>= 0), percentage (0-100).

#### ON DELETE SET NULL vs CASCADE vs RESTRICT

Look at these two foreign keys in the tasks table:

```sql
-- project_id: ON DELETE CASCADE
project_id UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE

-- assigned_to: ON DELETE SET NULL
assigned_to UUID REFERENCES users(id) ON DELETE SET NULL
```

**Why different strategies?**

```
project_id → ON DELETE CASCADE:
  "If a project is deleted, delete ALL its tasks too."
  
  Intuition: Tasks cannot exist without a project.
  An orphan task (no project) makes no sense in our system.
  So when the project goes, its tasks go too.

assigned_to → ON DELETE SET NULL:
  "If the assigned user is deleted, keep the task but unassign it."
  
  Intuition: A task CAN exist without an assignee.
  Maybe the user left the company, but the task still needs to be done.
  So we keep the task, just set assigned_to = NULL (unassigned).
  
  Note: assigned_to does NOT have NOT NULL — it's intentionally nullable
  because a task might not have an assignee yet, or the assignee might 
  be removed.

owner_id (in projects) → ON DELETE RESTRICT:
  "Don't allow deleting a user if they own projects."
  
  Intuition: You MUST deal with the projects first.
  Either transfer ownership or delete the projects.
  Then you can delete the user.
  This prevents accidental deletion of critical relationships.
```

```
Summary of referential integrity strategies:

┌──────────────┬──────────────────────────────────────────────────┐
│ Strategy     │ When to Use                                      │
├──────────────┼──────────────────────────────────────────────────┤
│ CASCADE      │ Child can't exist without parent.                │
│              │ Delete parent → delete children too.             │
│              │ Example: project deleted → tasks deleted         │
├──────────────┼──────────────────────────────────────────────────┤
│ SET NULL     │ Child CAN exist without parent.                  │
│              │ Delete parent → set FK to NULL.                  │
│              │ Example: user deleted → task unassigned          │
├──────────────┼──────────────────────────────────────────────────┤
│ RESTRICT     │ Parent MUST NOT be deleted while children exist. │
│              │ Delete parent → ERROR (blocked).                 │
│              │ Example: user deleted → blocked if they own      │
│              │ projects. Deal with projects first.              │
├──────────────┼──────────────────────────────────────────────────┤
│ SET DEFAULT  │ Delete parent → set FK to a default value.       │
│              │ Rarely used. Only if a sensible default exists.  │
└──────────────┴──────────────────────────────────────────────────┘
```

#### Foreign Key Constraint — The Gatekeeper

```
Because assigned_to REFERENCES users(id):

  ✅ INSERT INTO tasks (assigned_to, ...) VALUES ('valid-user-uuid', ...);
     → PostgreSQL checks: does this UUID exist in users.id? YES → Success

  ❌ INSERT INTO tasks (assigned_to, ...) VALUES ('random-nonexistent-uuid', ...);
     → PostgreSQL checks: does this UUID exist in users.id? NO → ERROR!
     → "insert or update violates foreign key constraint"

This prevents orphan references — you can't assign a task to a user
that doesn't exist. The database catches it even if your application
code has a bug.
```

### 3. Many-to-Many (M:N) Relationship

**Example:** Users can be members of MULTIPLE projects. Projects can have MULTIPLE users.

```
How to identify a many-to-many relationship:

Ask the question BOTH ways:
  → "Can a user be part of multiple projects?" → YES
  → "Can a project have multiple users?" → YES

If BOTH answers are YES → many-to-many.

Compare with one-to-many:
  → "Can a project have multiple tasks?" → YES
  → "Can a task belong to multiple projects?" → NO (only one)
  → Only one YES → one-to-many.
```

#### The Linking Table Pattern

You **cannot** directly represent many-to-many in a relational database. You need an intermediate table (called a **linking table**, **junction table**, or **bridge table**):

```
users:              project_members:           projects:
┌────────────┐      ┌─────────────────────┐    ┌────────────┐
│ id: 1      │◄────│ user_id: 1          │───►│ id: 1      │
│ Alice      │     │ project_id: 1       │    │ "App"      │
└────────────┘     │ role: owner         │    └────────────┘
                   ├─────────────────────┤
┌────────────┐     │ user_id: 1          │    ┌────────────┐
│ id: 2      │◄───│ project_id: 2       │───►│ id: 2      │
│ Bob        │     │ role: member        │    │ "Website"  │
└────────────┘     ├─────────────────────┤    └────────────┘
                   │ user_id: 2          │
                   │ project_id: 1       │
                   │ role: admin         │
                   ├─────────────────────┤
                   │ user_id: 3          │
                   │ project_id: 1       │
                   │ role: member        │
                   └─────────────────────┘

Reading the linking table:
  → Alice (1) is OWNER of App (1)
  → Alice (1) is MEMBER of Website (2)    ← Alice is in 2 projects
  → Bob (2) is ADMIN of App (1)           ← App has 3 users
  → User 3 is MEMBER of App (1)
```

#### Implementing with Composite Primary Key

```sql
CREATE TABLE project_members (
  -- Two foreign keys from BOTH tables
  project_id UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,

  -- Extra data about this relationship
  role member_role NOT NULL DEFAULT 'member',
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),

  -- COMPOSITE PRIMARY KEY: combination of both foreign keys
  PRIMARY KEY (project_id, user_id)
);
```

#### Why Composite Primary Key?

```
Instead of creating a separate id column, we use the COMBINATION of
project_id + user_id as the primary key.

What this means:
  → PRIMARY KEY = NOT NULL + UNIQUE (for the combination)
  → The SAME combination can only appear ONCE

  ✅ (project: 1, user: 1) → Alice is in Project App
  ✅ (project: 2, user: 1) → Alice is in Project Website  (different project, OK)
  ✅ (project: 1, user: 2) → Bob is in Project App        (different user, OK)
  ❌ (project: 1, user: 1) → DUPLICATE! Alice already in Project App → ERROR

This prevents duplicate memberships. A user cannot be added to the
same project twice. The database enforces this constraint.
```

#### Why `role` Is in the Linking Table

```
The role (owner, admin, member) is specific to the RELATIONSHIP,
not to the user or the project independently.

  → Alice might be OWNER of Project App, but MEMBER of Project Website
  → The role depends on the COMBINATION of user + project
  → That's why role belongs in the linking table

If role were in the users table:
  → Alice would have ONE role for ALL projects → wrong!
  → She needs different roles for different projects

If role were in the projects table:
  → The project would have ONE role for ALL users → wrong!
  → Different users need different roles in the same project
```

#### Relationship Implementation Summary

```
┌─────────────┬──────────────────────────────────────────────────────┐
│ Relationship│ How to Implement                                     │
├─────────────┼──────────────────────────────────────────────────────┤
│ ONE-TO-ONE  │ Foreign key IS the primary key in child table.       │
│             │ user_id UUID PRIMARY KEY REFERENCES users(id)        │
│             │ → Guarantees exactly one child per parent.           │
├─────────────┼──────────────────────────────────────────────────────┤
│ ONE-TO-MANY │ Foreign key in the "many" table, NOT the PK.        │
│             │ project_id UUID NOT NULL REFERENCES projects(id)     │
│             │ → Many rows can share the same project_id.          │
├─────────────┼──────────────────────────────────────────────────────┤
│ MANY-TO-MANY│ Create a linking table with TWO foreign keys.        │
│             │ Use composite primary key: PRIMARY KEY (fk1, fk2)    │
│             │ → Each unique combination appears exactly once.     │
└─────────────┴──────────────────────────────────────────────────────┘
```

## Down Migrations — Reverting Changes

Every up migration needs a corresponding down migration to enable rollbacks:

```sql
-- +migrate Down

-- Drop tables in REVERSE order (respect foreign key dependencies)
DROP TABLE IF EXISTS project_members;  -- References projects AND users
DROP TABLE IF EXISTS tasks;            -- References projects AND users
DROP TABLE IF EXISTS projects;         -- References users
DROP TABLE IF EXISTS user_profiles;    -- References users
DROP TABLE IF EXISTS users;            -- No dependencies

-- Drop custom types AFTER tables that use them
DROP TYPE IF EXISTS member_role;
DROP TYPE IF EXISTS task_status;
DROP TYPE IF EXISTS project_status;
```

**Order matters!** You must drop tables in reverse dependency order:

```
Creation order:        Drop order:
1. enum types          5. project_members    ← depends on projects + users
2. users               4. tasks              ← depends on projects + users
3. user_profiles       3. projects           ← depends on users
4. projects            2. user_profiles      ← depends on users
5. tasks               1. users              ← no dependencies
6. project_members     0. enum types         ← used by tables above

If you try to drop users BEFORE tasks:
  → PostgreSQL ERROR: cannot drop table users because tasks depends on it
  → Foreign key constraints BLOCK the drop
```

## Database Seeding — Test Data for Development

### What Is Seeding?

**Seeding** = inserting test/sample data into your database for development and testing purposes.

```
In production:
  → Data comes from real users (sign-ups, form submissions)
  → Tables populate naturally over time

In development:
  → You just created empty tables
  → You need data to test your APIs, UI, queries
  → Manually inserting data each time is tedious
  → Solution: Write a seed migration that inserts test data
```

### Creating a Seed Migration

```bash
# Create a separate migration file for seed data
dbmate new seed_data
# Creates: db/migrations/20240315150000_seed_data.sql
```

### Seed Data Example

```sql
-- +migrate Up

-- Use CTEs (Common Table Expressions) for readable, chained inserts
WITH inserted_users AS (
  INSERT INTO users (email, full_name, password_hash) VALUES
    ('alice@example.com', 'Alice Johnson', '$2b$10$hashedpassword1'),
    ('bob@example.com', 'Bob Smith', '$2b$10$hashedpassword2'),
    ('charlie@example.com', 'Charlie Brown', '$2b$10$hashedpassword3')
  RETURNING id, email  -- Return the auto-generated IDs for use below
),

-- Insert profiles using the IDs we just got back
inserted_profiles AS (
  INSERT INTO user_profiles (user_id, avatar_url, bio, phone)
  SELECT
    id,
    'https://example.com/avatar.jpg',
    CASE
      WHEN email = 'alice@example.com' THEN 'Full-stack developer'
      WHEN email = 'bob@example.com' THEN 'Backend specialist'
      ELSE 'Frontend developer'
    END,
    '+1-555-0100'
  FROM inserted_users
  RETURNING user_id
),

-- Insert projects owned by the first user
inserted_projects AS (
  INSERT INTO projects (name, description, status, owner_id)
  SELECT
    'Project Alpha',
    'Main application project',
    'active',
    id
  FROM inserted_users
  WHERE email = 'alice@example.com'
  RETURNING id AS project_id
)

-- Add all users as project members
INSERT INTO project_members (project_id, user_id, role)
SELECT
  p.project_id,
  u.id,
  CASE
    WHEN u.email = 'alice@example.com' THEN 'owner'::member_role
    WHEN u.email = 'bob@example.com' THEN 'admin'::member_role
    ELSE 'member'::member_role
  END
FROM inserted_projects p
CROSS JOIN inserted_users u;

-- +migrate Down
DELETE FROM project_members;
DELETE FROM tasks;
DELETE FROM projects;
DELETE FROM user_profiles;
DELETE FROM users;
```

#### Understanding the CTE Pattern

```
CTEs (Common Table Expressions) — WITH ... AS (...):

  WITH inserted_users AS (
    INSERT INTO users (...) VALUES (...)
    RETURNING id, email       ← Give me back the auto-generated IDs
  ),
  inserted_profiles AS (
    INSERT INTO user_profiles (...)
    SELECT ... FROM inserted_users  ← Use those IDs here!
  )

Why CTEs?
  → Users get auto-generated UUIDs
  → We need those UUIDs to create profiles, projects, memberships
  → CTEs let us chain inserts: insert users → get IDs → insert profiles using those IDs
  → All in a single SQL statement (atomic — either all succeed or none)
```

After applying the seed migration with `dbmate up`, all tables have data to test with.

## Writing Queries — Fetching Data with JOINs

Now that we have tables and data, let's write the actual queries that our backend APIs will use.

### Query 1: Get All Users (with Profile Data)

**API endpoint:** `GET /api/v1/users`

**Requirement:** Return all users along with their profile information in a single API call. The frontend shouldn't need a separate call to fetch profile data.

#### Building the Query Step by Step

**Pro tip:** When writing SQL queries, always **start from the FROM clause**, not SELECT. Know WHERE your data is coming from first, then decide WHAT to select.

```sql
-- Step 1: Start with FROM — where is the data?
FROM users u

-- Step 2: We also need profile data → JOIN with user_profiles
FROM users u
LEFT JOIN user_profiles up ON u.id = up.user_id

-- Step 3: Now SELECT what we want
SELECT
  u.*,                        -- All columns from users
  to_jsonb(up.*) AS profile   -- All columns from user_profiles, as JSON
FROM users u
LEFT JOIN user_profiles up ON u.id = up.user_id

-- Step 4: Add ordering (without ORDER BY, results are in RANDOM order!)
SELECT
  u.*,
  to_jsonb(up.*) AS profile
FROM users u
LEFT JOIN user_profiles up ON u.id = up.user_id
ORDER BY u.created_at DESC;  -- Most recently created first
```

#### Why LEFT JOIN (Not INNER JOIN)?

```
INNER JOIN:
  → Returns rows ONLY when BOTH tables have matching data
  → If a user has NO profile → that user is EXCLUDED from results
  → You'd get 3 users back when you actually have 5 ❌

LEFT JOIN:
  → Returns ALL rows from the LEFT table (users)
  → If there's a matching row in user_profiles → include it
  → If there's NO matching row → profile is NULL, but user still appears
  → You always get ALL users, regardless of profile status ✅
```

```
Visual:

users:           user_profiles:
┌─────────┐      ┌──────────────────┐
│ id: 1   │──────│ user_id: 1, bio  │  ← Match!
│ id: 2   │──────│ user_id: 2, bio  │  ← Match!
│ id: 3   │      │                  │  ← No profile!
└─────────┘      └──────────────────┘

INNER JOIN result:        LEFT JOIN result:
┌─────────┬─────────┐    ┌─────────┬─────────┐
│ user 1  │ profile │    │ user 1  │ profile │
│ user 2  │ profile │    │ user 2  │ profile │
└─────────┴─────────┘    │ user 3  │ NULL    │  ← Still included!
  (user 3 is MISSING!)   └─────────┴─────────┘
```

> **Rule of thumb:** Use `LEFT JOIN` when you want ALL records from the main table regardless of whether related data exists. Use `INNER JOIN` when you only want records that have matching data in BOTH tables. For most "get all" APIs, `LEFT JOIN` is correct.

#### Understanding `to_jsonb()`

```sql
to_jsonb(up.*) AS profile
```

This is a PostgreSQL-specific function that converts a row into a JSONB object:

```
What to_jsonb(up.*) does:

  Input (row from user_profiles):
    user_id: 'abc-123'
    bio: 'Full-stack developer'
    avatar_url: 'https://example.com/avatar.jpg'
    phone: '+1-555-0100'

  Output (JSONB object):
    {
      "user_id": "abc-123",
      "bio": "Full-stack developer",
      "avatar_url": "https://example.com/avatar.jpg",
      "phone": "+1-555-0100"
    }

Why do we convert to JSON?
  → Our API returns JSON to the frontend
  → Profile data is nested INSIDE the user object
  → to_jsonb() creates that nested structure at the database level
  → Less processing needed in application code
```

#### Why ORDER BY?

```
WITHOUT ORDER BY:
  → PostgreSQL returns rows in WHATEVER order it finds them
  → This order can change between queries (no guarantees!)
  → Users see items jumping around randomly in the UI ❌

WITH ORDER BY u.created_at DESC:
  → Most recently created users appear first
  → Consistent, predictable ordering ✅
  → DESC = descending (newest first)
  → ASC = ascending (oldest first) — this is the default
```

> **Rule:** ALWAYS add `ORDER BY` to any query that returns a list. Without it, the order is undefined and unpredictable. For most lists, ordering by `created_at DESC` (newest first) is a good default.

### Query 2: Get a Single User by ID

**API endpoint:** `GET /api/v1/users/:userId`

**Requirement:** Given a user ID (from the URL parameter), return that specific user with their profile.

#### Parameterized Queries — Preventing SQL Injection

Before writing the query, we need to understand **parameterized queries** — one of the most important security concepts in database interactions.

**The Problem — SQL Injection:**

```
Imagine your backend constructs the query by concatenating strings:

// ❌ DANGEROUS — string concatenation
const query = `SELECT * FROM users WHERE id = '${userIdFromUrl}'`;

What if the attacker sends this as the userId?
  userId = "'; DROP TABLE users; --"

The query becomes:
  SELECT * FROM users WHERE id = ''; DROP TABLE users; --'

PostgreSQL executes TWO statements:
  1. SELECT * FROM users WHERE id = ''     ← Returns nothing (harmless)
  2. DROP TABLE users                       ← DELETES YOUR ENTIRE TABLE! 💀
  -- The -- comments out the remaining quote

This is SQL injection — one of the most common and devastating attacks.
```

**The Solution — Parameterized Queries:**

```sql
-- ✅ SAFE — parameterized query
SELECT * FROM users WHERE id = $1;
-- $1 is a PLACEHOLDER (slot)
-- The actual value is passed SEPARATELY from the query
```

```
How parameterized queries work:

1. You send the QUERY TEMPLATE to PostgreSQL:
   "SELECT * FROM users WHERE id = $1"

2. You send the PARAMETER VALUES separately:
   [$1 = "abc-123-def-456"]

3. PostgreSQL KNOWS that $1 is just a VALUE, never SQL code
   → Even if someone sends "'; DROP TABLE users; --"
   → It's treated as a LITERAL STRING, not SQL
   → PostgreSQL looks for a user with id = "'; DROP TABLE users; --"
   → Finds nothing. Returns empty result. Database is SAFE. ✅
```

```
In your backend code (Node.js example):

// ❌ WRONG — string concatenation (vulnerable to SQL injection)
const query = `SELECT * FROM users WHERE id = '${userId}'`;
db.query(query);

// ✅ CORRECT — parameterized query (safe)
const query = 'SELECT * FROM users WHERE id = $1';
db.query(query, [userId]);  // userId passed as parameter, not in the query string
```

> **Rule:** NEVER construct SQL queries by concatenating user input. ALWAYS use parameterized queries. Every database driver and ORM supports them. There is zero reason not to use them.

#### The Single User Query

```sql
SELECT
  u.*,
  to_jsonb(up.*) AS profile
FROM users u
LEFT JOIN user_profiles up ON u.id = up.user_id
WHERE u.id = $1;  -- $1 = the userId parameter
```

The only difference from the "get all users" query is the `WHERE` clause that filters by ID.

```
How this flows in a backend system:

1. Frontend makes request: GET /api/v1/users/abc-123-def-456

2. Router extracts the dynamic parameter: userId = "abc-123-def-456"

3. Controller passes it to the service layer

4. Service calls the repository with the userId

5. Repository constructs the parameterized query:
   query: "SELECT u.*, to_jsonb(up.*) AS profile
           FROM users u
           LEFT JOIN user_profiles up ON u.id = up.user_id
           WHERE u.id = $1"
   params: ["abc-123-def-456"]

6. Database executes safely, returns ONE row

7. Result is serialized to JSON and sent to the frontend
```

### Dynamic Filtering and Sorting

**Common API pattern:** `GET /api/v1/users?sort=email&order=asc&search=alice`

Most "list" APIs need to support:
- **Dynamic sorting** — sort by any column (name, email, date)
- **Dynamic ordering** — ascending or descending
- **Dynamic filtering** — search by name, filter by status, etc.

#### The Challenge

You can't easily represent dynamic queries in a SQL editor — this is where your **application code** and **query builder** come in.

#### How It Works in Practice

```javascript
// In your backend code (conceptual example):

async function getUsers({ sort, order, search, status }) {
  let query = `
    SELECT u.*, to_jsonb(up.*) AS profile
    FROM users u
    LEFT JOIN user_profiles up ON u.id = up.user_id
  `;

  const params = [];
  const conditions = [];

  // Dynamic filtering
  if (search) {
    params.push(`%${search}%`);
    conditions.push(`u.full_name ILIKE $${params.length}`);
    // ILIKE = case-insensitive search
    // %search% = contains the search term anywhere in the name
  }

  if (status) {
    params.push(status);
    conditions.push(`u.status = $${params.length}`);
  }

  // Apply WHERE clause if any conditions exist
  if (conditions.length > 0) {
    query += ` WHERE ${conditions.join(' AND ')}`;
  }

  // Dynamic sorting (with whitelist for safety!)
  const allowedSortColumns = ['full_name', 'email', 'created_at'];
  const sortColumn = allowedSortColumns.includes(sort) ? sort : 'created_at';
  const sortOrder = order === 'asc' ? 'ASC' : 'DESC';
  query += ` ORDER BY u.${sortColumn} ${sortOrder}`;

  return db.query(query, params);
}
```

#### Why Whitelist Sort Columns?

```
// ❌ DANGEROUS — using user input directly in ORDER BY:
query += ` ORDER BY ${req.query.sort}`;
// User could send: sort=id; DROP TABLE users; --

// ✅ SAFE — whitelist approach:
const allowedColumns = ['full_name', 'email', 'created_at'];
const sortColumn = allowedColumns.includes(sort) ? sort : 'created_at';
// Only allow predefined column names
// Anything else defaults to 'created_at'
```

> **Important:** Parameterized queries protect VALUES (`WHERE id = $1`), but they do NOT protect column names or table names in `ORDER BY` or `FROM` clauses. For those, you MUST use whitelisting.

#### The SQL That Gets Generated

```sql
-- User requests: GET /api/v1/users?search=alice&sort=email&order=asc

-- Generated query:
SELECT u.*, to_jsonb(up.*) AS profile
FROM users u
LEFT JOIN user_profiles up ON u.id = up.user_id
WHERE u.full_name ILIKE $1        -- $1 = '%alice%'
ORDER BY u.email ASC;

-- User requests: GET /api/v1/users?sort=created_at&order=desc

-- Generated query (no search filter):
SELECT u.*, to_jsonb(up.*) AS profile
FROM users u
LEFT JOIN user_profiles up ON u.id = up.user_id
ORDER BY u.created_at DESC;
```

## Pagination — OFFSET and LIMIT

### Why Paginate?

When you have thousands or millions of rows, you **cannot** return all of them in a single API call:

```
Without pagination:
  GET /api/v1/users → Returns ALL 500,000 users
  → Huge database load (reading 500K rows)
  → Massive network transfer (megabytes of JSON)
  → Frontend crashes trying to render 500K items
  → Terrible user experience

With pagination:
  GET /api/v1/users?page=1&limit=20 → Returns 20 users
  GET /api/v1/users?page=2&limit=20 → Returns next 20 users
  → Small, fast database queries
  → Tiny network payloads
  → Smooth UI rendering
```

> **Rule:** ALWAYS paginate list APIs. Unless you have a very specific reason (like a dropdown with at most 50 items), every API that returns a list should support pagination.

### How OFFSET and LIMIT Work

```sql
SELECT * FROM users
ORDER BY created_at DESC
LIMIT 20      -- Return at most 20 rows
OFFSET 40;    -- Skip the first 40 rows
```

```
LIMIT = how many rows to return (page size)
OFFSET = how many rows to skip before starting to return

Page 1: OFFSET 0, LIMIT 20  → rows 1-20
Page 2: OFFSET 20, LIMIT 20 → rows 21-40
Page 3: OFFSET 40, LIMIT 20 → rows 41-60
Page 4: OFFSET 60, LIMIT 20 → rows 61-80

Formula: OFFSET = (page - 1) × limit
```

### Page Calculation in Backend Code

```
User sends:     page=3, limit=20
Backend computes: offset = (3 - 1) * 20 = 40

Important distinction:
  → User thinks in pages: page 1, page 2, page 3...
  → Database thinks in offsets: skip 0, skip 20, skip 40...
  → Your backend code converts between the two:
    offset = (page - 1) × limit
```

```sql
-- The parameterized query:
SELECT u.*, to_jsonb(up.*) AS profile
FROM users u
LEFT JOIN user_profiles up ON u.id = up.user_id
ORDER BY u.created_at DESC
LIMIT $1 OFFSET $2;

-- $1 = limit (from query param, default 20)
-- $2 = offset (calculated: (page - 1) * limit)
```

## Putting It All Together — Dynamic Query Construction

Let's see how a real "Get All Users" API combines filtering, sorting, AND pagination into a single dynamic query:

### The API Contract

```
GET /api/v1/users?letter=J&sort_by=email&sort_order=asc&page=1&limit=10

Query Parameters:
  letter     → Filter: users whose name starts with this letter (optional)
  sort_by    → Sort by: full_name, email, or created_at (default: created_at)
  sort_order → Order: asc or desc (default: desc)
  page       → Page number (default: 1)
  limit      → Items per page (default: 10)
```

### The Defaults

```
If user sends NO query parameters:
  → letter: null (no filter, return all users)
  → sort_by: created_at
  → sort_order: desc
  → page: 1
  → limit: 10

Result: First 10 users, sorted by newest first, no filtering.
```

### The Filter: ILIKE with Pattern Matching

```sql
-- Filter by first letter of name:
WHERE u.full_name ILIKE $1 || '%'

-- How it works:
-- $1 = 'J'
-- $1 || '%' = 'J%' (J followed by anything)
-- ILIKE = case-insensitive matching

-- So ILIKE 'J%' matches:
--   'John Doe'    ✅ (starts with J)
--   'jane Smith'  ✅ (starts with j — ILIKE is case-insensitive)
--   'Alice Brown' ❌ (doesn't start with J)
```

```
The % wildcard:
  'J%'   → Starts with J, anything after       → "John", "Jane", "Jim"
  '%son' → Ends with "son", anything before     → "Johnson", "Jackson"
  '%oh%' → Contains "oh" anywhere               → "John", "Mohit"

LIKE vs ILIKE:
  LIKE  → Case-sensitive   → 'j%' matches 'jane' but NOT 'Jane'
  ILIKE → Case-insensitive → 'j%' matches 'jane' AND 'Jane'
```

### The Complete Constructed Query

```sql
-- When user sends: ?letter=J&sort_by=email&sort_order=asc&page=1&limit=10

SELECT
  u.*,
  to_jsonb(up.*) AS profile
FROM users u
LEFT JOIN user_profiles up ON u.id = up.user_id
WHERE u.full_name ILIKE $1 || '%'    -- Filter: names starting with 'J'
ORDER BY u.email ASC                  -- Sort: by email, ascending
LIMIT $2 OFFSET $3;                   -- Pagination: 10 items, page 1

-- Parameters: $1 = 'J', $2 = 10, $3 = 0

-- When user sends: ?page=2&limit=10 (no filter, no custom sort)

SELECT
  u.*,
  to_jsonb(up.*) AS profile
FROM users u
LEFT JOIN user_profiles up ON u.id = up.user_id
ORDER BY u.created_at DESC           -- Default sort
LIMIT $1 OFFSET $2;                   -- Pagination: 10 items, page 2

-- Parameters: $1 = 10, $2 = 10
-- (No WHERE clause because no filter was provided)
```

## Query 3: Create a User (INSERT)

**API endpoint:** `POST /api/v1/users`

**Requirement:** Create a new user with email, full_name, and password (hashed in backend code before reaching the database).

```sql
INSERT INTO users (email, full_name, password_hash)
VALUES ($1, $2, $3)
RETURNING *;
```

### Breaking It Down

```
INSERT INTO users    → Which table to insert into
(email, full_name, password_hash) → Which columns we're providing values for
VALUES ($1, $2, $3)  → The actual values (parameterized)
RETURNING *          → Return the newly created row back to us

-- $1 = 'test@gmail.com'
-- $2 = 'Test User'
-- $3 = '$2b$10$hashedpasswordstring...'
```

### Why RETURNING *?

```
Without RETURNING:
  INSERT INTO users (...) VALUES (...);
  → Row is inserted. But you get NO information back.
  → You don't know the auto-generated UUID.
  → You'd need a SECOND query to fetch it: SELECT * FROM users WHERE email = '...';
  → Two database calls instead of one.

With RETURNING *:
  INSERT INTO users (...) VALUES (...) RETURNING *;
  → Row is inserted AND returned in ONE operation.
  → You immediately get the id, created_at, and all other fields.
  → Send this directly to the frontend as the API response.
  → One database call. Efficient. ✅
```

### What About Fields We Didn't Provide?

```
We only provided: email, full_name, password_hash

What about id, created_at, updated_at?
  → id: DEFAULT gen_random_uuid() → PostgreSQL generates it automatically
  → created_at: DEFAULT NOW() → PostgreSQL sets to current timestamp
  → updated_at: DEFAULT NOW() → PostgreSQL sets to current timestamp

That's the power of DEFAULT values — you don't need to provide them.
The database fills them in automatically.
```

## Query 4: Update a User Profile (UPDATE)

**API endpoint:** `PATCH /api/v1/users/:userId/profile`

**Requirement:** Update specific fields of a user's profile. The user can send any combination of fields — only update what they send, leave everything else unchanged.

### Why PATCH (Not PUT)?

```
PUT  = Replace the ENTIRE resource with new data
       → You must send ALL fields, even unchanged ones
       → Missing fields are treated as "set to null"

PATCH = Partially update the resource
        → Send ONLY the fields you want to change
        → Unmentioned fields stay unchanged

For profile updates, PATCH is correct:
  → User only wants to update their bio? Send just { bio: "new bio" }
  → Phone and avatar_url stay unchanged
```

### Partial Update Logic

```
The frontend sends:
{
  "bio": "Updated bio text",
  "phone": "+1-555-9999"
}

Note: avatar_url is NOT included — don't touch it!

In your backend code:
1. Check which fields were sent
2. Only include those fields in the UPDATE query
3. Construct the query dynamically

The resulting SQL:
UPDATE user_profiles
SET bio = $1, phone = $2
WHERE user_id = $3
RETURNING *;

-- $1 = 'Updated bio text'
-- $2 = '+1-555-9999'
-- $3 = 'abc-123-user-uuid'

-- avatar_url is NOT in the SET clause → stays unchanged ✅
```

### The Full Query

```sql
UPDATE user_profiles
SET
  bio = $1,          -- Only fields the user actually sent
  phone = $2
WHERE user_id = $3   -- Which user's profile to update
RETURNING *;         -- Return the updated row
```

```
What happens:
1. PostgreSQL finds the row WHERE user_id = $3
2. Updates ONLY bio and phone (avatar_url untouched)
3. Returns the full updated row (including unchanged fields)
4. Backend sends this as the API response
```

## The `updated_at` Problem — Database Triggers

### The Problem

Look at this scenario:

```
After updating a user's profile:

user_profiles:
┌──────────────┬──────────────────┬──────────────────────┐
│ bio          │ phone            │ updated_at           │
├──────────────┼──────────────────┼──────────────────────┤
│ "Updated bio"│ "+1-555-9999"    │ 2024-03-15 14:30:00  │ ← WRONG!
└──────────────┴──────────────────┴──────────────────────┘

The bio and phone were updated just now, but updated_at still shows
the original time from when the row was first created.
That's because we didn't explicitly update it!
```

### Solution 1: Manual Update (Tedious)

```sql
-- Add updated_at to every UPDATE query manually:
UPDATE user_profiles
SET bio = $1, phone = $2, updated_at = NOW()  -- ← Must remember this EVERY time
WHERE user_id = $3
RETURNING *;
```

**Problems with manual approach:**
- You must remember to add `updated_at = NOW()` to EVERY update query
- If you forget once, the timestamp is wrong
- Across dozens of tables and hundreds of queries, forgetting is inevitable
- More code to write and maintain

### Solution 2: Database Triggers (Automatic) ✅

A **trigger** is a function that runs automatically when a specific event occurs in the database.

```
The idea:
  → "Every time ANY row in ANY table is updated,
     automatically set updated_at to the current timestamp."
  → No manual work needed. No risk of forgetting.
  → The DATABASE handles it, not your application code.
```

#### Step 1: Create the Trigger Function

```sql
-- Create a reusable function that sets updated_at to NOW()
CREATE OR REPLACE FUNCTION update_timestamp()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

```
What this function does:

  CREATE OR REPLACE FUNCTION update_timestamp()
    → Creates a new function (or replaces if it already exists)

  RETURNS TRIGGER
    → This function is designed to be used as a trigger

  BEGIN ... END
    → The function body (a transaction block)

  NEW.updated_at = NOW()
    → NEW is the row AFTER the update
    → Set its updated_at field to the current timestamp

  RETURN NEW
    → Return the modified row (with updated timestamp)
    → This becomes the actual data that gets saved

  LANGUAGE plpgsql
    → Written in PL/pgSQL (PostgreSQL's procedural language)
```

#### Step 2: Attach Triggers to Tables

```sql
-- Trigger for users table
CREATE TRIGGER set_updated_at_users
  BEFORE UPDATE ON users
  FOR EACH ROW
  EXECUTE FUNCTION update_timestamp();

-- Trigger for user_profiles table
CREATE TRIGGER set_updated_at_user_profiles
  BEFORE UPDATE ON user_profiles
  FOR EACH ROW
  EXECUTE FUNCTION update_timestamp();

-- Trigger for projects table
CREATE TRIGGER set_updated_at_projects
  BEFORE UPDATE ON projects
  FOR EACH ROW
  EXECUTE FUNCTION update_timestamp();

-- Trigger for tasks table
CREATE TRIGGER set_updated_at_tasks
  BEFORE UPDATE ON tasks
  FOR EACH ROW
  EXECUTE FUNCTION update_timestamp();
```

```
Breaking down the trigger syntax:

  CREATE TRIGGER set_updated_at_users
    → Name the trigger (for referencing in DROP later)

  BEFORE UPDATE ON users
    → Run this BEFORE any UPDATE operation on the users table
    → BEFORE means: modify the data before it's saved
    → (AFTER would run after saving — too late to change updated_at)

  FOR EACH ROW
    → Run the trigger for EVERY row that's being updated
    → (Not just once per query, but once per affected row)

  EXECUTE FUNCTION update_timestamp()
    → Call our function that sets updated_at = NOW()
```

#### How It Works Now

```
BEFORE trigger setup:
  → Any UPDATE on users/user_profiles/projects/tasks
  → AUTOMATICALLY sets updated_at = NOW()
  → No manual work needed in your queries

Your UPDATE query can be simple:
  UPDATE user_profiles SET bio = $1 WHERE user_id = $2;
  → The trigger AUTOMATICALLY adds: updated_at = NOW()
  → You DON'T need to include it in your SET clause

Result:
  updated_at is ALWAYS accurate. Zero maintenance. Zero risk of forgetting.
```

## Database Indexes — Deep Dive

### What Is an Index?

An index is a **lookup table** that helps the database find data faster, similar to an index at the back of a book.

### The Book Analogy

```
Book without index:
  "Find where Chapter 4 starts"
  → Open the book
  → Go page by page: page 1... page 2... page 3...
  → Keep going until you find "Chapter 4" heading
  → Finally found it on page 54 (after scanning 54 pages!)

Book with index:
  "Find where Chapter 4 starts"
  → Open the index at the back
  → Look up: Chapter 4 → Page 54
  → Go directly to page 54
  → Done! (Looked at 1 page in the index, then jumped directly)
```

### How It Works in Databases

```
WITHOUT index (Sequential Scan):

  Query: SELECT * FROM tasks WHERE id = 'abc-123';

  Database process:
  ┌──────────────────────────────────────────────┐
  │ Task stored at disk location A               │
  │ → Read task, check: is id = 'abc-123'? NO    │
  │                                               │
  │ Task stored at disk location B               │
  │ → Read task, check: is id = 'abc-123'? NO    │
  │                                               │
  │ Task stored at disk location C               │
  │ → Read task, check: is id = 'abc-123'? NO    │
  │                                               │
  │ Task stored at disk location D               │
  │ → Read task, check: is id = 'abc-123'? YES!  │
  │ → Return this task.                           │
  │                                               │
  │ ... but must continue checking remaining rows │
  │ ... because there MIGHT be more matches       │
  └──────────────────────────────────────────────┘

  → Scans EVERY row in the table
  → O(n) — gets slower as table grows
  → With 1 million rows → scans 1 million locations 💀


WITH index (Index Lookup):

  The index is a separate, organized data structure:
  ┌────────────────────────────────────┐
  │         INDEX on tasks.id          │
  │                                    │
  │  id: 'aaa-111' → Disk Location A  │
  │  id: 'abc-123' → Disk Location D  │  ← Found it!
  │  id: 'def-456' → Disk Location B  │
  │  id: 'xyz-789' → Disk Location C  │
  └────────────────────────────────────┘

  Database process:
  1. Look up 'abc-123' in the index
  2. Index says: "It's at Disk Location D"
  3. Go directly to Location D
  4. Return the task

  → Uses organized lookup (B-tree)
  → O(log n) — stays fast even with millions of rows
  → With 1 million rows → ~20 lookups instead of 1 million ✅
```

### Two Properties of Indexes

```
Property 1: DIRECT LOOKUP
  → Instead of scanning all rows, go directly to the right one
  → Like looking up a word in a dictionary vs reading every page

Property 2: ORDERING
  → Index entries are stored in a specific order (ASC or DESC)
  → If your query needs sorted results, the index provides them pre-sorted
  → No need to sort at query time — already sorted in the index

Example:
  Index on created_at DESC:
  ┌─────────────────────────────────────┐
  │ 2024-03-15 → Location A            │  ← Newest
  │ 2024-03-14 → Location C            │
  │ 2024-03-13 → Location B            │
  │ 2024-03-12 → Location D            │  ← Oldest
  └─────────────────────────────────────┘

  Query: SELECT * FROM users ORDER BY created_at DESC;
  → Database reads the index top-to-bottom
  → Results are ALREADY in the right order
  → No sorting step needed → much faster
```

### When to Create Indexes

```
The THREE situations where indexes help:

1. JOIN CONDITIONS
   → LEFT JOIN user_profiles up ON u.id = up.user_id
   → Index user_profiles.user_id for faster joins

2. WHERE CLAUSES
   → WHERE status = 'pending'
   → Index tasks.status for faster filtering

3. ORDER BY / SORTING
   → ORDER BY created_at DESC
   → Index users.created_at DESC for pre-sorted results
```

### Primary Keys Are Automatically Indexed

```
Important: PostgreSQL AUTOMATICALLY creates an index on PRIMARY KEY columns.

  CREATE TABLE users (
    id UUID PRIMARY KEY  ← Automatically indexed!
  );

You NEVER need to manually index a primary key.

But FOREIGN KEYS are NOT automatically indexed!

  CREATE TABLE tasks (
    project_id UUID REFERENCES projects(id)  ← NOT automatically indexed!
  );

You SHOULD manually index foreign keys that appear in JOIN conditions.
```

### The Index Maintenance Tradeoff

Indexes are **not free**. They have a cost:

```
The tradeoff:

  READ operations (SELECT, JOIN):
    → With index: MUCH FASTER ✅
    → This is why we create indexes

  WRITE operations (INSERT, UPDATE, DELETE):
    → With index: SLIGHTLY SLOWER ❌
    → Every time you insert/update/delete a row,
      the index must ALSO be updated to stay accurate
    → More indexes = more maintenance overhead on writes

Think of it like a book's index:
  → Reading: Having an index makes finding chapters FAST
  → Writing: If you add a new chapter, you must ALSO update the index
  → If you have 20 indexes in the book, adding a chapter means
    updating 20 different index entries
```

```
When to create an index:
  ✅ The field is used in JOIN conditions frequently
  ✅ The field is used in WHERE clauses frequently
  ✅ The field is used in ORDER BY frequently
  ✅ The table has many rows (thousands+)
  ✅ The query is called frequently (many times per second)

When NOT to create an index:
  ❌ The table has very few rows (index overhead > benefit)
  ❌ The query is rarely called
  ❌ The table has heavy write traffic and light read traffic
  ❌ You're indexing every single column (too much write overhead)

Rule of thumb:
  → Start with indexes on foreign keys and frequently filtered/sorted fields
  → Monitor query performance
  → Add more indexes if needed, remove unused ones
```

### Creating Indexes — Migration Example

```sql
-- +migrate Up

-- ============================================================
-- INDEXES
-- ============================================================

-- Index on users.email (used in WHERE: find user by email)
CREATE INDEX idx_users_email ON users(email);

-- Index on users.created_at DESC (used in ORDER BY: list users newest first)
CREATE INDEX idx_users_created_at ON users(created_at DESC);

-- Index on tasks.project_id (FOREIGN KEY used in JOIN: get all tasks of a project)
CREATE INDEX idx_tasks_project_id ON tasks(project_id);

-- Index on tasks.assigned_to (FOREIGN KEY used in JOIN: get all tasks of a user)
CREATE INDEX idx_tasks_assigned_to ON tasks(assigned_to);

-- Index on tasks.created_at DESC (used in ORDER BY: list tasks newest first)
CREATE INDEX idx_tasks_created_at ON tasks(created_at DESC);

-- Index on tasks.status (used in WHERE: filter tasks by status)
CREATE INDEX idx_tasks_status ON tasks(status);

-- Index on project_members.project_id (used in JOIN: get all members of a project)
CREATE INDEX idx_project_members_project_id ON project_members(project_id);

-- Index on project_members.user_id (used in JOIN: get all projects of a user)
CREATE INDEX idx_project_members_user_id ON project_members(user_id);
```

### Understanding Each Index Decision

```
idx_users_email:
  → Why? We have APIs that look up users by email (login, search)
  → WHERE email = $1 → index makes this O(log n) instead of O(n)

idx_users_created_at (DESC):
  → Why? GET /api/v1/users returns users sorted by newest first
  → ORDER BY created_at DESC → pre-sorted in the index, no sorting needed

idx_tasks_project_id:
  → Why? "Get all tasks of a project" joins on this field
  → LEFT JOIN tasks ON projects.id = tasks.project_id
  → projects.id is PRIMARY KEY (auto-indexed) ✅
  → tasks.project_id is FOREIGN KEY (NOT auto-indexed) ❌
  → So we manually index task.project_id

idx_tasks_assigned_to:
  → Why? "Get all tasks assigned to a user" joins on this field
  → Same reasoning — foreign key, not auto-indexed

idx_tasks_status:
  → Why? Filter tasks by status: WHERE status = 'pending'
  → Frontend lets users filter by pending/in_progress/completed
  → Frequently used WHERE clause → worth indexing

idx_project_members_project_id / user_id:
  → Why? Many-to-many lookups go through this linking table
  → "Get all members of project X" → filter by project_id
  → "Get all projects of user Y" → filter by user_id
```

### The Trigger + Index Migration (Complete)

```sql
-- +migrate Up

-- ============================================================
-- INDEXES
-- ============================================================

CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_users_created_at ON users(created_at DESC);
CREATE INDEX idx_tasks_project_id ON tasks(project_id);
CREATE INDEX idx_tasks_assigned_to ON tasks(assigned_to);
CREATE INDEX idx_tasks_created_at ON tasks(created_at DESC);
CREATE INDEX idx_tasks_status ON tasks(status);
CREATE INDEX idx_project_members_project_id ON project_members(project_id);
CREATE INDEX idx_project_members_user_id ON project_members(user_id);

-- ============================================================
-- TRIGGER FUNCTION (reusable across all tables)
-- ============================================================

CREATE OR REPLACE FUNCTION update_timestamp()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- ============================================================
-- TRIGGERS (one per table that has updated_at)
-- ============================================================

CREATE TRIGGER set_updated_at_users
  BEFORE UPDATE ON users
  FOR EACH ROW EXECUTE FUNCTION update_timestamp();

CREATE TRIGGER set_updated_at_user_profiles
  BEFORE UPDATE ON user_profiles
  FOR EACH ROW EXECUTE FUNCTION update_timestamp();

CREATE TRIGGER set_updated_at_projects
  BEFORE UPDATE ON projects
  FOR EACH ROW EXECUTE FUNCTION update_timestamp();

CREATE TRIGGER set_updated_at_tasks
  BEFORE UPDATE ON tasks
  FOR EACH ROW EXECUTE FUNCTION update_timestamp();


-- +migrate Down

-- Drop triggers (reverse order)
DROP TRIGGER IF EXISTS set_updated_at_tasks ON tasks;
DROP TRIGGER IF EXISTS set_updated_at_projects ON projects;
DROP TRIGGER IF EXISTS set_updated_at_user_profiles ON user_profiles;
DROP TRIGGER IF EXISTS set_updated_at_users ON users;

-- Drop the trigger function
DROP FUNCTION IF EXISTS update_timestamp();

-- Drop all indexes
DROP INDEX IF EXISTS idx_project_members_user_id;
DROP INDEX IF EXISTS idx_project_members_project_id;
DROP INDEX IF EXISTS idx_tasks_status;
DROP INDEX IF EXISTS idx_tasks_created_at;
DROP INDEX IF EXISTS idx_tasks_assigned_to;
DROP INDEX IF EXISTS idx_tasks_project_id;
DROP INDEX IF EXISTS idx_users_created_at;
DROP INDEX IF EXISTS idx_users_email;
```

After running `dbmate up`, triggers and indexes are active. Now every UPDATE automatically refreshes `updated_at`, and all JOIN/WHERE/ORDER BY operations are optimized.

## What a Backend Engineer Does with Databases — Summary

```
The daily workflow:

1. RECEIVE an API request
   → GET /api/v1/users?status=active&sort=email&page=2

2. PARSE the request
   → Extract query parameters, URL params, request body
   → Validate inputs (check types, ranges, allowed values)

3. CONSTRUCT the database query dynamically
   → Build SELECT/INSERT/UPDATE/DELETE based on what was requested
   → Use parameterized queries for ALL user-provided values
   → Whitelist sort/filter columns for safety
   → Add pagination (OFFSET/LIMIT)

4. EXECUTE the query
   → Send to PostgreSQL via your database driver
   → Receive the result set

5. SERIALIZE and RETURN
   → Convert database rows to JSON
   → Send the response to the frontend

This covers ~80% of what you'll do with databases as a backend engineer.
```
