# Caching - Making Things Fast

> **Core idea:** Caching is keeping a **subset** of data in a **faster-to-access location** so you can decrease the time and effort it takes to retrieve it. Instead of recomputing or re-fetching data every time, you store the result and reuse it. This single technique is a massive factor in every high-performance application - from Google Search to Netflix to Twitter.

## What Is Caching?

```
One-line definition:
  Caching = decreasing the time and effort it takes to perform some work.

Technical definition:
  Caching = keeping a SUBSET of data from a primary storage
            in a FASTER location, so retrieval takes less time.

Key words:
  SUBSET  → Not ALL the data. Only the frequently accessed part.
  FASTER  → A location that takes less time to access (RAM vs disk).
  REUSE   → Once computed, serve the result again without recomputing.
```

### Cache Hit vs Cache Miss

```
When you look for data in the cache:

  CACHE HIT:  Data is found in the cache → return it immediately ✅
              Fast. No need to go to the primary source.

  CACHE MISS: Data is NOT in the cache → fetch from primary source,
              store it in cache for next time, return to client.
              Slow this time, but fast for all future requests.
```

### When Does Caching Make Sense?

```
Two common scenarios where caching is essential:

  1. HEAVY COMPUTATION
     → The result takes significant CPU/memory to compute
     → Example: Google ranking billions of web pages for a search query
     → You don't want to recompute this for every single request

  2. HEAVY DATA TRANSFER
     → The data is large and expensive to transfer
     → Example: Netflix streaming terabytes of video content
     → You don't want every user downloading from the origin server
```

## Real-World Examples

### Example 1: Google Search

```
Without caching:
  User searches "what is the weather today"
  → Google crawls, indexes, and ranks BILLIONS of web pages
  → Extremely expensive (lots of CPU, memory, time)
  → This query is searched MILLIONS of times per day
  → Doing this computation for every single query = servers crash

With caching:
  First user searches "what is the weather today"
  → Google runs the full algorithm → gets results
  → CACHES the results in a distributed in-memory caching system

  Second user (and millions after) searches the same thing:
  → Check cache: results found? YES (cache hit)
  → Return cached results INSTANTLY
  → No need to re-run the ranking algorithm

  Result: Billions of searches served per day with sub-second latency.
```

### Example 2: Netflix (CDN)

```
The problem:
  → Netflix stores movies in different resolutions (1080p, 720p, 480p)
  → Originating servers are in the US
  → A user in India requests a movie
  → Without caching: data travels from US → India for EVERY user
  → Terabytes of data, massive latency, huge bandwidth cost

The solution: CDN (Content Delivery Network)
  → Netflix places EDGE SERVERS all over the world
  → Popular content is CACHED on servers geographically close to users
  → User in India → served from nearest edge server in India
  → User in Japan → served from nearest edge server in Japan

  → Machine learning algorithms decide WHAT to cache at each location
    (based on regional viewing trends, popularity, etc.)
  → Result: minimal buffering, low latency, reduced origin server load
```

### Example 3: Twitter/X Trending Topics

```
The problem:
  → Trending topics are computed by analyzing MILLIONS of tweets
  → Machine learning algorithms, GPUs, terabytes of data
  → If every user opening the "Trending" tab triggered this computation
    → servers crash in seconds (billions of users)

The solution:
  → Twitter computes trending topics periodically (every few minutes)
  → Stores the result in an in-memory cache (like Redis)
  → When users open trending → served from cache instantly
  → Trends don't change every second - safe to cache for minutes

  → This is the pattern: if data doesn't change frequently
    and computation is expensive → CACHE IT
```

## Network-Level Caching - CDN

### How CDN Works

CDN = Content Delivery Network. Caches content on servers **geographically close** to users.

```
Step-by-step CDN flow:

  1. USER enters a URL in the browser
     → Browser sends a DNS query to resolve the domain name

  2. CDN DNS ROUTES the request to the nearest POP
     → POP = Point of Presence = a collection of edge servers in a region
     → Routing based on:
       - User's geographic location
       - Network conditions (bandwidth, latency)
       - Server health and load

  3. EDGE SERVER checks its cache
     → Cache HIT: content found → return it to user immediately ✅
     → Cache MISS: content NOT found → proceed to step 4

  4. EDGE SERVER fetches from ORIGIN
     → Fetches content from the originating server (e.g., US data center)
     → Caches it locally at the edge
     → Returns it to the user
     → Next request for same content = cache hit (step 3)

  5. TTL (Time to Live) controls freshness
     → Each cached item has a TTL (e.g., 2 hours)
     → After TTL expires: edge server fetches fresh content from origin
     → Ensures users don't get stale data forever
```

```
Visualization:

                    ┌──────────┐
                    │  ORIGIN  │  (US Data Center - has ALL content)
                    │  SERVER  │
                    └─────┬────┘
                          │
          ┌───────────────┼───────────────┐
          │               │               │
     ┌────▼────┐    ┌─────▼────┐    ┌─────▼────┐
     │ POP     │    │ POP      │    │ POP      │
     │ India   │    │ Europe   │    │ Japan    │
     │(edge    │    │(edge     │    │(edge     │
     │servers) │    │ servers) │    │ servers) │
     └────┬────┘    └─────┬────┘    └─────┬────┘
          │               │               │
     Users in        Users in         Users in
     India           Europe           Japan
     (low latency)   (low latency)    (low latency)

Without CDN: ALL users → US server → high latency for distant users
With CDN:    Each user → nearest POP → low latency for everyone
```

### What Gets Cached on CDN?

```
  → Video files (Netflix, YouTube)
  → Images (product photos, profile pictures)
  → Static web assets (JavaScript, CSS, HTML files)
  → Web pages (Vercel, Cloudflare Pages)
  → API responses (if configured)

Companies using CDN: Netflix, Amazon, Vercel, Cloudflare, YouTube, etc.
Any company serving content globally uses CDN.
```

## Network-Level Caching - DNS

DNS (Domain Name System) **heavily depends on caching** because the full resolution process is expensive - it involves querying multiple servers recursively.

### Full DNS Resolution Flow (Without Cache)

```
User types: example.com

  1. USER'S DEVICE sends DNS query
     → Goes to a RECURSIVE RESOLVER
     → Provided by ISP (Jio, Airtel) or public DNS (Google 8.8.8.8, Cloudflare 1.1.1.1)

  2. RECURSIVE RESOLVER checks local cache
     → Cache HIT? → Return IP immediately ✅ (skip everything below)
     → Cache MISS? → Start the recursive lookup ↓

  3. Query ROOT SERVER (13 root servers worldwide)
     → Root doesn't know the IP of example.com
     → But it knows WHERE TO FIND .com domains
     → Returns: "Go ask the .com TLD server"

  4. Query TLD SERVER (.com)
     → TLD server doesn't know the exact IP either
     → But it knows the authoritative server for example.com
     → Returns: "Go ask ns1.example.com (the authoritative name server)"

  5. Query AUTHORITATIVE NAME SERVER
     → THIS server knows the actual IP address
     → Returns: "example.com → 93.184.216.34"

  6. RECURSIVE RESOLVER returns IP to the browser
     → Also CACHES the result for future queries

That's 4+ network hops just to find an IP address!
```

### DNS Caching Layers

```
To avoid doing this expensive lookup for every request,
DNS caching happens at MULTIPLE levels:

Level 1: OPERATING SYSTEM CACHE
  → Windows, Mac, Linux all maintain local DNS caches
  → Before anything else, OS checks: "Do I already know this IP?"
  → If YES → skip everything. Fastest possible resolution.

Level 2: BROWSER CACHE
  → Chrome, Firefox maintain their own DNS cache
  → If OS doesn't have it, browser checks its own cache
  → If found → skip the recursive resolver

Level 3: RECURSIVE RESOLVER CACHE
  → The ISP/Google/Cloudflare resolver caches results
  → Serves millions of users, so popular domains are almost always cached
  → If resolver has it → skip root/TLD/authoritative servers

Level 4: AUTHORITATIVE NAME SERVER CACHE
  → Some authoritative servers also cache
  → Can skip further lookups in certain configurations

Typical flow WITH caching:
  User types example.com
  → OS cache? YES → done in <1ms
  → If not: Browser cache? YES → done in <1ms
  → If not: Resolver cache? YES → done in ~5ms
  → If not: Full recursive lookup → ~50-200ms (rare for popular domains)
```

## Hardware-Level Caching - CPU and RAM

```
Inside your computer, there's a caching hierarchy:

  ┌─────────────────┐
  │    CPU CORE      │
  │  ┌───────────┐  │  ← L1 Cache: ~1-2 ns access, smallest, fastest
  │  │  L1 Cache │  │    Stores the most immediately needed data
  │  └───────────┘  │
  │  ┌───────────┐  │  ← L2 Cache: ~3-10 ns access, larger, slightly slower
  │  │  L2 Cache │  │
  │  └───────────┘  │
  └────────┬────────┘
           │
  ┌────────▼────────┐  ← L3 Cache: ~10-30 ns access, shared between CPU cores
  │    L3 Cache     │
  └────────┬────────┘
           │
  ┌────────▼────────┐  ← RAM: ~50-100 ns access, gigabytes of storage
  │   RAM (Main     │    This is where Redis stores data!
  │    Memory)      │
  └────────┬────────┘
           │
  ┌────────▼────────┐  ← Disk (HDD/SSD): ~1,000,000+ ns (HDD), ~100,000 ns (SSD)
  │  Secondary      │    This is where PostgreSQL/MySQL store data!
  │   Storage       │
  └─────────────────┘

Speed comparison:
  L1 Cache:     ~1 ns
  RAM:          ~100 ns          (100x slower than L1)
  SSD:          ~100,000 ns      (1,000x slower than RAM)
  HDD:          ~10,000,000 ns   (100,000x slower than RAM)

This is why Redis (RAM) is so much faster than PostgreSQL (disk).
```

### Why Arrays Are Fast (CPU Cache in Action)

```
When you traverse an array: [1, 2, 3, 4, 5]

  → You access element 1
  → CPU's predictive algorithm says: "You'll probably want 2, 3, 4, 5 too"
  → CPU loads the ENTIRE array into L1/L2 cache
  → Accessing elements 2, 3, 4, 5 is now extremely fast (cache hits)

This is called SPATIAL LOCALITY - data near what you accessed
is loaded into fast cache preemptively.

This is why sequential array access is one of the fastest operations
in computer science - hardware-level caching makes it so.
```

## Why In-Memory Is Fast (RAM vs Disk)

```
HARD DISK (HDD):
  → Mechanical: a physical read head moves across a spinning platter
  → Has to physically MOVE to the right location
  → Speed is limited by physical mechanics
  → Cheap. Lots of capacity. But SLOW.

SSD (Solid State Drive):
  → No moving parts. Uses flash memory chips.
  → Much faster than HDD, but still slower than RAM.
  → Non-volatile (data survives power off).

RAM (Random Access Memory):
  → Electrical signals directly access memory addresses
  → "Random Access" = any location accessed in CONSTANT time
  → No seeking, no spinning, no mechanical delays
  → Incredibly fast. But:
    - LIMITED capacity (GBs, not TBs)
    - VOLATILE (data disappears when power is off)

This is the fundamental tradeoff:

  ┌──────────────────────────────────────────────────────────┐
  │               SPEED ←──────────────→ CAPACITY            │
  │                                                          │
  │  RAM:   Very fast    |  Limited capacity  |  Volatile    │
  │  Disk:  Slower       |  Huge capacity     |  Persistent  │
  │                                                          │
  │  You CANNOT replace disk with RAM entirely.              │
  │  RAM = fast access for a subset (cache)                  │
  │  Disk = permanent storage for everything (database)      │
  └──────────────────────────────────────────────────────────┘
```

## Software-Level Caching - Redis and In-Memory Databases

### What Is Redis?

```
Redis is an IN-MEMORY, KEY-VALUE, NoSQL database.

Let's break down each word:

  IN-MEMORY:
    → Stores data in RAM (not disk)
    → That's why it's extremely fast
    → Uses disk ONLY for persistence (background snapshots)
    → When Redis starts: loads data from disk into RAM
    → All read/write operations happen in RAM

  KEY-VALUE:
    → Data structure: key → value
    → Simple. No tables, no rows, no columns, no schemas.
    → Set a key, get a key. That's it.
    → Example:
        SET "user:123:profile" '{"name":"Aashish","age":22}'
        GET "user:123:profile"  → '{"name":"Aashish","age":22}'

  NoSQL:
    → No SQL queries, no JOINs, no aggregations
    → No strict schema enforcement
    → Values can be: strings, numbers, lists, sets, hashes, JSON
    → Much simpler than relational databases

Compared to PostgreSQL:
  PostgreSQL: Data on disk. SQL queries. Complex schemas. Rich features. Slower.
  Redis:      Data in RAM. Key-value. Simple. Limited features. FAST.

They're not competitors - they complement each other.
PostgreSQL = primary storage (persistent, complex queries)
Redis = cache layer (fast access, simple lookups)
```

## Caching Strategies

### 1. Lazy Caching (Cache-Aside)

```
"Don't cache anything proactively. Only cache when someone asks for it."

Flow:

  Client requests data
       │
       ▼
  Check cache: is data there?
       │
       ├── YES (cache hit) → return cached data ✅
       │
       └── NO (cache miss)
             │
             ▼
       Fetch from database (primary storage)
             │
             ▼
       Store result in cache (for next time)
             │
             ▼
       Return data to client


Example:
  1. User A requests profile of User 123
     → Cache miss → fetch from PostgreSQL → store in Redis → return
  2. User B requests profile of User 123
     → Cache HIT → return from Redis instantly (no DB query)
  3. User C requests profile of User 123
     → Cache HIT → return from Redis instantly

  First request: slow (DB hit)
  All subsequent requests: fast (cache hit)


Why "lazy"?
  → You don't predict what to cache
  → You only cache after someone actually requests it
  → Simple to implement, works great for read-heavy workloads
```

### 2. Write-Through Caching

```
"Every time data changes, update BOTH the database AND the cache."

Flow:

  Client creates/updates data
       │
       ▼
  Write to DATABASE
       │
       ▼
  Write to CACHE (same data, same time)
       │
       ▼
  Return success to client


Example:
  1. User updates their name
     → UPDATE users SET name='New Name' WHERE id=123  (database)
     → SET "user:123:profile" '{"name":"New Name"}'   (cache)
     → Both updated in the same API call

  Next read request:
     → Cache always has the LATEST data
     → No stale data, ever


Tradeoff:
  ✅ Cache is always fresh (never serves outdated data)
  ❌ Every write is slower (double write: DB + cache)
  ❌ More overhead on write operations

  Best for: Data that is read frequently AND updated occasionally.
  Not great for: Write-heavy workloads (too much overhead).
```

### Comparison

```
┌────────────────────┬────────────────────────┬──────────────────────────┐
│                    │ Lazy (Cache-Aside)     │ Write-Through            │
├────────────────────┼────────────────────────┼──────────────────────────┤
│ When is data       │ Only when requested    │ Every time data changes  │
│ cached?            │ (on cache miss)        │ (on every write)         │
├────────────────────┼────────────────────────┼──────────────────────────┤
│ Cache freshness    │ Can be stale           │ Always fresh             │
├────────────────────┼────────────────────────┼──────────────────────────┤
│ Write overhead     │ None                   │ Extra write per update   │
├────────────────────┼────────────────────────┼──────────────────────────┤
│ Implementation     │ Simpler                │ More complex             │
├────────────────────┼────────────────────────┼──────────────────────────┤
│ Best for           │ Read-heavy, rarely     │ Read-heavy, data must    │
│                    │ changing data          │ always be up-to-date     │
└────────────────────┴────────────────────────┴──────────────────────────┘
```

## Eviction Policies

Since RAM is limited, your cache WILL fill up. Eviction policies decide **which data to remove** when the cache is full and new data needs to be stored.

### No Eviction

```
→ Don't remove anything
→ When cache is full → ERROR on next insert
→ Not useful in practice, but it's the "what if you configure nothing" default
```

### LRU - Least Recently Used

```
"Remove the item that hasn't been accessed for the LONGEST time."

Cache is full: [Key1, Key2, Key3, Key4]

Access timestamps:
  Key1: accessed TODAY
  Key2: accessed TODAY
  Key3: accessed TODAY
  Key4: accessed YESTERDAY  ← least recently used

Key5 arrives, cache is full.
  → LRU says: Key4 was accessed the longest ago → EVICT Key4
  → Cache becomes: [Key1, Key2, Key3, Key5]

Best for: When recent data is more likely to be accessed again.
Most popular eviction policy. Good default choice.
```

### LFU - Least Frequently Used

```
"Remove the item that has been accessed the FEWEST times overall."

Cache is full: [Key1, Key2, Key3, Key4]

Access frequencies:
  Key1: accessed 5 times   ← least frequently used
  Key2: accessed 10 times
  Key3: accessed 6 times
  Key4: accessed 23 times

Key5 arrives, cache is full.
  → LFU says: Key1 was accessed fewest times → EVICT Key1
  → Cache becomes: [Key2, Key3, Key4, Key5]

Best for: When frequently accessed data should always stay cached,
regardless of when it was last accessed.
```

### TTL-Based Eviction (Time to Live)

```
"Each key has an expiration time. Expired keys get evicted first."

Cache: [Key1(TTL: 5min), Key2(TTL: 2hr), Key3(TTL: 30s), Key4(TTL: 1hr)]

Key5 arrives, cache is full.
  → Check: which key expires soonest?
  → Key3 has only 30 seconds left → EVICT Key3
  → Cache becomes: [Key1, Key2, Key4, Key5]

In Redis, you SET the TTL when storing data:
  SET "weather:delhi" '{"temp": 32}' EX 3600   ← expires in 1 hour
  SET "trending:india" '...' EX 300             ← expires in 5 minutes

After TTL expires, key is automatically deleted.
Next request → cache miss → fetch fresh data → cache with new TTL.
```

### Quick Comparison

```
┌──────────────┬────────────────────────────────────┬────────────────────┐
│ Policy       │ What it evicts                      │ Best for           │
├──────────────┼────────────────────────────────────┼────────────────────┤
│ No Eviction  │ Nothing (errors on insert)          │ Never              │
│ LRU          │ Least recently accessed item        │ General purpose    │
│ LFU          │ Least frequently accessed item      │ Hot/cold data      │
│ TTL-based    │ Item closest to expiration          │ Time-sensitive data│
└──────────────┴────────────────────────────────────┴────────────────────┘
```

## Backend Use Cases for Redis

### 1. Database Query Caching

```
Problem:
  → Complex SQL query with multiple JOINs and aggregations
  → Processes millions of rows
  → Called frequently (dashboard page, landing page)
  → Puts heavy load on database

Solution:
  → First request: run the query, cache the result in Redis with TTL
  → Subsequent requests: serve from Redis (skip the database entirely)
  → When data changes: invalidate/update the cache

  // Pseudocode
  const cacheKey = "dashboard:stats";
  let data = await redis.get(cacheKey);

  if (!data) {
    // Cache miss - run the expensive query
    data = await db.query("SELECT ... complex query with JOINs ...");
    await redis.set(cacheKey, JSON.stringify(data), "EX", 3600); // TTL: 1 hour
  }

  return JSON.parse(data);

Real-world examples:
  → Amazon: caches product details, prices, inventory
    (product info doesn't change every second)
  → Twitter/Facebook: caches user profile data
    (profiles change maybe once a year)
  → Any dashboard: caches aggregated statistics
    (recomputing every request is wasteful)

Pattern: READ-HEAVY + DATA CHANGES INFREQUENTLY = perfect for caching
```

### 2. Session Storage

```
Problem:
  → After authentication, server generates a session token
  → Every API call needs to verify the session
  → Storing sessions in PostgreSQL = database hit on EVERY request
  → Adds latency to every single API call
  → Floods the database with session lookups

Solution:
  → Store session tokens in Redis
  → Session lookup = Redis GET (sub-millisecond)
  → No database hit for authentication on every request

  // On login
  const sessionId = generateSessionId();
  await redis.set(`session:${sessionId}`, JSON.stringify(userData), "EX", 86400);
  // TTL: 24 hours - session auto-expires

  // On every API request (middleware)
  const session = await redis.get(`session:${req.cookies.sessionId}`);
  if (!session) return res.status(401).send("Unauthorized");
  req.user = JSON.parse(session);

Why Redis for sessions?
  → Every API call checks the session → needs to be FAST
  → Sessions are temporary → perfect for volatile storage
  → Sessions expire → TTL handles auto-cleanup
```

### 3. External API Caching

```
Problem:
  → Your backend calls an external API (weather API, exchange rates, etc.)
  → Each call to external API = network latency + potential billing
  → If 1000 users hit your API → 1000 calls to external API
  → Rate limits hit quickly. Bills skyrocket.

Solution:
  → Fetch from external API ONCE
  → Cache the result in Redis with appropriate TTL
  → Serve all subsequent requests from cache

  // Pseudocode
  const cacheKey = "weather:delhi";
  let weather = await redis.get(cacheKey);

  if (!weather) {
    weather = await fetch("https://api.weather.com/delhi");
    await redis.set(cacheKey, JSON.stringify(weather), "EX", 3600); // 1 hour TTL
  }

  return JSON.parse(weather);

  → 1000 users → 1 external API call + 999 cache hits
  → Weather doesn't change every minute → safe to cache for an hour
  → Saves money (fewer API calls) and latency (cache is faster than network)
```

### 4. Rate Limiting

```
Problem:
  → Need to limit: max 50 requests per minute per client
  → Must check and increment a counter on EVERY request
  → Storing counters in PostgreSQL = slow + floods the database

Solution:
  → Use Redis to store request counters per IP address

  How it works:

  Client makes a request
       │
       ▼
  Rate limit middleware extracts IP from X-Forwarded-For header
       │
       ▼
  Check Redis: GET "ratelimit:{IP}"
       │
       ├── Counter < 50 → INCR the counter, let request through ✅
       │
       └── Counter >= 50 → BLOCK the request, return 429 Too Many Requests ❌

  Redis key:  "ratelimit:10.0.0.1"
  Value:      counter (incremented with each request)
  TTL:        60 seconds (counter auto-resets every minute)

  // Pseudocode
  const ip = req.headers["x-forwarded-for"];
  const key = `ratelimit:${ip}`;
  const count = await redis.incr(key);   // Increment counter

  if (count === 1) {
    await redis.expire(key, 60);         // Set TTL on first request
  }

  if (count > 50) {
    return res.status(429).send("Too many requests. Try again later.");
  }

  next();  // Allow the request through

Why Redis and not PostgreSQL?
  → Rate limiting checks happen on EVERY request
  → Must be as fast as possible (sub-millisecond)
  → Even 20-30ms of DB latency adds up across all requests
  → Redis: <1ms. PostgreSQL: 5-50ms. That difference matters.
  → Also avoids flooding the database with rate limit queries
```

```
│                 │ minutes and cached - not computed per ride request.       │
├─────────────────┼───────────────────────────────────────────────────────────┤
│ Spotify         │ Your "Discover Weekly" playlist is generated once a       │
│                 │ week using heavy ML algorithms, then cached. Every time   │
│                 │ you open it, you're reading from cache, not triggering    │
│                 │ an ML pipeline.                                           │
├─────────────────┼───────────────────────────────────────────────────────────┤
│ WhatsApp        │ Caches user online/offline status and "last seen" in      │
│                 │ Redis. Checking DB for 2B+ users' status on every         │
│                 │ chat open would destroy any database.                     │
├─────────────────┼───────────────────────────────────────────────────────────┤
│ GitHub          │ README renders, repository file trees, and contribution   │
│                 │ graphs are cached. Parsing markdown and computing         │
│                 │ graphs for millions of repos on every page view           │
│                 │ would be extremely wasteful.                              │
├─────────────────┼───────────────────────────────────────────────────────────┤
│ Swiggy/Zomato   │ Restaurant menus, ratings, delivery time estimates        │
│                 │ are cached. Menu doesn't change every minute, but         │
│                 │ millions of users browse it every hour.                   │
├─────────────────┼───────────────────────────────────────────────────────────┤
│ Razorpay/Stripe │ Exchange rates, merchant configs, and plan details        │
│                 │ cached in Redis. Every payment request checking           │
│                 │ merchant config from DB = unnecessary latency on          │
│                 │ a time-sensitive operation.                               │
├─────────────────┼───────────────────────────────────────────────────────────┤
│ YouTube         │ Video metadata (title, description, view count) is        │
│                 │ cached. The actual video is served via CDN edge           │
│                 │ servers. View count is updated in cache and flushed       │
│                 │ to DB periodically (that's why counts lag sometimes).     │
├─────────────────┼───────────────────────────────────────────────────────────┤
│ LinkedIn        │ Profile views, connection counts, and feed posts          │
│                 │ are cached. "Who viewed your profile" is a cached         │
│                 │ aggregation, not a live DB query every time you check.    │
├─────────────────┼───────────────────────────────────────────────────────────┤
│ Slack           │ Channel membership lists, workspace settings, and         │
│                 │ user presence (online/away/offline) are all cached.       │
│                 │ With millions of messages per day, every presence         │
│                 │ check hitting the DB would be a disaster.                 │
└─────────────────┴───────────────────────────────────────────────────────────┘

The pattern across ALL of these:
  → Data that is READ frequently but WRITTEN rarely → cache it
  → Expensive computations done periodically → cache the result
  → Data that doesn't change every second → safe to cache with TTL
```

## Summary

```
┌──────────────────────────┬──────────────────────────────────────────────┐
│ Concept                  │ Summary                                      │
├──────────────────────────┼──────────────────────────────────────────────┤
│ What is caching          │ Storing a subset of data in a faster         │
│                          │ location to reduce time and effort.          │
├──────────────────────────┼──────────────────────────────────────────────┤
│ Cache hit / miss         │ Hit = found in cache (fast). Miss = not      │
│                          │ found (fetch from source, then cache it).    │
├──────────────────────────┼──────────────────────────────────────────────┤
│ CDN                      │ Edge servers worldwide. Content served       │
│                          │ from geographically closest server.          │
├──────────────────────────┼──────────────────────────────────────────────┤
│ DNS caching              │ Cached at OS → browser → resolver →          │
│                          │ authoritative. Avoids recursive lookups.     │
├──────────────────────────┼──────────────────────────────────────────────┤
│ RAM vs Disk              │ RAM: <100ns, volatile, limited capacity.     │
│                          │ Disk: >100,000ns, persistent, huge capacity. │
├──────────────────────────┼──────────────────────────────────────────────┤
│ Redis                    │ In-memory, key-value, NoSQL database.        │
│                          │ Stores data in RAM for sub-ms access.        │
├──────────────────────────┼──────────────────────────────────────────────┤
│ Lazy caching             │ Only cache when someone requests it.         │
│ (cache-aside)            │ Simple. May serve stale data.                │
├──────────────────────────┼──────────────────────────────────────────────┤
│ Write-through            │ Update DB + cache together on every write.   │
│                          │ Always fresh. More write overhead.           │
├──────────────────────────┼──────────────────────────────────────────────┤
│ LRU eviction             │ Remove least recently accessed item.         │
├──────────────────────────┼──────────────────────────────────────────────┤
│ LFU eviction             │ Remove least frequently accessed item.       │
├──────────────────────────┼──────────────────────────────────────────────┤
│ TTL eviction             │ Remove items closest to expiration.          │
├──────────────────────────┼──────────────────────────────────────────────┤
│ Use case: DB caching     │ Cache expensive query results.               │
├──────────────────────────┼──────────────────────────────────────────────┤
│ Use case: Sessions       │ Store auth sessions for sub-ms lookups.      │
├──────────────────────────┼──────────────────────────────────────────────┤
│ Use case: API caching    │ Cache external API responses to reduce       │
│                          │ billing and avoid rate limits.               │
├──────────────────────────┼──────────────────────────────────────────────┤
│ Use case: Rate limiting  │ Store request counters per IP in Redis.      │
│                          │ Fast enough for every-request checking.      │
└──────────────────────────┴──────────────────────────────────────────────┘

The golden rule:
  → If data is READ-HEAVY and CHANGES INFREQUENTLY → cache it.
  → If computation is EXPENSIVE and REPEATABLE → cache the result.
  → Use Redis/in-memory DB for the cache layer.
  → Use PostgreSQL/MySQL for the permanent storage.
  → They complement each other.
```

### Exercise 2 â€” Add Caching to an Existing API
```javascript
// Take any GET endpoint and add Redis caching:

const redis = require('redis');
const client = redis.createClient();

app.get('/api/products', async (req, res) => {
  const cacheKey = 'products:all';

  // 1. Check cache
  const cached = await client.get(cacheKey);
  if (cached) {
    console.log('CACHE HIT');
    return res.json(JSON.parse(cached));
  }

  // 2. Cache miss -> fetch from DB
  console.log('CACHE MISS');
  const products = await db.query('SELECT * FROM products');

  // 3. Store in cache with TTL
  await client.set(cacheKey, JSON.stringify(products.rows), { EX: 300 });

  res.json(products.rows);
});

// Test: First request -> MISS (slow). Second request -> HIT (fast).
// Measure the time difference!
```

### Exercise 3 â€” Rate Limiter with Redis
```javascript
// Build a rate limiter that allows 10 requests per minute per IP:

async function rateLimiter(req, res, next) {
  const ip = req.ip;
  const key = 'ratelimit:' + ip;
  const count = await client.incr(key);

  if (count === 1) await client.expire(key, 60);  // 60s window

  if (count > 10) {
    return res.status(429).json({ error: 'Too many requests. Try again later.' });
  }
  next();
}

app.use(rateLimiter);

// Test: Hit an endpoint 11 times rapidly -> 11th request gets 429.
// Wait 60 seconds -> counter resets -> requests allowed again.
```
