## Express.js Fundamentals

Express.js is a minimal, unopinionated web framework for Node.js. It is by far the most popular Node.js framework with millions of weekly downloads. Express sits on top of Node's built-in `http` module and provides a clean, powerful API for handling routes, parsing requests, managing middleware, and sending responses. Think of Node's raw `http` module as the engine, and Express as the car built around it — you could drive the engine directly, but the car makes it practical. Express doesn't force any specific project structure or patterns on you (unlike Django or Rails), which means you have the freedom and responsibility to organize your own code.

### What Express Does

```
Express sits ON TOP of Node's built-in http module.
It wraps the raw (req, res) with better APIs and adds:

  1. Routing       -> map URLs to handler functions
  2. Middleware    -> pluggable functions that process requests
  3. Request parsing -> body, query, params extraction
  4. Response helpers -> res.json(), res.status(), res.redirect()
  5. Error handling -> centralized error pipeline

Without Express (raw Node):
  if (req.method === 'GET' && req.url === '/api/users') { ... }
  if (req.method === 'POST' && req.url === '/api/users') { ... }
  // 50 routes = 50 if/else = unmaintainable

With Express:
  app.get('/api/users', getUsers);
  app.post('/api/users', createUser);
  // Clean, declarative, maintainable
```

### Basic Setup

```javascript
const express = require('express');
const app = express();

// Built-in middleware:
app.use(express.json());                    // parse JSON bodies
app.use(express.urlencoded({ extended: true })); // parse form data

// Routes:
app.get('/', (req, res) => {
  res.json({ message: 'API is running' });
});

// Start server:
const PORT = process.env.PORT || 3000;
app.listen(PORT, () => {
  console.log(`Server running on port ${PORT}`);
});
```

## Middleware Deep Dive

Middleware is the backbone of Express.js architecture. A middleware function is any function that has access to the request object (`req`), the response object (`res`), and the `next` function. Middleware functions execute in a **pipeline** — each request passes through a chain of middleware functions, one after another, before reaching the final route handler. This is incredibly powerful because it lets you compose reusable behaviors: logging every request, parsing JSON bodies, checking authentication, validating input, handling CORS — each is a separate middleware that you plug into the pipeline. If any middleware detects a problem (invalid token, missing field), it can short-circuit the pipeline and send an error response immediately, preventing the request from ever reaching the handler.

```
Middleware = functions that run BETWEEN receiving a request and sending a response.
They form a PIPELINE. Each middleware can:
  -> Read/modify the request
  -> Read/modify the response  
  -> Call next() to pass control to the next middleware
  -> End the request by sending a response (don't call next)

Request arrives
     |
     v
  [Middleware 1: logger]     -> logs method + url
     |
     v
  [Middleware 2: auth]       -> checks JWT token
     |
     v
  [Middleware 3: validate]   -> validates request body
     |
     v
  [Route Handler]            -> processes business logic
     |
     v
  [Error Middleware]          -> catches and formats errors
     |
     v
  Response sent
```

### Types of Middleware

```javascript
// 1. APPLICATION-LEVEL (runs on every request):
app.use((req, res, next) => {
  console.log(`${req.method} ${req.url} at ${new Date().toISOString()}`);
  next();   // MUST call next() or request hangs forever
});

// 2. ROUTE-LEVEL (runs only on specific routes):
app.get('/api/admin', authMiddleware, adminOnly, (req, res) => {
  // authMiddleware runs first, then adminOnly, then this handler
  res.json({ data: 'admin stuff' });
});

// 3. ERROR-HANDLING (4 params - Express recognizes the signature):
app.use((err, req, res, next) => {   // <-- 4 params = error handler
  console.error(err.stack);
  res.status(err.statusCode || 500).json({
    error: { message: err.message || 'Internal Server Error' }
  });
});

// 4. THIRD-PARTY:
const cors = require('cors');
const helmet = require('helmet');
const morgan = require('morgan');

app.use(cors());          // enable CORS
app.use(helmet());        // security headers
app.use(morgan('dev'));   // request logging
```

### Middleware Execution Order

```javascript
// ORDER MATTERS! Middleware runs TOP to BOTTOM as defined.

app.use(morgan('dev'));        // 1st: log the request
app.use(cors());               // 2nd: handle CORS
app.use(helmet());             // 3rd: security headers
app.use(express.json());       // 4th: parse JSON body
app.use(rateLimiter);          // 5th: rate limiting

// Routes come AFTER middleware:
app.use('/api/v1/users', userRoutes);
app.use('/api/v1/products', productRoutes);

// 404 handler (after ALL routes - catches unmatched):
app.use((req, res) => {
  res.status(404).json({ error: 'Route not found' });
});

// Error handler (MUST be last):
app.use((err, req, res, next) => {
  res.status(500).json({ error: err.message });
});
```

### Custom Middleware Examples

```javascript
// Authentication middleware:
function authenticate(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: 'No token provided' });

  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    req.user = decoded;   // attach user to request
    next();                // proceed to next middleware/handler
  } catch (err) {
    return res.status(401).json({ error: 'Invalid token' });
  }
}

// Authorization middleware:
function authorize(...roles) {
  return (req, res, next) => {
    if (!roles.includes(req.user.role)) {
      return res.status(403).json({ error: 'Insufficient permissions' });
    }
    next();
  };
}

// Usage:
app.delete('/api/users/:id', authenticate, authorize('admin'), deleteUser);
//  1. authenticate runs -> checks token, sets req.user
//  2. authorize('admin') runs -> checks if user.role is 'admin'
//  3. deleteUser runs -> only if both passed


// Request timing middleware:
function requestTimer(req, res, next) {
  const start = Date.now();
  res.on('finish', () => {
    const duration = Date.now() - start;
    console.log(`${req.method} ${req.url} ${res.statusCode} - ${duration}ms`);
  });
  next();
}
```

## Routing Architecture

As your API grows beyond a handful of routes, putting everything in a single file becomes unmaintainable. Express provides the **Router** class to organize routes into separate files grouped by resource or feature. Each Router acts like a mini Express app with its own routes and middleware. You define routes in separate files (e.g., `users.js`, `products.js`), then mount them on specific path prefixes in the main app. This creates a clean, modular structure where each file is small, focused, and easy to understand. This pattern directly maps to the **Handlers/Controllers** layer from the layered architecture discussed in the ControllerServicesRoutes notes.

```javascript
// routes/users.js
const express = require('express');
const router = express.Router();
const userController = require('../controllers/userController');
const { authenticate, authorize } = require('../middleware/auth');
const { validateUserCreate } = require('../middleware/validators');

router.get('/',                          userController.getAll);
router.get('/:id',                       userController.getById);
router.post('/', validateUserCreate,     userController.create);
router.patch('/:id', authenticate,       userController.update);
router.delete('/:id', authenticate, authorize('admin'), userController.delete);

module.exports = router;


// routes/index.js - Central route aggregator:
const express = require('express');
const router = express.Router();

const userRoutes = require('./users');
const productRoutes = require('./products');
const authRoutes = require('./auth');

router.use('/users', userRoutes);
router.use('/products', productRoutes);
router.use('/auth', authRoutes);

module.exports = router;


// main.js:
const routes = require('./routes');
app.use('/api/v1', routes);

// Result:
//   GET /api/v1/users       -> userController.getAll
//   POST /api/v1/auth/login -> authController.login
```

## Request and Response Objects

Every Express route handler receives two objects: `req` (the incoming request) and `res` (the outgoing response). The **request object** contains everything the client sent: URL path parameters (`:id`), query string parameters (`?page=1`), the request body (JSON payload), headers (Authorization, Content-Type), and metadata (method, IP, protocol). The **response object** provides methods to send data back: `res.json()` for JSON, `res.status()` to set status codes, `res.redirect()` for redirects, `res.cookie()` for cookies, and even `stream.pipe(res)` for streaming large files. Understanding these two objects deeply is essential because they are the interface between your code and the HTTP protocol.

### Request (req)

```javascript
app.post('/api/users/:userId/posts', authenticate, (req, res) => {
  // PATH PARAMETERS (from URL):
  req.params.userId;        // '42' (string, from :userId)

  // QUERY PARAMETERS (from ?key=value):
  // GET /api/users/42/posts?page=2&limit=10
  req.query.page;           // '2' (string)
  req.query.limit;          // '10' (string)

  // BODY (from POST/PATCH/PUT payload):
  req.body;                 // { title: 'My Post', body: '...' }
  req.body.title;           // 'My Post'

  // HEADERS:
  req.headers.authorization;  // 'Bearer eyJhbGci...'
  req.headers['content-type']; // 'application/json'
  req.get('Content-Type');     // same, cleaner API

  // REQUEST INFO:
  req.method;               // 'POST'
  req.url;                  // '/api/users/42/posts?page=2'
  req.path;                 // '/api/users/42/posts'
  req.ip;                   // '192.168.1.1'
  req.hostname;             // 'localhost'
  req.protocol;             // 'http' or 'https'

  // CUSTOM DATA (set by middleware):
  req.user;                 // { id: 1, role: 'admin' } (set by auth middleware)
});
```

### Response (res)

```javascript
// Status + JSON (most common):
res.status(200).json({ users: [...] });
res.status(201).json({ id: 1, name: 'Created' });
res.status(204).end();   // no content

// Shortcuts:
res.json(data);           // 200 + JSON (default status)
res.send('text');          // 200 + text
res.sendFile('/path/to/file.html');

// Headers:
res.set('X-Custom-Header', 'value');
res.set({ 'X-RateLimit-Remaining': 99, 'X-Request-ID': 'abc' });

// Cookies:
res.cookie('session', 'abc123', { httpOnly: true, maxAge: 3600000 });
res.clearCookie('session');

// Redirects:
res.redirect('/new-url');           // 302 (temporary)
res.redirect(301, '/permanent');    // 301 (permanent)

// Download:
res.download('/path/to/file.pdf', 'report.pdf');

// Streaming:
const stream = fs.createReadStream('video.mp4');
res.set('Content-Type', 'video/mp4');
stream.pipe(res);  // stream large files without loading into memory
```

## Express + Database Integration

Almost every backend API needs to persist data in a database. In Node.js, you connect to databases using **driver libraries** (like `pg` for PostgreSQL, `mysql2` for MySQL, `mongodb` for MongoDB). The most important concept here is **connection pooling** — instead of opening a new database connection for every request (expensive: TCP handshake + authentication each time), you create a **pool** of pre-established connections at startup and reuse them. Each request borrows a connection, runs its query, and returns the connection to the pool. This dramatically improves performance. You should also always use **parameterized queries** (using `$1`, `$2` placeholders) instead of string interpolation to prevent SQL injection attacks.

```javascript
// Using PostgreSQL with node-postgres (pg):
const { Pool } = require('pg');

const pool = new Pool({
  host: process.env.DB_HOST || 'localhost',
  port: process.env.DB_PORT || 5432,
  database: process.env.DB_NAME || 'myapp',
  user: process.env.DB_USER || 'postgres',
  password: process.env.DB_PASS,
  max: 20,                // max connections in pool
  idleTimeoutMillis: 30000,
  connectionTimeoutMillis: 2000,
});

// Repository pattern:
class UserRepository {
  async findAll(page = 1, limit = 20) {
    const offset = (page - 1) * limit;
    const result = await pool.query(
      'SELECT id, name, email, created_at FROM users ORDER BY created_at DESC LIMIT $1 OFFSET $2',
      [limit, offset]
    );
    return result.rows;
  }

  async findById(id) {
    const result = await pool.query('SELECT * FROM users WHERE id = $1', [id]);
    return result.rows[0] || null;
  }

  async create(userData) {
    const { name, email, hashedPassword } = userData;
    const result = await pool.query(
      'INSERT INTO users (name, email, hashed_password) VALUES ($1, $2, $3) RETURNING id, name, email, created_at',
      [name, email, hashedPassword]
    );
    return result.rows[0];
  }

  async update(id, fields) {
    // Dynamic update - only update provided fields
    const keys = Object.keys(fields);
    const values = Object.values(fields);
    const setClause = keys.map((key, i) => `${key} = $${i + 1}`).join(', ');
    
    const result = await pool.query(
      `UPDATE users SET ${setClause} WHERE id = $${keys.length + 1} RETURNING *`,
      [...values, id]
    );
    return result.rows[0];
  }

  async delete(id) {
    const result = await pool.query('DELETE FROM users WHERE id = $1', [id]);
    return result.rowCount > 0;
  }
}

// PARAMETERIZED QUERIES ($1, $2) prevent SQL injection.
// NEVER do: `SELECT * FROM users WHERE id = ${id}` (SQL injection!)
```

## Security Best Practices

Security is not optional in backend development — your API is exposed to the entire internet, and attackers constantly probe for vulnerabilities. The most common attacks against Node.js APIs include: **SQL/NoSQL injection** (malicious queries through user input), **XSS** (injecting scripts), **CSRF** (forging requests), **brute force** (password guessing), and **DDoS** (overwhelming the server with requests). Fortunately, there are well-established npm packages that handle most of these: `helmet` adds security headers, `cors` controls cross-origin access, `express-rate-limit` prevents abuse, and parameterized queries prevent injection. The golden rule is: **never trust user input**. Always validate, sanitize, and limit everything that comes from the client.

```javascript
const helmet = require('helmet');       // security headers
const cors = require('cors');
const rateLimit = require('express-rate-limit');
const hpp = require('hpp');             // HTTP parameter pollution
const mongoSanitize = require('express-mongo-sanitize');

// 1. HELMET - sets security headers:
app.use(helmet());
// Adds: X-Content-Type-Options, X-Frame-Options, CSP, etc.

// 2. CORS - control who can call your API:
app.use(cors({
  origin: ['https://myapp.com'],  // NOT '*' in production
  credentials: true,
}));

// 3. RATE LIMITING - prevent abuse:
app.use(rateLimit({
  windowMs: 15 * 60 * 1000,    // 15 minutes
  max: 100,                     // 100 requests per window
  message: { error: 'Too many requests' },
}));

// 4. BODY SIZE LIMIT - prevent large payload attacks:
app.use(express.json({ limit: '10kb' }));  // reject bodies > 10KB

// 5. HPP - prevent parameter pollution:
app.use(hpp());
// Without: /api/users?sort=name&sort=email (array injection)
// With: uses the last value only

// 6. NEVER expose stack traces in production:
app.use((err, req, res, next) => {
  res.status(500).json({
    error: {
      message: process.env.NODE_ENV === 'production'
        ? 'Something went wrong'
        : err.message,
    }
  });
});

// 7. ENVIRONMENT VARIABLES for secrets:
// NEVER hardcode: JWT_SECRET, DB_PASSWORD, API_KEYS
// Use .env file + dotenv package:
require('dotenv').config();
const secret = process.env.JWT_SECRET;

// 8. SQL INJECTION prevention:
// ALWAYS use parameterized queries:
// GOOD: db.query('SELECT * FROM users WHERE id = $1', [userId]);
// BAD:  db.query(`SELECT * FROM users WHERE id = ${userId}`);
```

## Clustering and Scaling

Node.js runs JavaScript on a single CPU core by default. On a modern server with 4, 8, or 16 cores, this means you're using only a fraction of the available processing power. **Clustering** solves this by creating multiple copies ("workers") of your application, one per CPU core, all sharing the same port. The operating system distributes incoming connections across the workers using round-robin load balancing. This can multiply your throughput by roughly the number of CPU cores. For production deployments, **PM2** is a popular process manager that handles clustering automatically, along with log aggregation, auto-restart on crashes, and zero-downtime reloads.

```javascript
// Node.js runs on ONE CPU core by default.
// Modern servers have 4, 8, 16+ cores.
// Clustering uses ALL cores.

const cluster = require('cluster');
const os = require('os');

if (cluster.isPrimary) {
  const numCPUs = os.cpus().length;
  console.log(`Primary process ${process.pid} forking ${numCPUs} workers`);

  // Fork one worker per CPU core:
  for (let i = 0; i < numCPUs; i++) {
    cluster.fork();
  }

  // Restart worker if it dies:
  cluster.on('exit', (worker) => {
    console.log(`Worker ${worker.process.pid} died. Restarting...`);
    cluster.fork();
  });
} else {
  // Worker processes run the Express app:
  const app = require('./app');
  app.listen(3000, () => {
    console.log(`Worker ${process.pid} started`);
  });
}

// With 4 CPU cores:
//   Primary process (manages workers, doesn't handle requests)
//   Worker 1 -> handles requests on port 3000
//   Worker 2 -> handles requests on port 3000
//   Worker 3 -> handles requests on port 3000
//   Worker 4 -> handles requests on port 3000
// OS load-balances incoming connections across workers.
// Throughput: ~4x a single process.


// EASIER ALTERNATIVE: PM2 (production process manager)
// pm2 start app.js -i max    -> auto-cluster across all CPUs
// pm2 start app.js -i 4      -> 4 instances
// pm2 monit                  -> live monitoring dashboard
// pm2 logs                   -> aggregated logs
// pm2 restart app             -> zero-downtime restart
```

## Environment Configuration

Every application needs configuration values that change between environments: database URLs, API keys, JWT secrets, port numbers. These should **never** be hardcoded in your source code. Instead, they are stored in **environment variables** — key-value pairs set outside your code. In development, you use a `.env` file (loaded by the `dotenv` package). In production, these are set in the deployment environment (Docker, Kubernetes, cloud platforms). A best practice is to create a centralized `config.js` file that reads all environment variables in one place, applies defaults, and exports a clean configuration object that the rest of your code imports.

```javascript
// .env file (NEVER commit to git):
// PORT=3000
// NODE_ENV=development
// DB_HOST=localhost
// DB_PORT=5432
// DB_NAME=myapp
// DB_USER=postgres
// DB_PASS=secret123
// JWT_SECRET=super-secret-key
// REDIS_URL=redis://localhost:6379

// config.js - centralize all config:
require('dotenv').config();

module.exports = {
  port: parseInt(process.env.PORT, 10) || 3000,
  nodeEnv: process.env.NODE_ENV || 'development',
  db: {
    host: process.env.DB_HOST || 'localhost',
    port: parseInt(process.env.DB_PORT, 10) || 5432,
    name: process.env.DB_NAME || 'myapp',
    user: process.env.DB_USER || 'postgres',
    password: process.env.DB_PASS,
  },
  jwt: {
    secret: process.env.JWT_SECRET,
    expiresIn: '24h',
  },
  isDev: process.env.NODE_ENV === 'development',
  isProd: process.env.NODE_ENV === 'production',
};

// Usage anywhere:
const config = require('./config');
app.listen(config.port);
jwt.sign(payload, config.jwt.secret);
```

## Logging and Monitoring

In production, `console.log()` is not enough. You need structured, leveled, persistent logging. **Winston** is the most popular Node.js logging library — it supports log levels (error, warn, info, debug), multiple output destinations (console, files, external services), and structured JSON formatting. Good logging lets you trace issues, understand traffic patterns, and debug production problems. Combined with **Morgan** (HTTP request logger middleware), you get a complete picture of every request hitting your server: method, URL, status code, response time. In production, logs are typically shipped to centralized services like Elasticsearch, Datadog, or AWS CloudWatch for analysis and alerting.

```javascript
// Use winston for production logging (NOT console.log):
const winston = require('winston');

const logger = winston.createLogger({
  level: process.env.LOG_LEVEL || 'info',
  format: winston.format.combine(
    winston.format.timestamp(),
    winston.format.json()
  ),
  transports: [
    new winston.transports.File({ filename: 'logs/error.log', level: 'error' }),
    new winston.transports.File({ filename: 'logs/combined.log' }),
  ],
});

// In development, also log to console:
if (process.env.NODE_ENV !== 'production') {
  logger.add(new winston.transports.Console({
    format: winston.format.simple(),
  }));
}

// Usage:
logger.info('User created', { userId: 123, email: 'a@b.com' });
logger.error('Failed to connect to DB', { error: err.message });
logger.warn('Rate limit approaching', { ip: req.ip, count: 90 });

// HTTP request logging with morgan:
const morgan = require('morgan');
app.use(morgan('combined', { stream: { write: msg => logger.info(msg.trim()) } }));
```

## Rate Limiting

Rate limiting controls how many requests a client can make in a given time window. This is essential for protecting your API from abuse, brute-force attacks, and accidental runaway scripts. For example, you might allow 100 requests per 15 minutes for general endpoints, but only 5 login attempts per 15 minutes (to prevent password brute-forcing). For single-server deployments, in-memory rate limiting works fine. For clustered or multi-server deployments, you need a shared store like **Redis** so all instances see the same request counts. Most APIs return a `429 Too Many Requests` status code when the limit is exceeded.

```javascript
const rateLimit = require('express-rate-limit');
const RedisStore = require('rate-limit-redis');
const Redis = require('ioredis');

// Basic in-memory rate limiting (single server):
const generalLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,   // 15 minutes
  max: 100,                     // 100 requests per window
  standardHeaders: true,        // Return rate limit info in headers
  message: { error: 'Too many requests, please try again later' },
});

// Strict limiter for auth endpoints (prevent brute force):
const loginLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5,                       // only 5 login attempts per 15 min
  message: { error: 'Too many login attempts' },
});

// Redis-backed rate limiting (works across clustered servers):
const redisLimiter = rateLimit({
  store: new RedisStore({
    sendCommand: (...args) => new Redis().call(...args),
  }),
  windowMs: 15 * 60 * 1000,
  max: 100,
});

app.use('/api', generalLimiter);
app.use('/api/auth/login', loginLimiter);
```

## Testing Node.js APIs

Testing is what separates hobby projects from production-quality code. In Node.js, the most common testing stack is **Jest** (test runner and assertion library) combined with **Supertest** (HTTP testing library that lets you make requests to your Express app without starting a real server). You write tests that simulate real API calls, then assert on the status code, response body, and headers. Good test coverage gives you confidence to refactor code, add features, and deploy without fear of breaking things. Tests should cover happy paths (valid input), error paths (invalid input, missing auth), and edge cases (empty lists, duplicate entries).

```javascript
// Using Jest + Supertest:
// npm install --save-dev jest supertest

// tests/users.test.js
const request = require('supertest');
const app = require('../app');   // import Express app (not listening)

describe('Users API', () => {
  
  // GET /api/users
  test('should return list of users', async () => {
    const res = await request(app).get('/api/v1/users');
    
    expect(res.status).toBe(200);
    expect(res.body).toBeInstanceOf(Array);
    expect(res.body.length).toBeGreaterThan(0);
  });

  // POST /api/users - valid
  test('should create a user', async () => {
    const res = await request(app)
      .post('/api/v1/users')
      .send({ name: 'Test User', email: 'test@test.com', password: '12345678' });

    expect(res.status).toBe(201);
    expect(res.body).toHaveProperty('id');
    expect(res.body.name).toBe('Test User');
  });

  // POST /api/users - invalid
  test('should reject invalid user', async () => {
    const res = await request(app)
      .post('/api/v1/users')
      .send({ name: '' });   // missing required fields

    expect(res.status).toBe(400);
    expect(res.body).toHaveProperty('errors');
  });

  // GET /api/users/:id - not found
  test('should return 404 for non-existent user', async () => {
    const res = await request(app).get('/api/v1/users/99999');
    expect(res.status).toBe(404);
  });

  // Protected route without token
  test('should return 401 without auth token', async () => {
    const res = await request(app).get('/api/v1/profile');
    expect(res.status).toBe(401);
  });

  // Protected route with token
  test('should return profile with valid token', async () => {
    const loginRes = await request(app)
      .post('/api/v1/auth/login')
      .send({ email: 'test@test.com', password: '12345678' });

    const res = await request(app)
      .get('/api/v1/profile')
      .set('Authorization', `Bearer ${loginRes.body.token}`);

    expect(res.status).toBe(200);
    expect(res.body).toHaveProperty('name');
  });
});

## Production Project Structure

A well-organized project structure is crucial for maintainability as your codebase grows. The industry-standard pattern separates code into distinct layers: **Routes** (define URL endpoints and apply middleware), **Controllers/Handlers** (extract request data, call services, send responses), **Services** (contain business logic), and **Repositories** (handle database queries). Each layer is in its own directory. This separation means you can change your database without touching your business logic, or change your validation library without touching your routes. It also makes testing easier — you can test each layer in isolation.

```
my-api/
  |-- src/
  |   |-- index.js               # Entry point: create app, start server
  |   |-- app.js                 # Express app setup (exported for testing)

  |   |
  |   |-- routes/
  |   |   |-- index.js           # Route aggregator
  |   |   |-- auth.routes.js

  |   |-- controllers/           # (Handlers) Extract, validate, respond

## Real-World Patterns

Production Node.js applications need several patterns beyond basic CRUD. **Graceful shutdown** ensures that when your server receives a termination signal (from deployments, scaling, or crashes), it finishes handling in-flight requests and closes database connections cleanly instead of abruptly dropping everything. **Health check endpoints** let load balancers and monitoring services verify your app is alive and connected to its dependencies. **Pagination utilities** standardize how you return large datasets in pages. These patterns seem small individually but together they make the difference between a brittle prototype and a reliable production system.

### Graceful Shutdown

```javascript
const server = app.listen(PORT, () => {
  console.log(`Server running on port ${PORT}`);
});

function gracefulShutdown(signal) {
  console.log(`${signal} received. Shutting down gracefully...`);
  
  server.close(() => {
    console.log('HTTP server closed');
    pool.end(() => {    // close DB connections
      console.log('DB pool closed');
      process.exit(0);
    });
  });

  // Force shutdown after 10 seconds:
  setTimeout(() => {
    console.error('Forced shutdown');
    process.exit(1);
  }, 10000);
}

process.on('SIGTERM', () => gracefulShutdown('SIGTERM'));
process.on('SIGINT', () => gracefulShutdown('SIGINT'));
```

### Health Check Endpoint

```javascript
app.get('/health', async (req, res) => {
  try {
    await pool.query('SELECT 1');  // check DB connection
    res.json({
      status: 'healthy',
      uptime: process.uptime(),
      timestamp: new Date().toISOString(),
      memory: process.memoryUsage(),
    });
  } catch (err) {
    res.status(503).json({
      status: 'unhealthy',
      error: 'Database connection failed',
    });
  }
});
```

### Pagination Utility

```javascript
function paginate(query, page = 1, limit = 20) {
  const offset = (page - 1) * limit;
  return {
    query: `${query} LIMIT $LIMIT OFFSET $OFFSET`,
    limit,
    offset,
    buildResponse: (rows, totalCount) => ({
      data: rows,
      pagination: {
        page,
        limit,
        total: totalCount,
        totalPages: Math.ceil(totalCount / limit),
        hasNext: page * limit < totalCount,
        hasPrev: page > 1,
      },
    }),
  };
}
```

## Interview Questions - Part 2

### Q11: What is middleware in Express? How does the middleware chain work?
```
Middleware are functions with signature (req, res, next) that execute
sequentially in the order they're registered. Each middleware can:
  -> Modify req/res objects
  -> End the request-response cycle
  -> Call next() to pass control to the next middleware

The chain: Logger -> CORS -> Auth -> Validator -> Handler -> Error Handler

If a middleware doesn't call next() and doesn't send a response,
the request hangs forever (timeout).

Error middleware has 4 params: (err, req, res, next).
Express knows it's an error handler by the 4-param signature.
```

### Q12: How would you handle uncaught exceptions and unhandled promise rejections?
```javascript
// Uncaught synchronous exceptions:
process.on('uncaughtException', (err) => {
  logger.error('UNCAUGHT EXCEPTION:', err);
  process.exit(1);  // MUST exit - process state is unreliable
});

// Unhandled promise rejections:
process.on('unhandledRejection', (reason) => {
  logger.error('UNHANDLED REJECTION:', reason);
  process.exit(1);  // Node 15+ does this automatically
});

// Key point: After an uncaught exception, the process state is
// UNDEFINED. You MUST exit and let a process manager (PM2)
// restart the process. Never try to continue.
```

### Q13: Explain the difference between app.use() and app.get/post/etc.
```
app.use(middleware):
  -> Matches ALL HTTP methods (GET, POST, PUT, DELETE, etc.)
  -> Matches the path PREFIX (not exact match)
  -> app.use('/api', fn) matches /api, /api/users, /api/anything

app.get(path, handler):
  -> Matches ONLY GET method
  -> Matches the EXACT path (or pattern)
  -> app.get('/api', fn) matches ONLY GET /api, not /api/users
```

### Q14: How do you scale a Node.js application?
```
Vertical scaling (one machine):
  1. Clustering: fork workers across all CPU cores
  2. Worker Threads: for CPU-intensive tasks
  3. Increase memory: --max-old-space-size=4096

Horizontal scaling (multiple machines):
  1. Load balancer (Nginx, HAProxy, AWS ALB) in front
  2. Multiple Node instances behind the load balancer
  3. Stateless design: no in-memory sessions (use Redis)
  4. Shared DB and cache (PostgreSQL + Redis)

Tools:
  -> PM2: process manager with clustering built-in
  -> Docker + Kubernetes: containerized deployment
  -> AWS ECS / GCP Cloud Run: managed containers
```

### Q15: What is the difference between spawn, exec, and fork in child_process?
```
exec():
  -> Spawns a shell, runs command, buffers the entire output
  -> Returns output as a single string when done
  -> Good for: short commands with small output
  -> exec('ls -la', (err, stdout) => console.log(stdout))

spawn():
  -> Spawns a process WITHOUT a shell
  -> Returns a stream (data comes in chunks)
  -> Good for: long-running processes, large output
  -> const child = spawn('ffmpeg', ['-i', 'in.mp4', 'out.avi'])

fork():
  -> Special case of spawn for Node.js scripts
  -> Creates a new V8 instance
  -> Establishes IPC (Inter-Process Communication) channel
  -> Parent and child can send messages to each other
  -> Good for: CPU-intensive work in separate processes
```

### Q16: What is libuv and why is it important?
```
libuv is a C library that provides:
  1. The event loop implementation
  2. Async I/O (file, DNS, network)
  3. Thread pool (default 4 threads) for blocking operations
  4. Cross-platform support

It's the engine behind Node's non-blocking I/O.

Network I/O: uses OS-level async (epoll/kqueue/IOCP) - no threads
File I/O: uses the thread pool (OS file APIs are synchronous)
DNS: uses the thread pool
Crypto: uses the thread pool

UV_THREADPOOL_SIZE=8 node app.js  (increase thread pool)
```

### Q17: What is connection pooling and why is it important?
```
Without pooling:
  Each request: open DB connection -> query -> close connection
  1000 requests = 1000 connections opened and closed
  Each connection = TCP handshake + authentication = ~50ms overhead
  Total overhead: 50 seconds of just connection setup!

With pooling:
  At startup: open 20 connections and keep them alive
  Each request: borrow a connection -> query -> return to pool
  1000 requests = same 20 connections reused
  Total overhead: ~0 (connections are already established)

const pool = new Pool({ max: 20 });  // 20 connections in pool
```

### Q18: Explain the difference between process.nextTick(), setImmediate(), and setTimeout(fn, 0).
```
Execution priority (highest to lowest):

1. process.nextTick()
   -> Runs IMMEDIATELY after current operation
   -> Before ANY I/O events or timers
   -> Before even Promise microtasks!
   -> Can starve I/O if used recursively

2. Promise.resolve().then(fn)
   -> Microtask queue
   -> After nextTick, before next event loop phase

3. setTimeout(fn, 0)
   -> Timer phase (first phase of event loop)
   -> Minimum delay is actually ~1ms (not truly 0)

4. setImmediate(fn)
   -> Check phase (after poll/I/O phase)
   -> Runs after I/O events are processed
```

### Q19: How do you prevent SQL injection in Node.js?
```
SQL injection: attacker sends malicious SQL through user input.

VULNERABLE:
db.query(`SELECT * FROM users WHERE id = ${req.params.id}`)
// If id = "1; DROP TABLE users;" -> table deleted!

SAFE (parameterized queries):
db.query('SELECT * FROM users WHERE id = $1', [req.params.id])
// The $1 is replaced safely. Special characters are escaped.
// "1; DROP TABLE users;" is treated as a literal string, not SQL.

Also safe: ORMs (Sequelize, Prisma, TypeORM) use parameterized
queries internally. But ALWAYS verify when writing raw SQL.
```

### Q20: What is the purpose of the `cluster` module?
```
Node.js runs on a single CPU core by default.
The cluster module creates child processes (workers) that
share the same server port, utilizing all CPU cores.

Primary process: manages workers, doesn't handle requests
Worker processes: each runs a copy of your app, handles requests
OS load-balances connections across workers round-robin.

4-core machine:
  -> Without clustering: ~15,000 req/s (1 core)
  -> With clustering: ~55,000 req/s (4 cores, not exactly 4x due to overhead)

Alternative: PM2 does clustering automatically with `pm2 start -i max`
```
